#### 3.1.1 详细模块介绍

**RKNPU GEM生命周期管理**：团队发现RKNPU驱动中GEM（Graphics Execution Manager）对象存在资源泄漏风险。在`drivers/npu/rockchip-npu/src/gem.rs`中实现了显式的GEM create/destroy机制，并添加了`[RKNPU_STATS]`统计信息，可追踪每个GEM的创建和释放情况。该模块是团队在NPU驱动层面的核心优化工作。

**DMA heap & mmap后端**：在`os/StarryOS/kernel/src/pseudofs/dev/dma_heap.rs`中实现，为DMA缓冲区分配返回真实的mmap文件描述符，使`sys_mmap`能够识别匿名DMA/RKNPU文件描述符。这是打通RKNN推理链路的关键修复。

**文件级GEM清理**：在`os/StarryOS/kernel/src/pseudofs/dev/card1.rs`中实现，跟踪每个打开文件描述符关联的GEM句柄，在文件关闭时自动释放残留的GEM资源，输出`[RKNPU_CLEANUP]`日志。

**Submit cache同步优化**：在`rknpu-sync-tracked`分支开发，通过跟踪每个GEM的dirty/need标志，缩小cache同步范围，减少不必要的缓存刷新操作。

**离线资源监测平台**：位于`apps/starry/resource-monitor/`，包含Shell脚本`system-monitor.sh`采集CPU使用率、内存占用、系统事件等数据，输出CSV/JSONL格式日志。`offline-viewer/index.html`是一个静态HTML页面，可导入日志文件生成图表、摘要和Linux vs StarryOS对比分析。

**RKNN最小推理链路**：在Starry OS上成功运行`tennis.rknn`模型，完成`rknn_init`、`rknn_run`、`rknn_outputs_get`、`rknn_destroy`全流程，GEM资源完全释放。这是团队在比赛截止前取得的关键突破。

**Starry OS板级Bring-up**：通过USB启动（U-Boot → kernel → SD/MMC → ext4 RootFS → `/bin/sh`），成功在RK3588硬件上启动Starry OS并进入交互式Shell。过程中解决了OP-TEE绕过、SD/MMC驱动适配、RootFS挂载等问题。
