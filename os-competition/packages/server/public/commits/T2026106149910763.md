#### 3.1.1 详细模块介绍
async-unified 项目是一个为 StarryOS 内核设计的异步系统调用统一运行时。核心创新在于通过编译时 feature flag（`thread`）实现同一 API 在同步（线程）和异步（协程）两种模式间切换。核心库 `async-unified` 包含 executor（基于 AtomicBool 的 block_on 执行器）、sync（双模式 Mutex + WaitQueue）、syscall（SyscallFuture 实现 Deref+Future 双模式）、scheduler（FIFO/RR/CFS 三种调度器）、io（AsyncRead/AsyncWrite trait 及组合子）、transport（抽象传输层）和 futures（yield/sleep/block/join）等模块。StarryOS 内核集成层实现了基于 ring buffer 的 MPSC 通道、per-CPU 事件驱动 dispatcher、内核 async runtime（Executor + AsyncTask + JoinHandle）以及用户空间共享内存访问机制。测试套件 `async-unified-test` 包含 8 个测试模块共约 38-40 个测试用例，覆盖双模式运行。文档极其丰富，包含 24 章 mdbook 在线文档和一份 PDF 项目书，详细描述了架构设计、执行流程、性能分析和各模块设计决策。

### 3.2 开发目标
- 明确程度：明确
- 依据：README 和 overview.md 明确描述了项目目标：提供一个低侵入性的异步系统调用模块，通过编译时 feature flag 在线程模式和协程模式间切换，无需修改原始 StarryOS 内核代码。目标包括 ring buffer dispatcher、SyscallFuture 双模式、异步 Mutex/WaitQueue、三种调度器、I/O trait 等。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| AtomicBool spin-loop 存在 lost wakeup 风险（host 端 block_on） | 文档中已说明该风险，内核端使用 axtask 调度器阻塞机制避免 race |
| QEMU 单核环境下用户程序需主动调用 sched_yield 避免自旋 | 文档中明确要求用户调用 ecall(124) 让出 CPU |
| Mutex 死锁检测需要每任务唯一 ID | 通过 set_task_id_provider 注册 axtask::current().id() 解决 |
| 之前存在 contention_with_block_on 测试因 race condition 被移除 | 改用 mutex::contention_two_threads 替代 |
| 轻量 syscall（如 getuid）dispatch 开销达 26× native | 文档分析认为这是常数开销（~340ns）被极低 baseline 放大的假象，真实 IO 场景 overhead 被淹没 |
| QEMU TCG 模拟放大上下文切换开销约 100× | 文档提供了真实硬件估算值（3-5us vs 300us） |
