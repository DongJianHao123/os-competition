#### 3.1.1 详细模块介绍

**backend模块**：基于FastAPI框架实现，提供完整的RESTful API。核心功能包括：Agent注册（关联PID与Trace ID）、单条/批量事件注入（自动按PID/PPID关联Agent与Trace）、Agent日志注入、配置化规则引擎检测、告警生成与去重、Trace时间线构建、中文诊断报告生成、仪表盘聚合统计（事件/告警/Agent/趋势/分布）、采集器进程状态监控。代码约440行，结构清晰，使用SQLite数据库，支持WAL模式和索引优化。

**collector模块**：包含eBPF采集器（基于BCC框架，监控execve/openat/unlinkat/connect四类系统调用）、OpenSSL uprobe采集器（Hook SSL_write/SSL_read截获HTTPS明文）、批量上传器（BatchUploader支持缓冲批量上报和回退机制）、事件过滤器（EventFilter实现噪声过滤、去重、限流、采样）。采集器支持进程名白名单、动态PID过滤、关键安全事件强制保留。代码质量高，错误处理完善。

**analyzer模块**：规则引擎从default_rules.yaml加载6条检测规则（敏感文件访问、Shell启动、批量删除、异常网络连接、Agent循环调用、TLS明文敏感字段），支持事件和日志两种检测源。关联器按PID链将eBPF事件归属到Agent和Trace。报告生成器组装包含异常概览、应用层语义证据、系统调用证据、Trace时间线、归因结论和处置建议的中文诊断报告。

**agents模块**：模拟多智能体运行环境，包含Planner、Search、Executor、Writer四种Agent角色，通过run_workflow.py驱动工作流，产生Prompt/Response/Agent ID/Trace ID等应用层语义数据，用于触发异常检测和跨层归因。

**static前端**：单页应用仪表盘，包含侧边栏导航、指标卡片（事件/告警/Agent/高危告警）、趋势图（最近30分钟）、分布图（事件类型/告警等级/告警类型）、事件流表格（支持类型筛选）、告警列表（含报告和链路按钮）、Trace时间线、诊断报告展示。使用原生JS实现，支持3秒自动刷新，代码约270行，质量良好。

**tests测试**：包含5类异常场景脚本（sensitive_file_access、shell_spawn、mass_delete、network_burst、agent_loop）和1个normal工作流脚本，以及端到端测试（test_end_to_end.py）。每个场景脚本模拟特定异常行为，用于验证采集、检测、告警全链路。

**scripts脚本**：包含一键启动（start_backend.sh、start_collector.sh）、一键演示（run_demo.sh、run_all_demos.sh）、性能测试（perf_test.sh、perf_compare_v7.sh）、报告导出（export_report.py）、Trace回放（replay_events.py）、最终检查（final_check.sh）等14个脚本，工程化程度高。

**docs文档**：16个文档覆盖项目说明、架构设计、测试计划、性能测试、性能优化、规则引擎、Trace时间线、前端仪表盘、批量上报、OpenSSL uprobe、打包检查、最终演示指南等，文档体系完整。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| eBPF采集器硬编码进程白名单，扩展性受限 | 通过环境变量AGENTGUARD_PID_FILTER动态控制PID过滤 |
| 批量上传器回退失败时事件静默丢弃 | 设计为回退到逐事件上报，但失败事件无重试机制（文档记录为后续优化方向） |
| 事件过滤器状态管理使用清空策略而非LRU | 使用_maybe_trim_state在超出4096阈值时清空，简单有效 |
| 数据库find_agent_by_pid_or_ppid中ppid查询逻辑错误（使用pid=?而非ppid=?） | 代码中存在该问题，但实际运行中可能因pid和ppid相同而偶然正确 |
| 报告生成器存在索引越界风险（trace["alerts"][0]） | 当前假设alerts非空，未做防御处理 |
| 规则文件后缀为.yaml但实际使用JSON解析 | 可能造成维护者困惑，但功能正常 |
| 前端无加载状态指示和分页支持 | 当前显示固定数量（28条事件、70条时间线），适合演示场景 |
