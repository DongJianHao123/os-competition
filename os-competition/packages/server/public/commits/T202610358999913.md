#### 3.1.1 详细模块介绍
**StreamANN（读优化引擎）**：基于DiskANN框架，通过异步I/O（io_uring）和双缓冲机制，将图遍历搜索中的随机I/O与计算重叠执行。开发过程从底层异步文件读取（linux_aligned_file_reader）开始，逐步实现read_graph_async、load_vertices_async、fetch_node_async等异步原语，最终构建完整的pipeann流水线。提交历史显示从5月14日到6月30日持续演进，经历了pipeann 1.0到1.5等多个版本迭代。

**Ouroboros Store（写优化引擎）**：采用LSM-Tree思想的Delta-Base分层存储架构，由5棵BFTree组成增量层处理实时写入，后台异步合并到Vamana图索引。实现了流式归并（stream_merge_to_fbin）替代全量HashMap，峰值内存降低72%；通过主动drop旧Arc和fadvise回收page cache，稳态RSS降低29%。搜索采用两阶段策略：delta层暴力扫描+base层Vamana图搜索，保证数据新鲜度与性能平衡。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| tokio嵌套问题导致异步执行异常 | 使用spawn_blocking避免tokio运行时嵌套 |
| 并发安全：多线程访问共享状态 | 使用AtomicU32自增ID分配器保证并发安全 |
| 数据新鲜度与性能平衡 | 采用两阶段搜索：delta暴力扫描+base Vamana图搜索 |
| 合并时索引一致性 | 原子替换base索引保证一致性 |
| Compaction峰值内存过高 | 将collect_live_entries全量HashMap重写为stream_merge_to_fbin流式归并，峰值内存降低72% |
| 旧索引mmap内存未释放 | 主动drop旧Arc并调用fadvise回收page cache，稳态RSS降低29% |
| PQ压缩内存预算超标 | PQ64改为PQ32，严格15%内存预算，添加RSS实时监控 |
| id_map内存占用过高 | HashMap改为Vec，降低内存开销 |
