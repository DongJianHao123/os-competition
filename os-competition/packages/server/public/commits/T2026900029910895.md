#### 3.1.1 详细模块介绍
本项目实现了一个基于ptrace的轻量化用户态动态探针工具，最终版本为probe8.c（789行）。项目从最初的probe.c（232行，仅支持F1-F7基础功能）开始，经过probe_v2.c（增加ASLR兼容跳板分配）、probe_3.c（完善ptrace安全封装）、probe_4.c（引入A1架构自适应和A2条件过滤）、probe6.c（648行，引入A3/A4/A5高级功能）、probe7.c（668行，优化锁策略和信号安全），最终演进到probe8.c（789行，完整实现F1-F7+A1-A5全部功能）。

F系列基础功能采用分层设计：F1动态插桩为核心，通过ptrace PTRACE_POKEDATA将函数入口5字节替换为相对跳转指令（0xE9+offset），跳转到预先在目标进程mmap分配的跳板内存区执行用户回调。F2/F3通过PTRACE_GETREGS读取寄存器实现参数和返回值捕获。F4在探针卸载时恢复原始指令并munmap释放跳板内存。F5通过修改probe_t.enabled标志实现运行时开关。F6通过probe数组管理最多32个探针。F7使用自适应自旋锁保护全局数据结构，ptrace耗时操作置于锁外以减少锁竞争。

A系列进阶功能在F系列基础上构建：A1通过uname或预定义宏检测架构，x86_64完整实现，AArch64预留接口。A2支持全局+per-probe两级过滤回调。A3通过写入/sys/kernel/debug/tracing/trace_marker节点实现用户态探针日志与内核ftrace数据合流。A4通过PTRACE_GETREGS/SETREGS保存恢复寄存器上下文，支持远程调用任意函数。A5通过替换probe_t中的filter和callback函数指针实现运行时热更新。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| waitpid 5秒超时：目标进程持续运行不进入内核暂停状态 | target8.c使用raise(SIGSTOP)主动暂停，设置ptrace_scope为0 |
| PTRACE_ATTACH失败：权限不足或ptrace_scope限制 | 探针必须以sudo运行，/proc/sys/kernel/yama/ptrace_scope必须为0，PID必须有效 |
| 目标进程插桩后段错误：地址错误（变量地址而非函数代码地址）、跳板内存分配失败、未保存FPU寄存器 | 使用正确的函数虚拟地址，确保内存权限，保存/恢复FPU寄存器 |
| trace_marker打开失败：debugfs未挂载或非root运行 | 确保debugfs已挂载，以root权限运行 |
| 多线程数据竞争：全局探针数组和PID标志操作未加锁 | 使用内置自适应自旋锁保护所有全局操作，ptrace耗时操作置于锁外 |
| 信号中断导致ptrace调用失败 | 实现ptrace_safe封装，EINTR时自动重试 |
| 内存泄漏：跳板内存未释放 | 在probe_remove中增加munmap释放，统一在结束时清理 |
| 锁粒度过大：ptrace阻塞操作在锁内导致性能下降 | 将ptrace等待和远程内存读写操作移到锁外，仅保护全局共享变量 |
