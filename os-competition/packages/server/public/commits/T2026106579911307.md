#### 3.1.1 详细模块介绍
项目采用内存--SSD 分层架构，系统分为四层：索引检索层（外存图遍历、PQ 粗排和原始向量精排）、动态写入层（增量日志、有效版本映射、失效记录和删除对象维护）、I/O 优化层（用户态缓存、拓扑预取和 Linux io_uring 批量随机读）和外存文件层（保存 vectors.bin、graph.bin、pq_codes.bin、metadata.tsv、delta.log 和 manifest.txt）。核心引擎 VectorEngine 作为统一门面，通过 shared_mutex 实现读写并发控制，Search 持有共享锁，写入操作序列化执行。所有写入（Add/Update/Delete）通过 LSM-Delta 写路径顺序追加到 delta.log，保证写入性能的同时支持崩溃恢复。压缩合并（Compact）在后台将基线文件与增量日志合并为新基线快照，原子切换，不影响前台搜索。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| 外存图遍历产生大量不连续随机读，默认 Page Cache 和顺序预取难以准确工作 | 实现用户态 BlockCache 和拓扑感知预取器，结合 io_uring 异步批量读 |
| 内存资源受限，无法将所有向量和图索引常驻内存 | 采用内存--SSD 分层架构，内存仅保留 PQ 短编码、码本、入口点等，全精度向量和图邻接表保存在 SSD |
| 需要支持高频查询下的实时更新（新增/更新/删除） | 设计 LSM-Delta 写路径，写入顺序追加到 delta.log，搜索时合并基线和增量结果 |
| 崩溃后数据一致性保证 | 实现 WAL 风格的 delta.log，Recover 方法重放日志重建内存状态，日志尾部不完整记录被忽略 |
| 压缩合并期间不影响前台搜索 | Compact 方法在临时目录重新物化基线快照，原子切换 manifest，期间释放写锁允许搜索继续 |
| PQ 关闭时 recall 极低（medium/stress 上仅 0.5%-1%） | 默认启用 PQ，benchmark 验证 PQ 可将 recall 提升至 89%-99.6% |
