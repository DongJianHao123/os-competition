#### 3.1.1 详细模块介绍
项目整体架构采用用户态与内核态分离的设计。用户态（k3_aiRuntime）负责将AI框架的算子转换为稳定的UAPI，内核侧（k3_aiScheduler、k3_kernel_backend）只处理graph解析、依赖管理、队列调度、buffer映射和completion，具体算子执行由backend根据目标路径（CPU/X100/A100）完成。k3_aiUabi作为内核与用户态共享的稳定ABI层，定义了所有数据结构。k3_ggml_bridge作为测试版桥接层，用于将ggml/llama.cpp的F32 MatMul请求转换为runtime graph提交。k3_test提供了完整的集成测试验证。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| K3/K3x不是传统NPU提交模型，没有传统/dev/npu驱动 | 改为"用户态runtime + 内核资源管理 + 异构核心调度"方案 |
| K3真板启动依赖UFS驱动，迁移成本高 | 采用bindgen生成寄存器定义、c2rust翻译C代码、手工补胶水代码的方式降低风险 |
| DMA cache同步为空实现 | 补齐clean/invalidate/sync路径解决数据不一致 |
| V扩展上下文没有保存恢复 | 增加V状态保存区（堆内存），在QEMU和真板验证 |
| IME MatMul形状和文档理解存在偏差 | 通过错误输出反推数据布局，逐项对比CPU reference后修正 |
| QEMU上的向量程序性能异常 | 限制线程数（-t 1）用于功能验证，性能以真板为准 |
