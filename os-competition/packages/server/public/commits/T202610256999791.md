#### 3.1.1 详细模块介绍

**local_agent.py**：核心智能体类 LocalAgent，采用多轮 LLM 驱动架构。每轮构建 system_prompt + state_prompt，调用 LLM 生成 JSON 格式的 analysis + tool_call，循环直到完成或达到硬限制（300 轮）。支持技能系统（基础技能 kpatch-basic + 可选技能，LRU 淘汰）、经验学习（成功提取规则、失败记录 KnownFailure）、验证门控（evaluate_done_gate 自动校验证据）、失败自反馈（构建失败后强制诊断）。最终输出 RunReport 写入 workspace/report.json。

**tools.py**：ToolRegistry 集中管理 12 个工具。核心工具 _handle_kpatch_build 实现完整构建管线：获取候选补丁 → dry-run → 适配 → 打分 → 可行性分析 → 实际构建。安全方面，bash 和 write_file 均经过 AgentPolicy 策略检查。日志截断智能，能提取错误相关行。

**feasibility.py**：基于 USENIX 2020 Android kernel hot patch 和 quick-wins QW3 限制启发式，实现 4 类静态检查：__init/__exit 段检查（block 级别）、struct 布局变更检查（block 级别）、static/全局变量改动检查（high 级别）、内联/短函数检查（medium~high 级别）。

**verification_gate.py**：完成门控验证，对 agent 声明的 success 状态强校验构建成功、模块路径、load 状态及 enabled 证据、unload 状态，缺一降级为 failed。对 failed 状态若终态为 not_applicable 则改判。防御性强，对 None、空字符串、大小写、结构化/日志回退均有处理。

**fc-mcp/server.py**：基于 MCP (Model Context Protocol) 的 kpatch livepatch 代理网关服务器，支持 JSON-RPC 2.0，提供 start_livepatch_task、get_livepatch_status、get_livepatch_report、get_livepatch_artifact 四个工具。支持 CORS、SSE 端点、健康检查。通过 BackendClient 与后端 API 通信。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| DeepSeek 返回 Markdown 代码块包裹的 patch，导致 kpatch-build 解析失败 | 改用 JSON Output 模式，模型只返回 JSON，程序读取 patch 字段并执行 schema 校验 |
| CVE-2024-42081 候选补丁不匹配目标内核（文件不存在） | 不能仅用 "CVE → stable commit" 作为最终补丁，需验证适配性，失败归因为 not_applicable/candidate_mismatch |
| VM 环境 gcc 版本不匹配（12.3.0-17 vs 12.3.0-16） | 降级 gcc/cpp/libgomp 到匹配版本，沉淀为环境初始化脚本 |
| kpatch 机制限制：__init 函数、静态数据、fentry 缺失、ABI 变化、编译器优化 | 结构化改写：修改 init 函数为运行时回调；静态变量保留原引用；全局数据用 shadow variable；ABI 变化新增函数替换调用点；缺少 fentry 迁移到可 hook 的上层函数 |
| kernel-devel 不包含完整驱动源码，导致 patch 应用失败 | 使用 source RPM（rpm -ivh + rpmbuild -bp --nodeps）得到打过发行版补丁的完整源码树 |
| patch --dry-run --batch 遇到 Reversed 时退出码为 0 导致误分类 | 列为最高优先级工程债，需修复退出码判断逻辑 |
| 候选补丁落盘异常（fetch_cve_patches 返回路径但文件未写入） | 引入 OSV fallback 后已解决 |
