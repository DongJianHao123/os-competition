#### 3.1.1 详细模块介绍

**ArceOS**：作为 TGOSKits 的核心组件化单内核，ArceOS 采用 Cargo feature 驱动的系统装配机制，支持编译期可裁剪。其架构分为可复用 crate 层、平台与 HAL 层、内核服务模块层、API 聚合层、用户库层以及应用与测试层。核心机制包括三种 API 封装策略（ax-api、ax-posix-api、ax-std/ax-libc），以及 ax-std 直接调用模块而非通过 syscall 的设计。任务调度模型支持 FIFO/RR/CFS 调度算法，状态机包括 Created、Ready、Running、Blocked、Exited。开发环境基于 Linux、Rust 工具链和 QEMU，支持 x86_64、riscv64、aarch64、loongarch64 四种架构。

**AxVisor**：基于 ArceOS 的统一组件化 Type-I Hypervisor，其架构围绕"ArceOS 作为宿主运行时 + 虚拟化组件作为能力核 + AxVisor 运行时负责编排 + Guest 作为最终负载"展开。核心设计目标包括统一（多架构覆盖）、组件化（VM、vCPU、虚拟设备、地址空间、API 注入等独立组件）、可配置（板级与 VM 配置驱动）和可验证（xtask、QEMU workflow 闭环）。运行时主线简洁：main() 依次执行使能硬件虚拟化、VMM 初始化、VM 启动、进入交互 shell。支持多种硬件平台（QEMU、Orange Pi 5 Plus、Phytium Pi、ROC-RK3568-PC、EVM3588）和多种客户机系统（ArceOS、Starry-OS、NimbOS、Linux）。

**StarryOS**：建立在 ArceOS 基础能力之上的组件化宏内核系统，继承了 ArceOS 的模块化、跨平台和 Rust 安全性，同时引入了更接近 Linux 的进程、线程、syscall、文件系统和 rootfs 语义。架构上分为用户态层、启动包层、内核核心层、Starry 专用组件层、ArceOS 基础模块层和平台层。核心实现包括进程与线程分离（ProcessData/Thread）、syscall 分发机制（覆盖进程管理、文件操作、内存管理、信号处理、网络、IPC 等）、用户程序加载（ELF loader）、IPC 机制（管道、eventfd、signalfd、pidfd、System V 消息队列/共享内存、epoll）。
