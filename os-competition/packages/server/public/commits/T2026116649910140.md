#### 3.1.1 详细模块介绍

**接入层（integration/vllm）**：通过 vLLM V1 KV connector 机制实现透明接入，在尽量少改动推理框架主流程的前提下，将 KV Cache 视为 block 级资源进行管理。`MiniFlexConfig` 支持从环境变量、JSON 文件或 vLLM 配置自动加载，实现了低侵入式集成。

**任务层（kvtask.py）**：`KVTaskEngine` 和 `KVTaskManager` 负责 KV 任务的完整生命周期管理。支持 GET（从缓存读取）和 PUT（写入缓存）两种任务类型，提供异步提交、批处理合并、超时等待、取消等能力。任务通过 `TransferOpGraph` 描述数据传输依赖关系，由调度器按依赖顺序执行。

**缓存层（cache/）**：`GlobalCacheEngine` 协调 GPU→CPU→SSD 三层缓存。使用 `RadixTree` 实现最长前缀匹配，`Mempool` 管理 block 分配，`CacheEngine` 分别管理 CPU 和 SSD 缓存。支持 LRU/LFU/FIFO/MRU/FILO 多种淘汰策略和 hot-data 保护机制。

**存储层（storage/）**：`StorageEngine` 统一管理 GPU/CPU/SSD 三种物理存储。通过 `StorageHandle` 抽象（TENSOR/TENSOR_HANDLE/FILE）屏蔽底层差异，支持 layout 自适应（LAYERFIRST/LAYERBLOCK/BLOCKFIRST）。

**传输层（transfer/）**：`TransferEngine` 配合 `TransferScheduler` 实现异步传输调度。`GPUCPUTransferWorker` 使用 CUDA stream + cudaMemcpy2DAsync 实现 GPU↔CPU 传输，`SSDCPUTransferWorker` 使用 io_uring 实现 CPU↔SSD 异步 I/O。支持 intra-process 和 inter-process 两种运行模式。

**C++/CUDA 扩展**：`transfer.cu` 实现了高效的 GPU↔CPU 批量 block 传输，`ssd_io_uring.cpp` 实现了基于 io_uring 的异步 SSD 读写，支持 O_DIRECT 和 scatter-gather I/O，具备同步回退能力。
