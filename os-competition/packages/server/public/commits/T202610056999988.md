#### 3.1.1 详细模块介绍

**tgoskits (StarryOS移植)**：团队基于StarryOS/ArceOS框架，针对Orange Pi 5 Plus（RK3588）平台进行系统移植。核心工作包括：1) 内存布局适配——将bitmap page allocator从4GiB扩展至64GiB（page-alloc-4g→page-alloc-64g），解决8GB内存中高地址RAM导致的Allocator Panic；2) 构建TFTP启动链路，通过U-Boot加载uImage和dtb启动StarryOS；3) 准备rootfs并部署用户态资产。板端已成功启动StarryOS并进入shell。

**aka-rk3588/tennis (机器人应用)**：网球机器人主程序，实现两状态状态机（chase追球/grab抓取）。工作流程：UVC摄像头捕获MJPEG帧→libjpeg-turbo解码为RGB→letterbox填充→RKNN YOLOv8推理检测网球→根据检测结果（面积比、居中程度）控制电机。包含test-uvc（单帧捕获）、test-yolo（捕获+推理+画框）、test-image（图片推理验证）、test-motor（电机测试）四个测试命令。板端test-image已成功验证RKNN推理链路。

**capture模块**：封装UVC摄像头操作，提供getFrame()接口获取MJPEG帧数据。

**detect模块**：封装RKNN Runtime调用，提供detect_init()和detect_run()接口进行YOLOv8目标检测。

**motor模块**：支持UART和PWM两种电机驱动方式，提供drive()、forward()、backward()、standby()等控制接口。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| Allocator Panic：bitmap allocator仅管理4GB，8GB高地址RAM超出容量 | 将page-alloc-4g切换为page-alloc-64g，BitAllocUsed从BitAlloc1M变为BitAlloc16M |
| 大块连续内存注入byte allocator引发初始化异常 | add_memory()按1GiB粒度分块循环调用 |
| GLIBC版本不兼容（PC Ubuntu 24.04 GLIBC 2.38 vs 板端GLIBC 2.35） | 在RK3588板端本地编译，或使用Ubuntu 22.04 Docker/匹配sysroot交叉编译 |
| 动态库路径冲突（/usr/lib被覆盖） | 将项目库集中部署到/opt/starry/lib，运行前设置LD_LIBRARY_PATH |
| rootfs设备名不稳定（mmcblk0p2/mmcblk1p2变化） | 改用PARTUUID指定rootfs分区 |
| 板端网络不可访问外网 | 配置PC端IP转发与NAT，板端设置默认网关和DNS |
| eMMC/SD初始化失败 | 改用PARTUUID提高稳定性，eMMC作为主要验证路径 |
| 缺少AArch64交叉编译器 | 安装gcc-aarch64-linux-gnu |
| 缺少yolov8依赖目录 | 克隆rknn_model_zoo |
| 缺少libuvc头文件 | 安装libuvc-dev |
