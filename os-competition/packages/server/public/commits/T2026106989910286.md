#### 3.1.1 详细模块介绍

**LD_PRELOAD Hook模块**：这是项目的核心模块，通过`LD_PRELOAD`机制在用户态拦截文件I/O系统调用，实现透明压缩。拦截了包括open/open64/creat/openat、read/pread64/readv、write/pwrite64/writev、lseek/lseek64、close、dup/dup2/dup3、ftruncate/truncate、stat系列、sendfile、mmap/munmap/msync、rename等约30个系统调用。写路径采用"先写后压缩"策略，写入时直接写入原始数据（RAW格式），由后台守护进程异步压缩。读路径支持GPU批量解压（SGLZ或LZ4），包含自适应预读（最多16块）、io_uring异步I/O、多CUDA Stream流水线（16条流）、per-fd LRU块缓存、延迟Header写入、CPU自动回退（GPU利用率>80%时）、数据库/日志白名单、CRC32校验等优化。mmap路径支持按需解压（同步预解压前16MB+后台异步解压剩余部分），512MB mmap返回时间从2414ms优化至231ms（10.4倍提升）。

**离线压缩守护进程**：后台扫描指定目录，识别RAW状态文件，使用GPU批量压缩（SGLZ或LZ4），写入临时文件后原子替换原文件。支持递归扫描（-r）、周期性守护模式（-d，每60秒扫描）、文件锁避免冲突、自适应压缩率阈值（默认85%，低于阈值保持RAW）。小文件（≤1KB）内联存储不压缩。

**SGLZ GPU压缩算法**：自定义轻量LZ77变体，128KB块分为32个4KB块，每个CUDA线程处理一个4KB块。使用共享内存哈希表（stride布局避免bank冲突），3字节滚动哈希，2048B滑动窗口。输出格式：tag字节（0x00=字面量，0x01=回退引用）+长度+数据/偏移。若压缩未减小则存储原始数据（0xFFFF标记）。支持批量压缩/解压API、紧凑格式解压、异步流水线。

**nvCOMP LZ4 GPU后端**：封装NVIDIA nvCOMP 4.2 Batched LZ4，输出格式为[1B flag][payload]（flag=0原始，flag=1 LZ4压缩）。支持单块和批量压缩/解压，设备内存池复用避免重复cudaMalloc。在Linux headers数据集上，压缩率从SGLZ的82.5%降至38.6%，顺序读带宽从279 MB/s提升至581 MB/s。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| jobs=32高并发场景性能骤降 | 通过多CUDA Stream并行（2→16条流）、io_uring异步I/O、自适应预读优化解决 |
| mmap返回延迟高（512MB需2414ms） | 实现按需解压：同步预解压前16MB+后台pthread异步解压剩余部分，延迟降至231ms（10.4x提升） |
| nvCOMP LZ4 CompressAsync传入栈上指针数组导致nvcompErrorInvalidValue | 改为设备端持久化指针数组，修复参数传递问题 |
| GPU显存不足时崩溃 | 实现GPU负载自动检测（nvidia-smi），利用率>80%时自动fallback到CPU后端 |
| 数据库/日志文件被压缩导致性能下降 | 实现数据库白名单机制，对.sql/.db/.log等文件跳过压缩 |
| 低压缩率数据（如已压缩文件）浪费GPU资源 | 自适应压缩率阈值（默认85%），低于阈值保持RAW状态 |
| 写路径性能开销 | 采用"先写后压缩"策略，写入时直接写原始数据，由后台守护进程异步压缩 |
| 小文件压缩收益低 | 实现内联存储（≤1KB文件直接存储在Header后），跳过GPU压缩 |
| 并发写入时文件状态不一致 | 使用文件锁（flock LOCK_EX\|LOCK_NB）避免冲突，原子rename替换 |
| CRC32校验增加CPU开销 | 添加编译选项GPUC_NO_VERIFY_CRC，生产环境可跳过校验 |
