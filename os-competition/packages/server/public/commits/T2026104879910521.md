#### 3.1.1 详细模块介绍
项目采用四层架构：数据采集层（eBPF内核探针+SSL嗅探）→ 上下文关联层（Redis）→ 跨层分析层（单智能体+多智能体分析器）→ 审计输出层（规则+LLM+前端展示）。eBPF探针通过kprobe/tracepoint/uprobe实现零侵入监控，支持syscall级和SSL应用层数据捕获。Redis作为解耦中间件，使用pub/sub和stream传递事件。分析层分为intra-agent（单智能体行为异常检测）和inter-agent（多智能体协同异常检测），后者使用GNN图结构检测器。审计引擎结合规则和LLM（DeepSeek）进行二次确认，降低误报。前端基于Next.js 15展示实时监控数据和异常告警。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| IPv4格式化字节序未转换（网络序→主机序） | 代码中format_ipv4_str函数存在bug，需修复字节序转换 |
| pending_read/pending_recv使用TID作为key但submit_raw_event使用PID | 可能导致TID!=PID时数据不匹配，需统一key |
| sys_exit_read/recvfrom使用struct trace_event_raw_sys_exit | 部分内核可能不支持，需添加CO-RE fallback |
| intra_agent中EVENT_CHMOD错误计入fs_open_cnt | 应使用独立计数器或重命名 |
| intra_agent中EVENT_AGENT_COMM错误计入net_connect_cnt | 应使用独立计数器 |
| _path_is_workspace_or_tmp对相对路径返回True | 过于宽松，应要求绝对路径 |
| inter_agent默认循环仅sleep，不处理观测数据 | 需实现实际观测处理逻辑 |
| 缺少对EVENT_READ/WRITE/CLOSE/TCP_SEND/TCP_RECV的显式测试 | 需补充测试用例 |
| 缺少错误路径测试（失败系统调用、无效注册） | 需补充负面测试 |
| 缺少性能/压力测试 | 需补充高频率事件和大规模agent测试 |
