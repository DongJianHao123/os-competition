#### 3.1.1 详细模块介绍
**BPF 数据面**：单一 BPF 对象包含三个 SEC 程序——dns_accel（XDP ingress）在 XDP 层拦截 DNS 查询，若命中 LRU 缓存则直接构造 DNS 响应并返回，避免用户态往返；dns_learn（TC egress）在 DNS 响应路径上学习 A 记录并写入缓存，支持 TTL 和热点提升（hits>32 时按比例延长 TTL）；grpc_fastpath（TC ingress）识别 gRPC/HTTP 流量，统计包量、采样 RTT，并支持通过 bpf_redirect_peer 将流量直接导向 VM 接口。所有程序共享 runtime_config map 实现动态配置。

**BPF 加载器**：使用 cilium/ebpf 库加载 BPF 对象，自动匹配接口类型（lo/veth/其他）选择 XDP 挂载 flags，支持 generic 和 native 模式 fallback。通过 replaceBpfFilter 在 TC ingress/egress 挂载程序。提供线程安全的运行时配置读写和指标查询。

**控制面 HTTP 服务**：提供 /v1/health（健康检查）、/v1/config（运行时配置 GET/PUT）、/v1/metrics（指标查询）、/v1/dns/cache（DNS 缓存预热 PUT）四个端点。配置更新经 runtime.ApplyDefaults 和 runtime.Validate 校验后写入 BPF map，实现动态生效。

**指标导出器**：按可配置间隔（默认 5 秒）调用 ReadBundle 读取四类服务指标（DNS/gRPC/VM/HTTP），以 JSONL 格式追加写入文件，支持优雅退出。

**实验脚本体系**：提供完整的实验矩阵脚本，支持 E0（Baseline，不加载 eBPF）到 E4（Full，DNS+gRPC+双端缓存）共 5 个实验场景，每组至少运行 3 次取平均值和标准差。产物包括原始数据、汇总 CSV、matrix-table.md 和图表。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| BPF 中 DNS 缓存热点提升条件逻辑缺陷：hits 字段在更新时被重置为 0，导致热点提升条件（hits>32）几乎永远不满足 | 需修复为在更新缓存时保留 hits 计数或使用独立的热点计数器 |
| XDP dns_accel 中调用 bpf_xdp_adjust_tail 后重复解析所有头部，代码重复约 30 行 | 可提取为内联函数或宏以减少重复 |
| DNS 缓存键仅依赖 FNV-1a 哈希，存在哈希碰撞风险 | 可考虑使用完整域名作为键或增加哈希冲突检测 |
| grpc_fastpath 中流开始时间使用 BPF_ANY 直接覆盖，同一流多个数据包时开始时间被刷新，导致 RTT 测量偏小 | 应使用 BPF_NOEXIST 仅在首次插入，或使用 bpf_map_update_elem 的 flags 参数控制 |
| TC 程序未检查 skb 线性数据长度，直接使用 skb->data 和 skb->data_end 可能不安全 | 建议使用 bpf_skb_pull_data 确保数据在线性区域 |
| 指标导出器错误处理过于简单：ReadBundle、json.Marshal、f.Write 出错时仅 continue，静默丢弃错误 | 应至少记录日志，考虑重试或向上传播错误 |
| 指标导出器未调用 f.Sync()，程序崩溃可能丢失最近数据 | 定期调用 f.Sync() 或提供可配置的同步策略 |
| 控制面 HTTP 服务缺少中间件（日志、鉴权、超时、恢复） | 可添加标准中间件增强可观测性和稳定性 |
| 控制面缺少 handler 级别的单元测试 | 应补充单元测试覆盖各 API 端点 |
| 配置管理缺少热重载机制 | 可添加文件 watch 实现配置热更新 |
