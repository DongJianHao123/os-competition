#### 3.1.1 详细模块介绍
项目基于 rCore-Tutorial-v3 ch7 分支，在 RISC-V64 QEMU 平台上用 Rust no_std 开发。核心设计原则是「机制与策略分离」：内核提供最小原语（16 个私有 syscall 600-615），用户态 Agent 决定循环策略。

- **T1 Agent 身份与创建**：通过 fork+sys_agent_create 将普通进程提升为 Agent，分配独立的 Agent Context 区（带 guard page 防止越界），存储 AgentMeta（类型、心跳间隔、资源配额、循环状态等）。
- **T2 结构化工具调用**：设计行式文本协议（`tool_name key=value ...`），内核解析并分发到静态 TOOLS 注册表。9 个工具中 6 个零新增 syscall（复用现有文件系统调用），3 个需要内核协作。
- **T3 上下文路径**：每个 Agent 维护一个环形缓冲区的 Context Path，记录每轮循环的请求摘要+结果摘要+时间戳+状态。用户态可直接零 syscall 读取，内核为唯一写入者。
- **T4 属性化文件系统**：扩展 easy-fs 的 inode 结构，增加属性存储区。新增按属性查询的 syscall，避免遍历整个目录树。
- **T5 Agent Loop 运行机制**：引入 Blocked 状态实现真休眠（0 CPU 占用），三种唤醒源：心跳、消息、文件修改。Agent 可订阅感兴趣的事件组合。
- **T6 综合演示**：sysadmin demo 展示心跳驱动的主 Agent 巡检文件系统、发现问题、修复问题、上报操作员的完整流程；NPC demo 展示 3 个 Agent 通过消息传递自组织成环形拓扑，零内核修改。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| Agent Context 区越界访问风险 | 引入 guard page（不可访问页）保护 Context 区边界 |
| 工具调用协议选择（JSON/binary/文本） | 选择行式文本协议，平衡可扩展性和内核复杂度 |
| 上下文路径存储位置（内核/用户态） | 采用分层设计：内核维护元数据，用户态存储节点内容，零 syscall 读取 |
| 文件属性存储方式 | 属性内嵌 inode（128→256B），避免额外索引开销 |
| Agent Loop 空转占用 CPU | 引入 Blocked 状态和 agent_wait 实现真休眠 |
| 消息通知邮箱溢出 | 采用 FIFO 丢弃策略，固定容量 MAILBOX_CAP=8 |
