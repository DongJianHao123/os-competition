#### 3.1.1 详细模块介绍
**DNS监控模块**：基于XDP技术，在网卡驱动层拦截UDP/53端口的DNS数据包。内核态程序（dns_cache.bpf.c）仅做最小工作——判断DNS包并发送原始payload到Ring Buffer，用户态程序（dns_loader.c）负责DNS协议解析、哈希表缓存（最大1024条，TTL=30秒）和统计报告。设计上刻意避免在内核态做DNS解析以绕过eBPF verifier指令数限制。使用Ring Buffer（而非perf buffer）和XDP_FLAGS_SKB_MODE保证兼容性。开发过程包含环境搭建（内核≥5.15、BTF支持）、依赖安装、编译验证、运行测试，并提供常见问题排查表。

**gRPC监控模块**：使用eBPF uprobe技术对基于HTTP/2和TLS加密的gRPC服务进行无侵入式监控。选择uprobe而非XDP/socket filter（因加密无法解析方法名），且仅用入口探针避免uretprobe与Go goroutine栈管理冲突。Hook目标为gRPC Handler层函数（如_EchoService_Echo_Handler），通过函数名直接获取服务名和方法名。内核态程序记录调用时间戳和方法名，用户态程序打印事件和统计报告。开发过程包含Go环境安装、gRPC测试服务创建、Hook点定位、编译运行验证。

**双端缓存系统**：核心创新在于同时在客户端和服务端侧维护XDP缓存，通过eBPF BPF Map实现内核态实时同步。支持动态算法切换——用户态通过写入BPF Map实时切换LRU/LFU淘汰算法，无需重启服务。内核态程序（server_cache.bpf.c、client_cache.bpf.c）拦截DNS查询和响应包，用户态缓存管理器（cache_manager.c）实现DNS解析、算法切换、Ring Buffer回调处理和统计报告。验证流程显示第一轮查询全部MISS并写入双端缓存，第二轮全部C-HIT（客户端命中），命中率可达100%。

**性能测试报告**：2026-05-20在Ubuntu 24.04.1 LTS（内核6.8.0）VMware虚拟机上使用dnsperf 2.14.0进行测试。三组对比：基线（直接查询8.8.8.8）QPS 823/延迟120.6ms，dnsmasq本地缓存QPS 35,523/延迟2.8ms，eBPF双端缓存QPS 30,507/延迟3.2ms。eBPF相比基线QPS提升37倍、延迟降低97.3%；相比dnsmasq，最大延迟优86%（24.1ms vs 173.4ms），延迟标准差优63%（1.3ms vs 3.5ms），核心优势为延迟稳定性。

**Grafana可视化监控**：监控栈由Prometheus（端口9090）+ Grafana（端口3000）+ Python Exporter（端口8000）组成。Exporter每5秒读取/tmp/ebpf_cache_stats.txt，暴露7个Gauge指标（总查询数、客户端命中、服务端命中、未命中、命中率、QPS、缓存算法）。Grafana Dashboard包含4个Panel（DNS查询统计、缓存命中率趋势、当前缓存算法、命中率Gauge），2×2布局，5秒自动刷新。

**TC虚拟化优化模块**：在TC ingress/egress钩子实现热点IP检测与协议栈绕过。使用LRU哈希表记录热点IP及访问次数，超过阈值则记录绕过统计。用户态程序（tc_loader.c）定期读取BPF Map展示绕过率和节省时间。实测虚拟化网络协议栈绕过率约98.7%。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| BPF程序过大（指令数超限） | 内核态只做最小工作（判断DNS包、发送payload），用户态负责DNS解析和缓存判断 |
| uretprobe与Go goroutine栈管理冲突导致server panic | 仅使用uprobe（入口探针），不使用uretprobe（返回探针） |
| nslookup走loopback导致XDP无法捕获 | 使用dig @8.8.8.8命令触发DNS查询 |
| VMware VMXNET3不支持Native XDP | 使用XDP_FLAGS_SKB_MODE保证兼容性 |
| BPF verifier报错 | 使用bpf_probe_read_kernel安全读取内核内存，严格边界检查 |
| xdp_attach失败 | 确认网卡名正确，使用ip link show查看实际网卡名 |
| 缓存命中数为0 | 检查DNS查询是否经过目标网卡，确认dig命令使用正确 |
