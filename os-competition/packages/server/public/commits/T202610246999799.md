#### 3.1.1 详细模块介绍
VeriSkain 采用四层模块化架构：采集层（ingest）基于 eBPF（tracepoints + uprobes）在内核态采集进程、文件、网络与 TLS 明文数据，通过 ringbuf 传递到用户态；状态层（state）维护进程、fd、cwd、路径、挂载、session、workspace 等注册表，以及独立的连接/套接字身份引擎（state-net）；内容层（content）进行全局 TLS 片段流重组，生成 PromptObject / SourceArtifact；决策层（decision）派生信号 → 检测器 → 告警策略 → sink（NDJSON、stderr、IPC）。数据流为：BPF(内核) → ringbuf → collector → NormalizedEvent → state/content → decision → Alert。

各 crate 职责明确：veriskein-proto 定义事件镜像与 ID 类型；veriskein-bpf 封装 libbpf 骨架绑定与全局 OpenSSL uprobe attach；veriskein-collector 消费 ringbuf 并校验 per-CPU 序列号；veriskein-normalizer 维护进程表、路径/挂载注册表、fd/cwd 表，进行路径解析与 workspace/sensitive 判定；veriskein-state-net 管理连接/套接字身份与端点历史；veriskein-graph 进行 session 候选与选举、角色分类；veriskein-content 实现流重组、HTTP/1 与 JSON 边界解析、浅层 JSON-RPC 解析；veriskein-correlator 实现 prompt/artifact/chain 关联、MinHash 签名、传播门控与 CAPI 评分；veriskein-detectors 包含 5 类核心检测器（非预期 Shell、敏感文件访问、越界删除、死循环、跨 Agent 提示注入）及可选 MCP 伪造检测；veriskein-alert 负责 schema 校验、severity/confidence 策略映射、节流与 NDJSON 输出；veriskein-ipc 提供本地 UDS 只读协议；veriskein-retention 实现有界保留与压力下降级；veriskein-daemon 是主二进制入口，整合所有 crate 形成完整管线；veriskein-cli 提供 tail --ipc 等客户端功能；veriskein-test 提供场景断言与 replay 回放框架；veriskein-perf 提供性能基准测试。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| 路径归一化：热路径中频繁调用 readlink 导致性能下降 | 设计 PathResolution 类型，热路径无 syscall，仅敏感规则候选才走 readlink |
| fd 状态继承：fork 后子进程 fd 表需要独立 | 实现 fd 表 copy-on-write 策略，exec 后保留未关闭 fd |
| TLS 明文捕获：OpenSSL 动态链接库版本兼容性 | 采用全局 DSO attachment 而非按进程 attach，支持 libssl.so 自动发现 |
| 跨 session 关联误报：纯文本相似度匹配导致高误报 | 设计传播门控机制，跨 session 关联必须先有显式传播事实，再考虑文本相似度 |
| 死循环检测误报：正常循环被误判 | 设计 120s 滑窗多子规则，结合文件 I/O 特征综合判定 |
| 告警节流：相同告警重复输出 | 实现 key=(session_id, type, reason_code, fallback.mode, 主对象哈希) 的 60s 去重机制 |
| 性能基准方法论：daemon 启动时间计入导致测量偏差 | 采用同一 workload 比较不同 capture mode，daemon 启动不计入计时 |
