#### 3.1.1 详细模块介绍
项目采用单内核渐进式架构，通过 Cargo feature gate（lab1~lab5）控制功能启用。内核核心模块包括陷阱处理（trap.rs）、虚拟内存管理（mm.rs）、进程管理（process.rs）、任务调度（task.rs）、文件系统（fs.rs）和同步原语（sync.rs）。6 个独立组件 crate（os-sbi, os-context, os-syscall, os-alloc, os-vm, os-fs）提供底层抽象，可独立发布。用户态提供 lib.rs 封装系统调用和 7 个测试程序（hello, power, yield, fork_test, exec_test, fs_test, pipe_test）。开发过程采用 3 人团队并行开发，使用 Cursor IDE 和 AI 辅助，所有代码经人工审查和测试验证。教学文档包含 5 个 Lab 的渐进式实验指导、习题和答案，以及 VitePress Web 手册。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| ch6 link/unlink 死锁：unlink 调用 clear() 时持有 fs.lock() 导致自旋死锁 | 在修改目录项前更新 nlink，在单次 modify_disk_inode 调用中完成目录项删除 |
| lab4 getpid/wait 栈溢出：fork_user_space 深度调用导致父进程 trap 栈覆盖相邻 slot 的 pid 字段 | 将 per-process 内核栈从 PCB 内嵌字段改为静态数组 KERNEL_STACKS[slot] |
| Lab3 虚拟内存集成：共享内核页表导致三个 app 都运行 yield；sys_write 输出全 \0 | 排除用户 slot 的恒等映射，在切换到内核 satp 前在用户 satp 下复制用户缓冲区 |
| Lab2 yield 仅 1 轮：批处理调度限制 | 记录为已知简化，Lab4 进程调度解决此问题 |
| Lab1 文档 P0 bug：修改链接地址到 0x80100000 未崩溃 | 改用 0x88000000 可靠复现崩溃 |
| clippy static_mut_refs 警告：os-alloc、os-context、os-vm 中使用 static mut | 替换为 SyncUnsafeCell 包装器 |
| 文档重组：多个文档内容重叠、引用断裂 | 合并为项目总报告.md，更新所有交叉引用 |
