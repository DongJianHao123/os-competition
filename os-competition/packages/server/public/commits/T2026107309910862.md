#### 3.1.1 详细模块介绍

**Client 代理库（libcudart_proxy.so）**：这是项目的核心组件，通过 `LD_PRELOAD` 机制在用户程序启动时加载，导出与 CUDA Runtime API 同名的符号，从而接管所有 CUDA 调用。代理库为每个进程创建一个 session，通过 gRPC 将请求转发给 server。代码约2597行，采用单例模式设计 `RuntimeProxyClient` 类，封装了 gRPC 通信、共享内存数据平面、Ring Buffer 提交、会话管理、模块/内核注册等全部逻辑。支持的主要 API 包括：设备查询（cudaGetDeviceCount/Properties）、内存管理（cudaMalloc/Free/HostAlloc/Memcpy）、流管理（StreamCreate/Destroy/Synchronize）、事件管理（EventCreate/Record/Synchronize/Query/ElapsedTime）、内核启动（cudaLaunchKernel）等。未实现的 API 明确返回 `cudaErrorNotSupported`。代码结构清晰但缺少头部注释，存在硬编码地址和超时值、重复的错误处理模式、忙等待等可优化点。

**Server 服务端（vgpu_server）**：服务端负责接收 client 请求，通过 CUDA Driver API 在真实 GPU 上执行。核心组件包括 `session_manager.h`（会话管理器，基于 unordered_map 的线程安全会话生命周期管理）、`ring_worker.h`（Ring worker 辅助工具，支持 CPU 绑定和空闲退避策略）和 `main.cpp`（主程序，约87KB）。Session 管理器设计简洁，使用细粒度锁（manager 级锁 + session 级锁），支持会话的增删查和超时清理。每个 session 维护独立的资源表（allocations、modules、streams、events），实现 per-session 资源隔离。

**gRPC 协议层（proto/vgpu.proto）**：定义了 `VgpuRuntime` gRPC 服务，包含会话管理（CreateSession/DestroySession/Keepalive）、设备查询（GetDeviceCount/GetDeviceProperties）、内存管理（Malloc/Free/Memcpy/MemcpyShm）、流管理（StreamCreate/Destroy/Synchronize）、事件管理（EventCreate/Record/Synchronize/Query/ElapsedTime/Destroy）和内核执行（RegisterModule/LaunchKernel）等接口。所有请求和回复均包含 `cuda_error` 和 `message` 字段用于错误报告。

**Shared memory/ring（shared/vgpu_shm_ring.h）**：定义了基于共享内存的无锁环形缓冲区，用于 client 与 server 之间的命令通信。容量固定为1024个条目，使用 GCC/Clang 内置原子操作实现无锁并发。支持7种操作码（MemcpyShm、MemcpyD2D、DeviceSynchronize、StreamSynchronize、EventSynchronize、EventQuery、LaunchKernel），每个条目包含14个时间戳字段用于细粒度性能追踪。设计精巧，通过魔数和版本号进行运行时校验。

**测试工具（tools/）**：包含完整的测试体系，包括烟雾测试（vector_add、runtime_query、memcpy、host_alloc、stream_async、event_timing）、性能基准测试（matrix_mul_benchmark）、并发测试（concurrent_worker、cross_session_sync_test）、压力测试（fire_forget_stress、stability_negative_smoke）和 Python 验收验证脚本（run_acceptance_validation.py、run_cross_session_sync_test.py）。验收脚本覆盖矩阵乘法性能比较、并发工作负载、错误隔离、崩溃隔离和长期稳定性测试。

### 4.2 遭遇的代码问题
| 问题描述 | 解决方法 |
| --- | --- |
| 初始实现中缺少会话隔离，多个client进程共享同一资源空间 | 重构server为per-session架构，每个client进程独立session，资源挂在session下 |
| 大块数据传输通过gRPC效率低 | 引入POSIX shared memory数据平面，大块H2D/D2H走SHM |
| 高频小命令（kernel launch、sync）RPC开销大 | 实现per-session SPSC ring buffer，无锁原子操作，降低延迟 |
| 长时间运行可能产生僵尸会话 | 实现Keepalive心跳和Ring超时回退（默认5秒），Server端共享内存清理（shm_unlink） |
| 虚拟地址查找效率低 | 使用std::map + upper_bound实现O(log n)指针查找 |
| 多进程并发时同步隔离问题 | 实现private default stream，cudaDeviceSynchronize()只同步当前session的stream集合 |
| 代码中硬编码服务器地址和超时值 | 通过环境变量（VGPU_SERVER、VGPU_RING_TIMEOUT_US等）可配置 |
| 忙等待（spin-wait）消耗CPU | 实现RingIdleBackoff退避策略：忙等→yield→睡眠1us→睡眠20us |
