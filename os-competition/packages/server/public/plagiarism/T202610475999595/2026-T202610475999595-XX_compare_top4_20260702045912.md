### 4. 核心代码调用流对比剖析
* **【进程管理/调度路径】**
* **底层技术析理**: 两者均遵循 xv6 的进程调度模型，即 `yield` -> `sched` -> `swtch` 的调用流。这是由 xv6 的教学设计决定的，任何基于 xv6 的实现都必然遵循此路径。然而，目标项目在系统调用层引入了 `kexit`、`kkill`、`kwait` 等 `k-` 前缀的内核函数，将系统调用入口与内核实现分离，这与参考项目直接将 `exit`、`kill` 作为内核函数调用形成鲜明对比，体现了不同的模块化设计思想。
* **代码溯源链接**:
  - 目标项目: `kernel/sysproc.c#sys_exit` -> `kernel/proc.c#kexit`
  - 参考项目: `kernel/sysproc.c#sys_exit` -> `kernel/proc.c#exit`

* **【文件系统/对象管理模型】**
* **底层技术析理**: 这是双方架构差异的决定性证据。目标项目采用传统的 **inode 模型**，文件关闭操作 `fileclose` 会调用 `iput` 来释放 inode 引用。参考项目则采用 **FAT32/entry 模型**，其 `fileclose` 调用 `eput` 来释放 FAT32 目录项。这两种模型在内存管理、磁盘布局和并发控制上完全不同，不可能通过简单的“混淆”或“移植”实现转换。
* **代码溯源链接**:
  - 目标项目: `kernel/file.c#fileclose` -> `kernel/fs.c#iput`
  - 参考项目: `kernel/file.c#fileclose` -> `kernel/fat32.c#eput`

* **【内存管理/路径】**
* **底层技术析理**: 在实现 `sys_getcwd` 时，目标项目完全依赖文件系统的 inode 操作（`dirlookup`, `ilock`）来解析路径，内存仅用于临时缓冲。而参考项目则显式地使用了内存管理原语（`kalloc`, `mappages`），暗示其可能采用了不同的用户态/内核态数据传递机制或地址空间管理策略。这进一步证明双方在核心子系统交互方式上存在独立设计。
