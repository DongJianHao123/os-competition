### 4. 核心代码调用流对比剖析

*   **【进程管理/调度原语】**

    *   **代码溯源链接**:
        - 目标项目: `kernel/proc.c#yield`
        - 参考项目: `kernel/proc.c#yield`

*   **【任务调度/同步原语】**
    *   **底层技术析理**: 控方将自旋锁 `acquire` 视为核心证据。其调用流 `__sync_lock_test_and_set -> __sync_synchronize -> holding -> push_off` 是 RISC-V 架构下，利用 GCC 内置原子操作实现自旋锁的**标准化模板**。`__sync_synchronize` 是内存屏障，`push_off` 是 RISC-V 中断禁用，此组合是保证多核临界区正确性的唯一路径。任何独立开发者查阅 GCC 手册和 RISC-V 特权规范后，都会写出逻辑相同的代码。
    *   **代码溯源链接**:
        - 目标项目: `kernel/spinlock.c#acquire`
        - 参考项目: `kernel/spinlock.c#acquire`

*   **【文件系统/目录查找】**
    *   **底层技术析理**: 这是证明双方架构差异的关键。目标项目的 `dirlookup` 基于传统 inode 文件系统，调用 `iget`（获取内存 inode）和 `readi`（从 inode 读取数据）。参考项目的 `dirlookup` 基于 FAT32 文件系统，调用 `eget`（获取 FAT32 目录项）和 `enext`（遍历目录簇链）。两者虽然函数同名，但其**内存对象模型（inode vs. FAT32 entry）和磁盘访问策略完全不同**，是底层架构异构的铁证。

### 6. 保守结论

本次仲裁的核心发现是，17.54% 的低 AST 相似度以及双方在文件系统（inode vs. FAT32）、内存管理（是否含写时复制）、系统调用扩展（`sys_pause` vs. `sys_clone`）等核心板块上的根本性架构差异，排除了大规模代码抄袭的可能。控方基于少数“底层逻辑一致”的指控，实质上是将 RISC-V 硬件强制逻辑、POSIX 系统调用语义以及 MIT xv6 教学内核的公共范式，误判为抄袭证据。目标项目采用的 `kexit`/`kfork` 等独立封装风格，以及参考项目独有的 FAT32、DMA 等复杂功能，进一步印证了双方是在同一公开教材基础上，为不同教学或实验目标而进行的独立开发与演化。整体抄袭风险极低。
