#### 4. 核心代码调用流对比剖析
* **【进程管理/进程终止路径】**
* **底层技术析理**: 目标项目的 `kexit` 函数在进程终止时深度集成了文件系统的日志事务机制（`begin_op`/`end_op`），并显式释放 inode 引用（`iput`），表明其将进程终止视为一个需要保证原子性与资源回收的复杂事务。参考项目的 `exit` 函数则直接调用 SBI 固件接口 `sbi_shutdown` 进行关机，并内联了 `reparent` 逻辑，其终止路径更侧重于与底层固件的交互。两者在资源清理的抽象层次与同步策略上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#kexit`
  - 参考项目: `kernel/proc.c#exit`

* **【进程管理/进程创建路径】**
* **底层技术析理**: 目标项目通过 `sys_fork` -> `kfork` 的调用流创建进程，`kfork` 是典型的内核内部封装函数。参考项目则通过 `sys_fork` -> `clone` 的调用流，`clone` 是 Linux 兼容层的标准接口，支持更细粒度的资源共享选项。这反映了两者在进程模型上的根本分歧：一个坚持 xv6 风格的内核封装，另一个向 Linux 兼容层靠拢。
* **代码溯源链接**:
  - 目标项目: `kernel/sysproc.c#sys_fork`
  - 参考项目: `kernel/sysproc.c#sys_fork`

* **【任务调度/异步运行时/VirtIO 磁盘中断处理】**
* **底层技术析理**: 两者在 `virtio_disk_intr` 中均遵循 VirtIO 规范处理设备中断，调用 `acquire`/`release` 保护临界区并使用 `wakeup` 唤醒等待进程。然而，参考项目在中断处理中调用了 `free_chain` 来显式回收 VirtIO 描述符链资源，目标项目则未体现此调用，暗示其在描述符管理上可能采用了不同的分配/回收策略或更简化的模型。
* **代码溯源链接**:
  - 目标项目: `kernel/virtio_disk.c#virtio_disk_intr`
  - 参考项目: `kernel/virtio_disk.c#virtio_disk_intr`

#### 5. 单方模块与降权项
- **单方模块**: 参考项目独有的 `文件系统/第三方库` 模块（`third/ext4_rs`，323 个函数）是双方架构差异的最显著标志。目标项目完全未包含此 Rust 实现的 ext4 文件系统库，若存在抄袭，遗漏如此大规模且可直接复用的代码库不合逻辑。
- **降权项**: 控方指控中多次出现的 `acquire`、`release`、`myproc`、`printf`、`panic`、`NELEM` 等函数或宏，属于操作系统内核的基础原语、标准库函数或通用工具宏。它们在所有基于 xv6 的项目中必然出现且调用模式固定，不能作为指向特定参考项目的同源指纹。跨板块出现的同名函数（如将 `kernel/fs.c#namex` 归入任务调度板块）亦被降权，因其分类失当，夸大了特定板块的相似度。

本案客观 AST 相似度仅 16.76%，目标项目对参考项目的覆盖率仅 21.29%，且目标项目拥有 277 个独有 AST 指纹，表明其存在大量独立实现。双方相似性高度集中于受 RISC-V 硬件 ABI、xv6 教学内核标准范式严格约束的底层函数（如自旋锁、PLIC 中断初始化、系统调用入口）。在核心的进程管理路径（`kexit` vs `exit`）和进程创建模型（`kfork` vs `clone`）上，双方展现出截然不同的架构设计与外部依赖选择。特别是参考项目独有的 `third/ext4_rs` 第三方库，目标项目完全缺失，这是否定单向抄袭的有力反证。综合判定，现有证据无法支持抄袭指控，项目间相似性可合理解释为基于共同开源基线的独立开发与方向性分化。
