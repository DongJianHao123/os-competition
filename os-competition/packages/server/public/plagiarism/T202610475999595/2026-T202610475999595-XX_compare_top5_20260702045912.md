## 4. 核心代码调用流对比剖析

### 【进程管理/进程退出路径】
- **底层技术析理**: 目标项目 `sys_exit` → `kexit` 调用链与参考项目 `sys_exit` → `exit` 调用链在系统调用分发层一致（`argint` 解析参数后调用内部函数），但在底层资源释放路径上存在实质性差异。目标项目 `kexit` 调用 `fileclose` → `iput` → `begin_op`/`end_op`（xv6 原始文件系统 inode 释放 + 日志事务），参考项目 `exit` 调用 `freeproc` + `copyout2`（进程结构释放 + 用户态数据拷贝），且参考项目 `forkret` 额外调用 `fat32_init`（FAT32 文件系统初始化）。两者在文件系统抽象层和资源管理模型上存在显著架构差异。
- **代码溯源链接**:

  - 参考项目: `kernel/sysproc.c#sys_exit` → `kernel/proc.c#exit`

- **底层技术析理**: 目标项目 `sys_fork` → `kfork` 与参考项目 `sys_fork` → `fork` 在系统调用分发层一致，但目标项目使用 `kfork` 命名（内核内部实现），参考项目直接使用 `fork` 命名。两者均遵循 xv6 标准进程创建流程（分配进程结构 → 复制页表 → 设置陷阱帧 → 返回 PID），调用图谱相似性由 xv6 规范决定。

### 【任务调度/异步运行时/自旋锁获取】
- **底层技术析理**: 双方 `acquire` 函数调用图谱均为 `__sync_lock_test_and_set` → `__sync_synchronize` → `holding` → `mycpu` → `panic` → `push_off`。这是 RISC-V 架构 + GCC 编译器环境下实现自旋锁获取的唯一最优序列：GCC 原子操作实现 TAS 锁、内存屏障保证顺序一致性、关中断防止死锁、`holding` 检测重复获取。任何独立实现都会收敛至此调用序列，属于架构约束收敛，不构成抄袭证据。
- **代码溯源链接**:
  - 目标项目: `kernel/spinlock.c#acquire`
  - 参考项目: `kernel/spinlock.c#acquire`

### 【任务调度/异步运行时/内核陷阱处理】
- **底层技术析理**: 双方 `kerneltrap` 函数均调用 `r_scause`/`r_sepc`/`r_sstatus` 读取 RISC-V CSR 寄存器，这是 RISC-V 特权架构规范规定的陷阱处理必要操作。`devintr` 设备中断分发、`intr_get` 中断状态检查、`panic` 异常处理是 xv6 标准陷阱处理流程。参考项目调用图谱多出 1 个节点（`r_stval` 读取陷阱值寄存器），说明参考项目有额外的陷阱信息记录逻辑。
