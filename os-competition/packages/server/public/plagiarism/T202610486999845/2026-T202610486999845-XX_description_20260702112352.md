## 第四部分：分模块架构说明

### MM（内存管理）
**底层证据部分体现**。证据主要来自用户态地址校验与拷贝原语：
- `task::copy_from_user`、`task::copy_to_user`
- `task::is_user_range_readable`、`task::is_user_range_writable`、`task::is_user_range_mapped`
- `activate_address_space`、`restore_user_memory_snapshot`、`discard_user_memory_snapshot`

这些函数说明内核具备用户地址空间访问控制、fork/clone/exec 时的内存快照切换能力。但**物理页分配器、页表建立/解除映射、TLB 刷新、内核堆分配器等底层内存管理子系统未在提供的 Call Map 中直接出现**。

### PROC（进程管理/任务调度）
**底层证据较强**。覆盖内容广泛：
- **进程生命周期**：`exec_resolved_inode`、`reset_for_program`、`finish_child`、`finish_child_context`
- **信号机制**：`sys_rt_sigaction`、`sys_sigaltstack`、`sys_rt_sigtimedwait`、`write_minimal_siginfo`、`write_waitid_siginfo`
- **定时器**：`sys_setitimer`、`sys_timer_settime`、`sys_clock_nanosleep`
- **IPC**：`sys_msgsnd`、`sys_msgrcv`、`sys_msgctl`、`sys_semctl`、`sys_semtimedop`、`write_shmid_ds`、`write_semid_ds`
- **同步**：`sys_futex`、robust futex list
- **权限与资源**：`sys_capset`、`sys_capget`、`sys_prlimit64`、`sys_getrusage`
- **调度相关**：`sys_sched_setattr`、`run_pending_thread_after_syscall`

但**真正的抢占式调度器（时间片轮转、时钟中断驱动的上下文切换）证据不足**；目前更像“单线程/协程式 pending thread 切换 + 系统调用返回时调度”。

### FS（文件系统）
**底层证据最强**。实现了大量 POSIX 文件系统调用：
- 打开/创建：`sys_openat`、`sys_openat2`、`sys_open_by_handle_at`、`sys_mq_open`
- 读写：`sys_read`、`sys_write`、`read_fd_to_kernel`、`write_fd_from_kernel`
- 元数据：`sys_fstatat`、`sys_statfs`、`sys_readlinkat`、`sys_renameat2`
- 挂载：`sys_mount`、`sys_umount2`
- 设备与控制：`sys_ioctl`、loop device、eventfd、inotify、epoll
- 后端：内存文件表（`MEM_FILES`、`FdEntry`）、ext4 只读解析（`fs/ext4.rs#lookup_child`、`read_next_ext4_dirent`）、dummy mount、proc 伪文件

### DEV（设备驱动）
**底层证据很弱**。仅见：
- `drivers/virtio.rs#read_sector`：VirtIO-blk 块设备扇区读取
- `sbi.rs#console_putchar`：SBI 串口控制台输出

**中断控制器（PLIC/CLINT）、定时器设备、网络设备、磁盘缓存、DMA 等均未体现**。

### SYSCALL（系统调用）
**证据强但形态集中**。`syscall/mod.rs#syscall` 是超大规模分发器（调用数 275），将请求路由到 `syscall/fs.rs`、`syscall/process.rs` 中的具体实现。用户态侧有极简库：`user/src/lib.rs#_start`、`user/src/syscall.rs#syscall`（内联汇编）。

### TRAP/IRQ（中断异常）
**底层证据弱**。仅：
- `trap/mod.rs#init` → `arch.rs#write_stvec`
- `trap/mod.rs#trap_handler` → `read_scause`、信号/退出处理

未见 PLIC/CLINT 初始化、外部中断分发、定时器中断处理等代码。

### NET（网络协议栈）
**底层证据未体现**。提供的 Call Map 中没有任何 socket 协议栈、TCP/IP、网络设备驱动或 netfilter 相关实现。虽然 `sys_accept`/`sys_accept4` 出现在 `syscall/mod.rs#syscall` 的调用链中，但具体实现未在证据池中展开，不能判定为完整网络栈。

## 第六部分：边界与未体现内容

1. **没有底层证据支撑的模块**：
   - 完整的物理内存管理器（buddy/slab/页框分配）
   - 页表建立、解除映射、TLB 刷新等虚拟内存映射层
   - 网络协议栈（TCP/IP、socket 层具体实现未展开）
   - 中断控制器驱动（PLIC/CLINT 等）
   - 定时器硬件驱动与时钟中断处理
   - SMP/多核调度与核间中断
   - 块缓存/page cache 层
   - 安全模块（除 capabilities 外的 LSM、seccomp 等）

2. **设备驱动证据极弱**：仅见 `VirtIO-blk` 扇区读取与 SBI 控制台输出，不能推断项目实现了完整的设备驱动框架。

3. **`os/` 与 `os-la/` 的关系**：两者文件结构、函数名、调用链几乎一致，仅个别函数规模有微小差异。现有证据只能说明它们是两套高度同构的代码树，**无法确定它们是不同架构、不同板级配置还是实验分支**。

4. **ext4 与第三方/公共代码**：`fs/ext4.rs` 中的 `lookup_child`、`for_each_child`、`read_next_ext4_dirent` 表明项目包含 ext4 文件系统解析逻辑，但应单独说明，**不宜直接等同于原创内核主体**。

5. **“内存管理”模块的判定边界**：本分析将大量涉及 `copy_from_user`/`copy_to_user`、用户内存快照的函数归入内存管理证据，但这仅代表用户态内存访问控制，**不等同于完整的内核 MM 子系统**。

6. 本文档所有判断均基于 AST Call Map、函数名、路径锚点与调用链，**仅为技术方向推测，不构成最终身份或原创性结论，也不做抄袭/违规判定**。
