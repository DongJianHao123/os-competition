## 4. 核心代码调用流对比剖析

### 【进程管理 / 线程退出路径】

**底层技术析理**：

目标项目与参考项目在进程/线程退出路径上呈现根本性的架构差异：

- **目标项目**采用基于全局调度器 `SCHEDULER` 的显式上下文切换模型。退出流程为：`SCHEDULER.lock` → `Thread::current().expect` → `current.release_kernel_fs_on_exit`（释放内核文件系统资源）→ `current.set_state(Exited)` → `next.set_state(Running)` → `processor::switch_to_thread`（主动切换到下一线程）。该模型体现了完整的 Linux 兼容内核设计——在退出时需清理内核文件系统引用、通过调度器选择下一就绪线程、执行显式栈切换。

- **参考项目**采用基于退出任务队列 `EXITED_TASKS` 的异步清理模型。退出流程为：`EXITED_TASKS.lock` → `crate::current` → `curr.inner.lock().set_state` → `axhal::misc::terminate`（直接终止执行）。该模型体现了协程式运行时的设计——退出任务被放入队列等待后续清理，当前执行流直接终止，无显式上下文切换。

**关键差异总结**：
| 维度 | 目标项目 | 参考项目 |
|------|---------|---------|
| 调度器抽象 | `SCHEDULER`（全局抢占式调度器） | `EXITED_TASKS`（退出任务队列） |
| 任务抽象 | `Thread`（内核级线程） | `async_task`（协程任务） |
| 切换机制 | `processor::switch_to_thread`（显式栈切换） | `axhal::misc::terminate`（直接终止） |
| 清理行为 | `release_kernel_fs_on_exit`（内核资源释放） | 无内核文件系统清理 |

- 目标项目: `os/src/thread/scheduler.rs#exit_current`

### 【内存管理 / 堆分配路径】

**底层技术析理**：

- **目标项目**使用分级 slab 分配器。分配路径为：`CommonSizeClass::from_size(size)` → `GLOBAL_HEAP_ALLOCATOR.lock()` → `alloc_from_class`（从对应大小类的 slab 缓存分配）→ 若失败则 `HeapSlot::alloc_large`（大块直接分配）。该模型通过预分配固定大小的 slab 页减少碎片，适用于需要精细内存控制的宏内核场景。

- **参考项目**使用字节级分配器。分配路径为：`balloc.add_memory` → `balloc.alloc(layout)` → `balloc.total_bytes`。该模型基于字节级分配器（类似 buddy allocator 变体），适用于异步运行时场景的简单内存管理。

**关键差异总结**：
| 维度 | 目标项目 | 参考项目 |
|------|---------|---------|
| 分配策略 | 分级 slab（`CommonSizeClass`） | 字节级分配（`balloc`） |
| 元数据结构 | `HeapSlot`、`SlabCache` | `balloc` 内部位图 |
| 碎片管理 | 按大小类隔离，减少外部碎片 | 字节级合并，依赖分配器算法 |
