#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 在`do_exit`函数中，双方均需完成futex清理、共享内存清理、信号发送等步骤，这是Linux进程退出的标准流程。然而，**同步策略与调用顺序**揭示了架构差异。目标项目在`axtask::yield_now`之后才调用`crate::syscall::SHM_MANAGER.lock().clear_proc_shm`，而参考项目在`yield_now`之前就通过`SHM_MANAGER.lock().clear_proc_shm`完成清理。这种核心流程中资源清理顺序的差异，是不同开发者独立设计的结果，而非代码抄袭。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/ops.rs#do_exit`
  - 参考项目: `api/src/task.rs#do_exit`

* **【进程管理/pidfd类型模型】**
* **底层技术析理**: `sys_pidfd_open`系统调用展示了双方在**对象管理模型**上的根本不同。目标项目对pidfd进行了更精细的抽象，区分了`PidFd::new_process`和`PidFd::new_thread`，并引入了`PidFdFlags`和`fd.set_nonblocking`处理。而参考项目则使用单一的`PidFd::new`构造器。这种类型体系的设计差异无法通过简单的“换皮”实现，是独立架构设计的明证。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/fs/pidfd.rs#sys_pidfd_open`
  - 参考项目: `api/src/syscall/fs/pidfd.rs#sys_pidfd_open`

* **【任务调度/异步运行时/futex等待】**
* **底层技术析理**: 在`wait_if`核心同步原语中，双方选择了不同的调度策略。目标项目调用`block_on`进行阻塞等待，而参考项目调用`block_on_interruptible`，表明后者考虑了信号中断的场景。这一差异反映了双方在**异步运行时调度粒度与可中断性**设计上的不同考量，是底层架构异构的体现。

#### 6. 保守结论
本次审计的核心结论是**独立开发**。20.11%的客观AST相似度从根本上否定了系统性抄袭的可能。控方提出的“100%同构咬合”指控，其证据高度依赖于对Linux/POSIX标准接口、公共开源框架API以及Rust语言通用模式的过度解读。辩方成功论证了这些相似性源于**领域必然性、框架共性和规范强制**。目标项目拥有远超参考项目的独立AST指纹，并在设备驱动、网络协议栈等关键模块上存在结构性缺失，在pidfd类型模型、异步原语、错误处理体系等核心实现上展现出明确的架构差异。因此，整体风险极低，不构成实质性代码抄袭。
