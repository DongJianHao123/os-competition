## 4. 核心代码调用流对比剖析

### 【进程管理/进程退出路径】

* **底层技术析理**: 
  - **目标项目**采用**扁平化内联退出模型**：`sys_exit` 直接调用 `curproc` 获取当前进程、`wait_queue_wakeup_one` 唤醒等待者、`sched_yield` 让出 CPU，最后 `panic` 终止。整个退出逻辑在系统调用入口一次性完成，无分层抽象。
  - **参考项目**采用**分层委托退出模型**：`sys_exit` 仅调用 `argint` 解析参数，然后委托给通用 `exit` 函数。`exit` 函数内部调用 `__futex_wake`（Futex 唤醒）、`fdtbl_closeall`（关闭所有文件描述符）、`pstate_migrate`（进程状态迁移）、`memset_user`（清零用户内存）等细粒度资源回收函数。
  - **异构性**：目标无 Futex、无文件描述符表关闭、无进程状态机迁移、无用户内存清零等参考项目的核心退出原语。目标使用 `wait_queue_wakeup_one`（等待队列原语），参考使用 `__futex_wake`（Futex 原语），同步机制不同。二者在退出路径的抽象层次、资源回收粒度、同步原语选择上存在根本性架构差异。

* **代码溯源链接**:
  - 目标项目: `src/syscall.c#sys_exit`
  - 参考项目: `src/kernel/sysproc.c#sys_exit` → `src/kernel/proc.c#exit`

---

### 【进程管理/当前进程获取】

* **底层技术析理**:
  - **目标项目**：`curproc` 仅调用 `thiscpu`，直接从 CPU 私有结构体中获取当前进程指针。实现极简，无中断屏蔽/开启操作，无断言检查。
  - **参考项目**：`myproc` 调用 `mycpu` 获取 CPU 结构体，**且必须包裹 `push_off`/`pop_off` 中断屏蔽对**，并包含 `assert` 断言检查进程状态有效性。这是 xv6 风格的中断安全设计。
  - **异构性**：目标无中断屏蔽机制（`push_off`/`pop_off`），参考强制要求中断屏蔽以保证获取当前进程的原子性。二者在并发安全模型上存在根本差异——目标可能依赖不同的同步策略（如 `thiscpu` 内部已处理），参考显式依赖中断屏蔽。

* **代码溯源链接**:
  - 目标项目: `src/syscall/syscall.c#curproc`
  - 参考项目: `src/kernel/proc.c#myproc`

---

### 【任务调度/进程入队】

* **底层技术析理**:
  - **目标项目**：`enqueue_proc` 使用 `list_add_tail` 将进程加入链表尾部，配合 `r_sstatus`/`w_sstatus` 读取/写入 RISC-V `sstatus` 寄存器，以及 `spinlock_acquire`/`spinlock_release` 自旋锁保护。这是直接操作 CSR 寄存器+链表的标准模式。
  - **参考项目**：调度相关函数（如 `sched_timeout`）使用 `set_timer`/`remove_timer` 定时器机制和 `sched` 调度函数，不直接操作 `sstatus` CSR 寄存器，也不使用 `list_add_tail` 链表操作。
  - **异构性**：目标直接操作 RISC-V 特权寄存器实现调度同步，参考通过定时器中断驱动调度。链表操作 vs 定时器驱动，底层机制完全不同。

* **代码溯源链接**:
  - 目标项目: `src/task/sched.c#enqueue_proc`
  - 参考项目: `src/kernel/proc.c#sched_timeout`

---

### 【内存管理/内存映射】

* **底层技术析理**:
  - **目标项目**：`sys_mmap` 调用 `PAGE_ALIGN_UP`（页对齐）、`find_free_user_range`（查找空闲用户地址空间）、`shared_vm_owner`/`shared_vm_lock`（共享虚拟内存所有者/锁机制）、`file_read`（文件读取）等。内存管理围绕**共享虚拟内存（shared VM）** 概念构建，支持父子进程间共享内存映射同步（`sync_shared_mmap_base_children`）。
  - **参考项目**：`sys_mmap` 调用 `mmap_map`（内存映射核心）、`fdtbl_getfile`（文件描述符表获取文件）、`argaddr`/`argint`（参数解析）。内存管理基于 **VMA（Virtual Memory Area）** 模型（`__vma_find_strict`），无共享虚拟内存所有者概念。
  - **异构性**：目标使用 `shared_vm_owner` 共享内存所有权模型，参考使用 VMA 区间查找模型。二者在虚拟内存管理的抽象层次和数据结构上完全不同。

* **代码溯源链接**:
  - 目标项目: `src/syscall/syscall.c#sys_mmap`
  - 参考项目: `src/kernel/sysfile.c#sys_mmap`

---

### 【中断/异常处理/Trap 初始化】

* **底层技术析理**:
  - **目标项目**：`trap_init` 直接操作 RISC-V CSR 寄存器：`r_sie`/`w_sie`（读写中断使能）、`r_sstatus`/`w_sstatus`（读写状态寄存器）、`set_kerneltrap`（设置内核 Trap 处理函数）。这是裸机 CSR 编程风格。
  - **参考项目**：`trap_init` 仅调用 `init_timer` 初始化定时器，不直接操作 `sie`/`sstatus` 寄存器。Trap 处理函数通过其他路径设置。
  - **异构性**：目标在 `trap_init` 中集中完成所有 CSR 配置，参考将定时器初始化分离。初始化策略和寄存器操作粒度不同。

* **代码溯源链接**:
  - 目标项目: `src/trap/trap.c#trap_init`
  - 参考项目: `src/kernel/trap.c#trap_init`

## 6. 保守结论

本案双方项目客观 AST 相似度仅 1.82%，双向覆盖率均不足 5%，处于独立开发项目的正常统计波动范围内。共同 AST 指纹高度集中于第三方 `printf` 公共库（19/25），核心内核模块（进程管理、内存管理、任务调度、中断处理）的共同指纹均为 0~1 个，且均源于 POSIX 标准接口同名或通用宏使用，非原创性逻辑复用。双方在进程退出路径（内联扁平 vs 分层委托）、当前进程获取（无中断屏蔽 vs 强制 `push_off`/`pop_off`）、调度机制（CSR 直接操作+链表 vs 定时器驱动）、内存管理模型（共享虚拟内存所有权 vs VMA 区间查找）等核心架构维度上存在根本性差异，调用图谱零重叠或仅共享公共库节点。控方证据存在跨板块硬凑（将 Bootloader Rust 代码归入进程管理）、功能等价混淆为代码复用、证据底座 API 认证失败等问题。综合判定：**无充分证据支持抄袭指控，双方项目为独立开发**。
