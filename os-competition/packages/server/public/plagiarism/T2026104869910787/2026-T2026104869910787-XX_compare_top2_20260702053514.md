* **【内存管理 / ELF加载】**
* **底层技术析理**: 两者均实现了从ELF文件到内存映射的`from_elf`函数，但展现出不同的设计深度。参考项目采用简单的段映射策略。目标项目则引入了`DL_INTERP_OFFSET`，显示其支持动态链接器，内存布局更复杂。这表明目标项目并非简单复用，而是在参考项目基础上进行了深度的架构扩展与自研。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/memory_set.rs#from_elf`
  - 参考项目: `os/src/mm/memory_set.rs#from_elf`

* **【进程管理 / 程序执行】**
* **底层技术析理**: `sys_execve`是进程创建的核心。参考项目通过`app_inode.read_all`将可执行文件完整读入内存后解析，是一种简单直接的实现。目标项目则通过`file.get_path`获取路径，再调用`extract_cstrings`处理参数，更接近标准Unix的路径解析与参数传递模型。两者在对象管理模型（文件vs路径）和调用流上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/task.rs#sys_execve`
  - 参考项目: `os/src/syscall/process.rs#sys_execve`

* **【文件系统 / FAT32兼容】**
* **底层技术析理**: 在`write_dentry`中，双方调用流高度一致（`(1..=ldir_count).rev`, `s_lname!`, `shortname_checksum`）。这是因为FAT32长文件名条目必须按倒序写入，且校验和算法是公开且固定的。这种相似性是遵循外部技术规范的必然结果，属于“规范收敛”，而非内部逻辑抄袭。
* **代码溯源链接**:

综合全模块证据，目标项目与参考项目的客观AST相似度极低(6.42%)，远未达到同源性判定门槛。控方提出的“直接抄袭”指控，其核心证据`forktree.rs`系公共用户态模板，其余证据则源于对POSIX标准、FAT32规范、RISC-V架构强制指令等外部约束的共同遵循，属于典型的“规范收敛”现象。辩方关于独立实现、架构扩展（如自研网络栈、LA64支持）的辩护理由充分。目标项目在内存管理、进程模型等核心子系统上展现出与参考项目不同的设计选择与实现路径。因此，判定两项目为**独立开发**，不存在违规抄袭或套壳复用行为。
