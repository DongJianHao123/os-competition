- **核心依据**: 双向AST覆盖率极低（目标项目9.46%，参考项目8.99%），共同AST指纹仅175个，远低于双方各自拥有的唯一AST指纹（1850与1946）。该数据表明两个项目在底层代码实现上存在根本性、系统性的差异。控方关于“进程管理”板块存在“系统性复用”的指控，与该板块仅2个共同唯一AST指纹、覆盖率不足2%的客观事实严重不符。双方在关键子系统上的架构选型、同步原语、对象模型均表现出显著异构性，符合独立开发的典型特征。

### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 目标项目的`sys_exit`调用`kernel_exit`和`schedule`，显式地执行内核退出和调度逻辑，而参考项目则封装为`exit_current_and_run_next`。`sys_exit_group`的差异更为显著，目标项目通过`task.op_thread_group_mut`遍历线程组并逐个调用`kernel_exit`，而参考项目则使用`exit_group_and_run_next`，其内部通过`VecDeque`管理退出列表。这体现了完全不同的进程生命周期管理模型和并发控制策略。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/task.rs#sys_exit`
  - 参考项目: `os/src/syscall/process.rs#sys_exit`

* **【内存管理/内核内存初始化】**
* **底层技术析理**: 目标项目的`new_kernel`采用手动构建`MapArea`并逐个插入`MemorySet`的方式，映射关系通过`(VirtAddr, PhysAddr, MapPermission)`元组数组显式定义。参考项目则使用`anonymous_identical_map!`宏进行批量匿名映射，并通过`should_map_trampoline!`条件编译宏处理trampoline页。前者是显式、过程式的构建，后者是声明式、宏驱动的构建，内存映射的抽象层次和实现哲学截然不同。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/memory_set.rs#new_kernel`
  - 参考项目: `os/src/mm/memory_set.rs#new_kernel`

* **【系统调用/分发机制】**
* **底层技术析理**: 目标项目的`syscall`分发函数使用一个预定义的`CARELESS_SYSCALLS`黑名单来过滤需记录日志的系统调用，然后通过一个巨大的`match`语句分发。参考项目则使用一个白名单`[...].contains`来检查是否应记录日志，其分发逻辑可能不同。两者在日志策略（黑名单 vs 白名单）和分发实现上存在结构性差异。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/mod.rs#syscall`
  - 参考项目: `os/src/syscall/mod.rs#syscall`

### 5. 单方模块与降权项
- **单方模块**: 参考项目独有的“网络协议栈/第三方库”和“第三方库”模块（共890个函数），目标项目完全没有。这表明双方在项目依赖和代码复用策略上存在根本性分歧，是独立开发的有力证据。
- **降权项**:
  - **POSIX/标准接口同名**: `sys_getpid`、`sys_execve`等函数名由Linux系统调用标准强制定义，同名不构成同源证据。
  - **最小实现/trait模板**: `readable`、`writable`等“内部闭环逻辑”函数，因其实现体量极小，AST指纹极易趋同，属于“最小实现必然相似”，已从强证据中降权。
  - **跨板块同名**: `dentry.rs#insert`与`file_descriptor.rs#insert`虽同名，但分属文件系统内部的不同子模块，操作对象完全不同，不构成同板块主证据。
  - **硬件驱动代码**: LoongArch寄存器操作（如`estat.rs#cause`）的相似性源于对同一硬件手册的标准实现，不视为软件架构复用。

### 6. 保守结论
综合全模块AST证据，目标项目与参考项目的双向代码覆盖率均低于10%，共同AST指纹仅175个，远低于双方各自数千个的唯一指纹。控方对“进程管理”板块的指控与不足2%的覆盖率严重矛盾，其列举的`readable`/`writable`等函数属于因实现体量过小而极易趋同的“最小实现”，不能作为抄袭证据。双方在进程调度、内存映射、系统调用分发等核心子系统上展现出截然不同的同步原语、抽象层次和实现路径。因此，整体代码相似性风险极低，不存在系统性复用或抄袭行为，应认定为独立开发。
