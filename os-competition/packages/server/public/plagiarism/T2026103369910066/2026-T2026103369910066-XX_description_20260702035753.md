## 第四部分：分模块架构说明

### MM（内存管理）

**有底层证据支持**：

- 地址空间管理：`os/src/mm/memory_set.rs#map_elf_loads`、`os/src/mm/address.rs`
- 系统调用：`os/src/syscall/process.rs#sys_mmap`、`sys_munmap`、`sys_mprotect`、`sys_sbrk`
- 懒加载页缺失：`os/src/trap/mod.rs#trap_handler` → `memory_set.handle_lazy_page_fault`
- 资源释放：`os/src/task/id.rs#dealloc_user_res` → `remove_area_with_start_vpn`
- fork 时地址空间克隆：`os/src/task/process.rs#fork` → `MemorySet::from_existed_user`
- 页表实现依赖 `third_party/riscv/src/paging/page_table.rs` 与 `multi_level.rs`，非原创页表代码。

**未体现**：交换（swap）、NUMA、大页、完整的缺页统计/调试接口。

### PROC（进程管理/任务调度）

**有底层证据支持**：

- 进程创建：`os/src/task/process.rs#init_main_task`、`fork`、`exec_with_interp_env`
- 系统调用：`os/src/syscall/process.rs#sys_exec`、`sys_clone`、`sys_fork`、`sys_waitpid`、`sys_spawn`
- 线程：`os/src/syscall/thread.rs#sys_thread_create`、`os/src/task/id.rs#dealloc_user_res`
- 调度器：`os/src/task/processor.rs#run_tasks`、`schedule`、`__switch`；`os/src/task/manager.rs#add_task/fetch_task`
- 阻塞/让出：`os/src/task/mod.rs#block_current_and_run_next`、`suspend_current_and_run_next`
- 信号与定时器：`sys_rt_sigaction`、`sys_rt_sigreturn`、`sys_rt_sigtimedwait`、`sys_setitimer`、`check_process_alarms`
- 命名空间/会话：`namespaced_time_us`、`sys_setns`、`sys_unshare`、`sys_setpgid`、`sys_setsid`
- 测试框架：`os/src/task/judge.rs#add_initproc`、`build_queue`、`expand_script`

**判断**：这是传统 PCB/TCB 进程线程模型，不是“异步运行时”。  
**未体现**：SMP 多核、实时调度、cgroup、内核线程独立栈管理细节。

### FS（文件系统）

**有底层证据支持**：

- VFS-like 入口：`os/src/fs/inode.rs#open_file`、`write`、`read`、`link_path`、`rename_path`、`remove_path`、`chmod_path`、`chown_path`
- ext4 包装层：`os/src/fs/ext4.rs#with_ext4`、`ext4_read_file`、`ext4_list_dir`、`ext4_metadata`
- 简单文件系统：`easy-fs/src/layout.rs#clear_size`
- 系统调用：`os/src/syscall/fs.rs#sys_openat`、`sys_read`、`sys_write`、`sys_pread64`、`sys_pwrite64`、`sys_getdents64`、`sys_newfstatat`、`sys_readlinkat`、`sys_mount`、`sys_umount2`、`sys_fcntl`、`sys_ioctl`、`sys_utimensat`
- 元数据与缓存：`FILE_META`、`LINK_GROUPS`、`DELETED_PATHS`、`MEM_FILES`、`MEM_DIRS`、`FILE_TIMES`、`MOUNT_TABLE`
- procfs：`os/src/fs/inode.rs#proc_status_data`、`proc_stat_data`
- 块设备读取：`os/src/fs/ext4.rs#read` → `TEST_BLOCK_DEVICE.read_block`

**未体现**：原创日志文件系统、块缓存层原创实现、硬链接/软链接完整语义底层细节（部分通过包装层实现）。

### DEV（设备驱动）

**有底层证据支持**：

- 块设备：`os/src/drivers/block/virtio_blk.rs#read_block`、`write_block`、`try_new`、`new`（包装 `third_party/virtio-drivers` 的 `VirtIOBlk`）
- 控制台：`os/src/sbi.rs#console_putchar`、`console_getchar`（直接 UART0 MMIO）
- 探测示例：`third_party/virtio-drivers/examples/riscv/src/main.rs#virtio_probe`（仅第三方示例，未在 `os/` 中体现集成）

**未体现**：网络设备、GPU、输入设备、声卡、PCI/CLINT/PLIC 完整驱动；`os/` 中仅 VirtIO blk 包装与 UART 控制台。

### SYSCALL（系统调用）

**有底层证据支持**：

- 分发器：`os/src/syscall/mod.rs#syscall`（规模 10017，内部调用 121 个 `sys_*`）
- 实现分散在：
  - `os/src/syscall/process.rs`：exec/clone/fork/wait/spawn/mmap/signal/timer/namespace/resource
  - `os/src/syscall/fs.rs`：open/read/write/mount/fcntl/ioctl/...
  - `os/src/syscall/sync.rs`：futex/mutex/semaphore/condvar
  - `os/src/syscall/thread.rs`：thread_create/waittid

**说明**：模块画像中“系统调用”仅 4 个函数，是因为大量 syscall 实现被分类到进程/文件/同步/线程模块；实际系统调用接口覆盖较广。

### TRAP/IRQ（中断异常）

**有底层证据支持**：

- `os/src/trap/mod.rs#init`、`set_kernel_trap_entry`、`set_user_trap_entry`（`stvec::write`）
- `os/src/trap/mod.rs#enable_timer_interrupt`（`sie::set_stimer`）
- `os/src/trap/mod.rs#trap_handler`：统一 U-mode 陷阱处理，调用懒加载页缺失、计时器、信号、alarm
- `os/src/trap/mod.rs#trap_from_kernel`：内核态异常直接 panic

**未体现**：完整 PLIC/CLINT 驱动、外部中断向量表、多种异常分类处理。

### NET（网络协议栈）

**底层证据未体现**。

- 虽然 `os/src/syscall/mod.rs#syscall` 的分发链中出现了 `sys_accept`、`sys_bind` 等名称，但没有找到 socket 实现、TCP/IP 协议栈、网络设备驱动或相关数据结构。
- 因此不能判定网络协议栈已实现。

## 第六部分：边界与未体现内容

- **没有底层证据支撑的模块**：
  - 网络协议栈 / socket 完整实现（仅有 `sys_accept`/`sys_bind` 等分发名，无实现函数）
  - SMP 多核调度、CPU 亲和性、实时调度器
  - cgroup、namespace 完整实现（仅有 time namespace 与 `setns`/`unshare` 少量接口）
  - 除 VirtIO blk 包装与 UART 控制台外的其他设备驱动（如 GPU、输入、声卡、NIC、PCI 枚举）
  - 原创页表实现（使用 `third_party/riscv` 的页表代码）
  - 原创 ext4 / easy-fs 内部实现（`os/src/fs/ext4.rs` 为包装层，`easy-fs` 为独立目录但规模较小）
  - Bootloader、用户态 libc、完整测试用例原创性均不在本证据范围内

- **“异步运行时”判断边界**：
  - 模块画像中“任务调度/异步运行时”名称包含“异步运行时”，但底层证据显示的是传统 `TaskControlBlock` + `__switch` 的线程/进程调度，没有 `async/await`、`waker`、`Future` 运行时证据。因此应理解为“传统任务调度”。

  - 以上“可能的系统方向”仅基于 AST Call Map、函数名、路径锚点与调用链推断，不是对作品最终身份、原创性或竞赛归属的结论。
