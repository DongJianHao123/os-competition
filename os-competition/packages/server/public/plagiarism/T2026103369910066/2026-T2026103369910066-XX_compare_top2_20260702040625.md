### 4. 核心代码调用流对比剖析

*   **【进程管理 / `sys_exec` 系统调用】**
    *   **底层技术析理**: 目标项目采用通用的 `all_data` 缓冲区进行路径和参数解析，通过迭代器查找空格来分割参数，体现了从用户态字符串自行解析的通用实现。参考项目则直接依赖内嵌的 `BUSYBOX_BIN` 二进制数据和 `args_vec` 构建，是一种更专用、更紧密耦合的实现。两者在**程序加载与参数传递的抽象层次**上截然不同。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/process.rs#sys_exec`
        - 参考项目: `os/src/syscall/process.rs#sys_exec`

*   **【进程管理 / `sys_uname` 系统调用】**
    *   **底层技术析理**: 目标项目构建了一个通用的字段数组 `fields`，通过迭代和 `copy_nonoverlapping` 将各字段拷贝到用户缓冲区，展现了数据驱动的通用设计。参考项目则直接在代码中硬编码了 `"KernelCraft"`、`"riscv64"` 等特定字符串，是一种静态、专用的实现。两者在**数据组织与系统信息管理模型**上存在本质区别。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/process.rs#sys_uname`
        - 参考项目: `os/src/syscall/process.rs#sys_uname`

*   **【任务调度 / `run_tasks` 调度循环】**
    *   **底层技术析理**: 两者均实现了 `fetch_task` -> `__switch` 的核心循环，这是 rCore 调度器的基础框架。然而，目标项目使用 `PROCESSOR.exclusive_access()` 获取锁并显式 `drop`，日志使用 `warn!` 宏；参考项目则使用 `PROCESSOR.exclusive_session()` 这一不同的同步原语，并使用 `println!`。这体现了在**同步策略**上的具体实现差异。
    *   **代码溯源链接**:
        - 目标项目: `os/src/task/processor.rs#run_tasks`
        - 参考项目: `os/src/task/processor.rs#run_tasks`

    *   **底层技术析理**: 调用流均包含回收 `TaskUserRes`、初始化 `TaskContext`、处理子进程等步骤，这是进程退出时资源清理的标准 POSIX 语义流程。目标项目显式调用 `PageTable::from_token` 进行地址空间清理，而参考项目调用 `Arch::shutdown`，表明在**内存回收的具体策略**上存在差异。
    *   **代码溯源链接**:

### 6. 保守结论

综合全案证据，目标项目与参考项目的客观 AST 相似度仅为 16.32%，远未达到可判定为抄袭或系统性复用的阈值。所谓的“铁证”主要集中在进程管理板块的机械访问器和单例包装函数上，这些是由 Rust 语言特性及双方共同遵循的 rCore 教学框架所决定的通用写法，不构成独创性表达。在核心系统调用（如 `sys_exec`、`sys_uname`）的实现上，双方展现出截然不同的设计逻辑与外部依赖。此外，一方独有网络协议栈，另一方独有评测框架，这种根本性的功能分野是独立开发的强有力证明。因此，本庭裁定，现有证据不足以支持抄袭指控，认定该项目为独立开发。
