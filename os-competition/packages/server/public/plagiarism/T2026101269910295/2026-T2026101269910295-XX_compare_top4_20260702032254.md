### 4. 核心代码调用流对比剖析
* **【进程管理/调度路径】**
* **底层技术析理**: 双方均实现了`yield` -> `sched` -> `swtch`的调度链，这是xv6-riscv的经典协作式调度模型。然而，在获取当前执行实体的关键步骤上，两者展现出架构级差异：目标项目使用`myproc`，返回进程控制块（PCB），其内部实现涉及`mycpu`、`push_off`/`pop_off`中断禁用操作，体现了以进程为调度单元、并严格管理中断的模型。参考项目使用`mythread`，返回线程控制块（TCB），其`myproc`仅是`mythread`的简单包装。这并非简单的重命名，而是反映了“进程调度”与“线程调度”两种不同并发模型的设计选择。目标项目的`myproc`实现更为复杂，直接操作CPU结构并关中断，显示出独立的设计演化。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#yield` -> `kernel/proc.c#sched` -> `kernel/proc.c#myproc`
  - 参考项目: `kernel/proc.c#yield` -> `kernel/proc.c#sched` -> `kernel/proc.c#myproc`

* **【进程管理/进程退出路径】**
* **底层技术析理**: 在`sys_exit`系统调用入口，目标项目调用`exit`，参考项目调用`thread_exit`。这再次印证了双方的抽象层次差异：进程级退出 vs 线程级退出。更关键的是，在`sys_exit_group`的实现中，目标项目加入了`acquire`/`release`锁保护，并调用`exit`和`myproc`，形成了一个带同步的进程组退出逻辑；而参考项目仅是简单地调用`argint`和`thread_exit`。目标项目的实现明显更复杂，增加了并发控制，这是独立演化的特征，而非抄袭的简化。
* **代码溯源链接**:
  - 目标项目: `kernel/sysproc.c#sys_exit` -> `kernel/syscall.c#sys_exit_group`
  - 参考项目: `kernel/sysproc.c#sys_exit` -> `kernel/sysproc.c#sys_exit_group`
