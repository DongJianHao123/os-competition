## 4. 核心代码调用流对比剖析

### 4.1 【内存管理/页表遍历路径】

* **底层技术析理**: 
  - **目标项目** (`walk`): 实现 RISC-V Sv39 三级页表遍历。使用 `VPN_INDEX` 提取虚拟地址索引，`pte_is_leaf` 判断大页/叶节点，`page_ref_inc` 在分配新页表时增加引用计数。映射建立由上层 `vmm_map` 完成，采用 `vmm_map → walk → 填充 PTE` 的分层模型。
  - **参考项目** (`pgdir_walk`): 实现 MIPS 风格两级页表遍历。使用 `PDX`/`PTX` 提取页目录/页表索引，`KADDR` 转换物理地址为内核虚拟地址。映射建立由 `page_insert` 完成，采用 `page_insert → pgdir_walk → 设置 PTE` 的模型。
  - **异构性**: 两者页表层级不同（Sv39 三级 vs MIPS 两级），术语体系不同（`VPN_INDEX` vs `PDX/PTX`），职责划分不同（目标 `walk` 负责引用计数，参考 `pgdir_walk` 仅负责查找）。外部调用图谱无共同节点。

* **代码溯源链接**:
  - 目标项目: `kernel/vmm.c#walk`
  - 参考项目: `lib/pmap.c#pgdir_walk`

### 4.2 【内存管理/物理页分配路径】

* **底层技术析理**:
  - **目标项目** (`page_alloc`): 调用 `memset` 清零 + `page2kva` 获取内核虚拟地址。空闲页管理由更高层 `pmm_free_pages` 负责，`page_free` 仅调用 `panic`（正常释放路径不经过该函数）。
  - **参考项目** (`page_alloc`): 调用 `LIST_FIRST`/`LIST_REMOVE` 从空闲链表取页 + `bzero` 清零 + `page2kva` 获取内核虚拟地址。`page_free` 调用 `LIST_INSERT_HEAD` 将页归还链表。
  - **异构性**: 目标项目未暴露链表操作，暗示其空闲页管理可能采用位图或伙伴系统等不同数据结构。`memset` 与 `bzero` 虽语义等价，但属于不同 C 库习惯，且清零操作是物理页分配器的通用需求。

* **代码溯源链接**:
  - 目标项目: `kernel/pmm.c#page_alloc`
  - 参考项目: `lib/pmap.c#page_alloc`

### 4.3 【内存管理/内存映射系统调用路径】

* **底层技术析理**:
  - **目标项目** (`sys_mem_map`): 调用 `syscall_get_process` 获取进程对象 → `user_perm_to_pte`/`user_perm_valid` 显式转换并校验权限 → `IS_ALIGNED`/`ROUND_DOWN` 地址对齐 → `pa2page`/`page_ref_inc`/`page_ref_dec` 管理引用计数 → 最终由 `vmm_map` 建立映射。
  - **参考项目** (`sys_mem_map`): 调用 `envid2env` 获取环境对象 → `ROUNDDOWN` 地址对齐 → `page_lookup` 查找物理页 → `page_insert` 建立映射 → `printf` 输出调试信息。
  - **异构性**: 目标项目具备独立的权限转换层（`user_perm_to_pte`、`user_perm_valid`），参考项目无此设计。目标使用引用计数机制（`page_ref_inc/dec`），参考使用 `page_insert` 内隐式管理。两者进程/环境抽象对象不同（`process` vs `env`）。

* **代码溯源链接**:
  - 目标项目: `kernel/syscall.c#sys_mem_map`
  - 参考项目: `lib/syscall_all.c#sys_mem_map`

### 4.4 【进程管理/进程资源释放路径】

* **底层技术析理**:
  - **目标项目** (`process_release_resources`): 采用高层封装设计——调用 `vmm_destroy_user_pagetable` 一次性销毁用户页表，`pmm_free_pages` 批量释放物理页，`printk` 输出日志。
  - **参考项目** (`env_free`): 采用底层手动遍历——直接使用 `PDX`/`PMX`/`PTX` 逐级索引页目录/页表，`PTE2PA` 解析物理地址，`page_decref` 逐页减引用，`LIST_INSERT_HEAD` 回收空闲页。
  - **异构性**: 两者抽象层级截然相反。目标项目进行了独立的模块化设计，将页表销毁封装为独立函数；参考项目在 `env_free` 中内联所有底层操作。外部调用图谱无共同节点。

* **代码溯源链接**:
  - 目标项目: `kernel/process.c#process_release_resources`
  - 参考项目: `lib/env.c#env_free`

## 6. 保守结论

本案客观 AST 相似度仅 1.33%，双向覆盖率均低于 6%，共同 AST 指纹仅 5 个。控方重点指控的【任务调度/异步运行时】板块共同指纹为 0，【内存管理】板块仅 1。证据底座中的大量配对属于跨板块、跨抽象层、跨语言的无效硬凑；同名函数（`sys_mem_map`、`page_alloc`、`memmove` 等）源于操作系统教学规范与标准 C 库，不构成独创性复用证据。

双方在核心架构维度呈现本质差异：目标项目采用 RISC-V Sv39 + `process` 抽象 + `vmm_map/vmm_unmap` 映射原语 + 显式权限转换层；参考项目采用 MIPS 风格页表 + `env` 抽象 + `page_insert/page_remove` 映射原语。目标项目完全缺失设备驱动与文件系统模块，若存在系统性复用不可能整个子系统缺失。

综合判定：目标项目系独立开发，不存在实质性代码复用或抄袭行为。建议驳回全部复用指控。
