### 4. 核心代码调用流对比剖析
*   **【内存管理/页表遍历】**
    *   **底层技术析理**: 两者均实现了 RISC-V SV39 标准的页表遍历（walk）算法，这是功能上的必然。然而，其底层内存管理模型完全不同。目标项目采用基于引用计数的物理页管理器，在遍历过程中通过 `page_alloc` 和 `page_ref_inc` 显式管理页帧的生命周期。参考项目则采用经典的 xv6 式分配器，通过 `kalloc` 分配并 `memset` 清零。这种在核心资源管理策略上的根本性差异，是独立实现的强有力证明。
    *   **代码溯源链接**:
        -   目标项目: `kernel/vmm.c#walk`
        -   参考项目: `kernel/vm.c#walk`

*   **【进程管理/进程派生】**
    *   **底层技术析理**: 目标项目的 `process_exofork` 与参考项目的 `clone` 代表了两种截然不同的进程派生哲学。`process_exofork` 是 exokernel 风格的“外核 fork”，其核心操作是 `page_ref_inc/dec` 对物理页引用的精确复制，不涉及文件系统或 VFS 层。而 `clone` 是 xv6/Linux 风格，其调用流深入 `uvmcopy`（地址空间复制）、`filedup`（文件描述符复制）和 `edup`（VFS 节点复制）。两者在对象管理模型和同步策略上存在架构级差异。
    *   **代码溯源链接**:
        -   目标项目: `kernel/process.c#process_exofork`
        -   参考项目: `kernel/proc.c#clone`

*   **【任务调度/格式化输出】**
    *   **底层技术析理**: 目标项目的 `vprintk` 和参考项目的 `lp_Print` 都实现了格式化字符串输出，并使用了 C 标准宏 `va_arg`。但二者的底层输出抽象不同。目标项目直接调用 `console_putchar` 等硬件相关接口，而参考项目通过 `OUTPUT` 宏向抽象缓冲区输出。此外，辅助函数的拆分粒度完全不同，表明这是对同一公共需求的独立实现。
    *   **代码溯源链接**:
        -   目标项目: `kernel/console.c#vprintk`
        -   参考项目: `kernel/printf_wyx.c#lp_Print`

### 6. 保守结论
本次审计的核心结论是，目标项目与参考项目在底层代码实现上存在显著差异，应认定为独立开发。客观 AST 相似度仅为 0.84%，共同指纹仅 4 个，且均可被标准 C 库、RISC-V 规范或 trivial 匹配所解释。在内存管理、进程派生等核心板块，两项目展现出截然不同的设计哲学与资源管理模型（引用计数 vs. 简单分配，exokernel vs. xv6/Linux 风格），且参考项目包含目标项目所没有的设备驱动等完整子系统。控方提出的指控多基于跨板块硬凑、功能泛化或对标准实现的误读，缺乏客观 AST 证据支持。综上，不存在足以认定代码复用或抄袭的实质性证据。
