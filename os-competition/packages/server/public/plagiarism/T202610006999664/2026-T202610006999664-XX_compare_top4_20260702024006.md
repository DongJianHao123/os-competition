### 4. 核心代码调用流对比剖析

*   **【内存管理/页表遍历】**
    *   **底层技术析理**: 双方均实现了 RISC-V 架构下的多级页表遍历（Page Table Walk）。这是 MMU 硬件定义的标准流程，其高层逻辑必然收敛。然而，在物理内存分配这一关键环节，两者展现出根本性的**内存模型差异**。参考项目采用传统的 xv6 模式，通过 `kalloc` 分配一个物理页并用 `memset` 清零。目标项目则实现了一套带引用计数的物理内存管理器（PMM），调用 `page_alloc` 获取页面后，必须调用 `page_ref_inc` 增加引用计数，并通过 `page2kva` 和 `page2pa` 在物理地址与内核虚拟地址之间转换。这种引用计数机制的引入，使得内存的分配、共享和回收（如 COW 场景）路径与参考项目完全不同，是底层架构异构的有力证明。
    *   **代码溯源链接**:
        - 目标项目: `kernel/vmm.c#walk`
        - 参考项目: `kernel/vm.c#walk`

*   **【进程管理/进程创建】**
    *   **底层技术析理**: 双方在进程创建这一核心原语上存在**抽象层次的根本差异**。参考项目使用经典的 Unix `fork` 语义，通过 `uvmcopy` 复制父进程地址空间。目标项目则使用 `process_exofork`，这是 exokernel 或微内核风格的进程创建原语，通常只共享地址空间而不立即复制，依赖于后续的 COW 或缺页处理。目标项目的 `syscall_dispatch` 中包含了 `sys_ipc_try_send` 和 `sys_ipc_recv` 等 IPC 系统调用，表明其进程间通信模型也与参考项目基于文件描述符的模型截然不同。这种进程模型和通信机制的选择是操作系统内核最核心的架构决策之一，其差异足以否定同源性。
    *   **代码溯源链接**:
        - 目标项目: `kernel/process.c#process_exofork`
        - 参考项目: `kernel/proc.c#fork`

*   **【中断/异常处理/Trap 分发】**
    *   **底层技术析理**: 双方 Trap 处理流程的**同步与分发策略**不同。参考项目的 `usertrap` 是一个典型的 monolithic 内核入口，在一个大函数内通过 `r_scause` 读取异常原因，然后直接调用 `devintr`、`handle_excp` 或 `syscall`。目标项目则抽象出了一个独立的 `trap_dispatch` 函数，它更像一个分发器，根据异常类型路由到 `syscall_dispatch`、`timer_handle_interrupt` 或 `process_handle_cow_fault`。目标项目的 `process_current` 和 `process_schedule` 调用表明其调度逻辑与 Trap 处理紧密耦合，而参考项目则在 `usertrap` 末尾通过 `yield` 或 `sched` 触发调度。这种控制流拓扑的差异是独立设计的体现。
    *   **代码溯源链接**:
        - 目标项目: `kernel/trap.c#trap_dispatch`
        - 参考项目: `kernel/trap.c#usertrap`
