#### 4. 核心代码调用流对比剖析
* **【进程管理/任务退出路径】**
* **底层技术析理**: 两者均遵循“获取当前任务引用 -> 清理退出任务队列 -> 通知等待者 -> 终止系统”的流程。核心差异在于最终终止系统的调用：目标项目使用 `axhal::power::system_off`，将关机视为电源管理操作；参考项目使用 `axhal::misc::terminate`，将其视为通用终止。这表明目标项目在硬件抽象层（HAL）的架构划分上进行了独立思考和改动，而非简单的标识符替换。
* **代码溯源链接**:

  - 参考项目: `arceos/modules/axtask/src/run_queue.rs#exit_current`

* **【进程管理/等待队列同步机制】**
* **底层技术析理**: 两者的 `WaitQueue` 实现均基于 `SpinNoIrq` 自旋锁保护 `VecDeque` 双端队列。核心阻塞路径 `wait_until` 均遵循“加锁 -> 检查条件 -> 将当前任务加入队列 -> 阻塞式重调度”的模式。唤醒路径 `notify_one` 均采用“加锁 -> 从队首弹出任务 -> 解锁任务”的模式。这种同步原语、数据结构和调度API的组合是 ArceOS 框架下实现条件等待的标准范式，其调用流高度一致是框架约束的必然结果。
* **代码溯源链接**:
  - 目标项目: `arceos/modules/axtask/src/wait_queue.rs#wait_until`
  - 参考项目: `arceos/modules/axtask/src/wait_queue.rs#wait_until`

* **【网络协议栈/DNS查询】**
