## 第四部分：分模块架构说明

### MM（内存管理）
底层证据较强。启动阶段在 `os/src/main.rs#main` 中初始化堆并注册物理内存区域；运行时通过 `os/src/mm/frame_allocator.rs#frame_alloc` / `frames_alloc` 进行物理页框分配，使用 `FrameTracker` 管理生命周期。地址空间构建由 `os/src/mm/memory_set.rs` 负责，支持 ELF 加载（`from_elf`）、进程克隆（`from_existed_user`）以及按范围拆分映射区域（`split_area_by_range`）。

### PROC（进程管理/任务调度）
传统进程管理证据充分。`os/src/task/task.rs` 与 `os/src/syscall/process.rs` 包含 `sys_clone`、`sys_exec`、`sys_exit`、`sys_exit_group`、wait 状态编码、PCB 父子关系、`pid2task` 映射等。`user/src/bin/initproc.rs#main` 中出现 `fork`/`exec`/`wait`，说明用户态已能使用这些语义。调度方面以 `current_task`、`suspend_current_and_run_next`、`add_task` 等原语为主；**轻量任务调度或异步运行时（async/await、waker、reactor）的底层证据未明确体现**。

### FS（文件系统）
文件系统是本作品代码量最大的方向。`os/src/fs/` 实现了 VFS 包装、`devfs`、管道（`pipe.rs`）、挂载表（`MNT_TABLE`）、`/proc` 风格虚拟文件（`vfs_registry.rs#parse_proc_pid_leaf`、`create_proc_dir`）以及 ext4 的 Rust 包装层（`os/src/fs/ext4_lw/`）。实际 ext4 逻辑大量来自第三方 `lwext4_rust/c/lwext4/` C 库，包括块分配、inode 分配、extent、目录索引、journal 等。

### DEV（设备驱动）
设备驱动证据集中在 virtio 系列。`os/src/drivers/virtio_blk.rs#new` 显示内核侧对 MMIO/PCI virtio 传输的初始化；`patch/virtio-drivers/` 则包含对 virtio-blk、virtio-sound、virtio-console、virtio-socket 的修改/测试代码，以及 virtio queue 的 `fake_read_write_queue`。需要注意：`patch/` 中的代码属于被修改的第三方 crate，不应直接等同于原创内核主体。

### SYSCALL（系统调用）
`os/src/syscall/mod.rs#syscall` 是统一分发入口，AST 显示其分发超过 100 个系统调用。具体实现分散在 `os/src/syscall/fs.rs`、`process.rs`、`mm.rs`、`signal.rs`、`other.rs` 中，覆盖文件、进程、内存、信号、时钟等类别。

### TRAP/IRQ（中断异常）
中断/异常基础设施有中等强度证据。`patch/polyhal` 提供了 x86_64 IDT、Local/IO APIC、aarch64 GIC/timer 的初始化与开关中断封装；`os/src/trap/interrupts.rs#inc_irq_count` 提供 IRQ 计数。但**具体 page fault / 系统调用异常 / 时钟 tick 等 trap 处理函数的底层证据未充分体现**。

### NET（网络协议栈）
**底层证据未体现**。虽然 `patch/virtio-drivers/src/device/socket/connectionmanager.rs` 出现 virtio-socket 设备驱动相关函数，且系统调用分发入口包含 `sys_accept`、`sys_bind` 等名称，但**没有 TCP/IP 协议栈、套接字层、网络缓冲区管理等内核网络栈的底层证据**。

## 第六部分：边界与未体现内容

### 1. 没有底层证据支撑的模块/能力
- **完整网络协议栈（NET）**：仅有 virtio-socket 设备驱动和若干 socket 系统调用名称，无 TCP/IP、套接字缓冲区、协议状态机等实现证据。
- **完整异步运行时**：调度原语存在，但 async/await executor、waker、reactor、epoll/kqueue 等底层证据未体现。
- **SMP/多核调度**：`CPU_NUM` 初始化有证据，但 per-CPU 调度、核间中断、自旋锁/RCU 等多核机制底层证据未体现。
- **详细 trap/异常处理**：IDT/GIC/IO-APIC 初始化有证据，但 page fault 处理、系统调用异常返回、时钟 tick 调度等具体处理函数未充分体现。
- **电源管理、热插拔、安全/LSM、容器/cgroup、调试/perf** 等：底层证据未体现。
- **块 I/O 调度器、VFS dentry/inode 缓存、文件系统日志上层封装**：除第三方 lwext4 外，原创内核侧证据有限。

### 2. 第三方/依赖边界
- `lwext4_rust/c/lwext4/` 是第三方 C 语言 ext4 实现，仅通过 Rust 绑定被内核使用，不应视为原创内核主体。
- `patch/virtio-drivers/` 与 `patch/polyhal/` 是对第三方 crate 的修改/补丁，属于依赖层改造，不等同于从零实现的内核驱动/平台抽象。
- `user/` 为用户态程序，仅用于验证内核接口，不属于内核本身。
