### 4. 核心代码调用流对比剖析

* **【进程管理/调度器核心】**
* **底层技术析理**: 这是控方指控的核心。两者虽在 `current` 和 `take_current` 函数上有相同的 Rust 惯用调用，但**调度器的整体架构截然不同**。目标项目采用**全局单例 `PROCESSOR`** 模型，通过 `PROCESSOR.exclusive_access()` 获取当前 CPU 的处理器结构，这是一种单核或粗粒度锁的多核模型。参考项目则采用 **per-HART 的 `get_proc_by_hartid(hart_id())`** 模型，为每个硬件线程维护独立的处理器结构，是典型的细粒度多核设计。这两种并发架构决定了任务切换、负载均衡等核心机制的根本分歧。将 `Option::take` 这种语言层面的微观共性，置于如此巨大的宏观架构差异之上来指控抄袭，在技术上是不成立的。
* **代码溯源链接**:

* **底层技术析理**: 在 `sys_getuid` 等系统调用中，双方实现了相同的 POSIX 接口，但**权限模型完全不同**。目标项目将 `uid`、`gid` 等凭证内嵌在 `TaskControlBlock` 中，通过 `current_task().unwrap().inner_exclusive_access()` 动态获取。这是一个支持每任务凭证隔离的现代 OS 模型。参考项目则使用全局变量 `current_uid`，这是一个简化但缺乏隔离性的模型。目标项目的实现更复杂、更安全，是对参考项目简化模型的一次**架构升级**，而非“批量洗稿”。
