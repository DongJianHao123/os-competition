## 4. 核心代码调用流对比剖析

### 4.1 【进程管理 / 进程退出路径】

**底层技术析理**:
两项目均实现 Linux `sys_exit` → `do_exit` 的退出路径，但资源清理策略存在架构差异：

- **文件描述符清理**: 目标使用 `FD_TABLE.clear`（全局表直接清空），参考使用 `close_all_filelike`（逐文件关闭）。前者为批量操作，后者为迭代操作，反映不同的资源管理模型。
- **子线程清理**: 目标通过 `curr_ext.process_data` 访问进程数据，参考通过 `clear_child_tid.query_paddr` 查询物理地址后操作。两者对 `clear_child_tid` 的访问路径不同。
- **共同调用**: `SignalInfo::new`、`axtask::exit`、`axtask::yield_now` 为 Linux 进程退出必须执行的信号发送与调度器切换操作，属于领域强制调用。

**代码溯源链接**:
- 目标项目: `api/src/imp/task/exit.rs#do_exit`
- 参考项目: `api/src/imp/task.rs#do_exit`

### 4.2 【进程管理 / 进程等待路径】

**底层技术析理**:
两项目均实现进程等待语义，但等待对象管理模型不同：

- **目标**: `sys_waitpid` 使用 `WaitOptions::from_bits_truncate` + `WaitPid::Pid`/`WaitPid::Pgid` 枚举，通过 `all_children().into_iter().filter(|child| pid.apply(child))` 进行子进程筛选。等待对象为全局进程表。
- **参考**: `sys_wait4` 使用 `WaitFlags::from_bits(option).unwrap_or` + 直接遍历子进程列表检查 `child.is_zombie` 和 `child.exit_code`。等待对象为进程组内部结构。
- **差异**: 目标使用类型安全的枚举（`WaitPid`）和 `WaitOptions` 位标志，参考使用原始位标志 `WaitFlags`。两者 API 设计风格不同。

**代码溯源链接**:
- 目标项目: `api/src/imp/task/wait.rs#sys_waitpid`
- 参考项目: `src/syscall_imp/task/thread.rs#sys_wait4`

### 4.3 【内存管理 / ELF 加载路径】

**底层技术析理**:
两项目均实现 ELF 文件加载，但错误处理策略不同：

- **目标**: `CStr::from_bytes_with_nul(interp).map_err(|_| AxError::InvalidData)?.to_str().map_err(...)` 使用 `map_err` 显式转换错误类型，错误处理链路完整。
- **参考**: `CStr::from_bytes_with_nul(interp).ok().and_then(|it| it.to_str().ok()).ok_or(...)` 使用 `ok()` + `and_then` 链式处理，风格更接近 Rust 标准库惯用法。
- **差异**: 两者对同一逻辑采用了不同的错误处理组合子链，反映独立的编码风格。

**代码溯源链接**:
- 目标项目: `core/src/mm.rs#load_user_app`
- 参考项目: `core/src/mm.rs#load_user_app`

### 4.4 【系统调用 / clone 路径】

**底层技术析理**:
两项目均实现 Linux `clone` 系统调用，但进程初始化流程差异显著：

- **目标**: `sys_clone` 调用链含 `CURRENT_DIR.deref_from(&process_data.ns).init_new`、`CURRENT_DIR.copy_inner` 等命名空间初始化操作，共 84+ 节点。
- **参考**: `sys_clone` 调用链含 `CloneFlags::from_bits((flags & !FLAG_MASK) as u32).unwrap`、`FD_TABLE.deref_from(&process.ns).init_new` 等标志位处理与文件描述符表初始化，共 57+ 节点。
- **差异**: 目标侧重命名空间（`CURRENT_DIR`）初始化，参考侧重克隆标志位（`CloneFlags`）解析。两者初始化流程和关注点不同。

**代码溯源链接**:
- 目标项目: `api/src/imp/task/clone.rs#sys_clone`
- 参考项目: `src/syscall_imp/task/thread.rs#sys_clone`

## 6. 保守结论

本案两项目客观 AST 相似度仅 5.1%，远低于实质性相似阈值。173 个共同 AST 指纹经逐项审查，均可归因于共享开源框架（ArceOS/nimbos）的公共代码、POSIX/Linux 系统调用语义的领域强约束、或 Rust/C 标准库惯用法。目标项目近 70% 的 AST 指纹为独有，且包含进程定时器管理、Shebang 解析、BusyBox 兼容等参考项目未实现的独立功能模块。参考项目 94.27% 的 AST 指纹未被目标覆盖，其包含的 1182 个第三方库函数（lwext4、smoltcp、visionfive2-sd 等）目标项目完全未包含，构成独立开发的最有力反证。在设备驱动与网络协议栈两个核心模块，双方共同 AST 指纹为零。综上，现有证据不足以证明目标项目对参考项目存在直接复制或实质性相似抄袭，判定为**独立开发**。
