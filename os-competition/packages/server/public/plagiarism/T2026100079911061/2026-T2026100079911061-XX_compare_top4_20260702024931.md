### 4. 核心代码调用流对比剖析
* **【进程管理 / clone3 系统调用】**
* **底层技术析理**: 两者均实现了 Linux `clone3` 系统调用，但采用了完全不同的用户态参数解析策略。目标项目采用**手动、版本化的指针偏移解析**，通过 `(uargs as usize).checked_add` 进行边界检查，再调用 `clone_arg_u64` 等辅助函数逐字段读取，最终汇入统一的 `sys_clone_common` 入口。参考项目则采用**类型化的一次性读取**，通过 `UserPtr::<CloneArgs>::new` 和 `cl_args.read` 将整个结构体从用户态拷贝到内核态，再传入 `self.task.do_fork`。前者是 C 风格的手动内存操作，后者是 Rust 风格的类型安全抽象，底层数据流和外部调用对象无交集。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process_ops.rs#sys_clone3`
  - 参考项目: `NoAxiom/kernel/src/syscall/process.rs#sys_clone3`

* **【进程管理 / procfs 文件系统实现】**
* **底层技术析理**: 双方均实现了 `procfs`，但架构风格截然不同。目标项目采用**过程式渲染器**，核心是 `render_file` 函数，它通过一个巨大的 `match` 或 `if-else` 分支调用 `render_cpuinfo`、`render_loadavg` 等具体渲染函数，使用自研的 `ProcBuf` 和 `ProcDirent` 对象。参考项目采用**面向对象的 VFS 挂载**，通过 `init` 函数创建 `RamFsDentry`、`ExeDentry`、`FdDentry` 等 dentry/inode 对象树，将每个 proc 文件抽象为一个 VFS 节点。前者是扁平化的函数派发，后者是层次化的对象树，架构设计哲学完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/procfs.rs#render_file`
  - 参考项目: `NoAxiom/kernel/src/fs/vfs/impls/proc/mod.rs#init`

* **【任务调度 / futex 等待与重排队机制】**
* **底层技术析理**: 两者对 futex 等待队列的管理模型不同。目标项目的 `requeue_futex_waiters` 位于调度器内部，直接遍历 `PROCESS_LIST`，操作进程私有的 `futex_waitv_keys` 数组，通过 `futex_waitv_index` 查找并移动等待者。参考项目的 `sys_futex` 是系统调用入口，操作全局的 `FUTEX_SHARED_QUEUE` 对象，调用其 `requeue` 或 `wake_waiter` 方法。前者是调度器中心的、基于进程结构的模型，后者是对象中心的、基于全局队列的模型。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/scheduler.rs#requeue_futex_waiters`
  - 参考项目: `NoAxiom/kernel/src/syscall/process.rs#sys_futex`
