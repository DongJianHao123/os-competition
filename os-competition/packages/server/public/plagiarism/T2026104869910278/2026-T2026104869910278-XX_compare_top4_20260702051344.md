#### 1. 宏观判定
- **定性结论**: **独立开发，底层架构显著差异**
- **核心依据**: 客观 AST 相似度仅为 **19.61%**。在扣除双方共同依赖的第三方开源库 `lwext4` 后，核心内核模块（进程管理、内存管理、任务调度等）的共同 AST 指纹覆盖率极低（目标项目 2.15%~11.48%，参考项目 4.06%~11.65%）。双方在调度架构、同步原语、内存访问模型及文件系统抽象层存在根本性异构。控方所指控的“铁证”均为 Rust 语言标准库 `Option<T>` 的惯用单行代码或操作系统领域的通用访问范式，不构成实质性代码抄袭。

#### 2. 全模块覆盖矩阵解读

- **进程管理**
  1. 双方均存在该模块。
  2. 存在极少量共同 AST 指纹（4 个）。
  3. 覆盖率极低（目标 2.15%，参考 5.48%），表明双方在进程创建、退出、状态查询等核心逻辑上实现方式迥异。
  4. **证据强度：弱**。共同指纹集中在 `current`、`take_current` 等 Rust 标准库单行调用，属于 trivial code。

- **任务调度/异步运行时**
  1. 双方均存在该模块。
  2. 存在极少量共同 AST 指纹（5 个）。
  3. 覆盖率低（目标 2.55%，参考 4.07%），揭示调度器实现差异巨大。
  4. **证据强度：弱**。共同指纹多为 POSIX 标准系统调用入口（如 `sys_nanosleep`），内部实现完全不同。

- **内存管理**
  1. 双方均存在该模块。
  2. 存在少量共同 AST 指纹（29 个）。
  3. 覆盖率较低（目标 7.0%，参考 11.65%），说明内存管理核心算法与数据结构独立。
  4. **证据强度：中**。共同指纹包含 `sys_mmap` 等系统调用入口，但目标项目使用 `translated_byte_buffer`，参考项目使用 `put_data/get_data`，内存访问模型异构。

- **中断/异常处理**
  1. 双方均存在该模块。
  2. 存在极少量共同 AST 指纹（2 个）。
  3. 覆盖率低（目标 5.56%，参考 10.0%），Trap 处理流程独立。
  4. **证据强度：弱**。共同指纹为 `set_kernel_trap_entry` 等寄存器操作，属于硬件平台强制要求的指令序列。

- **系统调用**
  1. 双方均存在该模块。
  2. 存在少量共同 AST 指纹（7 个）。
  3. 覆盖率中等（目标 11.48%，参考 8.64%），主要源于 POSIX 接口同名。
  4. **证据强度：弱**。共同指纹为用户态侧的 `syscall` 汇编指令及标准接口名，内核态分发逻辑完全不同。

- **文件系统**
  1. 双方均存在该模块。
  2. 存在少量共同 AST 指纹（8 个）。
