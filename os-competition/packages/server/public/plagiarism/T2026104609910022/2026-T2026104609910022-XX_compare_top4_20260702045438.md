### 4. 核心代码调用流对比剖析
* **【进程管理/execve 系统调用】**
* **底层技术析理**: 两者均实现了 `sys_execve`，但**程序加载策略**存在根本性分歧。目标项目硬编码了 `b"./busybox"` 和 `b"/busybox"` 作为默认执行路径，其参数处理流为 `String::from` -> `Vec::new` -> `arg.as_str`，呈现出以 busybox 为单一入口的嵌入式思路。参考项目则使用 `DEFAULT_SHELL.to_string` 作为默认 shell，并通过 `arg_ptr.is_null`、`argv.add` 等指针算术遍历参数列表，遵循标准的 Unix shell 启动流程。两者在用户态内存拷贝抽象上亦不同（目标用 `UserBuffer`，参考用 `copy_from_user`），表明其执行体加载与参数传递的底层数据流完全异构。
* **代码溯源链接**:
  - 目标项目: `src-la/syscall/process.rs#sys_execve`
  - 参考项目: `os/src/syscall/process.rs#sys_execve`

* **【进程管理/clone 系统调用】**
* **底层技术析理**: 两者均实现了 `sys_clone`，但**进程/线程创建模型**截然不同。目标项目直接调用 `TaskControlBlock::clone_thread` 和 `TaskControlBlock::fork`，并在创建过程中集成了 `crate::fs::manager::FS_MANAGER.lock`，表明其文件系统上下文与任务创建深度耦合。参考项目则采用更传统的 Unix 风格，通过 `CloneFlags::from_bits` 解析标志位，并结合 `Signals::empty`、`Signals::from_signum` 等信号机制来配置新任务。这种从“TCB 方法分发”到“标志位+信号配置”的模型差异，是架构设计层面的根本不同。
* **代码溯源链接**:
  - 目标项目: `src-la/syscall/process.rs#sys_clone`
  - 参考项目: `os/src/syscall/process.rs#sys_clone`

* **【进程管理/futex 系统调用】**
* **底层技术析理**: 两者均实现了 `sys_futex`，但**同步原语实现**完全不同。目标项目的 futex 操作直接关联物理内存，调用 `virt_to_phys` 进行地址转换后执行 `wake_futex`，暗示其可能在内核空间直接操作物理页。参考项目则采用了标准的 `FutexOption::from_bits_truncate` 解析选项，并调用 `do_futex_wait` 函数，该函数内部使用 `WaitQueue::new` 和 `Arc::downgrade` 构建了一套基于虚拟地址的等待队列机制。一个偏向物理地址操作，一个偏向虚拟地址队列管理，同步策略存在本质区别。
* **代码溯源链接**:
  - 目标项目: `src-la/syscall/process.rs#sys_futex`
  - 参考项目: `os/src/syscall/process.rs#sys_futex`

* **【任务调度/主调度循环】**
* **底层技术析理**: 两者均有 `run_tasks` 函数，但**调度循环的辅助逻辑**差异显著。目标项目在循环中频繁调用 `crate::fs::manager::FS_MANAGER.lock` 进行文件系统同步，并包含 `crate::test_runner::on_root_task_exit` 和 `crate::sbi::shutdown` 等测试与关机钩子。参考项目的循环则关注 `do_wake_expired` 来处理超时唤醒的任务，并通过 `task.acquire_inner_lock` 操作任务内部锁。目标项目将调度器与文件系统和测试框架紧耦合，而参考项目则与定时器机制紧耦合，体现了不同的内核架构设计哲学。
* **代码溯源链接**:
  - 目标项目: `src-la/task/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

### 5. 单方模块与降权项
*   **单方模块**: 无。双方在主要内核模块上均有覆盖，但实现路径不同。
*   **降权项说明**:
    *   **`[同板块同名]` 但调用图谱差异巨大的函数**: 如 `sys_execve`、`sys_clone`、`sys_futex`、`sys_kill` 等，已从主证据降级为弱信号。它们证明了双方在实现同一 POSIX 接口时，采用了完全不同的内部架构、数据流和第三方库依赖。
    *   **跨板块证据**: 如目标项目的文件系统代码与参考项目内存管理代码的对比，已依规排除，不构成同板块主证据。
    *   **通用接口与极简封装**: `sys_getpid`、`sys_yield`、`timer::sub` 等函数，因其功能极度简单或由 POSIX 标准定义，其调用图谱的相似性不具备证明力，已降权处理。
    *   **第三方库/公共依赖**: 双方均使用了 `rCore` 教学框架的抽象（如 `Processor`、`TaskContext`、`__switch`），由此产生的相似性属于共享上游来源的领域必然性，不能作为抄袭证据。

### 6. 保守结论
本庭裁定，目标项目与参考项目在代码实现层面构成**独立开发**。全项目 2.2% 的极低 AST 相似度，以及文件系统、设备驱动、中断处理等核心特色模块的**零共同指纹**，是独立开发的决定性证据。控方基于进程管理模块 7 个极简包装函数（合计不足 20 行代码）的指控，在技术上无法成立。这些函数均为 rCore 教学框架下 `Processor` 单例的标准访问器或 `__switch` 切换模板，其调用图谱的相似性是领域必然性（scènes à faire）和 Rust 语言惯用法的体现，而非代码抄袭。更重要的是，在 `sys_execve`、`sys_clone`、`sys_futex` 等关键系统调用路径上，双方在内存抽象、进程模型、同步策略等方面展现出**体系性架构差异**，彻底排除了“在同一个代码基座上魔改”的可能性。综合判定，目标项目原创性成立，复用指控予以驳回。
