### 4. 核心代码调用流对比剖析
* **【进程管理/`sys_clone` 系统调用路径】**
* **底层技术析理**: 控方认为双方 `sys_clone` 的调用流“获取当前任务 -> 执行 fork -> 添加至调度器”一致，构成复用。然而，深入分析 AST 图谱发现，两者在**对象管理模型**上存在根本差异。目标项目将 `fork` 实现为 `TaskControlBlock` 的关联函数 (`TaskControlBlock::fork`)，并显式操作文件描述符管理器 (`FS_MANAGER.lock`)；而参考项目将 `fork` 实现为任务实例的方法 (`current_task.fork`)，且未在 `sys_clone` 层面直接涉及文件系统。这种从“实例方法”到“关联函数”的转变，以及文件系统管理职责的引入，表明目标项目对进程创建流程进行了重构，而非简单复制。
* **代码溯源链接**:
  - 目标项目: `src-la/syscall/process.rs#sys_clone`
  - 参考项目: `kernel/src/syscall/process.rs#sys_clone`

* **【任务调度/异步运行时/`sys_getdents64` 系统调用路径】**
* **底层技术析理**: 双方均实现了 `sys_getdents64` 系统调用，但底层**数据序列化策略**截然不同。目标项目的 AST 图谱显示其直接操作原始字节序列（如 `(e.ino as u64).to_le_bytes`），手动构建 dirent 结构；而参考项目则定义了一个 `Dirent::default` 结构体，并通过 `dirent.as_bytes` 方法进行序列化。这反映了两种完全不同的编程风格和抽象层次：一种偏向底层字节操控，另一种偏向结构化抽象。
* **代码溯源链接**:
  - 目标项目: `src-la/syscall/fs.rs#sys_getdents64`
  - 参考项目: `kernel/src/syscall/fs.rs#sys_getdents64`

* **【内存管理/`fork` 核心逻辑路径】**
* **底层技术析理**: 双方 `fork` 的核心逻辑均涉及地址空间复制，但**内存复制模型**不同。目标项目的 `MemorySet::from_existed_user` 通过 `core::ptr::copy_nonoverlapping` 逐页复制物理内存内容；而参考项目的 `AddrSpace::from_existed_user` 则通过 `Section::from_another` 复制内存区域（Section）对象。前者是物理页级别的拷贝，后者是逻辑区域对象的复制，表明双方在内存管理抽象上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `src/task/task.rs#fork`
  - 参考项目: `kernel/src/task/task.rs#fork`

### 5. 单方模块与降权项
*   **单方模块**:
    *   **文件系统/第三方库 (ext4)**: 目标项目独有的 `scan_ext4_directory`、`scan_ext4_root` 等函数，表明其具备独立的 ext4 文件系统解析能力，是独立开发的强有力证据。
    *   **系统调用 (用户库)**: 目标项目在 `user/src/main.rs` 中使用内联汇编 (`asm!`) 直接执行 `ecall`，而参考项目在 `user/src/syscall.rs` 中封装了 `syscall` 函数。这是两种不同的系统调用封装策略。
*   **降权项**:
    *   **跨板块证据**: 控方引用的参考项目中 `oscomp/*.c` 测试程序、`user/src/bin/initproc.rs` 用户态程序等，与目标项目的内核态 `syscall/process.rs` 不在同一抽象层，不能作为同板块主证据。
    *   **公共接口/泛型函数名**: `current_task`、`add_task`、`suspend_current_and_run_next`、`translated_refmut` 等均为 rCore 教学 OS 框架中的标准 API 或惯用命名，其出现是框架依赖的自然结果，不具独创性。
    *   **架构变体内部一致性**: 目标项目 `src-la` 与 `src` 目录下的代码重复，是同一项目为适配 LoongArch 和 RISC-V 架构的正常工程实践，与是否抄袭外部项目无关。

### 6. 保守结论
本次审计的客观 AST 相似度仅为 2.24%，这是一个极低的数值，强烈指向独立开发。尽管在内存管理等模块观察到少量共同指纹，但这完全在基于同一公共教学框架（rCore）进行独立开发的可解释范围内。控方提出的“复用指控”，如 `sys_yield` 的单调用、`busybox` 的硬编码、`current_user_token` 样板代码等，均属于操作系统内核开发中由 POSIX 标准、测试环境或框架惯用法导致的语义级收敛，而非受保护的独创性表达。双方在文件系统、设备驱动等核心子系统上 AST 指纹覆盖率为零，且在内存管理、进程创建等关键路径上展现出不同的对象模型和实现策略。综上，本案不存在足以判定代码抄袭的实质性证据，整体风险极低。
