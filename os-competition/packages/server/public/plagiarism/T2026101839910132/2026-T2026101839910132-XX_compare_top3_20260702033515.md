#### 4. 核心代码调用流对比剖析
* **【进程管理/sys_sigreturn 信号上下文恢复】**
* **底层技术析理**: 两者均实现了从用户态 `ucontext_t` 结构恢复 `MachineContext` 的逻辑。该过程涉及通过 `copy_from_user` 从用户空间拷贝数据，其偏移量计算 `ucontext_addr + 2 * size_of::<usize>() + size_of::<SignalStack>() + size_of::<Signals>() + crate::hal::UserContext::PADDING_SIZE` 严格遵循了 `ucontext_t` 的内存布局。此布局由 Linux ABI 定义，但将 `PADDING_SIZE` 这一 HAL 层细节纳入计算，显示出高度的实现一致性。
* **代码溯源链接**:
  - 目标项目: `GCore/os/src/syscall/process.rs#sys_sigreturn`
  - 参考项目: `os/src/syscall/process.rs#sys_sigreturn`

* **【任务调度/异步运行时/run_tasks 调度主循环】**
* **底层技术析理**: 两者均实现了一个协作式调度循环，核心流程为 `fetch_task` -> `__switch` -> `do_wake_expired`。关键差异在于同步原语与架构适配：目标项目使用 `PROCESSORS.lock` 并传入 `hart_id`，表明其面向多核（SMP）设计，且调用了 `crate::hal::arch::riscv::time::set_next_trigger` 等架构特定接口；参考项目使用 `PROCESSOR.lock`，是典型的单核调度器实现。这表明目标项目在参考项目的调度骨架基础上，进行了向多核环境的实质性架构演进。
* **代码溯源链接**:
  - 目标项目: `GCore/os/src/task/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

* **【文件系统/directory_tree rename 重命名操作】**
