## 第四部分：分模块架构说明

### MM（内存管理）
底层证据较强。核心能力包括：
- **物理页框分配**：`GCore/os/src/mm/frame_allocator.rs#frame_alloc_uninit` 使用全局 `FRAME_ALLOCATOR` 分配未初始化页框。
- **用户地址空间布局**：`GCore/os/src/mm/memory_set.rs#map_elf` 加载 ELF 段，`dealloc_user_res` 回收线程栈与 trap 上下文，`mmap`/`sbrk`/`mprotect` 支持系统调用层面的内存请求。
- **页表与异常处理**：`GCore/os/src/hal/arch/loongarch64/laflex.rs` 提供 `LAFlexPageTable`，`GCore/os/src/hal/arch/loongarch64/trap/mod.rs#trap_handler` 在缺页时调用页表接口。

### PROC（进程管理/任务调度）
传统进程管理证据充分：
- **PCB 与生命周期**：`GCore/os/src/task/task.rs#sys_clone` 构造子任务（`MemorySet::from_existing_user`、`Futex`、`ITimerVal`、`ProcClock`）；`GCore/os/src/task/mod.rs#exit_group_and_run_next` 处理线程组退出；`GCore/os/src/syscall/process.rs#sys_execve` 实现 exec。
- **调度器**：`GCore/os/src/task/processor.rs#run_tasks` 维护就绪队列与可中断队列，执行 `__switch`；`GCore/os/src/task/processor.rs#schedule` 切到 idle 上下文。
- **同步与原语**：`GCore/os/src/task/threads.rs#do_futex_wait` 实现 futex 等待；`GCore/os/src/task/signal.rs#do_signal` 处理信号投递。
- **关于“异步运行时”**：代码中仅见阻塞/唤醒模型与定时器队列，**未见 waker、executor、async/await 运行时等底层证据**。

### FS（文件系统）
证据强，呈现多层结构：
- **VFS/目录树层**：`GCore/os/src/fs/directory_tree.rs#open`、`rename`、`mkdir`、`delete` 负责路径解析与节点管理，并初始化 `/proc`、`/dev` 等伪文件系统。
- **具体文件系统**：FAT32（`GCore/os/src/fs/fat32/fat_inode.rs`）、ext4（`GCore/os/src/fs/ext4/ext4_inode.rs`、`extent.rs`、`ialloc.rs`、`balloc.rs`）。
- **挂载与系统调用**：`GCore/os/src/syscall/fs.rs#sys_mount` 提供 mount 接口；`sys_openat`、`sys_readv`/`writev`、`sys_sendfile`、`sys_splice` 等覆盖常见文件 IO。

### DEV（设备驱动）
原创驱动代码较薄，大量依赖第三方 crate：
- **块设备**：`GCore/os/src/drivers/block/virtio_blk.rs#new` / `read_block` / `write_block` 封装 `virtio-drivers` 的 VirtIOBlk；`GCore/os/src/drivers/block/sata_blk.rs#pci_init` 初始化 AHCI。
- **网络设备**：`GCore/os/src/net/config.rs#new` 配置 smoltcp 接口；实际网卡驱动（ixgbe/e1000）位于 `dependency/dep_iso`，VirtIO socket 位于 `dependency/virtio-drivers`。
- **其他设备**：`dependency/virtio-drivers/src/device/sound.rs` 等属于第三方示例/库代码，**不能视为该作品原创驱动主体**。

### SYSCALL（系统调用）
证据强，呈现 Linux 兼容风格：
- **分发器**：`GCore/os/src/syscall/mod.rs#syscall` 包含大量系统调用号匹配与日志黑名单。
- **进程类**：`GCore/os/src/syscall/process.rs#sys_execve`、`sys_clone`、`sys_futex`、`sys_prlimit`、`sys_nanosleep`、`sys_process_vm_readv`/`writev` 等。
- **文件类**：`GCore/os/src/syscall/fs.rs#sys_openat`、`sys_read`/`write`、`sys_pread`/`pwrite`、`sys_readv`/`writev`、`sys_mount`、`sys_sendfile`、`sys_splice`。
- **网络类**：`GCore/os/src/syscall/net.rs#sys_sendmsg`。

### TRAP/IRQ（中断异常）
双架构 HAL 证据可见：
- **LoongArch64**：`GCore/os/src/hal/arch/loongarch64/trap/mod.rs#trap_handler` 处理异常/中断/系统调用；`trap_from_kernel` 处理内核态异常；`enable_timer_interrupt` 配置定时器中断；`GCore/os/src/hal/arch/loongarch64/register/base/estat.rs#cause` 解析异常原因。
- **RISC-V**：`GCore/os/src/hal/arch/riscv/trap/mod.rs#trap_handler` 同样进入调度与信号处理。

### NET（网络协议栈）
socket 层证据较充分，但协议栈主体为第三方：
- **socket 抽象**：`GCore/os/src/net/tcp.rs#accept`/`connect`/`_read`/`listen`/`shutdown`、`GCore/os/src/net/udp.rs#connect`/`_read`、`GCore/os/src/net/icmp.rs#bind`。
- **地址与接口**：`GCore/os/src/net/address.rs`、`GCore/os/src/net/config.rs#new`、`GCore/os/src/net/mod.rs#can_bind`。
- **底层栈**：调用链中出现 `socket::tcp::Socket`、`Interface`、`PacketBuffer`、`NET_INTERFACE.poll`，**表明核心协议栈大概率来自 smoltcp 等第三方库**，本作品主要提供 socket 与系统调用适配层。

- **网络协议栈主体为第三方**：TCP/UDP/ICMP socket 适配层存在，但核心协议对象（`socket::tcp::Socket`、`Interface`、`PacketBuffer`）指向 smoltcp 等外部库，**原创协议栈实现证据不足**。
- **未见虚拟化/超管证据**：`dependency/riscv/src/register/hypervisorx64/hcounteren.rs` 属于第三方寄存器定义 crate，OS 主体未体现 hypervisor、VMM、嵌套页表等。
