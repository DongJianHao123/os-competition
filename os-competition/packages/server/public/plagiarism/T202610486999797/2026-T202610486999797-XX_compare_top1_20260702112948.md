#### 4. 核心代码调用流对比剖析
* **【进程管理/进程身份获取】**
* **底层技术析理**: 两者均实现了 `sys_getpid` 系统调用，调用流均为 `sys_getpid -> myproc`。然而，`myproc` 的内部实现揭示了本质差异。目标项目的 `myproc` 引入了 `proc_current` 抽象层，将“获取当前 CPU”和“获取当前进程”解耦，这是面向多核、多架构扩展的设计。参考项目的 `myproc` 则直接在函数内完成从 CPU 到进程的查找，是典型的单核教学内核实现。这种抽象层次的差异是架构设计上的根本不同。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/sysproc.c#sys_getpid`
  - 参考项目: `kernel/sysproc.c#sys_getpid`

* **【内存管理/VirtIO 磁盘驱动】**
* **底层技术析理**: 在 `virtio_disk_rw` 的实现中，两者都需要将虚拟地址转换为物理地址以进行 DMA 操作。目标项目使用 `PTE_TO_PA` 宏，暗示其采用基于页表项的直接转换模型。参考项目则调用 `kwalkaddr` 函数，这是一个遍历页表的软件实现。这代表了两者在内存管理单元（MMU）抽象和地址转换策略上的不同选择，是核心内存模型的异构。
* **代码溯源链接**:
  - 目标项目: `kernel/dev/virtio.c#virtio_disk_rw`
  - 参考项目: `kernel/virtio_disk.c#virtio_disk_rw`
