* **底层技术析理**: 目标项目 `sys_exit` 调用 `arg_uint32` 解析参数，随后进入 `proc_exit`。`proc_exit` 的调用图谱包含 `proc_signal_deliver_pending` 等信号递送逻辑，说明进程退出前需要处理 pending 信号，这是一个更接近现代 Linux 的复杂信号模型。参考项目 `sys_exit` 调用 `argint` 解析参数，随后进入 `exit`。`exit` 是 xv6 经典实现，不涉及信号递送。两者的参数解析抽象（类型化 vs 通用）和进程退出生命周期管理模型完全不同。

  - 参考项目: `kernel/sysproc.c#sys_exit`

* **底层技术析理**: 目标项目 `sys_kill` 调用 `arg_uint32, arg_uint64` 解析信号和 PID，随后进入 `proc_kill`。`proc_kill` 的调用图谱包含 `proc_signal_action_owner, proc_signal_bit, proc_signal_is_terminating`，表明其基于信号位图和信号动作所有者的模型实现。参考项目 `sys_kill` 调用 `argint` 解析 PID，随后进入 `kill`。`kill` 的调用图谱仅包含 `acquire, release`，是简单的设置 `p->killed` 标志的 xv6 模型。两者的信号模型和进程状态管理机制完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/sysproc.c#sys_kill`
  - 参考项目: `kernel/sysproc.c#sys_kill`

* **【进程管理/获取当前进程路径】**
* **底层技术析理**: 目标项目 `myproc` 调用 `mycpu, pop_off, proc_current, push_off`。其中 `proc_current` 是一个独立的抽象层，将“获取当前进程”与“获取当前 CPU”解耦。参考项目 `myproc` 调用 `mycpu, pop_off, push_off`，直接通过 `mycpu` 推导当前进程。目标项目引入了额外的抽象层，且 `mycpu` 底层调用 `arch_cpu_id`（多架构抽象），参考项目 `mycpu` 底层调用 `r_tp`（直接读 RISC-V 寄存器）。两者的 CPU/进程上下文模型和硬件抽象层设计不同。
* **代码溯源链接**:

  - 参考项目: `kernel/proc.c#myproc`
