## 第四部分：分模块架构说明

### MM（内存管理）
**已实现（有底层证据）**：

- 内核/用户页表初始化：`kernel/arch/riscv64/boot/main.c#main` 调用 `kvm_init`/`kvm_inithart`；LoongArch 侧有 `la_mm_alloc_begin_direct` 等启动期内存发现。
- 地址空间布局与加载：`kernel/proc/exec.c#proc_exec` 调用 `arch_proc_mmap_begin`、`arch_proc_mmap_pages`、`arch_proc_trapframe_alloc`。
- 用户态拷贝与缺页：`uvm_copyin`/`uvm_copyout` 在大量 syscall 中出现；`kernel/arch/loongarch64/trap/trap.c#la_trap_handle_shared_process_fault` 调用 `uvm_mmap_handle_fault`。
- 返回用户态时的地址空间切换：`kernel/trap/trap_user.c#trap_user_return` 设置 `kernel_stack`/`kernel_token`/`kernel_trap` 并通过 `trampoline_userret` 返回。

**未体现/证据有限**：显式的 swap、按需分页（demand paging）完整策略、大页/NUMA 内存管理、内存 cgroup 等高级特性底层证据未体现。

### PROC（进程管理/任务调度）
**已实现（有底层证据）**：

- 传统进程生命周期：`proc_clone`、`proc_exec`、`proc_exit`、`proc_wait`、`fork_return` 均存在，且 `proc_wait` 调用 `proc_reap_vm_group_zombies_locked`、`proc_free`，说明有 zombie 回收机制。
- PCB 与进程表：`myproc`、`proc_table_begin/end`、PID 分配与保留（`proc_pid_reserve_min`、`proc_table_reserve_locked`）。
- 信号机制：`proc_signal_deliver_pending`、`sys_rt_sigaction`、`proc_kill`。
- Futex/线程同步：`sys_futex`、`futex_light_wait`/`wake`/`requeue`、robust futex 相关函数。
- 调度与上下文切换：`proc_scheduler`、`proc_sched`、`proc_scheduler_run_selected` 调用 `swtch`。
- 进程组/会话：`proc_setpgid`、`proc_getpgid`、`proc_getsid`、`sys_setsid`。

**关于“轻量任务调度/异步运行时”**：存在 `kernel/fs/light_runtime.c#light_runtime_install`、`sysfile_light_*`、`sysproc_*_light_*` 等轻量快速路径与运行时 copyout 层，但**没有** `task`、`channel`、`waker`、`ring_fifo` 等典型 async runtime 抽象，因此不能判定为完整异步运行时，只能视为传统调度器上的 syscall 快速路径/测试辅助层。

### FS（文件系统）
**已实现（有底层证据）**：

- VFS 抽象层：`kernel/fs/file.c#file_open_mode`、`file_read`、`file_write`、`file_close`、`file_dup` 等。
- 多后端支持：FAT32（`kernel/fs/fat32.c`）、ext4（`kernel/fs/ext4.c`、`ext4_probe*`）、memfile（`kernel/fs/memfile.c`）、procfs（`kernel/fs/file_procfs_light.c`、`proc_build_text`）、pipe、RTC/设备文件。
- 挂载与路径解析：`kernel/fs/mount.c#resolve_mount_dir` 同时调用 `ext4_path_resolve` 与 `fat32_namei`；`file_resolve_vfs_path_at`。
- 文件系统初始化：`kernel/fs/fs.c#fs_init` 调用 `vfs_init`、`vfs_probe_filesystem`、`buf_init`。
- 镜像构建工具：`mkfs/mkfs.c` 用于生成初始文件系统镜像。

**第三方/协议约束代码说明**：`kernel/arch/loongarch64/storage/ext4_probe*.c` 与 `kernel/fs/ext4.c` 中的 ext4 解析逻辑属于文件系统/第三方库模块，应单独说明为 ext4 格式的适配/探测代码，而非原创内核主体。

### DEV（设备驱动）
**已实现（有底层证据）**：

- 串口：`la_uart_puts`、`la_uart_put_hex`、`uart_putc_sync`、`uart_intr`。
- 块设备：LoongArch 侧 VirtIO-PCI 块设备驱动（`la_virtio_pci_block_probe`、`la_virtio_pci_transfer_sector`、`la_virtio_pci_negotiate_features`）。
- 中断控制器：PLIC（RISC-V）、EXTIOI/LS7A（LoongArch）、timer。
- PCI 枚举：`la_pci_find_virtio_block`、`la_pci_enable_device`。

**未体现/证据有限**：GPU、声卡、网卡（除 VirtIO 块设备外）、USB、SATA/AHCI 等驱动底层证据未体现。

### SYSCALL（系统调用）
**已实现（有底层证据）**：

- 文件类：`sys_openat`、`sys_newfstatat`、`sys_unlinkat`、`sys_readv`、`sys_writev`、`sys_readlinkat`、`sys_utimensat` 等。
- 进程/信号/时间/身份/控制类：`sys_futex`、`sys_nanosleep`、`sys_gettimeofday`、`sys_getpid/ppid/tid/uid/gid`、`sys_setpgid`、`sys_prlimit64`、`sys_getrandom` 等。
- 轻量分发层：`sysfile_light_dispatch_number`、`syscall_dispatch_light_number`、`sysproc_*_light_*` 系列。
- Socket 轻量接口：`sysfile_socket_light_*`（bind/connect/accept/sendto/recvfrom）。

### TRAP/IRQ（中断异常）
**已实现（有底层证据）**：

- 统一 trap 入口：`la_trap_handle`（LoongArch）、`trap_user_handler`/`trap_kernel_handler`（公共层）。
- 系统调用、缺页、时钟中断、外设中断的分发。
- 中断使能与清除：`arch_intr_on/off`、`arch_trap_clear_software_interrupt`、`plic_claim/complete`、`la_extioi_claim` 等。

### NET（网络协议栈）
**底层证据未体现完整网络协议栈**。

仅有 `kernel/syscall/sysfile_socket_light.c` 中的 socket 轻量系统调用分发（`sysfile_socket_light_recvfrom`、`sysfile_socket_light_bind/connect/accept` 等），其内部调用 `dgram_get`、`sock_ring_alloc`、`proc_sleep`/`proc_wakeup`，看起来是本地/测试用的 datagram ring 或 socket shim，**没有 TCP/IP、UDP、ARP、路由、网卡驱动等完整网络栈证据**。

## 第六部分：边界与未体现内容

### 没有底层证据支撑的模块/特性

- **完整网络协议栈**：仅有 socket-light shim，无 TCP/IP/UDP/ARP/路由/网卡驱动证据。
- **高级虚拟化/ hypervisor**：无 VMX/SVM、KVM、设备模拟等证据。
- **SMP/NUMA 深度支持**：虽有 `mycpu`、`arch_cpu_id`，但未体现多核调度策略、NUMA 内存分配、CPU 亲和性完整实现。
- **安全机制**：无 MAC（SELinux/AppArmor）、 capabilities、seccomp、ASLR、KASLR、栈保护等证据。
- **动态链接加载器**：内核内未体现 ELF interpreter、`ld.so` 加载逻辑。
- **高级文件系统**：无 NFS、Btrfs、ZFS、日志/快照/COW 等证据。
- **多媒体/GPU/声卡/USB 等驱动**：底层证据未体现。
- **电源管理、热插拔、调试器（KGDB）、跟踪（ftrace/perf）**：底层证据未体现。

### 需要单独说明的代码边界

- `kernel/arch/loongarch64/storage/ext4_probe*.c` 与 `kernel/fs/ext4.c` 属于**文件系统/第三方库或强协议约束代码**，其功能是实现 ext4 格式的解析与挂载，不应被夸大为原创内核主体。
- `user/initcode.c` 是用户态测试启动器，用于运行 Busybox/glibc/musl/LTP 等测试用例，不属于内核核心，但可作为系统调用兼容性的验证入口。
- `mkfs/mkfs.c` 是文件系统镜像构建工具，属于配套工具链。

### 判断性质说明

- 本作品中“多架构支持”是基于存在 `arch/riscv64` 与 `arch/loongarch64` 目录及 `arch_*`/`la_*` 抽象接口的**可能方向推断**，并非对硬件支持完整度的最终结论。
- “轻量任务调度/异步运行时”仅体现为轻量 syscall 快速路径与运行时 copyout 层，**不等同于**完整 async runtime。
- 本描述仅基于 AST Call Map、函数名、模块名、路径锚点与调用链，不做抄袭、违规或原创性最终判定。
