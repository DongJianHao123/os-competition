#### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建路径】**
* **底层技术析理**: 目标项目 `sys_fork` 调用 `proc_fork`，参考项目 `sys_fork` 调用 `fork`。两者均为系统调用入口的单节点派发，但目标项目的 `proc_fork` 封装在独立的进程子系统中，与参考项目全局 `fork` 的命名空间与模块化设计存在根本差异。目标项目在进程创建路径上还实现了 `sys_clone`（调用 `proc_clone`）以支持 Linux 兼容的线程创建，参考项目无此抽象。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/sysproc.c#sys_fork`
  - 参考项目: `kernel/sysproc.c#sys_fork`

* **【进程管理/进程上下文获取】**
* **底层技术析理**: 目标项目 `myproc` 调用链为 `mycpu -> pop_off -> proc_current -> push_off`，参考项目为 `mycpu -> pop_off -> push_off`。目标项目通过 `proc_current` 将“当前进程”抽象为独立接口，解耦了 CPU 结构与进程调度器的直接依赖，这是支持多架构（LoongArch64/RISC-V64）与更复杂调度模型（如 futex 优先级队列）的架构设计。参考项目直接通过 `mycpu()->proc` 访问，是 xv6 教学内核的简化实现。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/cpu.c#myproc`
  - 参考项目: `kernel/proc.c#myproc`

* **【进程管理/休眠与同步】**
* **底层技术析理**: 目标项目 `sys_sleep` 调用 `arg_uint32, myproc, proc_exit, proc_sleep, spinlock_acquire, spinlock_release`，参考项目调用 `acquire, argint, myproc, release, sleep`。目标项目使用显式类型的 `spinlock_*` 锁原语与 `proc_sleep` 封装，并引入 `proc_exit` 节点处理休眠中被终止的边界情况。参考项目使用通用 `acquire/release` 和 `sleep`。两者在锁粒度、异常安全与休眠抽象上存在实质性差异。
