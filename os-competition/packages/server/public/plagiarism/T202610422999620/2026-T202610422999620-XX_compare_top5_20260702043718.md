- **共同 AST 指纹**：1 个
- **覆盖率**：目标 0.4%，参考 2.5%
- **解读**：目标 587 个函数，参考仅 45 个函数，规模差异达 13 倍。唯一共同指纹为 `sys_setpgid` 同名函数，但参考实现已退化为 `log::debug!`，无实际可比逻辑。目标独有的 `sys_wait4`、`sys_waitid`、`futex_requeue_common`、`sys_sched_setattr`、`sys_capset`、`sys_prlimit64` 等函数在参考项目中无任何对应。
- **证据强度**：弱

### 任务调度/异步运行时
- **双方均存在**：是
- **共同 AST 指纹**：4 个
- **覆盖率**：目标 1.42%，参考 4.6%
- **解读**：共同指纹集中在 `sys_kill`、`sys_utimensat`、`sys_getdents64`、`sys_faccessat` 四个同名系统调用。这些函数名由 POSIX/Linux ABI 强制规定，且双方调用的外部对象集合差异显著（目标使用 `TASK_MANAGER.for_each`、`SigInfo::new`，参考使用 `pid2task`、`inner.signals.add_sig`）。
- **证据强度**：弱

### 内存管理
- **双方均存在**：是
- **共同 AST 指纹**：28 个
- **覆盖率**：目标 6.81%，参考 14.14%
- **解读**：这是双方覆盖率最高的模块，但 28 个共同指纹中多数为 `sys_fcntl`、`sys_brk`、`from_existed_user`、`write_at` 等 POSIX 系统调用或 Rust 惯用命名。目标使用 `MemorySet`/`MapArea`/`VPNRange` 抽象，参考使用 `AddrSpace`/`Section` 抽象，底层内存管理模型不同。
- **证据强度**：中（需进一步剖析架构差异）

### 中断/异常处理

- **共同 AST 指纹**：1 个
- **覆盖率**：目标 5.0%，参考 20.0%
- **解读**：共同指纹为 `enable_timer_interrupt` 同名函数，但目标支持 LoongArch64 和 RISC-V 双架构（`ecfg::enable_timer_interrupt` / `sie::set_stimer`），参考仅 RISC-V 单架构。覆盖率数值因参考模块函数总数仅 8 个而被放大。
- **证据强度**：弱

### 系统调用

- **证据强度**：弱

## 4. 核心代码调用流对比剖析

### 【进程管理 / exec 系统调用路径】

**底层技术析理**：

目标项目 `sys_execve` 与参考项目 `sys_exec` 虽然同为 exec 系统调用入口，但底层调用流揭示了完全不同的内核架构设计：

| 维度 | 目标项目 | 参考项目 |
|------|----------|----------|
| 地址空间抽象 | `MemorySet::try_from_elf_data` | `AddrSpace::create_user_space` |
| 内核空间访问 | 无显式锁 | `KERNEL_SPACE.lock()` |
| 内存映射清理 | 无（由 MemorySet 重建隐式处理） | `remove_mmap_area_with_start_vpn` |
| 参数传递 | `args_vec.as_slice`（切片引用） | `(0..=args.len()).collect`（索引向量构造） |
| 陷阱上下文 | `TrapContext::init_app_context` | 无显式陷阱初始化（由 Section 隐式处理） |
| 信号栈 | `SignalStack::default` | 无 |
| 文件执行路径 | `builtin_for_fs_exec` → `exec_fs_file` → `shebang_args` | `"/busybox".to_string` / `"sh".to_string` 硬编码 |

目标项目的 exec 路径包含完整的 shebang 脚本解析（`shebang_args`）、信号栈初始化（`SignalStack::default`）、以及通过 VFS 层的文件执行权限检查（`check_exec_permission`、`exec_looks_like_ltp_mkfs`）。参考项目的 exec 路径则围绕 `KERNEL_SPACE` 锁、mmap 区域清理、以及硬编码的 busybox/sh 路径展开。两者在地址空间管理、内核同步策略、文件执行流程三个核心维度上均采用不同的设计抽象。
