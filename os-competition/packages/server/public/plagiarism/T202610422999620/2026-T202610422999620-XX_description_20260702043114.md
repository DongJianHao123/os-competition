## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心文件为 `os/src/mm/memory_set.rs`，负责构建用户地址空间、ELF 加载、页表映射与缺页处理。

- `os/src/mm/memory_set.rs#try_from_elf_data`：按 ELF program header 创建 `MapArea`，建立用户态页表。
- `os/src/arch/loongarch64/trap/mod.rs#handle_user_page_fault`：读取 `badv`，调用 `memory_set.handle_page_fault`，失败时构造 `SigInfo` 发送信号。
- `os/src/drivers/virtio/mod.rs#dma_dealloc`：维护 `DMA_ALLOCATIONS`，管理 VirtIO 所需的连续物理页。
- 系统调用层面可见 `sys_msync`、`sys_fallocate`、`sys_pread64`/`sys_pwrite64` 中的用户缓冲区检查与 `copy_to_user`/`copy_from_user`。

### PROC（进程管理/任务调度）

底层证据强。PCB 集中在 `os/src/task/task.rs`，字段涵盖 tid/tgid/pgid/sid、uid/euid/suid、capabilities、nice/sched policy、cpu affinity、robust_list、thread_group、children/parent 等。

- `os/src/task/task.rs#clone_`：创建新任务，复制/共享内存集、文件表、信号状态、能力集等。
- `os/src/task/task.rs#execve`：替换地址空间、重置信号栈、初始化用户栈与陷阱上下文。
- `os/src/syscall/process.rs#sys_wait4` / `sys_waitid`：实现 `wait4/waitid`，处理等待选项与资源统计。
- `os/src/task/scheduler.rs#switch_to_next_task_after_exit` / `run_tasks`：基于 `PROCESSOR` 与 `__switch` 的协作式/抢占式上下文切换。
- `os/src/signal/mod.rs#handle_signal`：信号投递与 sigreturn 栈帧构造。
- 支持 futex（`task/futex/wait.rs`）、robust list、`setpgid/setsid`、capabilities、`sched_setaffinity` 等。

### FS（文件系统）

底层证据强，且代码量最大。采用 VFS 抽象：`Dentry`、`Inode`、`File`、`Path`、`Nameidata`。

- `os/src/fs/namei.rs#resolve_path_from` / `link_path_walk` / `open_last_lookups` / `filename_rename`：路径解析、打开、重命名。
- `os/src/fs/mount.rs#do_mount` / `remove_mount_tree_inner`：挂载树、backing root、mount point 管理。
- `os/src/fs/proc/dirs.rs#generate_pid_stat` 及 `fs/proc/{stat,maps,smaps,meminfo,mounts,...}`：procfs 实现。
- `os/src/fs/ext4/inode.rs`：对第三方 lwext4 的封装，使用 `EXT4_OP_LOCK` 串行化底层操作。
- `os/src/fs/pipe.rs` 与 `os/src/syscall/special_fd.rs`：管道、epoll、timerfd 等。

### DEV（设备驱动）

底层证据**弱**，仅 34 个函数。

- `os/src/drivers/mod.rs#new_device`：通过 MMIO 探测 VirtIO 设备。
- `os/src/drivers/disk.rs#read` / `write` / `seek`：VirtIO block 的块读写。
- `os/src/arch/loongarch64/sbi.rs#console_putchar`：串口/SBI 控制台输出。
- `os/src/mm/memory_set.rs#new_kernel`：将 `VIRTIO_MMIO` 区域映射到内核地址空间。

### SYSCALL（系统调用）

底层证据强。入口为 `os/src/syscall/mod.rs#syscall`，调用数达 217。

- 分发到 `syscall/fs.rs`、`syscall/process.rs`、`syscall/net.rs`、`syscall/signal.rs`、`syscall/special_fd.rs`。
- 覆盖文件、进程、网络、信号、epoll/timerfd、内存（brk/msync）等 Linux 兼容调用。
- 大量调用包含 `current_task().expect(...)` 获取当前任务上下文，并进行用户态指针校验。

### TRAP/IRQ（中断异常）

底层证据**弱**，仅 21 个函数。

- `os/src/arch/loongarch64/trap/mod.rs#trap_handler`：异常/中断统一入口，解析 `estat.cause`，处理定时器中断、系统调用、缺页。
- `os/src/arch/rv64/trap/mod.rs#init`：设置 `stvec` 与 `sstatus`。
- `os/src/arch/loongarch64/register/mod.rs#cause` / `interrupt_from_index`：异常原因解码。
- 可见定时器中断使能，但完整中断控制器（PLIC/IOIE/LIOINTC）驱动证据不足。

### NET（网络协议栈）

底层证据中强。

- `os/src/net/socket.rs`：统一 socket 抽象，区分 TCP/UDP/Unix。
- `os/src/net/tcp.rs#listen` / `connect` / `recv`：TCP 监听、连接、接收，使用 `SocketSetWrapper` 与 `block_on`。
- `os/src/net/udp.rs#bind` / `send_impl`：UDP 绑定与发送。
- `os/src/net/socket.rs#connect_path`：Unix domain socket 通过 `UNIX_LISTENERS` 查找监听端。
- `os/src/syscall/net.rs#read_sockaddr_for_domain` / `write_sockaddr_for_domain`：用户态 socket 地址拷贝。
- 与 epoll/poll 集成：`sys_pselect6` 中出现 `PollRegistration`、`block_for_poll`。

## 第六部分：边界与未体现内容

- **启动/早期初始化代码底层证据未体现**：现有证据主要集中在 trap 初始化与用户态 `initproc`，未看到 bootloader 交接、页表早期建立、BSS 清零等完整启动链。
- **多核/SMP 支持底层证据不足**：虽然出现 `sched_setaffinity` 等 CPU 亲和性字段，但未看到 IPI、per-CPU 调度器、核间同步等强证据。
- **设备驱动范围狭窄**：仅见 VirtIO block 与串口控制台，网络底层网卡驱动（如 virtio-net）、PLIC/IOIE/LIOINTC 中断控制器、时钟源等驱动证据不足。
- **独立块缓存/缓冲区层底层证据未体现**：块设备读写直接通过 ext4 封装层完成，未见通用 buffer cache。
- **安全/隔离增强模块底层证据未体现**：未见 LSM、seccomp、cgroup、完整 namespace（仅 `net_ns` 的 loopback tag 写回）、KASLR 等。
- **第三方依赖需单独说明**：`os/src/fs/ext4/inode.rs` 明显是对 lwext4 的封装，不应视为原创文件系统实现主体。
- **“可能的系统方向”仅为基于 AST 与目录名的技术推断**，不代表最终作品身份、原创性或合规性结论。
