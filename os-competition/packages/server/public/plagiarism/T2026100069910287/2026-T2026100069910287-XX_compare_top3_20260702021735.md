### 4. 核心代码调用流对比剖析
* **【进程管理 / `sys_execve` 系统调用】**
* **底层技术析理**: 两者均实现了 `execve` 系统调用，但路径解析和程序加载策略存在根本性架构差异。目标项目采用通用化的文件系统抽象，通过 `<dyn File>::open` 和 `Path::resolve_symlink_through` 动态解析可执行文件路径，体现了与文件系统层解耦的设计。参考项目则硬编码了针对特定架构（`/loongarch/musl/busybox`, `/riscv/musl/busybox`）的路径字符串，其实现与特定二进制文件强耦合。这表明两者在文件系统抽象层次和系统可移植性设计上存在本质不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_execve`
  - 参考项目: `os/src/syscall/process.rs#sys_execve`

* **【任务调度/异步运行时 / 执行器队列操作】**
* **底层技术析理**: 两者均实现了任务执行器，但调度策略和数据结构截然不同。目标项目在 `lib/executor/src/lib.rs` 中实现了双队列模型：`push`/`fetch` 操作普通任务队列 `tasks`，而 `push_prio`/`fetch_prio` 操作优先级任务队列 `pritasks`，`fetch` 会优先从优先级队列获取任务。参考项目在 `os/src/executor/mod.rs` 中采用单队列模型，通过 `push`（尾部插入）和 `push_preempt`（头部插入）在同一 `queue` 上实现抢占。这种从单队列抢占式到双队列优先级式的设计差异，是调度器核心架构的不同。
* **代码溯源链接**:
  - 目标项目: `lib/executor/src/lib.rs#fetch`
  - 参考项目: `os/src/executor/mod.rs#fetch`

* **【内存管理 / 用户态指针安全抽象】**
* **底层技术析理**: 两者在处理用户态内存访问时采用了完全不同的安全抽象模型。目标项目使用泛型封装 `UserReadPtr<T>` 和 `UserWritePtr<T>`，在类型层面区分读写权限，将安全检查融入类型系统。参考项目则使用 `UserPtrRaw` 结合运行时的 `ensure_read`/`ensure_write` 方法，并需要显式传入内存空间锁 `&mut task.get_vm_space().lock()`。这是两种截然不同的内存安全设计哲学：编译期类型强制 vs. 运行时检查。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/signal.rs#sys_rt_sigtimedwait`
  - 参考项目: `os/src/syscall/signal.rs#sys_rt_sigtimedwait`

### 5. 单方模块与降权项
- **单方特有模块/函数**:
  - **目标项目**: `[user/src/initproclib.rs]` 函数 `git_init`，`[kernel/src/task/taskf.rs]` 函数 `exit` 中调用的 `FutexHashKey`、`SHARED_MEMORY_MANAGER` 等，体现了目标项目独有的 futex 和共享内存管理机制。
  - **参考项目**: `[os/src/syscall/process.rs]` 函数 `sys_waitpid` 中调用的 `PROCESS_GROUP_MANAGER`，体现了其独特的进程组管理方式。
  这些单方存在的模块和调用链，是两者独立开发的有力证据。
- **降权项说明**:
  - **POSIX/Linux 接口强制同名函数**: `sys_getpid`, `sys_execve`, `sys_clone`, `sys_exit` 等函数名由标准接口定义，任何实现者都必须使用，不能作为同源证据。
  - **Rust 语言惯用法与标准库节点**: `Ok(…)`, `Some(…)`, `Arc::clone`, `VecDeque::new`, `iter().map().reduce()` 等是 Rust 生态的标准模式，其 AST 指纹必然相同。
  - **Linux ABI 常量**: `/proc/meminfo` 的字段名、信号名（`SIGCHLD`）、系统调用号等是公共接口常量，非原创表达。
  - **硬件驱动寄存器操作**: `uart8250`, `dw_mshc` 等驱动中的寄存器偏移量和初始化序列由硬件手册规定，实现空间有限，其 AST 趋同是领域约束所致。

### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度仅为 5.28%，处于极低水平。控方基于浅层外部调用图谱提出的“系统性复用”指控，被辩方从 POSIX 接口约束、Linux ABI 兼容、Rust 语言惯用法及硬件规范等角度有效驳斥。深入分析显示，两者在用户指针安全模型、任务调度器架构、文件系统模块化设计、`execve` 路径解析策略等核心子系统上存在根本性的设计差异。现有证据充分支持“独立开发”的结论，整体学术不端风险极低。共同 AST 指纹主要源于实现同类操作系统时不可避免的领域趋同，而非代码抄袭。
