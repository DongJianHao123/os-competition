### 4. 核心代码调用流对比剖析
* **【进程管理/系统调用路径】**
* **底层技术析理**: 以 `sys_getpid` 为例，两者调用流均为“获取当前任务 -> 返回进程ID”，但底层对象模型和并发处理不同。参考项目通过 `current_task().unwrap()` 返回 `Option`，并调用 `get_tgid()` 方法，体现了基于线程组（thread group）的进程模型。目标项目直接访问 `current_task().pid` 字段，`current_task` 返回不可空类型，模型更扁平。这种差异反映了双方在任务结构体设计和空安全处理上的不同选择。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_getpid`
  - 参考项目: `os/src/syscall/process.rs#sys_getpid`

* **【进程管理/并发控制】**
* **底层技术析理**: 在退出码管理上，两者采用了根本不同的同步原语。参考项目使用原子操作 `self.exit_code.store/load`，追求无锁化高性能。目标项目使用互斥锁 `self.exit_code.lock`，追求简单安全的并发控制。这是两种截然不同的并发编程范式，是架构设计层面的深层差异，绝非简单的“重命名”可以达成。
* **代码溯源链接**:

  - 参考项目: `os/src/task/task.rs#set_exit_code`

* **【内存管理/系统调用】**
* **底层技术析理**: 在 `sys_prlimit64` 实现中，双方展示了完全不同的内存安全模型。参考项目使用 `core::ptr::write` 等裸指针操作，属于不安全 Rust 范式。目标项目则封装了 `UserReadPtr` / `UserWritePtr` 类型，通过安全抽象进行用户态内存访问。这是内核设计中最核心的安全架构差异，表明双方在内存管理上走了完全不同的技术路线。
