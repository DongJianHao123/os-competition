### 4. 核心代码调用流对比剖析
* **【进程管理/进程时间统计】**
* **底层技术析理**: 控方认为 `get_process_ustime` 等函数是“重命名混淆”的抄袭。本庭剖析其调用流后发现，这是同一语义在 Rust 语言下的自然实现。两者均需遍历线程组、提取时间、求和。Rust 对此的标准惯用法即为 `iter().map().reduce().unwrap()`。目标项目使用 `thread.timer_mut()`，参考项目使用 `thread.time_stat()`，这恰恰揭示了双方在**对象抽象**上的根本差异：一个将时间视为定时器模块的可变状态，另一个将其封装为独立的统计结构体。这是不同架构设计的体现，而非简单的重命名。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/time.rs#get_process_ustime`
  - 参考项目: `kernel/src/task/resource.rs#get_process_ustime`

* **【设备驱动/DW_MSHC 初始化】**
* **底层技术析理**: `card_init` 和 `send_cmd` 函数在 AST 层面高度重合。本庭认定，这种重合是**硬件规范驱动**的必然结果。对 DW_MSHC 这类标准 IP 核进行初始化，必须按照其数据手册规定的顺序，向特定寄存器偏移（`CLKDIV::offset`、`CMD::offset`）写入特定命令字（`CMD::no_data_cmd_no_crc(card_idx, 2).with_response_length(...)`）。该调用序列是硬件的事实标准，任何驱动开发者都必须遵循，不具备代码表达的独创性。
* **代码溯源链接**:
  - 目标项目: `lib/driver/src/block/dw_mshc/mmc.rs#card_init`
  - 参考项目: `driver/src/blk/dw_mshc/mmc.rs#card_init`

### 5. 单方模块与降权项
*   **单方模块**: 参考项目中存在 `第三方库` 模块（`crates/ring-buffer`, `crates/range-map`），目标项目无此模块。这表明双方在基础库选型和依赖上存在差异，是独立开发的佐证。
*   **降权项**: 以下证据因不具备特异性，本庭不予采信为同源性证据：
    *   **所有 VFS Trait 实现** (`base_open`, `base_create`, `base_lookup`, `get_attr`): 这些是框架强制要求的模板代码，其调用图谱（`Arc::new`, `FileMeta::new`, `Ok`, `Err`）是任何基于该框架的实现都会产生的。

    *   **公共库/宏调用** (`log::info!`, `async_task::spawn`, `core::hint::spin_loop`): 这些是 Rust 生态的公共依赖，不能作为两个项目之间的特异性关联证据。
