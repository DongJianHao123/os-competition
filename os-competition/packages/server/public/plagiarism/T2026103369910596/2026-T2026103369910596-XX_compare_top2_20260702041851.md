  - 参考项目: `kernel/sysproc.c#sys_exit`

* **【进程管理/调度器核心循环】**
* **底层技术析理**: 参考项目 `scheduler` 的调用流为 `acquire` -> 遍历进程表 -> `swtch` 上下文切换 -> `release`。目标项目 `run_tasks` 的调用流为 `INITPROC.inner_exclusive_access` -> 查找CFS队列 -> `__switch` 上下文切换 -> 处理僵尸进程及控制台中断。两者在**对象管理模型**上截然不同：参考项目直接操作静态进程表，目标项目通过 `INITPROC` 单例和CFS红黑树管理任务。目标项目额外集成了 `console_receive`、`hartid` 等多核及设备轮询逻辑，体现了架构上的独立扩展。
* **代码溯源链接**:
  - 目标项目: `os/src/sched/processor.rs#run_tasks`
  - 参考项目: `kernel/proc.c#scheduler`

* **【内存管理/伙伴分配器】**
* **底层技术析理**: 两者 `alloc`/`dealloc` 函数的调用图谱高度相似，均包含 `layout.size().next_power_of_two`、`block.pop` 等核心步骤。这是**伙伴分配算法**的标准实现流程，属于公开的经典算法，其AST结构相似是算法逻辑本身决定的，不构成独创性抄袭。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/heap_allocator.rs#alloc`
  - 参考项目: `third/buddy_system_allocator/src/lib.rs#alloc`

### 5. 单方模块与降权项
- **网络协议栈**: 目标项目包含290个函数的完整TCP/IP协议栈，参考项目完全不存在。这是目标项目独立开发的强证据。
- **Futex/进程记账/Procfs等高级特性**: 目标项目实现了 `futex_wait_addr`、`write_process_accounting_on_exit`、`ProcKernelSysctlNode` 等复杂Linux兼容特性，在参考项目中无任何对应实现，体现了独立开发能力。
- **同名系统调用**: `sys_getpid`、`sys_clone` 等均为POSIX/Linux标准接口名称，任何兼容内核都必然使用，不能作为抄袭证据。
- **跨板块硬凑**: 控方将 `sys_futex` 与 `virtio_disk_intr`、`read_at_result` 与 `plicinithart` 等跨模块、跨功能的函数强行对齐，此类证据无效，已全部降权。

### 6. 保守结论
综合全模块AST指纹数据，目标项目与参考项目的整体相似度极低（4.3%），且重合点几乎全部集中于双方引用的开源第三方库（`ext4_rs`、`buddy_system_allocator`）。在操作系统内核的核心板块（进程管理、任务调度、文件系统、系统调用），客观结构指纹重合为0，底层架构（如从轮询调度到CFS、从xv6风格FS到ext4/FAT32、从无网络到完整TCP/IP协议栈）存在根本性差异。目标项目展现了大量独立实现的高级特性。因此，本庭裁定，无证据表明目标项目对参考项目构成实质性代码复用，两者为独立开发。
