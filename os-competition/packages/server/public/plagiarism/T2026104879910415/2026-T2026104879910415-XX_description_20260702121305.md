## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心实现分布在 `arceos/modules/axmm/`、`arceos/modules/axalloc/` 与 `kernel/src/syscall/memory.rs`。

- **页异常处理**：`kernel/src/trap.rs#handle_user_page_fault` 调用 `aspace.handle_page_fault`，最终进入 `arceos/modules/axmm/src/backend/alloc.rs#handle_page_fault_alloc`，涉及 `alloc_frame`、`cursor.map/remap`、`hardware_page_flags`。
- **用户态内存映射**：`kernel/src/syscall/memory.rs#syscall_mmap` 支持 `map_alloc`、`map_alloc_overwrite`、`map_shared_lazy`，并处理 `SharedPages`。
- **初始化**：`arceos/modules/axruntime/src/lib.rs#init_allocator` 调用 `axalloc::global_init` 与 `global_add_memory`。
- **进程相关内存**：`kernel/src/syscall/thread.rs#syscall_clone_fork` 调用 `aspace.fork`；`kernel/src/exec/mod.rs#map_initial_tls` 处理静态 TLS 布局。

> 未体现：底层 MMU/TLB 平台相关代码、NUMA、swap、大页/巨页管理的直接证据较少。

### PROC（进程管理）

底层证据充分，已超出“轻量任务”范畴，具备传统 Unix 进程管理特征。

- **进程创建**：`kernel/src/task/mod.rs#fork_from`、`kernel/src/task/mod.rs#new_child`、`kernel/src/syscall/thread.rs#syscall_clone_fork`。
- **进程执行**：`kernel/src/syscall/process.rs#syscall_execve`、`kernel/src/exec/mod.rs#load_exec_image`。
- **进程退出**：`kernel/src/syscall/process.rs#exit_thread_common`、`kernel/src/task/mod.rs#exit_thread_group`。
- **等待与回收**：`kernel/src/syscall/process.rs#syscall_wait4` / `syscall_waitid`、`kernel/src/task/wait.rs#wake_waiting_parent_for_child_event`、`kernel/src/task/wait.rs#reap_child_process`。
- **进程关系**：`reparent_children_to_init`、进程组与会话、`procfs` 中的 `pid`、`ppid`、`thread_ids`。
- **凭证与能力**：`syscall_capget`、`syscall_capset`、`syscall_prctl`、`credentials`、`capability_sets`。

### 任务调度/异步运行时

以 `kernel/src/task/wait.rs` 与 `kernel/src/task/fd.rs` 为核心，表现为基于可中断阻塞队列的调度模型。

- **调度器主循环**：`kernel/src/task/wait.rs#schedule_next_ready` 从 `READY_THREADS` 取出下一个任务，并轮询 `INTERRUPTIBLE_BLOCKS` 与 futex 超时。
- **阻塞与唤醒**：`INTERRUPTIBLE_BLOCKS` 全局队列，`wake_ready_pipe_reads`、`wake_signal_wait_thread_where`、`wake_blocked_thread_tid`。
- **CPU 亲和性**：`kernel/src/syscall/thread.rs#syscall_sched_setaffinity`。
- **ArceOS 线程支持**：`arceos/ulib/axstd/src/thread/multi.rs#spawn_unchecked`、`arceos/modules/axtask/src/run_queue.rs#exit_current`。

### FS（文件系统）

文件系统是代码量最大的原创模块之一，同时大量依赖第三方库。

- **原创层**：
  - `kernel/src/fs/rootfs.rs`：挂载点管理、容量受限 ramfs、`create_file`、`rename_path`、`check_open_vfs_real_write_capacity`。
  - `kernel/src/fs/procfs.rs`：`/proc` 与 `/sys` 伪文件系统，`get_child`、`render`、`write_at`。
  - `kernel/src/syscall/fs.rs`：`syscall_open_by_handle_at`、`read_file_handle_id`。
- **第三方库**：
  - `third_party/rust/git/rust-fatfs`：FAT 文件系统
  - `third_party/rust/git/ext4-view-rs`：EXT4 只读/解析支持
  - `third_party/rust/git/smoltcp` 不属 FS，但 `third_party` 整体占 FS 相关代码很大比例

### DEV（设备驱动）

证据相对薄弱，主要体现为框架级初始化与枚举。

- `arceos/modules/axdriver/src/lib.rs#init_drivers`：探测 block/net/display 设备并枚举。
- `arceos/tools/bwbench_client/src/device.rs`：基于 Linux raw socket 的测试工具，非内核驱动。
- `arceos/tools/raspi4/chainloader/src/bsp/device_driver/bcm/bcm2xxx_pl011_uart.rs#init`：树莓派 UART 初始化示例。

> 未体现：具体块设备控制器、NVMe/SATA/SD 卡控制器、网卡 DMA、显示控制器等的完整驱动实现。

### SYSCALL（系统调用）

系统调用层证据非常强，呈现 Linux ABI 兼容层。

- **分发中枢**：`kernel/src/syscall/mod.rs#dispatch_minimal_syscall` 调用数 227，覆盖 fstatfs/statfs/sysinfo/syslog 等。
- **按子系统分布**：
  - 进程：`kernel/src/syscall/process.rs`
  - 线程/调度：`kernel/src/syscall/thread.rs`
  - 内存：`kernel/src/syscall/memory.rs`
  - 文件：`kernel/src/syscall/fs.rs`
  - 信号：`kernel/src/syscall/signal.rs`
  - poll/epoll：`kernel/src/syscall/poll.rs`
  - 时间：`kernel/src/syscall/time.rs`
  - 网络：`kernel/src/syscall/net.rs`
- **ABI 解码**：`kernel/src/arch.rs#decode_syscall_abi`、`kernel/src/trap.rs#handle_minimal_syscall`。

### TRAP/IRQ（中断异常）

证据较弱，仅有入口级函数。

- `kernel/src/trap.rs#handle_minimal_syscall`：系统调用入口。
- `kernel/src/trap.rs#handle_user_page_fault`：用户态页异常。
- `kernel/src/trap.rs#resume_syscall_thread`：系统调用返回/信号恢复路径。
- `kernel/src/syscall/signal.rs#syscall_rt_sigreturn`：信号返回。
- `kernel/src/syscall/time.rs#syscall_clock_nanosleep`：定时睡眠。

> 未体现：完整中断控制器（GIC/PLIC/IO-APIC）、中断分发、设备中断处理链、时钟中断底层驱动等。

### NET（网络协议栈）

原生网络代码极少，协议栈主体来自第三方 `smoltcp`。

- **原生包装层**：
  - `kernel/src/task/fd.rs#sendto_socket_fd`、`connect_socket_fd`、`accept_socket_fd`
  - `kernel/src/syscall/net.rs#read_sockaddr_un_path`、`write_sockaddr_un`
  - `arceos/api/arceos_posix_api/src/imp/net.rs#recvfrom`、`connect`、`bind`
  - `arceos/ulib/axstd/src/net/tcp.rs#bind/connect`
- **第三方协议栈**：
  - `third_party/rust/git/smoltcp/src/socket/tcp.rs#process` / `dispatch`
  - `third_party/rust/git/smoltcp/src/storage/ring_buffer.rs`

> 说明：网络协议栈原创性主要体现在 socket fd 管理与 POSIX API 适配，TCP/IP 协议状态机由 `smoltcp` 提供。

## 第六部分：边界与未体现内容

### 1. 没有底层证据支撑的模块/能力

- **完整中断/异常子系统**：仅有系统调用入口、页异常、信号返回等少量函数；底层 IRQ 控制器（GIC/PLIC/IO-APIC）、中断向量表、设备中断注册链未体现。
- **原生设备驱动**：`axdriver` 仅体现设备探测与枚举；具体块设备、网卡 DMA、显示控制器、时钟中断驱动实现证据不足。
- **原生网络协议栈**：TCP/IP 协议状态机、拥塞控制、IP 分片等主要依赖 `smoltcp`；原生代码仅为 socket fd 与 POSIX API 包装。
- **电源管理、SMP 核间中断、NUMA、swap/虚拟内存换页、安全模块（LSM）、虚拟化/Hypervisor**：底层证据未体现。
- **文件系统日志与写回策略**：`ext4-view-rs` 为第三方库，原创文件系统层主要围绕 ramfs/procfs/VFS 挂载。

### 2. 第三方与公共依赖说明

- `third_party/rust/git/smoltcp/`：网络协议栈主体，不应视为该作品原创内核主体。
- `third_party/rust/git/ext4-view-rs/`、`third_party/rust/git/rust-fatfs/`：文件系统第三方实现。
- `arceos/` 框架代码（`axruntime`、`axtask`、`axmm`、`axalloc`、`axdriver`、`axstd`、`axlibc` 等）属于公共基础框架，评审时需注意区分“基于 ArceOS 的扩展/适配”与“完全原创实现”。

### 3. 判断性质说明
