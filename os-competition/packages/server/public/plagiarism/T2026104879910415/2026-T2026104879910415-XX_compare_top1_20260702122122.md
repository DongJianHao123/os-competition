### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出】**
* **底层技术析理**: 目标项目的 `exit_thread_common` 是一个复杂的内部函数，其调用流涉及 `release_advisory_locks_for_process`（文件锁释放）、`flush_all_stdio_line_buffers_for_tid`（标准IO刷新）、`ltp_trace::trace_exit_publish`（测试追踪）等，体现了对 Linux 兼容性的深度支持。参考项目的 `sys_exit` 是一个简单的系统调用处理函数，调用流仅为 `exit_current_and_run_next`，直接切换到下一个任务。两者在抽象层次、功能完备性和外部依赖上完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#exit_thread_common`
  - 参考项目: `os/src/syscall/process.rs#sys_exit`

* **【任务调度/异步运行时/锁原语】**
* **底层技术析理**: 目标项目的 `mutex::lock` 基于 `compare_exchange_weak` 乐观锁和 `ax_wait_queue_wait_until` 等待队列实现，是 ArceOS 异步运行时的一部分。参考项目的 `mutex::lock` 基于 `self.locked.exclusive_access` 互斥锁和 `suspend_current_and_run_next` 直接挂起当前任务，是传统的同步阻塞模型。两者的同步策略和与调度器的交互方式存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `arceos/ulib/axstd/src/sync/mutex.rs#lock`
  - 参考项目: `os/src/sync/mutex.rs#lock`

* **【内存管理/地址空间 Fork】**
* **底层技术析理**: 目标项目的 `aspace::fork` 调用流围绕 `Backend::new_alloc` 和 `MemoryArea::new` 展开，体现了基于 `Backend` 和 `MemoryArea` 抽象的内存管理模型。参考项目的 `process::fork` 调用流围绕 `MemorySet::from_existed_user` 和 `FdTable::from_another` 展开，体现了基于 `MemorySet` 和文件描述符表复制的传统模型。两者的内存抽象和对象管理模型异构。
* **代码溯源链接**:
  - 目标项目: `arceos/modules/axmm/src/aspace.rs#fork`

* **【系统调用/信号处理】**
* **底层技术析理**: 目标项目的 `syscall_rt_sigprocmask` 直接操作 `CurrentThread` 的信号状态，调用 `defer_cancel_signal_delivery_once` 等高级特性，表明信号处理是线程级的。参考项目的 `sys_sigprocmask` 通过 `current_process` 获取进程内部锁 `inner_exclusive_access` 来修改信号掩码，表明信号处理是进程级的。两者的信号分发粒度和并发控制模型不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/signal.rs#syscall_rt_sigprocmask`
