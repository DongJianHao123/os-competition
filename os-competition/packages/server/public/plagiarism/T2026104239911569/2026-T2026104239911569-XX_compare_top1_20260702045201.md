## 4. 核心代码调用流对比剖析

### 【进程管理 / 进程退出与回收路径】

**底层技术析理**:
目标项目与参考项目在进程退出与回收路径上展现了完全不同的对象管理模型与同步策略。

- **目标项目**采用**全局锁 + 包装器模式**：所有进程/线程操作必须通过 `TASK_MANAGER.lock` 获取全局互斥锁，然后通过 `manager_wrapper.rs` 中的包装函数（如 `exit_current_and_run_next`、`waitpid_current`）协调 `manager.rs` 中的核心逻辑。进程退出时调用 `exit_current_and_run_next` → 获取 `TASK_MANAGER.lock` → 标记线程为 Zombie → 唤醒等待该线程的父进程（`wake_threads_joining`）→ 唤醒等待子进程的父进程（`wake_parent_waiting_for`）→ 回收僵尸进程（`reap_process`）→ 调用 `__switch` 进行上下文切换。进程回收通过 `try_waitpid` 遍历 `process.threads` 检查子进程状态，使用 `WaitPidAction::Return` 枚举返回退出码。

- **参考项目**采用**当前任务句柄 + 信号机制**：进程操作通过 `get_current_task().unwrap()` 获取当前任务句柄，然后直接操作 `task.inner.lock` 中的 `children` 链表。进程退出时调用 `exit_current_task` → 标记退出状态 → 触发 `panic!`。进程回收通过 `waitpid` 函数遍历 `tcb_inner.children.iter().enumerate()` 查找子进程，调用 `tcb_inner.children.remove()` 移除僵尸子进程，并通过 `handle_zombie_task` 发送 `send_signal` 信号通知父进程。

**核心差异**:
1. **同步原语**: 目标项目使用全局 `TASK_MANAGER.lock`（单一锁保护所有进程/线程表），参考项目使用 `task.inner.lock`（每个任务独立锁） + `cpu_local`（CPU 局部状态）。
2. **对象管理**: 目标项目通过 `processes: Vec<Option<Process>>` + `threads: Vec<Option<Thread>>` 集中管理，参考项目通过 `children: Vec<Arc<Task>>` 链表 + `ORIGIN_USER_PROC` 全局根进程管理。
3. **回收策略**: 目标项目使用 `reap_process` 主动回收 + `free_pids`/`free_tids` 栈复用 PID/TID，参考项目使用 `handle_zombie_task` 信号驱动回收 + `global_logoff_signals` 全局注销信号。

### 【进程管理 / 调度器核心算法】

**底层技术析理**:
目标项目实现了完整的 **Stride 调度算法**，参考项目使用简单的**先进先出/轮转调度**。

- **目标项目**的调度核心位于 `manager.rs` 中：
  - `pick_ready_process_by_stride`: 遍历 `self.processes`，调用 `self.process_has_ready_thread(pid)` 检查就绪线程，调用 `self.update_process_stride_by_alpha(pid)` 更新进程 stride 值，选择 `pass` 最小的进程。
  - `pick_ready_thread_in_process`: 在选中进程内，遍历 `process.ready_threads.clone()`，调用 `process.ready_threads.iter().position()` 查找就绪线程，调用 `process.ready_threads.remove()` 取出线程。
  - `update_process_stride_by_alpha`: 使用 `base_tickets.saturating_mul(factor).max(1)` 计算动态 tickets，调用 `crate::task::process::stride_from_tickets` 计算 stride 值。
  - `get_process_sched_stat`: 调用 `crate::math::sched_thread_scale` 计算缩放因子，调用 `manager.count_sched_runnable_threads_in_process(pid).max(1)` 统计可调度线程数。

- **参考项目**的调度核心位于 `cpu_local.rs` 中：
  - `run_tasks`: 循环调用 `cpu_local.current()` 获取当前任务，调用 `__switch` 切换上下文，无显式调度算法选择逻辑。
  - `push_task_to_scheduler`: 将新任务推入调度队列，无优先级或 stride 计算。

**核心差异**:
1. **调度算法**: 目标项目实现了完整的 Stride 调度（tickets、stride、pass、alpha 缩放因子），参考项目无显式调度算法，依赖 `__switch` 被动切换。
2. **数据结构**: 目标项目维护 `ready_threads: Vec<Tid>` 就绪队列 + `tickets: usize` + `stride: usize` + `pass: usize` 调度状态，参考项目无对应数据结构。
3. **调度统计**: 目标项目包含 `sched_thread_scale`、`count_sched_runnable_threads_in_process` 等调度统计函数，参考项目无对应实现。

## 6. 保守结论

本案基于客观 AST 调用图谱进行裁决，整体相似度仅 15.13%，352 个共同指纹主要分布在任务调度、内存管理、文件系统等边缘板块，且占比极低（目标覆盖率 29.65%，参考覆盖率 23.61%）。**进程管理板块共同唯一 AST 指纹为 0，双方覆盖率均为 0.0%**，不存在可被客观度量的结构重合。目标项目采用集中式 `TASK_MANAGER` + Stride 多线程调度架构，参考项目采用分布式 `get_current_task` + `cpu_local` + 信号机制架构，二者在同步原语、对象管理模型、调度算法、进程回收策略等核心维度上存在根本性分歧。同名 POSIX 系统调用函数仅反映接口规范合规性，底层 AST 调用链完全不同。跨板块 libc 测试用例、跨模块同名函数、Rust 语言惯用表达式等均已降权或驳回。综合判定：**复用指控不成立，目标项目原创性应予认定，整体同源风险为低**。
