### 4. 核心代码调用流对比剖析
* **【进程管理/进程等待机制】**
* **底层技术析理**: 目标项目 `scheduler::wait` 与参考项目 `process::wait` 虽同名且同板块，但底层架构完全不同。目标项目在调度器内部维护 `self.processes` 向量，通过索引直接操作子进程状态；参考项目则通过全局单例 `ProcessList::get()` 获取 RCU 保护的双向链表，再调用 `remove_process` 进行复杂的状态同步与资源回收。两者在数据容器、并发控制、对象生命周期管理上存在根本性设计差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/scheduler.rs#wait`
  - 参考项目: `src/kernel/task/process.rs#wait`

* **【内存管理/物理内存分配】**
* **底层技术析理**: 目标项目 `heap::alloc` 基于空闲链表 (`alloc_from_free_list`) 和 CAS 无锁并发；参考项目 `allocator::alloc` 则基于 SLAB 分配器 (`SLAB_ALLOCATOR.alloc`) 和页分配器 (`Page::alloc_at_least`)。两者在内核内存分配的核心算法、数据结构（链表 vs. 多级 slab）和并发策略上无任何重合。
* **代码溯源链接**:
  - 目标项目: `kernel/src/mm/heap.rs#alloc`
  - 参考项目: `src/kernel/mem/allocator.rs#alloc`

* **【系统调用/信号处理】**
* **底层技术析理**: 目标项目 `sys_rt_sigaction` 通过 `USER_PROCESSES.lock` 锁住全局进程表，再通过 `table.tasks[current_index].signal_actions` 操作任务私有信号表；参考项目 `rt_sigaction` 则无全局进程表概念，直接通过 `UserPointer`/`UserPointerMut` 与用户态内存交互信号结构体。两者在用户态数据拷贝、信号集存储位置、并发控制粒度上完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process/signal/sys.rs#sys_rt_sigaction`
  - 参考项目: `src/kernel/syscall/procops.rs#rt_sigaction`
