## 4. 核心代码调用流对比剖析

### 【进程管理 / POSIX 信号等待路径】

**底层技术析理**: `sigwait` 在 POSIX 标准中定义为“等待信号集中的任一信号”，其语义等价于以空超时调用 `sigtimedwait`。目标项目（RT-Thread libc）与参考项目（musl libc）均采用这一标准薄封装模式，但底层 `sigtimedwait` 的实现所依赖的内核对象截然不同：
- **目标项目**: `sigtimedwait` 最终调用 RT-Thread 的 `rt_thread_suspend`/`rt_schedule` 等内核原语，基于 RT-Thread 的线程控制块（TCB）和信号等待队列。
- **参考项目**: `sigtimedwait` 最终通过 `__syscall` 陷入 ChCore 内核，基于 capability group 的信号挂起机制和 futex 同步。

这种“接口同名、底层异构”的特征是独立开发的典型标志。

**代码溯源链接**:
- 目标项目: `rt-thread/components/libc/posix/signal/posix_signal.c#sigwait`
- 参考项目: `user/system-services/chcore-libc/musl-libc/src/signal/sigwait.c#sigwait`

---

### 【进程管理 / 线程屏障同步路径】

**底层技术析理**: `pthread_barrier_wait` 是 POSIX 线程屏障的核心接口。目标项目与参考项目的实现模型存在根本性差异：
- **目标项目**: 采用传统的 **条件变量 + 互斥锁** 模型。调用 `pthread_mutex_lock` → `pthread_cond_broadcast`/`pthread_cond_wait` → `pthread_mutex_unlock`，屏障计数器由用户态维护，同步依赖于 pthread 条件变量的调度唤醒。
- **参考项目 (musl)**: 采用 **futex + 原子操作** 模型。调用 `a_fetch_add`（原子递减）→ `__wake`/`__wait`（futex 系统调用），屏障计数器由原子变量维护，同步依赖于 Linux futex 的内核态等待队列。

两种模型在内存模型、同步策略、系统调用路径上完全不同，不可能通过简单复制代码实现。

**代码溯源链接**:
- 目标项目: `rt-thread/components/libc/posix/pthreads/pthread_barrier.c#pthread_barrier_wait`
- 参考项目: `user/system-services/chcore-libc/musl-libc/src/thread/pthread_barrier_wait.c#pthread_barrier_wait`

---

### 【进程管理 / 进程退出路径】

**底层技术析理**: `sys_exit_group` 在双方项目中同名，但处于完全不同的系统层级：
- **目标项目**: 位于 `components/lwp/lwp_syscall.c`，属于 **用户态轻量级进程（LWP）的系统调用封装层**。调用 `lwp_self()` 获取当前 LWP 句柄，然后调用 `lwp_exit()` 设置退出状态并触发 LWP 回收。整个流程在 RT-Thread 的 LWP 子系统内完成。
- **参考项目**: 位于 `kernel/object/recycle.c`，属于 **内核态 capability group 回收模块**。调用 `__exit_cap_group` 释放进程的所有 capability，然后调用 `sched`/`switch_context` 进行调度切换，最后通过 `eret_to_thread` 返回用户态。整个流程在 ChCore 微内核的 capability 管理框架内完成。

两者虽然函数名相同（均由 POSIX `exit_group` 语义衍生），但调用链、数据结构、资源回收模型完全不同。

**代码溯源链接**:
- 目标项目: `rt-thread/components/lwp/lwp_syscall.c#sys_exit_group`
- 参考项目: `kernel/object/recycle.c#sys_exit_group`

---

### 【第三方库/公共依赖 / lwIP TCP 接收路径】

**底层技术析理**: `tcp_receive` 是 lwIP 协议栈中 TCP 段接收处理的核心函数，调用 39+ 个 lwIP 内部宏/函数（如 `LWIP_ASSERT`、`TCP_SEQ_BETWEEN`、`pbuf_*`、`tcp_*`）。双方项目中该函数的 AST 指纹完全一致，原因是 **双方均从 lwIP 官方上游（savannah.nongnu.org/projects/lwip）集成了相同版本的源码**。目标项目路径 `lwip-2.1.2` 与参考项目路径 `lwip` 均指向 lwIP 官方发布版本。这种一致性属于“共同祖先”而非“项目间复制”。

**代码溯源链接**:

| `epoll_wait` / `waitpid` / `sigprocmask` / `sigwaitinfo` | POSIX 标准接口同名，但底层调用对象不同（`epoll_do_wait` vs `epoll_pwait`；`lwp_waitpid` vs `chcore_waitpid`；`rt_hw_interrupt_disable` vs `pthread_sigmask`）。属于**同名异构**。 |
| `tcp_debug_print_pcbs` | lwIP 调试辅助函数，仅调用 `LWIP_DEBUGF` 和 `tcp_debug_print_state`，属于**通用调试宏调用**，不体现独创逻辑。 |
| `printf` / `strtol` / `strtoul` / `strrchr` | libc 标准函数，双方实现路径不同（目标使用 RT-Thread 内核打印，参考使用 musl 文件流），属于**独立实现的公共接口**。 |

## 6. 保守结论

本案客观 AST 相似度仅为 8.82%，双向覆盖率均低于 18%，远未达到可认定代码复用的阈值。控方所列“铁证”中，超过 20 项来自 lwIP/PolarSSL/FatFS/libfdt 等第三方开源组件——这些组件的函数名、调用图谱、AST 结构由上游源码决定，双方因共同引用而产生相同指纹，属于“共同祖先”而非“项目间复制”。控方将网络协议栈函数强行归入“进程管理”板块，制造了“同板块底层逻辑一致”的假象；一旦按真实技术领域归类，网络协议栈模块共同指纹为 0。在双方均存在的核心内核模块（进程管理、内存管理、系统调用）中，目标项目底层调用 RT-Thread 原语（`rt_*`/`lwp_*`），参考项目调用 ChCore/musl 原语（`__syscall`/`chcore_*`），同步模型（条件变量 vs futex）、内存模型（PMO vs 堆分配器）、调度机制均存在根本性架构差异。综合判定：**双方项目为独立开发，不存在违规代码复用或抄袭行为**。
