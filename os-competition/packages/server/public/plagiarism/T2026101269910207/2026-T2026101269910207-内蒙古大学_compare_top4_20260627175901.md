## 4. 核心代码调用流对比剖析

### 4.1 【进程管理 / 线程退出路径】

- **底层技术析理**:  
  目标项目 `pthread_exit` 基于 RT-Thread 内核对象模型：通过 `rt_enter_critical`/`rt_exit_critical` 关中断保护临界区，调用 `rt_thread_detach` 将线程从调度器中分离，`rt_schedule` 触发上下文切换。参考项目 `pthread_exit` 基于自研任务模型：通过 `atomic_dec` 原子递减引用计数，`task_block` 将任务置为阻塞态，`sys_exit` 触发内核退出流程。**两者在同步原语（关中断 vs 原子操作）、资源回收策略（detach vs 引用计数）、调度入口上完全不同**。

- **代码溯源链接**:
  - 目标项目: `rt-thread/components/libc/posix/pthreads/pthread.c#pthread_exit`
  - 参考项目: `src/task/pthread.c#pthread_exit`

### 4.2 【进程管理 / 信号掩码操作】

- **底层技术析理**:  
  目标项目 `sigprocmask` 直接操作硬件中断状态（`rt_hw_interrupt_disable`/`rt_hw_interrupt_enable`），将信号屏蔽实现为中断屏蔽的简单封装。参考项目 `sigprocmask` 通过 `expblock`/`expmask`/`expunblock` 操作进程控制块中的信号位图，与中断状态无关。**两者在抽象层次（硬件级 vs PCB 级）和实现策略上截然不同**。

- **代码溯源链接**:
  - 目标项目: `rt-thread/components/libc/posix/signal/posix_signal.c#sigprocmask`
  - 参考项目: `libs/xlibc/unistd/signal.c#sigprocmask`

### 4.3 【进程管理 / 进程退出路径】

- **底层技术析理**:  
  目标项目 `sys_exit` 通过 `lwp_thread_exit` 终止 LWP（轻量级进程）线程，日志记录使用 `LOG_E` 宏。参考项目 `sys_exit` 涉及完整的进程资源回收链：`adopt_children_to_init`（子进程托孤）、`proc_close_other_threads`（关闭同进程其他线程）、`proc_release`（释放进程结构）、`task_exit_hook`（调用退出钩子）。**两者在退出语义（线程级 vs 进程级）和资源回收复杂度上存在数量级差异**。

- **代码溯源链接**:
  - 目标项目: `rt-thread/components/lwp/lwp_syscall.c#sys_exit`
  - 参考项目: `src/task/exit.c#sys_exit`

### 4.4 【内存管理 / TCP 数据接收路径】

- **底层技术析理**:  
  双方 `tcp_receive` 函数均为 lwIP 官方实现，调用 `LWIP_ASSERT`、`LWIP_DEBUGF`、`TCPH_FLAGS`、`pbuf_*` 系列等 lwIP 标准 API。**此路径属于第三方开源组件，不反映双方原创内存管理架构**。目标项目原创内存管理基于 `rt_malloc`/`rt_mempool`（小内存池 + 大内存分配器），参考项目基于 `kmalloc`/`kfree`（slab + 伙伴系统），两者在原创内存路径上无任何 AST 重合。

  - 参考项目: `sbin/netserv/lwip/core/tcp_in.c#tcp_receive`

### 4.5 【系统调用 / select 多路复用】

- **底层技术析理**:

本庭经对双方 671 个共同 AST 指纹的逐项审查，确认其绝大部分可追溯至 **lwIP、FatFS、PolarSSL 等第三方开源组件**，属于双方合法集成公共上游代码所产生的正常 AST 重合。控方指控的 7 项“同板块底层逻辑一致铁证”全部源于 lwIP 协议栈，且存在将网络协议栈函数错误归类至进程管理板块的硬伤——正确归类后，网络协议栈模块共同指纹为 0。双方在操作系统内核原创模块（进程调度、内存管理、中断处理、系统调用）上的 AST 重合度趋近于零，且同名标准接口（`pthread_exit`、`sigprocmask`、`sys_exit` 等）的内部调用图谱呈现显著架构差异（RT-Thread 内核对象模型 vs 自研任务模型）。客观 AST 相似度 4.31% 及双向覆盖率（目标 5.98%，参考 13.33%）均处于独立开发项目的正常波动范围。**综合判定：双方不存在代码抄袭或直接搬运关系，底层架构显著差异，属于独立开发。**
