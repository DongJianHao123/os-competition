* **底层技术析理**: 这是控方指控的核心，但 AST 证据揭示了根本性的架构差异。目标项目采用**显式状态机 + 独立僵尸队列**模型，通过 `mark_exit`、`mark_zombie` 等函数进行状态转换，并使用独立的 `records` 队列和裸指针 (`addr_of_mut!`) 管理僵尸进程。参考项目则将退出和调度合并为 `exit_current_and_run_next`，并直接在任务控制块的 `inner.children` 上通过 `inner_exclusive_access` 互斥锁进行同步和遍历。两者在**同步原语、数据容器和职责分离**上采取了完全不同的设计策略。
* **代码溯源**:
  - 目标项目: `src/task/mod.rs#exit_current` -> `src/task/mod.rs#push_zombie_record` -> `src/task/mod.rs#take_zombie_child_record`
  - 参考项目: `os/src/syscall/process.rs#sys_exit` -> `os/src/syscall/process.rs#sys_waitpid`

* **【内存管理 / 分配器实现】**
* **底层技术析理**: 双方均实现了 Rust 的 `GlobalAlloc` trait，因此 `alloc` 和 `dealloc` 函数必然同名且签名一致。然而，底层管理模型截然不同。目标项目实现了一个基于链表的堆分配器 (`heap_allocator`)，管理内核堆空间。参考项目实现了一个基于栈式回收和引用计数的物理帧分配器 (`frame_allocator`)，管理物理内存页。一个管理虚拟堆，一个管理物理帧，其**管理对象、数据结构和分配算法**均无同源性。
* **代码溯源**:
  - 目标项目: `src/mm/heap_allocator.rs#alloc`
  - 参考项目: `os/src/mm/frame_allocator.rs#alloc`

* **【设备驱动 / 块设备读写】**
* **底层技术析理**: 尽管 `read_block` 和 `write_block` 是块设备驱动的通用接口，但底层硬件交互完全不同。目标项目直接操作 VirtIO MMIO 或 PCI 设备的队列和描述符环 (`avail_idx`, `DATA_BUF`)。参考项目则通过 SD 卡协议 (`send_cmd`, `get_response`) 进行通信。两者在**硬件抽象层和驱动模型**上无任何重合。
* **代码溯源**:
