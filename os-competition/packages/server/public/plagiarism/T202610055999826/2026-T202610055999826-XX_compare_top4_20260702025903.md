#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 目标项目 `sys_exit` 直接委派给 `exit_current`，该函数操作 `exited_task` 的 `mark_exit`、`mark_zombie` 等状态转换方法，并通过 `parent` 字段访问父进程。参考项目 `sys_exit` 调用 `exit_current_and_run_next`，该函数在退出后立即触发调度。目标项目将退出与调度解耦，参考项目将两者耦合，控制流拓扑不同。
* **代码溯源链接**:
  - 目标项目: `src/syscall/task.rs#sys_exit`
  - 参考项目: `kernel/src/syscall/process.rs#sys_exit`

* **【内存管理/页表创建路径】**
* **底层技术析理**: 目标项目 `find_pte_create` 使用 `LoongArchPte::new_table` 创建LoongArch架构的页表项，通过 `Self::vpn_indexes` 获取多级索引。参考项目 `find_pte_create` 使用 `PageTableEntry::new` 创建RISC-V架构的页表项，通过 `idxs.iter()` 遍历。两者页表项结构、索引层级、权限位定义均因硬件架构不同而存在根本差异。
* **代码溯源链接**:
  - 目标项目: `src/arch/loongarch64/mm.rs#find_pte_create`
  - 参考项目: `kernel/src/mm/page_table.rs#find_pte_create`

* **【设备驱动/块设备读取路径】**
* **底层技术析理**: 目标项目 `read_block` 直接操作virtio MMIO寄存器，通过构造描述符链、写入设备队列、轮询状态位完成I/O。参考项目 `read_block` 通过 `self.0.lock()` 获取互斥锁后调用内部驱动对象的 `read_block` 方法，采用封装与同步模型。两者在I/O抽象层次、同步原语使用、错误处理路径上均不同。
* **代码溯源链接**:
  - 目标项目: `src/drivers/virtio_mmio.rs#read_block`
  - 参考项目: `kernel/src/drivers/block/virtio_blk.rs#read_block`

#### 5. 单方模块与降权项
- **文件系统/第三方库**: 目标项目独有 `src/fs/ext4.rs` 模块，包含 `read_file_range`、`create_file` 等45个函数，属于引入的第三方ext4库封装。参考项目无此模块，使用自研RunFS。此差异源于文件系统选型不同，不能作为同源证据。
- **跨语言/跨层证据降权**: 参考项目中 `oscomp/src/oscomp/` 路径下的 `test_waitpid`、`test_fork` 等C语言用户态测试程序，与目标项目的内核态Rust实现不在同一抽象层，已降权为环境背景信号。
- **POSIX同名系统调用降权**: `sys_getpid`、`sys_getppid`、`sys_exit` 等函数名由POSIX标准强制规定，任何兼容POSIX的内核都必须使用这些名称。同名本身不构成复用证据，其内部AST结构已证实完全不同。

#### 6. 保守结论
本案客观AST相似度仅1.6%，双方在进程管理、中断处理、系统调用、文件系统、设备驱动等核心板块的共同指纹均为0。内存管理板块的12个共同指纹源于堆分配、页表遍历等通用算法模式，且因硬件架构（LoongArch/RISC-V）不同导致页表结构、分配器策略存在根本差异。目标项目独有ext4文件系统库、双架构支持、直接寄存器级驱动等特征，与参考项目的自研RunFS、RISC-V单架构、SBI封装驱动模型形成鲜明架构分野。综合判定，两项目为独立开发，不存在代码复用或抄袭行为。
