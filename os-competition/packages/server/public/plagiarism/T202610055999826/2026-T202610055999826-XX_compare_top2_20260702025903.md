### 4. 核心代码调用流对比剖析
* **【进程管理 / exec 系统调用】**
* **底层技术析理**: 这是控方指控的核心。参考项目的 `exec` 函数是一个单体函数，直接调用 `MemorySet::from_elf` 和 `TrapContext::app_init_context` 完成全部加载与上下文初始化工作。目标项目则将其拆分为 `exec_user_app` 和 `exec_elf_bytes` 两个函数，前者负责在内置应用表中查找，后者负责加载 ELF 并调用 `install_user_image`。这不仅是函数拆分，更反映了架构差异：目标项目有独立的“内置应用”加载路径 (`USER_APPS`, `find_user_app`)，而参考项目则统一通过文件系统 inode 加载。两者在内存模型（`install_user_image` vs `MemorySet`）和上下文初始化策略上完全不同，不存在 AST 级别的复用。
* **代码溯源链接**:
  - 目标项目: `src/task/task.rs#exec_user_app` -> `src/task/task.rs#exec_elf_bytes`
  - 参考项目: `os/src/task/task.rs#exec`

* **【进程管理 / 僵尸进程回收机制】**
* **底层技术析理**: 目标项目在内核态 `task/mod.rs` 中实现了一套完整的僵尸进程管理机制，包括 `push_zombie_record`、`take_zombie_child_record`、`has_zombie_child` 等函数，直接操作原生指针队列 (`*records`)。参考项目则无此内核态实现，其进程等待逻辑（如 `wait`）在用户态的 `initproc.rs` 中通过系统调用完成。这是两种完全不同的进程生命周期管理模型，目标项目的设计更为复杂和底层，不存在复用。

* **底层技术析理**: 双方都有 `alloc` 和 `dealloc` 函数，但目标项目实现的是内核**堆分配器** (`heap_allocator.rs`)，管理内核动态内存，使用链表和 `header_ptr` 操作。参考项目实现的是**物理帧分配器** (`frame_allocator.rs`)，管理物理内存页，使用 `recycled` 栈进行回收。两者管理的资源类型、数据结构和算法完全不同，属于内存管理子系统的不同层面，无同源性。
