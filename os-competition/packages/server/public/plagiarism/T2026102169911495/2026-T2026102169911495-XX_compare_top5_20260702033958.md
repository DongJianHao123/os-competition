#### 4. 核心代码调用流对比剖析
* **【进程管理 / `do_exit` 退出路径】**
* **底层技术析理**: 两者均处理进程退出，但内存模型与资源清理路径截然不同。目标项目直接操作虚拟内存（`clear_child_tid.vm_write(0)`）并主动清理共享内存（`SHM_MANAGER.lock().clear_proc_shm`），体现了独立内核的内存管理能力。参考项目则通过 `UserPtr` 抽象（`clear_child_tid.get_as_mut`）访问用户态内存，并调用 `axtask::exit` 将最终清理委托给调度器，体现了微内核/Unikernel 风格的委托机制。两者在同步策略上也有差异：目标项目使用 `FutexKey::new_current` 处理 robust futex，参考项目则直接清空文件表（`FD_TABLE.clear`）。
* **代码溯源链接**:

  - 参考项目: `api/src/imp/task/exit.rs#do_exit`

* **【进程管理 / `sys_execve` 程序加载】**
* **底层技术析理**: 两者均实现 `execve` 系统调用，但用户态数据读取方式完全不同。目标项目使用 `vm_load_until_nul` 直接从用户态虚拟内存中逐字节拷贝并以 null 结尾解析，属于内核态内存操作。参考项目使用 `argv.get_as_null_terminated()?.iter().map`，依赖 `UserPtr` 抽象和 `get_as_*` 系列方法，将用户态指针封装为高级对象进行操作。这反映了两种完全不同的用户态交互模型。

* **底层技术析理**: 两者均实现 `mmap`，但后端存储抽象层不同。目标项目使用 `Backend::new_alloc`、`Backend::new_file` 等统一后端枚举来管理物理内存、文件和匿名页，体现了清晰的策略模式。参考项目则通过 `MmapFlags::from_bits_truncate` 和 `MmapProt::from_bits_truncate` 解析标志，并直接操作 `VirtAddr`，其内存管理逻辑更贴近底层地址操作。两者在对象模型上存在本质区别。
