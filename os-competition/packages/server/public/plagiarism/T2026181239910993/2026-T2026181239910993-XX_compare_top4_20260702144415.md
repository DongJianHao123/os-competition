#### 4. 核心代码调用流对比剖析

* **【进程管理/任务调度】**
* **底层技术析理**: 双方在获取当前任务的核心调度器原语上存在根本性架构差异。目标项目采用全局 `PROCESSORS` 数组，通过 CPU ID 索引并加锁获取当前任务指针，体现了基于静态数组的多核管理模型。参考项目则通过 `get_current_processor()` 获取当前 CPU 的单例对象，再访问其 `current_task` 字段，体现了基于处理器对象的多核抽象。两者在内存模型、同步策略和对象管理上完全不同。
* **代码溯源链接**:

  - 参考项目: `pantheon/kernel/src/task/processor.rs#current_task`

* **【进程管理/系统调用】**
* **底层技术析理**: 以 `sys_getppid` 为例，虽然高层语义相同，但底层调用流揭示了不同的进程模型。目标项目通过 `current_process` 获取进程结构体，访问其 `inner.parent` 字段，并使用 `Weak::upgrade` 和 `and_then` 进行安全的链式错误处理。参考项目则通过 `current_task.pcb()` 访问任务控制块，直接对 `parent` 字段进行 `clone` 和多次 `unwrap` 操作。这反映了双方在进程-线程关系、错误处理哲学和对象封装粒度上的显著差异。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_getppid`
  - 参考项目: `pantheon/kernel/src/syscall/impls/process.rs#sys_getppid`

* **底层技术析理**: 在 `sys_mmap` 实现中，目标项目将文件类型判断（`InodeMode::FILE`, `BLOCK`, `CHAR`）与内存映射逻辑耦合，并使用 `MapPermission::from_prot` 进行权限转换。参考项目则使用独立的 `MmapFlags` 和 `MmapProts` 进行参数解析，逻辑分离更清晰。两者在内核内部抽象（Inode vs MmapFlags）和模块职责划分上存在明显差异。
