#### 4. 核心代码调用流对比剖析

* **【进程管理 / 进程调度核心】**
* **底层技术析理**: 目标项目采用 `PROCESSORS[id].as_mut().unwrap().lock()` 的 per-CPU 静态数组架构，通过 `get_tp()` 获取当前 CPU ID 后索引本地处理器结构体。参考项目采用 `get_proc_by_hartid(hart_id())` 的函数查询架构，通过 hart ID 动态获取处理器。两者在多核拓扑管理、处理器状态封装、上下文切换路径上存在根本性设计差异。目标项目的 Task 与 Process 解耦（`current_task().and_then(|t| t.process.upgrade())`），参考项目为紧耦合（`current_task().unwrap().pid`）。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

* **【内存管理 / ELF 加载】**
* **底层技术析理**: 目标项目在 `from_elf` 中使用 `GLOBAL_DCACHE` 全局缓存和 `KERNEL_VMSET` 内核虚拟内存集，支持动态链接器基址计算（`interp_base + ph.virtual_addr()`）。参考项目使用 `MapArea::new` 和 `Self::new_from_kernel` 的简单映射模型。目标项目的内存抽象层次更高，支持更复杂的地址空间布局。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/vm_set.rs#from_elf`
  - 参考项目: `os/src/mm/memory_set.rs#from_elf`

* **底层技术析理**: 目标项目 `syscall` 分发器覆盖 193+ 系统调用号，包含完整的网络协议栈调用（`sys_accept`、`sys_bind`、`sys_sendto` 等）。参考项目分发 107+ 系统调用，无网络相关调用。目标项目使用 `polyhal::instruction::shutdown` 跨架构关机，参考项目直接调用 `shutdown`。系统调用能力集和底层实现路径均不同。

**降权项（不构成强同源证据）**:

#### 6. 保守结论

本案客观 AST 相似度为 13.42%，但其中 84.84% 的共同指纹来自双方独立引用的第三方开源库 `lwext4_rust`，剔除后内核层面真实重合度不足 2.5%。控方所列“铁证”集中于用户态公开测试程序 `forktree.rs` 和一行式 Rust 标准方法调用，这些均属于公共教学资源或语言惯用法，不具备独创性，不能作为抄袭的有效证据。双方在进程调度架构（per-CPU 数组 vs 函数查询）、进程模型（Task-Process 解耦 vs 紧耦合）、内存管理（显式 TLB 操作 vs growproc 封装）、文件系统实现深度（完整 VFS vs 空实现）以及网络协议栈存在性等核心维度上呈现系统性差异。目标项目独有的网络协议栈和多架构支持能力进一步佐证其独立开发性质。综合判定：**无充分证据支持代码抄袭指控，目标项目应认定为独立开发。**
