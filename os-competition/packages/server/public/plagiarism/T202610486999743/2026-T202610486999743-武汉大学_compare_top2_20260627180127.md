### 4. 核心代码调用流对比剖析
* **【进程管理/调度核心】**
* **底层技术析理**: 双方 `run_tasks` 函数体现了根本性的架构分歧。目标项目构建了一个基于 `Future` 的协程调度循环，通过 `CoroutineUnit::new` 创建协程，`Pin::new` 固定内存，并调用 `poll` 驱动执行，这是典型的异步运行时模型。参考项目则采用传统的多道程序调度，在一个循环中通过 `hart_id` 获取当前硬件线程的处理器结构，检查 `futex_timer`，然后切换到下一个任务。两者在调度实体、状态管理和驱动模型上完全不同。
* **代码溯源链接**:
  - 目标项目: `os/src/processor/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

* **【进程管理/进程克隆】**
* **底层技术析理**: `sys_clone` 的实现展示了不同的资源隔离策略。目标项目显式地调用 `clone_mount_namespace` 和 `cloned_net_namespace_for`，表明其实现了更细粒度的命名空间隔离机制。参考项目则通过 `FdTable::from_another`、`FsInfo::from_another` 等 `from_another` 模式复制进程资源，这是一种更传统的资源复制方式。两者对“克隆”这一语义的工程实现路径选择不同。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_clone`
  - 参考项目: `os/src/syscall/process/clone.rs#sys_clone`

* **【内存管理/用户态数据交互】**
* **底层技术析理**: 在内核与用户态数据拷贝上，双方采用了不同的内存管理抽象。目标项目统一使用 `current_mm_activation().user_token` 获取当前内存空间的访问令牌，再配合 `write_user_value` 或 `translated_refmut` 进行数据写入。参考项目则直接通过 `process.get_locked_memory_set` 获取内存集，然后调用 `memory_set.put_data` 写入。前者引入了一层“内存激活”抽象，后者则更直接地操作内存集对象。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_sysinfo`
  - 参考项目: `os/src/syscall/process/others.rs#sys_sysinfo`

### 5. 单方模块与降权项
- **单方模块**: **网络协议栈** (`os/src/net/mod.rs`) 仅存在于目标项目中，包含 `disconnect`、`route_request` 等完整网络功能，而参考项目无此模块。这证明目标项目具备独立构建复杂子系统的能力。
- **公共接口与惯用法**: 所有同名系统调用（如 `sys_getpid`、`sys_exit`）均由 POSIX/Linux ABI 规范强制定义，函数名相同是行业标准。`Option::as_ref().map()`、`Option::take()` 等是 Rust 语言处理 `Option` 类型的标准惯用法，任何 Rust 开发者都会写出相同的 AST 节点。这些不能作为同源性证据。
- **第三方库**: `文件系统/第三方库` 模块中 501 个共同 AST 指纹均来自 `lwext4` 开源库，是双方共同依赖的第三方代码，与内核原创性无关，在抄袭判定中必须予以排除。

### 6. 保守结论
本次审计的整体 AST 相似度为 14.15%，处于极低水平。在排除第三方开源库 `lwext4` 的重合后，双方在操作系统内核的核心原创模块（进程管理、任务调度、内存管理、文件系统等）上几乎不存在共同 AST 指纹。尽管双方实现了相同的 POSIX 系统调用接口，但底层架构选择、内存管理模型、调度器设计和资源隔离策略均存在显著差异。特别是目标项目独有的协程调度器和网络协议栈，进一步证明了其独立架构设计能力。因此，综合所有证据，判定两个项目为独立开发，不存在同源性抄袭行为。
