## 4. 核心代码调用流对比剖析

### 【进程管理 / `sys_getppid` 系统调用路径】

**底层技术析理**：

目标项目与参考项目在获取父进程 ID 的系统调用中，虽然最终都需通过 `Weak::upgrade` 升级父进程弱引用指针，但两者的**错误处理策略、中间抽象层、性能监控机制**存在根本差异：

- **参考项目**采用直接式 `ok_or` → `unwrap` 路径：`inner.parent.as_ref().ok_or(...)` → `parent.upgrade().unwrap()`，遇到空父进程时通过 `Err` 返回错误码，逻辑线性且无额外埋点。
- **目标项目**采用函数式 `and_then` 组合子路径，并插入三个自定义性能探针函数 `getppid_prof_add` / `getppid_prof_delta` / `getppid_prof_now`，形成 `inner.parent.as_ref().and_then(|p| p.upgrade())` 的链式处理。这表明目标项目实现了独立的性能剖析子系统，且错误处理采用 `Option` 单子而非 `Result` 双子。

两者的 AST 调用图谱在 `inner.parent.as_ref` 之后即分叉为 `ok_or` vs `and_then`，属于不同的控制流抽象选择。

## 6. 保守结论
