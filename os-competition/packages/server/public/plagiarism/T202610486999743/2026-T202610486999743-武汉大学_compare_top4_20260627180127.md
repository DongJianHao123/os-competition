#### 4. 核心代码调用流对比剖析

* **【进程管理 / `sys_wait4` 系统调用】**
* **底层技术析理**: 两者均实现了 POSIX `wait4` 语义，但底层进程检索与阻塞机制存在根本性架构差异。参考项目采用全局进程数组 `proc_find` 查找子进程，并通过 `yield` 主动放弃 CPU 实现忙等或协作式阻塞。目标项目则基于 Rust 的所有权与内部可变性模型，通过 `Arc::strong_count` 管理任务生命周期，使用 `child.inner_exclusive_access` 安全获取子进程内部状态，并通过 `current_task_has_interrupting_signal` 检查信号队列来决定是否返回 `-EINTR`，而非直接调用调度让步。两者在内存安全模型、并发控制原语及阻塞语义实现上无共同 AST 路径。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_wait4`
  - 参考项目: `src/core/syscall.c#sys_wait4`

* **【任务调度/异步运行时 / `sys_setitimer` 系统调用】**
* **底层技术析理**: 该调用涉及定时器设置，两者处理管线截然不同。参考项目使用 `argraw` 从 Trap 帧直接读取参数，依赖 `assert` 做粗粒度断言，并以 `get_tick_by_ms` 获取毫秒级 Tick 作为时间基准。目标项目则使用 `copy_from_user_checked` 结合 `size_of::<ITimerVal>` 进行类型安全的用户态数据拷贝与大小校验，时间基准采用 `clock_gettime_us` 微秒级高精度时钟，并通过 `current_mm_activation().user_token` 操作地址空间。两者在用户态交互安全策略、时间精度及地址空间操作抽象上完全不同。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/time.rs#sys_setitimer`

* **底层技术析理**: 信号注册的入口逻辑体现了语言级安全模型的差异。参考项目为 C 风格，通过 `argraw` 获取参数后直接 `assert`。目标项目为 Rust 风格，通过 `action.is_null` 进行空指针检查，并通过 `core::mem::size_of::<SigSet>` 校验用户传入的 `sigsetsize` 参数是否符合 ABI 预期，随后通过 `current_task().unwrap` 获取任务上下文。目标项目的校验是 Rust 类型系统与 Linux ABI 合规性的体现，而非对参考项目 `assert` 的翻译。

  - 参考项目: `src/core/syscall.c#sys_rt_sigaction`
