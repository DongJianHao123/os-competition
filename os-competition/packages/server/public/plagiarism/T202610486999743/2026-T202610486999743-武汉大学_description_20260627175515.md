## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心文件包括 `os/src/mm/memory_set.rs`、`os/src/mm/frame_allocator.rs`、`os/src/trap/mod.rs`。

- 进程地址空间通过 `memory_set.rs` 管理，支持用户栈初始化（`os/src/task/task.rs#init_user_stack_with_args_env`）。
- 物理页框分配器支持常规页框以及 1G/2M 大页回收缓存（`os/src/mm/frame_allocator.rs#huge_1g_dealloc`、`huge_2m_dealloc`）。
- 页表节点使用引用计数回收（`os/src/mm/frame_allocator.rs#pt_node_dealloc`）。
- 用户态/内核态拷贝通过 `user_token` + `copy_between_tokens` 完成（`os/src/syscall/mm.rs#sys_process_vm_readv`）。
- 异常路径 `os/src/trap/mod.rs#trap_handler` 处理 TLB 刷新、缺页地址、信号检查等。

**未体现**：swap、页替换算法、NUMA 内存策略的完整实现底层证据未体现。

### PROC（进程管理/任务调度）

传统进程管理证据较充分。核心文件 `os/src/syscall/process.rs`、`os/src/mission/mod.rs`、`os/src/processor/processor.rs`。

- 支持 `clone`（`os/src/syscall/process.rs#sys_clone`）、`spawn`（`os/src/syscall/process.rs#sys_spawn`）、`exec`（`do_exec_common`）、`waitid/wait4`。
- 支持 `exit_group`（`os/src/mission/mod.rs#request_exit_group`）与 fd/futex 清理。
- 支持 futex（`sys_futex`）、robust list、epoll、pidfd、namespace（mount/net/ipc/time）、capability、credential、rlimit、priority、sched affinity。
- 调度器采用协程/异步运行时：`os/src/processor/processor.rs#run_tasks` 创建 `CoroutineUnit` 并轮询；`os/src/processor/coroutine.rs#poll` 使用 waker 唤醒；存在本地队列、任务迁移（`migrate_one_task`）、负载跟踪（`update_load_sum`）。

**未体现**：传统 CFS/O(1) 调度器的红黑树/vruntime 证据未体现；调度器身份是“基于协程的异步运行时”。

### FS（文件系统）

文件系统是本项目最强模块。核心文件 `os/src/syscall/fs.rs`、`os/src/fs/stdio.rs`、`third_party/ext4fs/c/lwext4`。

- 系统调用层实现 `openat/openat2/open_tree/fsopen/mount/read/write/pread64/pwrite64/fallocate/unlinkat/mkdirat/mknodat/utimensat/fcntl/ioctl` 等。
- 路径解析支持 mount namespace、传播属性（`apply_mount_propagation_change`）、BusyBox applet fallback。
- 虚拟文件系统支持 `/proc`（`ProcKpageflagsFile`、`ProcPagemapFile`、`ProcNetTagFile`）、`/sys/cgroup`（`CgroupFile`）、`memfd`、`eventfd`、`timerfd` 等。
- 实际存储后端通过第三方 C 库 `lwext4` 实现 ext4（`third_party/ext4fs/c/lwext4/src/ext4.c` 等）。
- 块设备层通过 `os/src/drivers/block/virtio_blk.rs` 读写。

### DEV（设备驱动）

底层证据较弱。仅见：

- `os/src/drivers/block/virtio_blk.rs#read_blocks/write_blocks/unshare`：VirtIO blk 块读写与共享缓冲区管理。
- `hal/src/component/virtio/loongarch64.rs#pci_assign_bars`：LoongArch64 下 PCI BAR 分配。
- `os/src/drivers/block/mod.rs`：通用块层批量读写封装。

**未体现**：除块设备外，字符设备、显示、输入、网络设备、时钟、中断控制器、DMA 等驱动底层证据未体现。

### SYSCALL（系统调用）

底层证据强。`os/src/syscall/mod.rs#syscall` 规模约 28180，调用数 252，分发到：

- `fs.rs`：文件系统调用
- `process.rs`：进程/命名空间/能力/模块调用
- `mm.rs`：进程间内存拷贝
- `net.rs`：socket 调用
- `time.rs`：时间/定时器
- `signal.rs`：信号
- `ipc.rs`：System V IPC
- `key.rs`：keyring
- `bpf.rs`：eBPF 程序加载与校验（`run_program`、`verify_program`）

### TRAP/IRQ（中断异常）

证据主要存在于 HAL 层与顶层 trap handler：

- `hal/src/component/trap/loongarch64.rs#init` 配置异常入口与 TLB refill 入口。
- `hal/src/component/trap/loongarch64.rs#la_tlb_refill_failed` 处理 TLB refill 失败。
- `hal/src/component/irq/loongarch64.rs#enable_timer_interrupt/disable_timer_interrupt` 控制定时器中断。
- `hal/src/component/trap/riscv64.rs#user_status_init_bits` 初始化 RISC-V 用户态状态位。
- `os/src/trap/mod.rs#trap_handler` 处理用户态异常、timer、缺页、信号投递。
- `os/src/trap/mod.rs#trap_from_kernel_handler` 处理内核态异常。

**未体现**：完整中断控制器（PIC/PLIC/IOAPIC）、多核 IPI、设备中断路由底层证据未体现。

### NET（网络协议栈）

底层证据较弱。核心文件 `os/src/net/mod.rs`（105 个函数）：

- 维护 unix domain socket 与 inet socket 的 `stream_map`、`udp_map`、`listeners`、`bound` 表。
- 支持 `bind_unix`、`unregister`、`disconnect`、`recv_datagram`、`try_push_from_peer_vec`。
- 支持 netlink 路由请求构造（`route_request`、`build_netlink_link_dump`、`build_netlink_addr_dump`）。

**未体现**：完整 TCP/IP 协议栈（连接管理、拥塞控制、IP 分片、ARP/NDP、路由表查找）底层证据未体现；现有证据更接近 socket 描述符兼容层与 netlink 消息封装。

## 第六部分：边界与未体现内容

### 1. 第三方库边界
- `third_party/ext4fs/c/lwext4` 是独立的 C 语言 ext4 实现，本项目通过封装/FFI 方式集成。该部分不应被夸大为本项目的原创内核主体。

### 2. 没有底层证据支撑的模块或能力
- **完整 TCP/IP 协议栈**：现有 56 个网络函数主要集中在 socket 描述符表、unix domain socket、netlink 消息构造，TCP/UDP/IP 协议状态机、拥塞控制、ARP/NDP、路由查找等底层证据未体现。
- **除 VirtIO-blk 外的设备驱动**：显示、输入、网络设备、字符设备、时钟、中断控制器、DMA 等驱动底层证据未体现。
- **完整中断控制器与多核 IPI**：仅有 timer/TLB refill/trap handler 证据，中断路由、IPI、PLIC/IOAPIC 等底层证据未体现。
- **swap 与页替换策略**：内存管理证据集中在页表、页框分配、大页回收，swap、LRU/Clock 等页替换策略底层证据未体现。
- **虚拟化、电源管理、安全模块（除 capability/key 外）、图形/声音等**：底层证据未体现。

### 3. 方向性判断说明
- 本作品“可能的系统方向”为面向 RISC-V64/LoongArch64 的通用类 Unix 内核，重点在文件系统兼容层、系统调用兼容层、协程调度与内存管理。
- 上述判断仅基于 AST Call Map、函数名、模块路径与调用链，属于技术方向推断，**不是最终原创性、完整性或合规性结论**。
- 不做抄袭/违规判断。
