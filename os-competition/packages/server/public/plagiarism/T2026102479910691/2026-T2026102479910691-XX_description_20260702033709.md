## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心函数包括：

- `kernel/mm.c#kalloc_page` / `kernel/mm.c#kalloc_zero`：物理页分配，后者对页面清零。
- `kernel/mm.c#walk_pgtbl`：页表遍历，缺失中间页表时会调用 `kalloc_zero` 分配。
- `kernel/mm.c#uvm_create` / `uvm_map` / `uvm_alloc` / `uvm_free`：用户虚拟地址空间创建、映射、分配与释放。
- `kernel/main.c#run_test`、`kernel/proc.c#proc_ready`、`kernel/elf.c#elf_load` 均调用 `uvm_map`/`kalloc_zero` 建立用户页表。

### PROC（进程管理/任务调度）

底层证据较弱。可确认的内容：

- `kernel/proc.c#proc_init`、`proc_alloc`、`proc_ready`、`proc_free`：进程结构初始化、分配、地址空间准备与释放。
- `kernel/proc.c#fd_alloc`、`fd_get`、`fd_close`：文件描述符表操作。
- `kernel/syscall.c#sys_getpid`、`sys_getppid`、`sys_exit` 存在，但 `sys_exit` 仅调用 `uart_puts`。
- `sys_clone`、`sys_wait4`、`sys_execve` 在画像中出现，但调用链为“内部闭环逻辑”，**未体现具体实现**。

**未体现**：fork/exec/wait/zombie、PCB 父子链表、调度器、上下文切换、多任务抢占。

### FS（文件系统）

可确认的是**只读 EXT4 解析**与基于 fd 表的系统调用接口：

- `kernel/ext4.c#ext4_init`、`ext4_open`、`ext4_list_root`、`ext4_scan_testcode`、`read_inode`、`read_extent`、`read_block`：通过 `virtio_read` 读取 EXT4 超级块、inode、extent 数据。
- `kernel/syscall.c#sys_openat`、`sys_read`、`sys_close`、`sys_dup3_impl`、`sys_fstat`、`sys_getdents64`：用户态文件接口。
- `kernel/syscall.c#syscall_register_file`：syscall 层面的文件注册入口。

**未体现**：VFS 抽象层、文件写入、创建、删除（`sys_unlinkat` 为内部闭环逻辑，无底层证据）、缓存/缓冲区管理。

### DEV（设备驱动）

证据较强：

- **UART**：`kernel/uart.c#uart_init`、`uart_putc`、`uart_puts`，用于内核早期打印与 `sys_write` 输出。
- **virtio-blk**：`kernel/virtio.c#virtio_init`、`virtio_read`、`virtio_open`、`blk_dev_ready`、`alloc_desc`、`free_chain`，通过 MMIO 寄存器 `R`/`W` 与描述符环操作块设备。

### SYSCALL（系统调用）

证据较强：

- `kernel/syscall.c#syscall_handle`：读取系统调用号后通过 `syscall_table[nr]` 分派。
- `kernel/syscall.c#syscall_init_table`：初始化系统调用表。
- 大量 `sys_*` 处理函数：`sys_read`、`sys_write`、`sys_openat`、`sys_mmap`、`sys_brk`、`sys_exit`、`sys_getcwd`、`sys_uname` 等。
- 字符串辅助函数 `str_eq`、`str_len`、`str_cpy` 也归入系统调用/工具层。

### TRAP/IRQ（中断异常）

证据中等：

- `kernel/trap.c#trap_init`：设置 RISC-V `stvec`。
- `kernel/trap.c#trap_handler`：读取 `scause`，对系统调用异常调用 `syscall_handle`。
- `kernel/main.c#launch_user`、`kernel/proc.c#proc_switch_to_user`：涉及返回用户态路径 `user_trap_ret`。
- `r_scause`、`r_sstatus`、`w_sstatus`、`w_sie`、`w_stvec` 等 CSR 读写封装存在，但多为内部闭环或简单封装。

**未体现**：定时器中断、外部设备中断、多核核间中断、完整 IRQ 处理框架。

### NET（网络协议栈）

**底层证据未体现**。未出现任何网络设备驱动、协议栈（TCP/IP、socket 接口）相关函数或文件。
