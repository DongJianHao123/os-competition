### 4. 核心代码调用流对比剖析
* **【内存管理/页表映射】**

  - 参考项目: `proc.c#fork` -> `vm.c#mappages` -> `vm.c#walkaddr`

* **【设备驱动/Virtio 块设备】**
* **底层技术析理**: 这是控方指控的核心，但双方在同步模型上存在根本性差异。目标项目的 `virtio_read` 是一个简化的、无锁的轮询模型，调用流为 `R` -> `alloc_desc` -> `free_chain`。参考项目的 `virtio_disk_rw` 是一个完整的多线程、带锁的读写驱动，调用流包含 `R`, `__sync_synchronize`, `acquire`, `alloc3_desc`, `free_chain`, `release`, `sleep`。目标项目剥离了所有同步原语，这不仅是“删除”，更代表了从多核安全到单核轮询的**架构降级**，两者不是同一实现。
* **代码溯源链接**:
  - 目标项目: `kernel/virtio.c#virtio_read`
  - 参考项目: `virtio_disk.c#virtio_disk_rw`

* **底层技术析理**: 双方文件系统实现完全不同。目标项目包含一个独立的 ext4 子系统，调用流为 `ext4_init` -> `read_extent` -> `read_inode` -> `read_block`。参考项目则基于 xv6 的传统 inode/bcache 模型，调用流为 `ilock` -> `bread` -> `brelse`。这是两个完全不同的文件系统架构，不存在同源性。

本庭基于客观 AST 数据进行裁决。目标项目与参考项目的整体相似度仅为 0.83%，所有核心模块的共同唯一 AST 指纹均为 0。这意味着，在排除公共接口命名后，双方在代码结构层面没有表现出任何实质性的同源性。控方提出的“函数重命名”、“内联伪装”等指控，未能得到 AST 结构指纹的支持，其证据主要建立在领域公共名词和跨模块联想之上。相反，目标项目在文件系统（ext4）、内存管理 API（`kalloc_page`/`kalloc_zero`）、日志子系统（`uart_*`）以及 Virtio 驱动的同步模型上，均展现出与参考项目（xv6 风格）截然不同的独立架构选择。因此，本庭认定，**无证据表明目标项目对参考项目存在代码复用，应判定为独立开发。**
