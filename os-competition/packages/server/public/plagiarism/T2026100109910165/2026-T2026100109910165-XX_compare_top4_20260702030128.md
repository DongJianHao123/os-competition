#### 4. 核心代码调用流对比剖析
* **【进程管理/进程等待与回收】**
* **底层技术析理**: 两者均实现 POSIX `wait4` 语义，但底层对象管理模型截然不同。目标项目采用自研的 `proc_find_waitable_zombie` 函数，通过 `proc_is_child_of` 进行父子关系判定，回收路径为 `proc_reap_child` -> `write_user32`。参考项目则基于 BSD 风格的 `LIST_FOREACH` 宏遍历进程链表，通过 `wait` 函数集中处理，回收路径为 `wait_exit` -> `copy_out`。目标项目无任何 `LIST_*`/`TAILQ_*` 宏调用，同步原语亦非 `mtx_lock` 体系。
* **代码溯源链接**:
  - 目标项目: `arch/riscv/trap.c#sys_wait4`
  - 参考项目: `kern/syscall/sys_proc.c#sys_wait4`

* **【任务调度/futex 同步机制】**
* **底层技术析理**: 两者均实现 `futex` 系统调用，但内部拆分与数据结构完全不同。目标项目将唤醒操作拆分为 `futex_wake_waiters` 和 `futex_requeue_waiters`，核心匹配逻辑依赖 `futex_bitset_matches` 和 `proc_table_base`。参考项目则统一通过 `futex_wake`/`futex_wait`/`futex_requeue` 处理，内部使用 `TAILQ_*` 队列管理等待者。目标项目的 `futex_bitset_matches` 在参考项目中无对应 AST 节点。
* **代码溯源链接**:
  - 目标项目: `arch/riscv/trap.c#sys_futex`
  - 参考项目: `kern/syscall/sys_futex.c#sys_futex`

* **【内存管理/动态内存分配】**
* **底层技术析理**: 两者均提供 `kmalloc`/`kfree` 接口，但堆管理策略异构。目标项目 `kmalloc` 调用 `align_size` 进行对齐分配，`kfree` 调用 `heap_insert_block` 回收。参考项目 `kmalloc` 调用 `extend_heap` 扩展堆空间，并使用 `mtx_lock`/`mtx_unlock` 进行同步保护，`kfree` 调用 `list_insert_head` 回收。目标项目未出现 `mtx_*` 锁调用，同步机制不同。
* **代码溯源链接**:
  - 目标项目: `kernel/mm.c#kmalloc`
