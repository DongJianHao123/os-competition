## 4. 核心代码调用流对比剖析

### 【进程管理 / sys_wait4 系统调用路径】

* **底层技术析理**: 
  - **目标项目**: 实现完整 POSIX `wait4` 语义。调用链 `proc_find_waitable_zombie → proc_reap_child → wait_to_kernel → write_user32` 体现：① 专门的僵尸进程查找器（非通用进程查找）；② 显式的子进程回收操作；③ 内核态等待原语；④ 用户态 32 位返回值写回。进程模型支持僵尸状态、父子回收关系、等待队列。
  - **参考项目**: 实现教学内核简化版。调用链 `argraw → memset → my_proc → proc_find → yield` 体现：① 通用参数解析器；② 内存清零初始化；③ 当前进程获取；④ 通用进程查找（非僵尸专用）；⑤ 主动让出 CPU（无等待队列）。进程模型为简单的运行/就绪二态模型。
  - **异构性**: 目标项目具备僵尸进程状态机、内核等待原语、用户态写回机制；参考项目无僵尸状态、无等待原语、通过 yield 主动让出。两者在进程状态模型、同步原语、用户态交互机制上存在本质差异。

* **代码溯源链接**:
  - 目标项目: `arch/riscv/trap.c#sys_wait4`
  - 参考项目: `src/core/syscall.c#sys_wait4`

---

### 【任务调度/异步运行时 / kmain 内核入口】

* **底层技术析理**:
  - **目标项目**: 调用 `devfs_init → ext4_run_test_scripts → mm_init → mount_procfs → mount_root_filesystem → platform_paging_init → platform_shutdown → proc_system_init`。体现：① 设备文件系统（devfs）独立初始化；② ext4 测试脚本运行（内置测试框架）；③ procfs 挂载（进程信息文件系统）；④ 多架构平台分页初始化（`platform_paging_init`）；⑤ 显式关机路径（`platform_shutdown`）。
  - **参考项目**: 调用 `cpuinfo_init → disk_init → dmw_init → fs_init → kinit → pmm_init → proc_init`。体现：① CPU 信息展示初始化；② 磁盘驱动初始化；③ DMW（直接映射窗口）初始化（LoongArch 特有）；④ 简单 FS 初始化；⑤ 物理内存管理器初始化。
  - **异构性**: 目标项目面向多架构（RISC-V/LoongArch），具备 devfs/procfs/ext4 完整文件系统栈、内置测试框架、显式关机路径；参考项目面向 LoongArch 教学环境，使用 DMW 直接映射、简单 FS、无测试框架。两者在文件系统栈深度、架构抽象层、测试基础设施上完全不同。

* **代码溯源链接**:
  - 目标项目: `kernel/main.c#kmain`
  - 参考项目: `src/core/kmain.c#kmain`

---

### 【内存管理 / kmalloc 分配器路径】

* **底层技术析理**:
  - **目标项目**: 调用 `align_size → align_up`。实现简单的对齐分配器，基于堆插入块（`heap_insert_block`）管理。内存模型为单调增长的堆分配，无页面级管理、无 slab 缓存。
  - **参考项目**: 调用 `PFN_TO_PADDR → kalloc_bytes_slab → kalloc_pages_buddy`。实现多级分配器架构：伙伴系统（buddy）管理物理页面，slab 分配器管理小对象。内存模型为页面级 + 对象级两级管理。
  - **异构性**: 目标项目为简单堆分配器（单级、无页面抽象），参考项目为伙伴系统+slab 多级分配器。两者在内存管理粒度、分配策略、碎片管理机制上存在根本性架构差异。

* **代码溯源链接**:
  - 目标项目: `kernel/mm.c#kmalloc`
  - 参考项目: `src/mm/mm.c#kmalloc`

---

### 【文件系统 / ext4_mount 第三方库引用】

* **底层技术析理**:
  - **目标项目**: 调用 `kfree → kmalloc → kmemset → vfs_alloc_generic_dentry → vfs_alloc_super_block → vfs_make_inode_common`。在 lwext4 库调用前后使用自研 VFS 层（通用 dentry/超级块/inode 分配）进行适配。
  - **参考项目**: 调用 `ext4_assert → ext4_bcache_cleanup → ext4_bcache_fini_dynamic → ext4_bcache_init_dynamic → ext4_block_bind_bcache → ext4_block_fini`。直接使用 lwext4 库的块缓存（bcache）接口进行管理。
  - **异构性**: 双方均引用同一开源第三方库 lwext4，但适配层完全不同：目标使用自研 VFS 层封装，参考直接使用库内块缓存接口。这是对同一公共依赖的独立集成，非双方之间的代码复用。

* **代码溯源链接**:
  - 目标项目: `kernel/vfs.c#ext4_mount`
  - 参考项目: `src/fs/lwext4/src/ext4.c#ext4_mount`
