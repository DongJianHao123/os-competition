## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心机制是**缺页驱动的懒映射（lazy mmap）**：

- `arch/riscv/trap.c#riscv_trap_handler` 直接调用 `add_mmap_region`、`align_down/up_u64`、`handle_lazy_mmap_fault`
- `arch/riscv/trap.c#handle_lazy_mmap_fault` 调用 `find_mmap_region`、`map_user_existing_page`、`page_alloc`、`ext4_read_inode_range`
- `arch/riscv/user.c#user_space_build` 负责用户态地址空间初始化：`map_page`、`map_initial_tls`、`map_trampoline`、`map_vdso`、`page_alloc`、`page_free`
- `arch/riscv/trap.c#sys_execve` 在加载新镜像前调用 `clear_all_mmap_regions`

未体现的内容：swap、页替换算法、TLB shootdown、NUMA、多级页表细节。

### PROC（进程管理/任务调度）

底层证据支持**传统进程管理**，而非独立异步运行时：

- 进程表/PCB：`proc_table_base`、`find_proc_by_pid`、`proc_current`
- 生命周期：`kernel/proc.c#proc_exit`、`proc_reap_child`、`arch/riscv/trap.c#sys_wait4`、`arch/riscv/sched.c#fork_ret`
- 信号与定时器：`sys_rt_sigtimedwait`、`proc_next_signal`、`proc_check_real_timer`、`proc_arm_real_timer`
- 同步原语：`futex_wake_waiters`、`futex_requeue_waiters`
- FD 表：`copy_fd_table`、`proc_close_fds`、`get_file_by_fd`

“任务调度/异步运行时”模块统计中的大量函数（如 `sys_read`、`sys_write`）实为系统调用处理，被分类算法归入该模块，**不能作为独立 async runtime 的证据**。实际调度证据为 `return_to_scheduler`、`wait_to_kernel`、`fork_ret` 等传统上下文切换路径。

### FS（文件系统）

文件系统是该作品证据最强的子系统之一：

- **VFS 层**：`kernel/vfs.c` 提供通用 dentry/inode、mount、lookup、fd 分配（`alloc_vfs_fd`、`vfs_create_file_from_dentry`、`ext4_lookup`、`ext4_mount`）
- **EXT4**：`kernel/ext4.c` 实现 inode 读取、目录查找、extent 树遍历、重命名（`ext4_read_inode_range`、`ext4_lookup_path`、`ext4_rename_path`、`visit_dir_inode`）
- **特殊文件系统**：
  - `devfs`：`kernel/devfs.c#devfs_init`，支持 memory file 读写
  - `tmpfs`：`kernel/tmpfs.c#tmpfs_init`
  - `procfs`：`kernel/vfs.c` 中 `read_pid_status`、`read_pid_stat`、`procfs_make_child_node` 等
- **ELF 加载**：`kernel/elf.c#elf_load_user_space`、`load_inode_to_stage`、`parse_header`

注意：`kernel/ext4.c` 在模块画像中被归入“文件系统/第三方库”，应单独说明其边界。

### DEV（设备驱动）

设备驱动证据较弱，种类单一：

- `kernel/virtio_blk.c`：virtio-blk MMIO 驱动，包含初始化、队列配置、扇区读写（`virtio_blk_init`、`setup_modern_queue`、`virtio_blk_read_sector`）
- `arch/riscv/platform.c#platform_putc` / `arch/loongarch/platform.c#platform_putc`：串口/控制台字符输出

未体现：网络设备、显示、键盘、磁盘缓存控制器、非 virtio 块设备等。

### SYSCALL（系统调用）

模块统计中“系统调用”仅 2 个函数，**但实际系统调用接口大量实现在 `trap.c`/`trapbk.c` 中**，通过 RISC-V 异常向量进入：

- 文件类：`sys_read`、`sys_write`、`sys_openat`、`sys_fstatat`
- 进程类：`sys_execve`、`sys_wait4`、`sys_rt_sigtimedwait`、`sys_rt_sigprocmask`
- IO 多路复用：`sys_ppoll`、`sys_pselect6`
- 时间：`sys_clock_nanosleep`
- 内存：`sys_execve` 间接涉及 mmap 清理

实现上依赖 `copy_to_user`、`copy_from_user_bytes`、`read_user64`、`read_user_timespec` 等用户态拷贝辅助函数。

### TRAP/IRQ（中断异常）

异常处理是代码量最大的模块：

- 主入口：`arch/riscv/trap.c#riscv_trap_handler`（242 函数文件）与 `arch/riscv/trapbk.c#riscv_trap_handler`（235 函数文件）高度重复
- 时钟中断：`arch/riscv/timer.c#timer_system_init`、`sbi_set_timer`
- 系统调用分发、缺页处理、信号投递均在异常入口中完成

未单独体现外部中断控制器（如 PLIC）的清晰封装；timer 中断证据明确。

### NET（网络协议栈）

**底层证据未体现**。未出现任何网络协议、套接字、网卡驱动相关函数。

## 第六部分：边界与未体现内容

### 没有底层证据支撑的模块

- **网络协议栈（NET）**：未出现任何 socket、TCP/IP、网卡驱动相关函数
- **独立异步运行时**：模块统计中的“任务调度/异步运行时”大量函数实为 trap 中的系统调用处理，未出现 `waker`、`ring_fifo`、`async/await` 等独立原语
- **高级调度器**：未体现 CFS、SMP、多核调度、调度类框架
- **swap 与页替换**：仅有按需缺页加载，未体现交换分区、LRU 等
- **丰富设备驱动**：仅 virtio-blk 与串口输出，无网络、显示、键盘、RTC 等
- **安全/权限框架**：仅见 `proc_effective_umask` 等零散点，无完整 ACL/Capability/LSM
- **动态内核模块/热插拔/电源管理/声音/GPU**：均无证据

### 需要单独说明的边界

- `kernel/ext4.c` 被模块画像归入“文件系统/第三方库”，其代码应视为文件系统实现或基于 ext4 规范的实现，**不能仅凭目录名直接等同于原创内核主体**
- `arch/riscv/trapbk.c` 与 `arch/riscv/trap.c` 高度重复，可能是备份实现、双异常向量或历史版本，需作者说明二者关系
- `arch/loongarch/platform.c#platform_putc` 仅出现一处，不能说明作品支持 LoongArch 完整内核，只能证明存在跨平台串口输出代码
