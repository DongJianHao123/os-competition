#### 4. 核心代码调用流对比剖析
*   **【进程管理 / `sys_execve` 系统调用】**
    *   **底层技术析理**: 双方均实现 POSIX `execve`，但**程序加载与路径解析策略根本不同**。目标项目采用 `Nameidata` 进行 VFS 路径漫游，并集成 `fanotify` 通知机制，体现了更复杂的文件访问控制与监控架构。参考项目则通过解析 `argv` 指针数组并直接使用 `DEFAULT_SHELL` 回退，实现相对简化。两者在内存管理、对象模型上存在显著差异。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/task.rs#sys_execve`
        - 参考项目: `os/src/syscall/process.rs#sys_execve`

*   **【进程管理 / `run_tasks` 调度核心】**
    *   **底层技术析理**: 双方**任务调度器的同步与并发模型截然不同**。目标项目使用 `PROCESSOR[hart_id]` 静态数组实现 per-hart 调度器，通过索引直接访问当前核心的处理器结构，是无锁或细粒度锁设计。参考项目使用 `PROCESSOR.lock` 全局互斥锁保护整个调度器状态，是典型的粗粒度锁设计。这体现了在多核同步策略上的根本性架构差异。
    *   **代码溯源链接**:
        - 目标项目: `os/src/task/processor.rs#run_tasks`
        - 参考项目: `os/src/syscall/process.rs#sys_exit_group`

*   **【内存管理 / `new_kernel` 内核内存初始化】**
    *   **底层技术析理**: 双方**内核地址空间的构建方法存在差异**。目标项目显式地使用 `DTB_BASE` 解析设备树来获取物理内存信息，并手动创建 `MapArea` 和 `PageTableEntry`，展现了基于设备树的动态内存发现机制。参考项目则使用 `anonymous_identical_map!` 宏进行静态或简化的等值映射，并依赖 `should_map_trampoline!` 等条件编译，其内存抽象层次和初始化流程不同。
    *   **代码溯源链接**:
        - 目标项目: `os/src/mm/memory_set.rs#new_kernel`
        - 参考项目: `os/src/mm/memory_set.rs#new_kernel`

*   **【中断/异常处理 / `trap_handler` 陷阱处理】**
    *   **底层技术析理**: 双方均处理 LA64 架构的 Trap，**处理流程的侧重点和复杂度不同**。目标项目在 Trap 入口处即进行 `BadV` 读取和 `Instruction` 解析，并直接处理 `PageFaultCause`，逻辑更集中于内存访问异常。参考项目则额外处理 `ERA` 寄存器的保存与恢复，并涉及浮点上下文切换，其 Trap 处理流程更完整，覆盖了更多异常类型。这表明双方对 LA64 Trap 处理的理解和实现粒度不同。
    *   **代码溯源链接**:
        - 目标项目: `os/src/arch/la64/trap/mod.rs#trap_handler`
        - 参考项目: `os/src/hal/arch/loongarch64/trap/mod.rs#trap_handler`

#### 5. 单方模块与降权项
*   **单方模块**:
    *   **网络协议栈/第三方库** 与 **第三方库** 模块仅存在于参考项目。这源于参考项目集成了 `isomorphic_drivers`、`pci`、`rustsbi` 等外部依赖，而目标项目未采用。这是**项目架构选型与依赖管理策略的根本性差异**，是证明双方独立开发的有力证据。
*   **降权项**:
    *   所有标记为“内部闭环逻辑”的函数（如 `readable`、`writable`、`can_lookup`、`get_size`、`is_store`），因其实现通常为单行或极简逻辑，不具备可保护的独创性，**不能作为同源性证据**。
    *   所有标记为“同板块同名”的 POSIX 标准系统调用（如 `sys_getpid`、`sys_exit`、`sys_getppid`），其函数名由标准强制规定，**同名不能作为同源性证据**。其内部实现的差异（如目标使用 `current_task().tgid`，参考使用 `current_task().unwrap()`）反而指向独立开发。
    *   所有与特定硬件架构（LA64）绑定的操作（如 `sbi_call`、`console_getchar`、`get_timer_freq_first_time`），其实现逻辑由硬件手册或 SBI 标准严格规定，**调用流相似是遵循标准的必然结果，不能作为代码复用证据**。

#### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度仅为 4.27%，远低于同源性阈值。控方关于“进程管理”板块的复用指控，经技术析理，其证据或源于 POSIX/Linux ABI 标准、或源于 LA64 硬件架构规范、或为 Rust 语言通用实践，均不构成核心逻辑的抄袭。双方在任务调度器同步模型（per-hart vs 全局锁）、内存初始化策略（设备树动态映射 vs 静态等值映射）、第三方库依赖等关键架构决策上存在根本性差异。因此，本次审计的最终结论为：**两个项目为独立开发，不存在同源性抄袭**。
