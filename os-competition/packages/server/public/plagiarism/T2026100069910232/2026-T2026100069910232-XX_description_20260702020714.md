## 第四部分：分模块架构说明

### MM（内存管理）

**已体现证据**：
- 系统调用：`sys_mmap`、`sys_msync`（`src/kernel/src/syscall/mm/mmap.rs`）
- 地址空间后端：`src/kernel/src/mm/aspace/backend/cow.rs#alloc_new_at` 实现 COW 后端，从文件读取并映射到物理页
- ELF 加载：`src/kernel/src/mm/loader.rs#load_user_app`、`load`
- 用户态内存访问原语：`src/kernel/src/mm/access.rs`，以及调用链中大量 `vm_read`、`vm_write`、`vm_read_uninit`

**可说明的架构**：用户虚拟地址空间按区域（area）管理，每个区域可绑定不同后端（匿名分配、COW、文件映射、线性、共享）。支持 ELF 可执行文件加载、用户指针安全读写。

**未体现**：物理页框分配器、页回收/交换、TLB/页表底层刷新、大页/hugepage、NUMA 内存策略等底层证据未体现。

**已体现证据**：
- 进程创建：`src/kernel/src/syscall/task/clone.rs#do_clone` 实现 clone/fork
- 程序执行：`src/kernel/src/syscall/task/execve.rs#execve_inner`
- 进程退出：`src/kernel/src/task/ops.rs#do_exit`
- 等待/回收：`src/kernel/src/syscall/task/wait.rs#reap_child`
- 进程表/进程组：`PROCESS_TABLE`、`PROCESS_GROUP_TABLE`
- pidfd：`src/kernel/src/syscall/fs/pidfd.rs#sys_pidfd_open`、`sys_pidfd_getfd`、`sys_pidfd_send_signal`
- procfs：`src/kernel/src/pseudofs/proc.rs#lookup_child`

- futex/robust list：`src/kernel/src/task/futex.rs#wait_if`、`exit_robust_list`

**可说明的架构**：实现了传统 Unix 进程模型（PID、TID、父子关系、进程组、会话）、fork/exec/exit/wait、信号投递、pidfd、procfs 信息导出，以及与 futex 相关的线程同步。

**未体现**：真正的 CPU 调度器内部数据结构（如 runqueue、CFS、时间片计算）在 `axtask` 中，当前证据仅体现对其的调用；SMP 调度、负载均衡、上下文切换汇编等底层证据未体现。

## 第六部分：边界与未体现内容

### 明确没有底层证据支撑的模块/能力

1. **中断与异常处理**：除模板 `irq.rs` 中的 `todo!()` 桩和少量信号/termios 函数外，无中断向量、异常入口、上下文切换汇编、中断控制器驱动的证据。
2. **硬件设备驱动**：无 PCI、USB、NVMe、SATA、块设备、网卡、GPU、声卡、DMA、设备树/ACPI 等真实硬件驱动证据。
3. **块设备与存储栈**：无块层、磁盘缓存、真实磁盘文件系统（ext4/xfs 等）实现。
4. **网络协议栈内部**：仅有 socket 系统调用封装，无 TCP/IP 状态机、路由、ARP、网卡驱动等。
5. **物理内存管理底层**：无页框分配器、伙伴系统、页回收、swap、TLB 管理、hugepage 等直接证据。
6. **调度器内部**：证据主要体现对 `axtask` 的调用，而非独立实现的 CPU 调度器。
7. **安全/审计机制**：无 LSM、seccomp、capabilities、audit 等证据。
8. **电源管理、SMP 启动、时钟中断架构**：无相关底层证据。

### 需要谨慎的“可能方向”判断

- “轻量任务调度或异步运行时”仅基于 `axtask::yield_now`、`block_on`、`FutexKey` 等调用链推断，不代表项目独立实现了完整调度器。
- “传统进程管理”基于 `do_clone`、`execve_inner`、`do_exit`、`wait`、`pidfd`、`PROCESS_TABLE` 等证据，但父子关系、zombie、会话等细节需结合运行时测试进一步验证。
- 文件系统能力强主要反映在 VFS 与伪文件系统层，真实持久化存储支持证据不足。
