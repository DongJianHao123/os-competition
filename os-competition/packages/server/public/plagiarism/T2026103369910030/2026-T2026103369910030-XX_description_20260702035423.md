## 第四部分：分模块架构说明

### MM（内存管理）
**证据较强**。核心抽象为 `AddrSpace`，在任务创建、ELF 加载、系统调用中频繁出现。`kernelx/src/kernel/task/tcb.rs#new_inittask` 与 `new_exec` 创建地址空间并加载 ELF；`kernelx/src/kernel/syscall/mm.rs#mmap` 实现匿名/文件映射；`kernelx/src/arch/riscv/task/traphandle.rs#usertrap_handler` 处理用户态页故障与 `copy_from_user`。内核堆使用 `kernelx/clib/lib/tlsf/tlsf.c` 的 TLSF 分配器。

### PROC（进程管理/任务调度）
**证据较强**。存在完整的 PCB/TCB 生命周期：`kernelx/src/kernel/task/pcb/lifecycle.rs#new / exit / recycle`，`kernelx/src/kernel/task/tcb.rs#new_inittask / new_exec / exit`。父子关系、等待队列、僵尸回收在 `kernelx/src/kernel/task/pcb/wait.rs` 中体现（`wait_child`、`waitable_child`、`reap_child`）。`kernelx/src/kernel/scheduler/processor.rs#schedule` 完成上下文切换。`ptrace`、`execve`、`waitid/wait4` 等系统调用进一步佐证多进程语义。

### FS（文件系统）
**证据非常强**。VFS 层包含 dentry、mount、inode 抽象，`kernelx/src/fs/vfs/mount.rs` 实现了 mount/bind/move/unmount 及挂载传播。EXT4 有两条实现线：原生 Rust 的 `kernelx/src/fs/ext4_native/` 与基于 `lwext4` C 库的 `kernelx/src/fs/ext4/`。此外还实现了 procfs（`/proc/[pid]/stat|maps|fd` 等）、devfs（null/zero/ptmx/rtc/kvm）、memtreefs。

### DEV（设备驱动）
**证据中等**。设备发现依赖 FDT，`kernelx/src/arch/{riscv,loongarch}/fdt.rs` 与 `kernelx/clib/lib/libfdt` 负责解析。已识别 PCI、`kernelx/src/driver/pci.rs`、VirtIO-Block、`kernelx/src/driver/block/virtio.rs`、NS16550A 串口、`kernelx/src/driver/char/tty.rs`、LoongArch RTC `ls7a`、RISC-V PLIC。驱动层函数总量不大，但覆盖启动所需关键设备。

### SYSCALL（系统调用）
**证据较强**。`kernelx/src/kernel/syscall/` 下按功能分文件：`fs.rs`（openat、copy_file_range、utimensat、fcntl、fanotify）、`event.rs`（select/poll/epoll/ppoll/timerfd/eventfd）、`ipc.rs`（sem/msg/signal/futex）、`task.rs`（exec/wait/ptrace）、`mm.rs`（mmap/process_vm）、`uid.rs`（capset/capget）。调用链中大量出现 `current::fdtable`、`current::pcb`、`current::tcb`、`current::schedule`。

### TRAP/IRQ（中断异常）
**证据中等**。RISC-V 与 LoongArch 各有一套 trap handler：`kernelx/src/arch/riscv/task/traphandle.rs#kerneltrap_handler / usertrap_handler`、`kernelx/src/arch/loongarch/task/traphandle.rs#kerneltrap_handler / usertrap_handler`。`kernelx/src/kernel/trap.rs#trap_return` 负责返回用户态前的调度与 SATP 校验。函数总量仅 57，说明中断分发可能较集中，未看到大量独立设备 ISR。

### NET（网络协议栈）
**证据中强**。自顶向下可见：`kernelx/src/net/interface/dispatch.rs#on_receive` 做链路层解析与协议分发；`kernelx/src/net/interface/arp.rs`、`dhcp.rs` 处理地址解析与配置；`kernelx/src/net/protocol/ipv4.rs` 构造 IPv4 包；`kernelx/src/net/socket/tcp.rs` 实现 TCP 状态机、重传、乱序合并；`kernelx/src/net/socket/netlink.rs#handle_getlink` 提供 Netlink 接口。整体为自研协议栈，非套接字转发到外部栈。

## 第六部分：边界与未体现内容

- **未体现多架构支持**：代码中仅出现 RISC-V 与 LoongArch 相关路径，没有 x86、AArch64 等架构的底层证据。
- **虚拟化仅为线索**：`kernelx/src/arch/riscv/kvm/vcpu.rs#goto_guest` 出现单函数证据，但缺少完整的 KVM/VMM 模块，不能判定为完整虚拟化支持。
- **无高级安全子系统证据**：未看到 LSM、SELinux、seccomp-bpf、Capabilities 之外的强制访问控制实现。
- **无图形/音频/电源管理证据**：未出现 GPU/DRM、声卡、ACPI/UEFI、休眠、CPU 热插拔、NUMA 等相关代码。
- **启动方式单一**：设备发现完全依赖 FDT（libfdt），没有 ACPI/UEFI 启动证据。
- **第三方库需单独区分**：`lwext4`、`tlsf`、`libfdt` 为外部 C 库，不应被夸大为主体内核原创实现；主体内核代码以 Rust 为主。
- **“异步运行时”判断有限**：任务调度/异步运行时模块函数数 429，但底层证据更多体现为传统阻塞-唤醒调度器，完整 async/await runtime 证据不足。
- **静态函数数量不等于功能完整性**：本报告仅基于 AST Call Map、函数名与路径锚点描述技术事实，不做运行时覆盖、正确性或原创性判定。
- **身份信息未推断**：未根据项目名、目录名或常理推测学校、队伍、作者、指导老师等身份信息。
