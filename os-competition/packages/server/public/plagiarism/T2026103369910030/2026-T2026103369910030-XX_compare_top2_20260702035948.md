### 4. 核心代码调用流对比剖析
* **【进程管理 / `exit` 系统调用】**
* **底层技术析理**: 两者均实现了进程退出逻辑，但抽象层次与同步策略截然不同。目标项目 (`kernelx`) 采用 Rust 的所有权与生命周期模型，通过 `State::Exited` 枚举状态机、`KSiFields::SigChld` 信号字段、`c.parent.lock` 锁同步以及 `child.recycle` 显式资源回收来管理进程退出。参考项目 (`2024`) 则采用传统的 C 语言面向过程风格，直接调用 `kill` 函数向父进程发送信号，并通过 `my_proc` 获取当前结构体指针进行状态修改。目标项目展现了更细粒度的对象管理，而参考项目是典型的宏内核过程调用。
* **代码溯源链接**:
  - 目标项目: `kernelx/src/kernel/task/pcb/lifecycle.rs#exit`
  - 参考项目: `src/core/proc.c#exit`

* **【任务调度/异步运行时 / `write` 阻塞写操作】**
* **底层技术析理**: 目标项目的 `write` 位于管道 IPC 模块 (`kernel/ipc/pipe/inner.rs`)，其阻塞机制依赖于 `current::schedule` 和 `current::task().take_wakeup_event()` 等异步事件循环框架。参考项目的 `write` 位于文件系统模块 (`fs/fs2.c`)，其阻塞机制是直接调用 `yield` 主动放弃 CPU。两者虽然都实现了写入阻塞，但一个基于事件驱动的异步运行时，一个基于主动让步的协程模型，在调度策略和实现哲学上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `kernelx/src/kernel/ipc/pipe/inner.rs#write`
  - 参考项目: `src/fs/fs2.c#write`

* **【内存管理 / `fork` 系统调用】**
* **底层技术析理**: 目标项目的 `fork` 实现 (`kernel/mm/maparea/filemap/private.rs`) 利用 Rust 的 `Box::new` 分配新结构，并通过 `FrameState::Cow` (写时复制) 和 `matches!` 模式匹配来管理物理页框的共享与复制。参考项目的 `fork` 实现 (`core/exec.c`) 则直接操作页表 (`copy_pagetable`)、进行虚拟地址转换 (`PA2VA`) 并拷贝文件描述符表 (`fdtable_copy`)。一个是基于类型系统和所有权的高层抽象，另一个是基于指针和宏的底层硬件操作，内存管理模型完全不同。
* **代码溯源链接**:
  - 目标项目: `kernelx/src/kernel/mm/maparea/filemap/private.rs#fork`
  - 参考项目: `src/core/exec.c#fork`

### 5. 单方模块与降权项
*   **单方模块**:
    *   **网络协议栈**: 目标项目拥有完整的 TCP/IP 协议栈实现（`kernelx/src/net/`），包含 TCP、UDP、DHCP、ARP 等模块。参考项目无此模块。这是目标项目独立开发、功能更丰富的显著标志。
*   **降权项**:
    *   **第三方库/公共依赖**: `lwext4`、`libfdt` 及通用 C 库函数 (`strcmp`, `strcpy` 等) 的重合，属于对公开上游代码的共同使用，不构成项目间的代码复用。
    *   **POSIX/Linux 标准接口**: `exit`、`wait4`、`write`、`dup3`、`fork` 等函数名相同，是因为它们都遵循 POSIX/Linux 标准。其内部实现逻辑的相似性由接口规范强制决定，不能作为抄袭证据。
    *   **跨板块配对**: 控方将目标项目的网络协议栈函数 (`process_segment`) 与参考项目的进程切换函数 (`switch_to`) 进行比对，此类跨模块证据违反“同板块”原则，予以降权排除。

### 6. 保守结论
综合全模块 AST 指纹分析，目标项目与参考项目在核心自研模块（进程、调度、内存、系统调用、驱动）的共同指纹为零，表明两者在底层架构、抽象层次和实现路径上存在根本性差异。高达 9.15% 的总体相似度几乎完全由双方共同依赖的 `lwext4` 等第三方开源库贡献，属于正常的公共代码引用。控方提出的“语义级复写”指控缺乏 AST 节点层面的直接证据支持，且存在跨模块硬凑证据的情况。因此，本庭裁定目标项目为独立开发，不存在对参考项目的代码复用，整体风险极低。
