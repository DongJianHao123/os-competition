#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 目标项目将进程退出逻辑拆分为两层。`process.rs#exit_current` 负责清理进程控制块（读取全局数组 `PROC_CLEARTID`, `PROC_PPID` 等），随后调用 `runtime.rs#exit_current` 执行任务状态切换（`task::transition_state`）并最终通过 `crate::sbi::shutdown` 停机。参考项目则在 `run_queue.rs#exit_current` 中集中处理，通过 `curr.inner.lock().set_state` 设置任务状态，并调用 `axhal::misc::terminate` 终止。两者在对象管理模型（全局数组 vs 任务对象内部锁）和硬件抽象层（sbi vs axhal）上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `modules/axruntime/src/process.rs#exit_current` -> `modules/axruntime/src/runtime.rs#exit_current`
  - 参考项目: `modules/axtask/src/run_queue.rs#exit_current`

* **【内存管理/物理帧分配路径】**
* **底层技术析理**: 目标项目 `mm.rs#alloc_frame` 通过一个全局的 `FRAME_ALLOC` 分配器进行帧分配，并检查 `FRAME_ALLOC_READY` 标志位。参考项目 `paging.rs#alloc_frame` 则通过 `global_allocator()` 分配页面，并调用 `virt_to_phys` 进行地址转换。两者的分配器后端和地址转换模型完全不同。
* **代码溯源链接**:
  - 目标项目: `modules/axruntime/src/mm.rs#alloc_frame`
  - 参考项目: `modules/axhal/src/paging.rs#alloc_frame`

* **【文件系统/FAT32读取路径】**
* **底层技术析理**: 目标项目 `fat32.rs#read_at` 通过 `inode_cluster` 和 `inode_is_dir` 等函数直接操作inode。参考项目 `fatfs.rs#read_at` 则使用 `SeekFrom::Start` 进行定位，并通过 `ans.is_err()` 进行错误处理。两者对FAT32库的封装和调用模式不同。
