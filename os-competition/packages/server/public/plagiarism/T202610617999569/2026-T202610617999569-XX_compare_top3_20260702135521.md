* **底层技术析理**: 这是控方核心指控点。目标项目 `sys_execve` 的调用流为 `vm_load_string` (安全用户态字符串加载) -> `FS_CONTEXT.lock().resolve` (通过文件系统上下文锁解析路径) -> `execve_common` (核心执行逻辑，包含 `Arc::new`, `AxError::from` 等 Rust 内存管理)。参考项目 `sys_execve` 的调用流为 `argraw` (直接寄存器读取) -> `get_path` (C风格路径处理) -> `exec_file` -> `exec` (直接操作进程结构体)。两者在**内存抽象**（Rust 所有权/Arc vs. C 裸指针）、**同步策略**（显式锁 `FS_CONTEXT.lock` vs. 隐式全局 `my_proc`）、**错误处理**（`Result` 类型 vs. 直接调用 `kill`）上存在根本性架构差异。唯一的共同点是遵循了 POSIX `execve` 的“解析-执行”标准语义。
