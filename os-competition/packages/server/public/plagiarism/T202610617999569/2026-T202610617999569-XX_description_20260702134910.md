## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。内核实现了基于地址空间区域的内存映射框架，支持多种 mmap 后端（匿名分配、COW、文件映射、线性映射、共享映射），并包含 ELF 加载器与 `/proc/[pid]/maps` 生成逻辑。

证据锚点：

- `kernel/src/syscall/mm/mmap.rs#sys_mmap`：构造 `Backend::new_alloc/new_cow/new_file/new_linear/new_shared`。
- `kernel/src/mm/aspace/backend/cow.rs#alloc_new_at`：COW 页分配时调用 `file.read_at`。
- `kernel/src/mm/loader.rs#load_user_app_from_source`：调用 `ELF_LOADER.lock().load(...)` 加载用户 ELF。
- `kernel/src/syscall/mm/mmap.rs#mmap_proc_maps`：生成 `/proc/self/maps`。

### PROC（进程管理）

底层证据强，具备传统 Unix 进程管理要素：进程表、PCB（`proc_data`）、fork/clone、execve、exit、wait、信号、ptrace、pidfd、进程组/会话、命名空间相关 proc 节点。

证据锚点：

- `kernel/src/syscall/task/clone.rs#do_clone`：创建任务扩展、复制 fd 表、写入 parent_tid。
- `kernel/src/syscall/task/execve.rs#execve_common`：路径解析、ELF 加载、替换地址空间。
- `kernel/src/task/ops.rs#do_exit`：清理 futex、共享内存、信号、子进程时间统计。
- `kernel/src/syscall/task/wait.rs#sys_waitid`：遍历子进程、处理 wait_status、生成 siginfo。
- `kernel/src/syscall/task/ptrace.rs#sys_ptrace`：`PTRACE_TABLE`、attach 权限检查、信号停止。
- `kernel/src/syscall/fs/pidfd.rs#sys_pidfd_open/sys_pidfd_send_signal`：pidfd 抽象。

### 任务调度/异步运行时

证据主要体现为对外部运行时原语的调用，而非本仓库内完整的调度器实现。多处使用 `axtask::yield_now`、`axtask::set_current_affinity`，以及 `block_on` 实现 `select`/`epoll_wait`。

证据锚点：

- `kernel/src/syscall/task/schedule.rs#sys_sched_setaffinity`：调用 `axtask::set_current_affinity`。
- `kernel/src/syscall/io_mpx/select.rs#do_select`：使用 `block_on(...)` 与 `FdPollSet`。
- `kernel/src/syscall/io_mpx/epoll.rs#do_epoll_wait`：使用 `block_on(...)`。

### FS（文件系统）

底层证据最强。内核实现了 VFS 层、路径解析、挂载表、fd 表、文件/目录操作、fanotify、procfs、devfs，并通过 `lwext4_rust` 提供 ext4 存储后端。

证据锚点：

- `kernel/src/syscall/fs/fd_ops.rs#sys_openat`：路径解析、权限检查、fd 分配。
- `kernel/src/syscall/fs/mount.rs#sys_mount/sys_umount2/sys_move_mount/sys_open_tree`：挂载、绑定挂载、移动挂载。
- `kernel/src/syscall/fs/fanotify.rs#sys_fanotify_init` / `queue_event`：文件事件通知。
- `kernel/src/pseudofs/proc.rs#lookup_child`：procfs 动态节点构造。
- `third_party/lwext4_rust/c/lwext4/src/ext4.c#ext4_dir_rm` / `ext4_extent.c#ext4_extent_remove_space`：ext4 后端实际块/索引操作。

### DEV（设备驱动）

仅有伪文件系统设备的证据，未见完整设备驱动框架（如 PCI、块设备层、DMA、中断驱动等）。

证据锚点：

- `kernel/src/pseudofs/dev/tty/mod.rs#ioctl`：处理 `Termios`/`WindowSize` 等用户内存读写。
- `kernel/src/pseudofs/dev/loop.rs#ioctl`：处理 `loop_info`/`loop_info64` 设置。
- `kernel/src/syscall/fs/fd_ops.rs#add_to_fd` 中出现 `"/dev/console"` 作为默认标准设备。

### SYSCALL（系统调用）

存在统一分发入口，覆盖 fs、net、mm、task、signal、time、bpf、io_mpx 等大类。

证据锚点：

- `kernel/src/syscall/mod.rs#handle_syscall`：总入口，按 `Sysno` 分发。
- `kernel/src/syscall/bpf.rs#sys_bpf`：BPF map/prog 操作。
- `kernel/src/syscall/time.rs#sys_setitimer`：间隔定时器。
- `kernel/src/syscall/net/opt.rs#sys_getsockopt/sys_setsockopt`：socket 选项。

### TRAP/IRQ（中断异常）

**底层证据未体现**。函数统计中“中断/异常处理”仅 27 个函数，且全部位于 `syscalls_code/*` 测试用例（如 `recvmsg01.c`、`sigaltstack01.c`），未见内核侧硬件中断、异常向量、页故障处理、时钟中断等代码。

### NET（网络协议栈）

**底层证据未体现**。仅发现 socket 系统调用封装（`sys_getsockopt`、`sys_setsockopt`、`recv_impl` 等），没有 TCP/IP 协议栈、网卡驱动、路由表、sk_buff 等实现证据。

## 第六部分：边界与未体现内容

1. **中断/异常处理（TRAP/IRQ）底层证据未体现**  
   统计中的 27 个函数均为 `syscalls_code` 测试用例，未见内核侧硬件中断向量、异常分发、页故障处理、时钟中断等代码。

2. **网络协议栈（NET）底层证据未体现**  
   仅有 socket 相关系统调用的封装（getsockopt/setsockopt/recv 等），没有 TCP/IP 协议栈、网卡驱动、路由、连接管理等实现。

3. **通用设备驱动框架底层证据未体现**  
   仅发现 `pseudofs/dev/` 下的 tty、loop 伪设备，以及 `/dev/console` 默认设备，没有 PCI、块设备层、DMA、中断驱动等。

4. **第三方库需与原创主体区分**  
   - `third_party/lwext4_rust` 提供 ext4 文件系统后端，是存储实现的关键依赖，但非本仓库原创内核主体。  
   - `third_party/syscalls` 主要是系统调用号定义与辅助测试代码。  
   - `syscalls_code` 目录是 LTP 风格的系统调用测试用例，用于兼容性验证，不是内核实现。

5. **任务调度/异步运行时的原创范围有限**  
   证据显示大量依赖 `axtask` 原语（`yield_now`、`set_current_affinity`）和 `block_on`，本仓库内更偏向 syscall 层对异步运行时的使用，而非完整自研调度器。

6. **“可能的系统方向”仅为基于目录与调用链的保守推断**  
   不能据此判定作品最终身份、原创性比例或是否违规；本文档仅用于说明“实现了什么、证据来自哪里、哪些模块没有体现”。
