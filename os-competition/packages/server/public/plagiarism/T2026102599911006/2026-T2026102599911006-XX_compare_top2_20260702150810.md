### 4. 核心代码调用流对比剖析
* **【进程管理/调度原语】**
* **底层技术析理**: 双方 `yield` 函数的调用流均为 `acquire -> myproc -> release -> sched`。此调用流是 xv6 进程调度模型的核心：必须持有进程锁 (`acquire`) 才能修改进程状态，通过 `myproc` 获取当前进程指针，设置状态为 `RUNNABLE` 后释放锁 (`release`)，最后调用 `sched` 进入调度器上下文切换。这是实现协作式调度的标准范式，其顺序由并发安全语义强制决定，并非可自由发挥的“创意表达”。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#yield`
  - 参考项目: `kernel/proc.c#yield`

* **【文件系统/目录查找】**
* **底层技术析理**: 双方均有 `dirlookup` 函数，但底层对象模型截然不同。目标项目调用 `iget`，表明其基于 inode 模型（符合 ext4 设计）；参考项目调用 `eget`、`enext` 等，表明其基于 FAT32 的目录项集群链模型。尽管顶层函数名相同，但核心数据结构和访问方法存在根本性架构差异，这是独立设计的典型特征。
