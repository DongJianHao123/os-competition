#### 4. 核心代码调用流对比剖析

* **【进程管理/调度原语】**
* **底层技术析理**: 两者均遵循 xv6 的调度模型：通过 `swtch` 切换上下文，`scheduler` 循环寻找可运行进程。但目标项目的 `scheduler` 在切换前后显式调用 `intr_off` 和 `intr_on` 管理中断状态，而参考项目仅调用 `intr_on`。这反映了不同的中断屏蔽策略。在进程退出方面，目标项目的 `kexit` 调用链包含 `begin_op, end_op, fileclose, iput`，表明其进程模型与文件系统日志及引用计数机制紧密耦合，而参考项目的 `exit` 无此特征，架构复杂度不同。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#scheduler`，`kernel/proc.c#kexit`
  - 参考项目: `kernel/proc.c#scheduler`，`kernel/proc.c#exit`

* **【系统调用/分发路径】**
* **底层技术析理**: 这是双方架构差异最显著的路径。目标项目的 `syscall` 函数在获取系统调用编号后，调用 `linux_dispatch` 函数进入一个独立的 Linux 兼容系统调用处理流程。而参考项目的 `syscall` 函数直接以编号为索引，在 `syscalls[num]` 函数指针数组中查找并调用对应的 xv6 原生处理函数。目标项目引入了全新的分发层和兼容层，而参考项目保持 xv6 的原始设计。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall.c#syscall`
  - 参考项目: `kernel/syscall.c#syscall`

* **【文件系统/目录查找】**
* **底层技术析理**: 两者均实现目录项查找，但底层文件系统模型完全不同。目标项目基于传统 inode 模型，通过 `iget` 获取 inode，再调用 `readi` 读取目录数据块，最后用 `namecmp` 比较文件名。参考项目基于 FAT32 模型，通过 `edup` 复制目录项，调用 `enext` 遍历簇链，并使用 `reloc_clus` 处理簇的定位。这是两种互不兼容的文件系统架构。
* **代码溯源链接**:
  - 目标项目: `kernel/fs.c#dirlookup`
  - 参考项目: `kernel/fat32.c#dirlookup`

#### 5. 单方模块与降权项

*   **单方模块（强独立证据）**:
    *   **文件系统/第三方库**: 目标项目独有的 `ext4_disk.c`、`ext4.c` 等文件，实现了完整的 ext4 文件系统支持。其函数如 `ext4_disk_init`、`ext4_read_block`、`ext4_init`、`read_file_data` 调用了 `probe_ext4_virtio`、`extent_lookup` 等参考项目完全不存在的对象。这证明目标项目进行了大规模独立开发。
    *   **系统调用**: 目标项目独有的 `linux_syscalls.c` 文件及其内部的 `l_getppid` 等函数，以及 `syscall` 中调用的 `linux_dispatch`，构成了一个独立的 Linux 系统调用兼容层，在参考项目中无任何对应。

*   **降权项（不构成强同源证据）**:
    *   **第三方库/公共依赖**: `user/printf.c#vprintf` 与 `xv6-user/printf.c#vprintf` 调用链相似，但 `printf` 系列函数是标准 C 库的公共实现，其格式化字符串解析逻辑由语言标准定义，不具备原创性，不能作为抄袭证据。
    *   **跨板块同名函数**: 控方将用户态 Shell 函数（`user/sh.c` 中的 `parseexec`, `fork1`, `gettoken` 等）归入【进程管理】或【任务调度/异步运行时】板块进行指控。Shell 解析器属于用户态应用程序，与内核进程调度属于不同抽象层次，跨板块同名/同调用链应降级为弱信号，不构成同板块主证据。
    *   **硬件/ABI 约束函数**: `cpuid` 调用 `r_tp`，`mycpu` 调用 `cpuid` 等，是 RISC-V 架构下读取 Hart ID 和获取 per-CPU 变量的唯一或标准方式，受硬件规范强制约束，其调用图一致不能视为代码复用。

#### 6. 保守结论

本次审计的宏观相似度仅为 17.38%，双向覆盖率均低于 35%，属于低度重叠。双方在进程调度、中断处理等基础模块的少量相似，根源于共同的 xv6 教学系统架构与 RISC-V 硬件 ABI 约束，属于领域共性实现。而双方在**系统调用分发**（Linux 兼容层 vs. 原生 xv6）、**文件系统模型**（ext4/inode vs. FAT32）等核心子系统上存在根本性的架构分歧。目标项目独有的 ext4 文件系统支持和 Linux 系统调用兼容层是其独立开发的有力证明。综合判定，现有证据无法支持“抄袭”或“重度复用”指控，双方相似性应认定为基于公开教学代码与领域通用架构的独立实现。
