#### 4. 核心代码调用流对比剖析
* **【内存管理/页表遍历与内存拷贝路径】**
* **底层技术析理**: 目标项目的 `pte_walk` 是一个纯粹的页表遍历辅助函数，仅调用 `page_alloc` 处理缺页，后续的 `copy_from_user` 通过 `memcpy` 完成数据拷贝。参考项目的 `pgdir_walk` 则深度耦合了 MIPS 架构的宏（`KADDR`, `PDX`, `PTX`），其调用者 `page_fault_handler` 使用 `bcopy` 并包含 `printf` 调试输出。两者在**地址翻译模型**（RISC-V Sv39 vs MIPS）、**拷贝原语**（`memcpy` vs `bcopy`）和**错误处理路径**上完全不同，体现了不同架构和设计哲学下的独立实现。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/user.c#pte_walk`, `kernel/proc/user.c#copy_from_user`
  - 参考项目: `lib/pmap.c#pgdir_walk`, `lib/traps.c#page_fault_handler`

* **【内存管理/ELF 加载路径】**
* **底层技术析理**: 目标项目的 `load_elf` 链路从 ext4 文件系统读取（`read_ext4_file_path`），并显式处理动态链接器（`elf_interp_path`），最终通过 `map_user_range` 映射内存。参考项目的 `disk_load_elf` 则通过自定义的 `eread` 从磁盘读取，使用 `page_insert` 逐页映射物理页框。两者在**文件系统后端**（ext4 vs 自定义磁盘IO）、**内存映射粒度**（区间映射 vs 逐页插入）和**对动态链接的支持**上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/user.c#load_elf`, `kernel/proc/user.c#load_elf_image`
  - 参考项目: `lib/exec.c#disk_load_elf`, `lib/exec.c#sys_exec`

* **【进程管理/进程克隆路径】**
* **底层技术析理**: 目标项目的 `proc_clone_current` 在内核态完成所有操作，通过 `copy_user_page_table` 递归复制整个页表层级。参考项目的 `fork` 则在用户态通过系统调用封装（`syscall_env_alloc`）实现，其页表复制策略是 `duppage` 配合 `syscall_mem_map`，更接近写时复制（COW）或按需映射。两者在**系统调用层次**（内核态一体 vs 用户态系统调用组合）和**页表复制策略**上截然不同。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/user.c#proc_clone_current`
  - 参考项目: `user/fork.c#fork`, `lib/syscall_all.c#sys_env_alloc`
