## 第四部分：分模块架构说明

### MM（内存管理）
- 用户地址空间管理由 `luwu-kernel/src/user_paging.rs` 承担，核心函数包括 `read_user_bytes`、`write_user_bytes`、`prepare_user_stack`、`read_user_cstr`。
- 物理页查找通过 `phys_for_va` → `phys_to_virt` → `read_volatile/write_volatile` 完成。
- 系统调用层提供 `sys_mmap`（`map_anonymous_range`）和 `sys_brk`（调整 `program_break`）。
- `process.rs#exec_current` 在加载 ELF 后调用 `UserMemory::create::<A>` 建立新地址空间。
- **未体现**：swap、页缓存、COW fork、slab/伙伴分配器细节、DMA 一致性框架。

### PROC（进程管理/任务调度）
- 传统进程管理证据充分：`process.rs` 维护全局 `PROCESS_TABLE`，字段包括 pid、parent、cwd、fds、program_break、alarm_deadline、context。
- `fork_current` 复制用户上下文并分配新 PID；`exec_current` 替换地址空间并进入用户态；`sys_wait4` 支持等待子进程；`terminate_current_with_signal` 处理退出与父进程唤醒。
- 异步运行时是内核内部机制：`async_rt.rs` 提供 `spawn`、`poll_task`、`run`、`wait`，维护 `ready` 队列与 `sleepers`，由定时器 tick 驱动。
- **注意**：异步运行时 ≠ 完整进程调度器；进程切换仍通过 trap/wait/switch_to_process 完成。

### FS（文件系统）
- 内核层存在两套文件系统：
  1. **内嵌内存文件系统**：`MEMORY_FILES` / `MEMORY_DIRS`，用于管道、`/dev`、测试脚本等，函数集中在 `syscall/mem.rs`、`syscall/dirents.rs`。
  2. **ext4 实现**：`luwu-ext4` 子 crate（306 个函数），包含 superblock、block group、inode/block 分配器、extent 树、目录读写、journal 提交。
- 内核通过 `fs::resolve_path`、`root.read/write/mkdir/rmdir/rename/symlink` 等接口调用 ext4。
- **未体现**：VFS 抽象层是否完整独立、缓存页管理、硬链接计数持久化细节需进一步确认。

### DEV（设备驱动）
- 仅见 **VirtIO-block** 驱动，支持 PCI 与 MMIO 两种 transport。
- `virtio/pci.rs#probe_first` 扫描 PCI 总线并识别 VirtIO-blk；`setup_queue` 配置 virtqueue。
- `virtio/mmio.rs#setup_queue_modern` 处理 MMIO 现代设备。
- `virtio/blk.rs#submit_request` / `poll_complete` 完成请求提交与完成轮询。
- `virtio/mod.rs#read_block` / `write_block` 提供块读写接口。
- **未体现**：网络设备、字符设备、GPU、USB、NVMe、通用中断框架。

### SYSCALL（系统调用）
- 入口集中在 `syscall/mod.rs#dispatch`，根据 ABI 解码参数后分发。
- 覆盖文件 IO（`read/write/pread64/pwrite64/readv/writev`）、文件操作（`openat/mkdirat/unlinkat/renameat/linkat/symlinkat`）、状态（`stat/newfstatat/statx`）、内存（`mmap/brk`）、进程（`execve/wait4/exit`）、时间（`gettimeofday/clock_gettime/nanosleep/setitimer`）、杂项（`ioctl/ppoll/sysinfo/readlinkat`）。
- **未体现**：完整信号机制（仅有 `signal_for_exception` 少量证据）、ptrace、ipc/shm/sem。

### TRAP/IRQ（中断异常）
- 架构相关代码在 `luwu-riscv/src/main.rs` 和 `luwu-loongarch/src/main.rs`：timer 初始化、`advance_syscall_epc`、trap 入口保存/恢复。
- 通用处理在 `luwu-kernel/src/trap.rs#handle_trap`：解码异常原因、分发 syscall、处理定时器中断、写回返回值、恢复用户态。
- `report_unhandled` 处理未识别异常。
- **未体现**：外部设备 IRQ 控制器抽象（如 PLIC/IOAPIC）、中断向量表动态注册。

### NET（网络协议栈）
- **底层证据未体现**：无 socket、TCP/IP、网卡驱动、网络缓冲区相关函数。

## 第六部分：边界与未体现内容

- **没有底层证据支撑的模块**：
  - 网络协议栈（NET）
  - 安全/权限机制（capabilities、ACL、namespace）
  - SMP/多核调度与核间中断
  - 电源管理、ACPI、设备树完整解析（仅见 `fdt::discover_memory` 内存发现）
  - 通用 IRQ 控制器与外部设备中断向量注册
  - swap、页缓存、COW fork、内核堆分配器细节
  - 完整信号机制、ptrace、System V IPC

- **需要单独说明的代码边界**：
  - `luwu-ext4` 子 crate 在统计中被单独归类为“文件系统/第三方库”，其 306 个函数构成 ext4 实现主体；评审时不应将其直接等同于 `luwu-kernel` 原创内核主体。
  - “第三方库”分类下的 2 个函数位于 `virtio/pci.rs`，仅为 PCI capability 遍历封装，不能代表完整驱动框架。
  - 异步运行时（`async_rt.rs`）是内核内部任务调度机制，不应被描述为完整的进程调度器；进程切换仍依赖 trap 与 `switch_to_process`。
