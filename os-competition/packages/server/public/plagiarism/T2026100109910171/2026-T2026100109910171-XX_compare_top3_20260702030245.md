### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 在 `do_exit` 的核心实现中，双方均执行了清理共享内存、写入清除子线程ID、让出CPU等标准 POSIX 退出流程。然而，**用户态内存访问模型**存在根本性差异。目标项目使用 `clear_child_tid.vm_write(0)` 直接进行虚存写入，而参考项目则通过 `UserPtr::<Tid>::from` 创建一个用户态指针抽象，再调用 `.get_as` 或 `.query_paddr` 进行访问。这反映了两种完全不同的内核-用户态内存交互哲学：一种直接映射操作，另一种通过抽象层隔离。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/ops.rs#do_exit`
  - 参考项目: `api/src/imp/task.rs#do_exit`

* **【内存管理/内存映射路径】**
* **底层技术析理**: 在 `sys_mmap` 的实现中，双方都处理了地址对齐、长度计算和标志位解析。但**内存映射后端（Backend）的抽象**截然不同。目标项目通过 `Backend::new_alloc`、`Backend::new_file` 等一组构造函数来创建不同的映射后端，体现了策略模式。参考项目则更侧重于 `MmapFlags` 的复杂位运算处理，将逻辑内聚在系统调用函数中。这显示出两者在内存管理模块的架构设计上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/mm/mmap.rs#sys_mmap`
  - 参考项目: `src/syscall_imp/mm/mmap.rs#sys_mmap`

* **【文件系统/伪文件系统构建路径】**
* **底层技术析理**: 在构建 `procfs` 时，双方都使用了字符串 `"proc"` 作为挂载点名，这是 POSIX 标准的约定。但**底层文件系统引擎**完全不同。目标项目基于 `SimpleFs` 构建，而参考项目基于 `DynamicFs` 构建。这两种文件系统在节点管理、查找逻辑和可扩展性上存在本质区别，表明双方在 VFS 层的技术选型和实现上走了不同的路线。
* **代码溯源链接**:
  - 目标项目: `kernel/src/pseudofs/proc.rs#new_procfs`
  - 参考项目: `core/src/vfs/proc.rs#new_procfs`

### 5. 单方模块与降权项
- **单方模块**: 参考项目独有的【设备驱动】、【网络协议栈】以及大量的【第三方库】（如 `lwext4_rust`, `visionfive2-sd`, `smoltcp` 等）模块，在目标项目中完全找不到对应实现。这构成了两个项目在功能范围和代码构成上的巨大差异，是独立开发的有力证据。
- **降权项（弱信号）**:
  - **跨板块配对**: 控方证据中，将 `lookup_child`（文件系统/伪文件系统）与 `sys_wait4`（进程管理）等跨模块函数强行配对，这违反了同板块比对原则，不构成有效证据。
  - **极短调用链**: `sys_exit` -> `do_exit`，`sys_getpid` -> `current().as_thread()...` 等调用链过短，是 POSIX 系统调用的标准薄封装模式，任何兼容内核都会如此实现，不具备唯一性。
  - **标准库惯用法**: `self.is_zombie.load`、`self.parent.lock().upgrade`、`INIT_PROC.get().unwrap()` 等 1-3 个节点的调用链，是 Rust 标准库 `AtomicBool`, `Weak<T>`, `Once<T>` 的标准操作范式，属于语言层面的共性，不能作为代码抄袭的证据。
  - **公共接口/宏**: `debug!`, `info!`, `assert!`, `Err`, `Ok` 等是 Rust 语言和日志库的公共接口，不具备任何独创性。

### 6. 保守结论
综合宏观相似度（2.72%）、极低的双向覆盖率以及全模块矩阵分析，两个项目在代码实现层面表现出根本性的架构差异。尽管在【进程管理】和【文件系统】等模块存在少量结构重合，但这些重合点经技术剖析，几乎全部可归因于 POSIX 标准语义、Linux 系统调用约定以及 Rust 标准库的惯用法，而非对独创性代码表达的复用。特别是参考项目包含大量目标项目所没有的设备驱动、网络协议栈和第三方库，进一步排除了系统性抄袭的可能。因此，整体风险极低，应认定为两个独立开发的作品。
