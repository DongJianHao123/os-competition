### 4. 核心代码调用流对比剖析
* **【进程管理/`sys_exec` 系统调用路径】**
* **底层技术析理**: 这是进程管理中最为核心和复杂的系统调用，双方实现路径完全不同。目标项目采用 `kernel_execve` 作为核心执行体，并调用 `parse_user_str_array` 和 `resolve_exec_path` 等辅助函数，显示出更模块化的设计。参考项目则直接在主函数中通过 `app_inode.read_all` 读取文件，并使用 `args_vec.push` 手动构建参数向量。两者在文件加载、参数解析和地址空间构建的抽象层次上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `oskernel/os/src/syscall/process.rs#sys_exec`
  - 参考项目: `os/src/syscall/process.rs#sys_exec`

* **【内存管理/`fork` 进程派生路径】**
* **底层技术析理**: 双方都遵循了 `fork` 的标准流程（获取父进程 -> fork -> 获取子进程 PID -> 获取 Trap 上下文），但在内存拷贝这一核心操作上选择了不同的策略。目标项目使用 `MemorySet::from_existed_user`，暗示其可能采用物理页共享或更复杂的写时拷贝机制。参考项目使用 `MemorySet::from_cow`，是经典的写时拷贝实现。这种底层内存管理模型的差异是独立设计的有力证明。
* **代码溯源链接**:
  - 目标项目: `oskernel/os/src/task/process.rs#fork`

### 5. 单方模块与降权项
- **单方模块**:
  - **网络协议栈**: 目标项目拥有完整的 TCP/UDP/Socket 实现（`oskernel/os/src/net/`），而参考项目完全缺失此模块。这是两者架构差异的决定性证据。

  - **薄封装函数与 Rust 惯用法**: `pid2process`、`take_current`、`thread_count` 等单行函数，以及 `unwrap()`、`clone()`、`panic!` 等标准库方法，在任何 Rust 项目中都普遍存在，其相似性不具备鉴别力。
  - **硬件/平台决定项**: `set_kernel_trap_entry` 等直接操作 RISC-V CSR 寄存器的函数，其实现由指令集架构规范唯一确定，必须相同。
