### 4. 核心代码调用流对比剖析
* **【进程管理/系统调用路径】**
* **底层技术析理**: 控辩双方争议焦点。以`sys_fork`为例，双方调用流均为`current_process.fork -> new_process.getpid -> new_process_inner.tasks[0].as_ref().unwrap() -> task.inner_exclusive_access().get_trap_cx`。此路径是rCore教学操作系统框架下，实现POSIX `fork`语义的标准“进程控制块复制 -> 获取新PID -> 获取主线程 -> 设置陷阱上下文返回值”流程。其相似性源于对同一公开教学框架的遵循，而非对参考项目的直接复制。目标项目在`sys_exec`、`sys_wait4`等系统调用中采用了`kernel_execve`、`wait4_core`等独立封装的实现，与参考项目的`app_inode.read_all`、`sys_waitpid`路径形成鲜明对比，证明了其在系统调用层的独立设计能力。

  - 参考项目: `os/src/syscall/process.rs#sys_fork`

* **【网络协议栈/中断处理路径】**
* **底层技术析理**: 这是本次审计中相似度最高的模块。`net_interrupt_handler`函数的调用图谱高度一致，从`NET_DEVICE.receive/transmit`到`LOSE_NET_STACK.0.exclusive_access().analysis`，再到`arp_packet.reply_packet`的完整处理链几乎完全相同。`socket.rs`中的`socket`表管理函数（`add_socket`, `get_socket`等）也呈现出高度结构一致性。这提示双方在网络协议栈的实现上可能存在比其它模块更强的关联性，例如参考了同一份开源网络栈实现（如smoltcp的某种封装模式）或存在直接的代码借鉴。然而，考虑到整体9.84%的极低相似度，这部分相似更可能指向对特定开源组件的共同依赖或参考，而非对整个内核项目的抄袭。

### 5. 单方模块与降权项
- **第三方库**: 参考项目独有`virtio-drivers`等第三方库模块，目标项目未体现。这属于正常的项目依赖差异，不构成同源证据。

- **Trivial访问器函数**: `current_task`、`take_current`、`inner_exclusive_access`、`get_user_token`等函数，其实现是Rust语言惯用法（如`Option::take`、`Mutex::exclusive_access`）的直接应用，表达空间极小，不能作为抄袭证据。

### 6. 保守结论
本次审计基于9.84%的客观AST相似度，判定两个项目为独立开发。尽管在进程管理、网络协议栈等模块观察到局部结构相似性，但这主要归因于双方共同遵循的RISC-V硬件规范、POSIX系统调用标准、rCore教学操作系统框架以及Rust语言惯用法。网络协议栈的较高相似度提示可能存在对同一开源组件的参考，但这不足以推翻整体独立开发的结论。控方“骨架级全盘克隆”的指控在宏观数据和模块差异分析面前无法成立。综合判定，目标项目原创性成立，抄袭风险低。
