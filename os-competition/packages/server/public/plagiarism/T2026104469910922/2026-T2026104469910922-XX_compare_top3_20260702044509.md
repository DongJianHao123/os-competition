#### 4. 核心代码调用流对比剖析
* **【进程管理/调度核心】**
* **底层技术析理**: 目标项目 `sched` 函数调用流为 `get_cpu -> get_proc -> holding -> intr_get -> panic -> swtch`，显式引入了 `get_cpu`/`get_proc` 抽象链来获取当前 CPU 和进程。参考项目 `sched` 调用流为 `holding -> intr_get -> mycpu -> myproc -> panic -> swtch`，使用 `mycpu`/`myproc` 直接访问。目标项目的抽象层（最终调用 `hal_get_cpu_id`）与参考项目的直接寄存器访问（`r_tp`）在架构分层和对象管理模型上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `kernel/core/proc.c#sched`
  - 参考项目: `kernel/proc.c#sched`

* **【内存管理/用户态数据拷贝】**
* **底层技术析理**: 目标项目 `copyin/copyout` 使用 `PA2VA`、`PTE2PA`、`PGROUNDDOWN` 等宏进行独立的页表遍历和地址转换，依赖 `get_proc` 获取进程上下文。参考项目则使用 `walkaddr` 函数和 `memmove` 完成相同功能。两者在页表解析的内存模型和函数依赖上选择了不同的实现路径。
* **代码溯源链接**:
  - 目标项目: `arch/riscv/mm/vm.c#copyin`
  - 参考项目: `kernel/vm.c#copyin`

* **【文件系统/文件关闭】**
* **底层技术析理**: 目标项目 `fileclose` 调用 `vfs_iput` 释放 inode，这是其 VFS 框架的标准接口。参考项目 `fileclose` 调用 `eput`，这是其 easyfs 特定实现的接口。两者依赖的文件系统后端和对象销毁模型完全不同，体现了顶层设计上的异构性。
* **代码溯源链接**:
  - 目标项目: `kernel/core/file.c#fileclose`
  - 参考项目: `kernel/file.c#fileclose`

#### 5. 单方模块与降权项
- **目标项目独有模块**: `文件系统/第三方库` (ext4fs, easyfs, devtmpfs 等多文件系统支持)、独立的日志框架 (`LOG_TRACE`, `LOG_INFO` 等)、硬件抽象层 (`hal_get_cpu_id`)。这些模块在参考项目中无对应实现，是目标项目独立开发的强有力正向证据。

- **降权项**: 控方指控的 `sys_getuid`、`sys_sched_yield`、`uartintr` 等函数，因属于 POSIX 标准接口、公开教材范式或 RISC-V ABI 决定的无创造性实现，其相似性不能作为抄袭证据，已降权处理。跨板块同名函数（如将 `sys_close` 归入任务调度）亦不构成同板块主证据。

#### 6. 保守结论
本次审计的客观 AST 相似度仅为 2.9%，双向覆盖率均低于 7%，表明两个项目在代码实现层面具有高度独立性。控方提出的“铁证”均可被合理解释为对 POSIX 标准、RISC-V 规范或 MIT xv6 等公开范式的共同遵循，此类函数无创造性表达空间，其相似性属于领域约束下的必然收敛。更为关键的是，双方在核心架构上存在根本差异：目标项目构建了 HAL 抽象层、VFS 框架并支持多文件系统，而参考项目则采用直接寄存器访问、easyfs/FAT32 后端及 RustSBI 引导。这些系统级的异构设计远非“换皮”或“宏替换”所能实现。综合判定，目标项目为独立开发，整体代码同源性风险极低。
