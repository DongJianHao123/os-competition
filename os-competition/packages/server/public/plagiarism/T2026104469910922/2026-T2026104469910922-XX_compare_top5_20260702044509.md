## 4. 核心代码调用流对比剖析

### 【进程管理 / 调度核心路径】

**底层技术析理**：
双方 `sched` 函数均实现 xv6 标准调度入口：检查锁持有状态 → 检查中断状态 → 获取当前进程指针 → panic 保护 → 调用 `swtch` 上下文切换。这是 xv6 教学内核的公开标准流程，控制流由调度语义决定。

关键差异在于进程/CPU 指针获取方式：
- **目标项目**：`get_cpu()` → `cpuid()` → `hal_get_cpu_id()`，采用硬件抽象层（HAL）风格，将 CPU ID 获取与进程指针获取显式分离。
- **参考项目**：`mycpu()` → `r_tp()`，直接读取 RISC-V `tp` 寄存器，采用内聚式实现，进程指针与 CPU 指针耦合在 `myproc()` 中。

这是架构抽象层次的差异，而非简单的文本替换。目标项目的 HAL 层设计使其具备更好的跨平台移植性，参考项目的内聚式设计则更紧凑。

**代码溯源链接**：
- 目标项目：`kernel/core/proc.c#sched`
- 参考项目：`kernel/proc.c#sched`

---

### 【任务调度/异步运行时 / UART 中断处理】

**底层技术析理**：
双方 `uartintr` 均实现 xv6 教科书标准 UART 中断处理流程：获取锁 → 从 UART 读取字符 → 交给控制台处理 → 释放锁 → 启动发送。该流程是 xv6 公开教学代码的标准范式，任何基于 xv6 的 RISC-V 内核都会采用。

关键差异：
- **目标项目**：位于 `devlog/2025_12/xv6_uart.c`，文件路径含开发日志标记，表明是独立移植学习 xv6 范式的产物。
- **参考项目**：位于 `kernel/uart.c`，为标准内核模块路径。

目标项目的文件组织方式（`devlog/` 目录）表明其 UART 驱动经历了独立的开发与调试过程，而非从参考项目直接复制。

**代码溯源链接**：
- 目标项目：`devlog/2025_12/xv6_uart.c#uartintr`
- 参考项目：`kernel/uart.c#uartintr`

---

### 【内存管理 / 用户空间拷贝】

**底层技术析理**：
双方 `copyin` 均实现用户空间到内核空间的数据拷贝，但页表遍历与地址转换策略不同：
- **目标项目**：使用 `PA2VA`/`PTE2PA` 宏链进行物理地址到虚拟地址的转换，页表遍历通过 `get_proc()` 获取进程页表基址后逐级 walk。
- **参考项目**：使用 `walkaddr` 函数进行页表遍历，返回物理地址后通过 `memmove` 直接拷贝。

目标项目的地址转换宏体系（`VA2PA`/`PA2VA`/`PTE2PA`）是其独立的内存管理抽象层设计，参考项目使用 `walkaddr` 单一函数。两者在内存模型上存在架构差异。

**代码溯源链接**：
- 目标项目：`arch/riscv/mm/vm.c#copyin`
- 参考项目：`kernel/vm.c#copyin`

### 【文件系统 / 文件关闭路径】

**底层技术析理**：
双方 `fileclose` 均实现文件关闭的标准流程，但底层文件系统架构根本不同：
- **目标项目**：调用 `vfs_iput`，通过 VFS（虚拟文件系统）抽象层释放 inode，支持 ext4、easyfs、devtmpfs 等多种文件系统。
- **参考项目**：调用 `eput`，直接操作 FAT32/easyfs 内聚式文件系统实现，无 VFS 抽象层。

这是文件系统架构的根本差异：目标项目采用 VFS + 多文件系统后端的设计，参考项目采用内聚式单文件系统设计。两者在文件系统层不存在复用基础。

**代码溯源链接**：
- 目标项目：`kernel/core/file.c#fileclose`
- 参考项目：`kernel/file.c#fileclose`

## 6. 保守结论

本案客观 AST 相似度仅 2.7%，双向覆盖率均低于 10%，共同 AST 指纹仅 19 个，而双方各自拥有超过 300 个唯一指纹。在操作系统内核这一高度规范化、大量继承 xv6 教学范式的领域中，该重合水平属于“领域共性噪声”范畴，远未达到“大规模直接复用”或“隐蔽式抄袭”的认定门槛。

被控重合函数集中于 POSIX 系统调用薄包装和 xv6 标准调度/同步原语，其结构相似性由接口语义和公开教学范式决定，而非相互复制。目标项目使用 `get_proc`/`get_cpu` 的 HAL 风格抽象和 `LOG_*` 结构化日志框架，与参考项目的 `myproc`/`mycpu` 内聚式实现和 `printf` 日志方式存在架构层面的根本差异。目标项目独立实现的 ext4 文件系统、devtmpfs、VFS 抽象层、VirtIO-blk 驱动等模块，在参考项目中完全不存在，构成独立开发的最强正面证据。

综合判定：双方为同一教学内核谱系下的独立演化项目，不存在足以认定抄袭的同源证据链。建议驳回控方全部指控。
