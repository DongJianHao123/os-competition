### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出与等待】**
* **底层技术析理**: 双方均实现了 `wait`/`exit` 语义，但底层模型截然不同。目标项目采用现代 Linux 内核风格的复杂退出链，涉及 `ptrace`、`futex`、`pidfd` 等多子系统回调，并使用哈希表管理等待队列，同步原语为 `spin_lock`/`arch_disable_interrupt`。参考项目采用教学 OS 的简单模型，直接遍历链表 (`list_traversal`)，通过 `thread_block` 实现阻塞，无任何现代内核特性。两者在对象管理模型和同步策略上存在代差。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/task_syscall.c#sys_wait4`
  - 参考项目: `userprog/wait.c#sys_wait`

* **【内存管理/内存映射】**
* **底层技术析理**: 目标项目的 `sys_mmap` 实现了完整的 VMA (Virtual Memory Area) 管理，调用链涉及 `alloc_mapping_vma`、`do_munmap_locked`、`check_address_space_limit` 等复杂地址空间操作。参考项目的 `sys_mmap` 仅调用 `do_mmap` 和 `get_page`，是简化的页级映射。两者在内存抽象层次（VMA vs. Page）和地址空间管理复杂度上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/mm/mm_syscall.c#sys_mmap`
  - 参考项目: `mm/mmap.c#sys_mmap`

* **底层技术析理**: 目标项目的 `sys_mount` 是一个完整的 VFS 挂载分发器，处理 bind mount、move mount、remount 等多种现代挂载标志，调用 `vfs_do_bind_mount` 等复杂逻辑。参考项目的 `sys_mount` 仅调用 `mount_fs` 和 `path_resolution`，是简单的文件系统挂载实现。两者在系统调用功能的完备性和 VFS 层的复杂性上完全不在一个量级。
