### 4. 核心代码调用流对比剖析
* **【进程管理 / 任务退出与调度】**
* **底层技术析理**: 目标项目采用全局运行队列锁 (`RUN_QUEUE.lock()`) 配合 `block_current` 进行任务阻塞，退出时通过 `EXITED_TASKS.lock()` 管理僵尸任务。参考项目则采用 per-CPU 免锁运行队列 (`current_run_queue::<NoPreemptIrqSave>`) 和 `blocked_resched`，退出时使用 `EXITED_TASKS.current_ref_mut_raw()`。两者在**同步策略**（全局锁 vs per-CPU 免锁）和**调度器抽象**上存在根本性架构差异，非简单“换皮”可达成。
* **代码溯源链接**:
  - 目标项目: `modules/axtask/src/run_queue.rs#exit_current` / `modules/axtask/src/wait_queue.rs#wait_until`
  - 参考项目: `arceos/modules/axtask/src/run_queue.rs#exit_current` / `arceos/modules/axtask/src/wait_queue.rs#wait_until`

* **【进程管理 / 时间与忙等待】**
* **底层技术析理**: 目标项目在 `busy_wait_until` 中使用 `monotonic_time` (单调时钟)，保证等待不受系统时间调整影响。参考项目则使用 `wall_time` (墙上时钟)。这一差异体现了对**时间语义**的不同设计选择，直接影响系统行为。
* **代码溯源链接**:
  - 目标项目: `modules/axhal/src/time.rs#busy_wait_until`
  - 参考项目: `arceos/modules/axhal/src/time.rs#busy_wait_until`

* **【内存管理 / 缺页异常处理】**
* **底层技术析理**: 目标项目的 `handle_page_fault` 直接尝试在 `KERNEL_SPACE` 中映射缺失页面。参考项目则构造 POSIX 信号 (`SignalInfo`)，并检查地址是否在用户栈范围内，走完整的信号递送流程。两者在**异常处理模型**（内核直接处理 vs 信号机制）上完全不同。
