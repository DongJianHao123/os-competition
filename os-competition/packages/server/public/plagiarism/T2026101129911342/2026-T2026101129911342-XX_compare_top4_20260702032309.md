### 4. 核心代码调用流对比剖析
* **【内存管理 / 物理帧分配】**
* **底层技术析理**: 双方均需实现物理内存分配，但采用了根本不同的分配器模型。目标项目 (`khal/src/riscv64.rs#frame_alloc`) 使用一个简单的 bump 分配器，通过原子加法 (`FRAME_NEXT.fetch_add`) 单调递增分配，永不回收，调用流为 `fetch_add -> assert! -> write_bytes`。参考项目 (`kernel/src/memory/frame_allocator.rs#frame_alloc`) 则实现了一个可回收的帧分配器，通过 `FRAME_ALLOCATOR.lock().alloc()` 进行分配，其内部涉及更复杂的回收管理逻辑。两者在内存模型（单调 vs 回收）和同步策略（原子操作 vs 锁）上存在根本性架构差异。
