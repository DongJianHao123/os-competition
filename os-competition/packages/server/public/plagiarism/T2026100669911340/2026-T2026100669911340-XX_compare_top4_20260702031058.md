## 4. 核心代码调用流对比剖析

### 【进程管理/上下文获取路径】

* **底层技术析理**: 目标项目 `get_kernel_context` 为 Cortex-M 平台上下文快照函数，标记为“内部闭环逻辑”，说明其通过内联汇编直接操作 `CONTROL`/`PSP`/`MSP` 寄存器，不依赖外部函数调用。参考项目 `copy_context` 同为“内部闭环逻辑”，但其操作的是 RISC-V `sscratch`/`sepc`/`sstatus` CSR 寄存器。两者虽然都是上下文操作，但**目标寄存器集完全不同**（ARMv7-M vs RISC-V Privileged ISA），汇编指令序列无任何可比性。目标项目进程模型为静态 `proc` 数组 + `proc_get_free_pid` 遍历分配，参考项目为 `allocproc` 动态分配 + `fork` 父子继承，进程生命周期管理模型根本不同。

* **代码溯源链接**:
  - 目标项目: `src/kernel/proc.c#get_kernel_context`
  - 参考项目: `proc.c#copy_context`

### 【任务调度/异步运行时/系统调用分发路径】

* **底层技术析理**: 目标项目 `trap_handler` 位于 `src/kernel/syscall.c`，调用图谱为 `printf, sys_brk, sys_close, sys_exit, sys_fstat, sys_openat, sys_read, sys_write`。这是一个**硬编码的显式分支分发器**，直接通过 `if/else` 或 `switch` 结构调用具体系统调用实现函数，无进程上下文参数（无 `myproc`），无函数指针表。参考项目 `syscall` 位于 `syscall.c`，调用图谱为 `NELEM, myproc, syscalls[num]`，通过**静态函数指针数组 `syscalls[num]` 进行间接查表分发**，依赖 `myproc()` 获取当前进程结构体。两者的分发机制在 AST 层面完全不同：一个是直接调用图（call graph 边指向具体函数），一个是间接调用图（通过数组索引 + 函数指针）。目标项目无 `syscalls` 表、无 `myproc`、无 `NELEM` 宏，参考项目无 `sys_openat`、`sys_fstat` 等显式调用。此外，目标项目运行于 Cortex-M 的 SVC 异常上下文（`SVC_Router` → `trap_handler`），参考项目运行于 RISC-V 的 `ecall` 指令陷阱上下文（`usertrap` → `syscall`），异常入口机制不同。

* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#trap_handler`
  - 参考项目: `syscall.c#syscall`

### 【内存管理/内存释放路径】

* **底层技术析理**: 目标项目 `kfree` 调用 `DATA_TO_BLOCK`、`SET_FREE`、`insert_free`、`try_merge`，这是典型的**伙伴分配器（Buddy Allocator）** 释放路径：将地址转换为块号，标记为空闲，插入空闲链表，尝试与相邻空闲块合并。参考项目 `kfree` 调用 `acquire`、`memset`、`panic`、`release`，这是典型的**空闲链表分配器**释放路径：获取自旋锁，清空内存内容（安全擦除），将页面插入空闲链表头部，释放自旋锁。两者在内存管理策略上存在根本分歧：伙伴系统通过 2^n 大小块的分裂与合并管理内存，空闲链表通过单链表管理固定大小页面。调用图谱无任何重叠。

* **代码溯源链接**:
  - 目标项目: `src/kernel/alloc.c#kfree`
  - 参考项目: `kalloc.c#kfree`

### 【文件系统/文件关闭路径】

* **底层技术析理**: 目标项目 `sys_close` 调用 `ext4_close`，直接进入 ext4 文件系统的 inode 释放与块回收路径，无参数解析步骤（参数已由上层 trap 框架解析）。参考项目 `sys_close` 调用 `argfd`（从当前进程文件描述符表解析 fd）、`fileclose`（减少引用计数，必要时释放文件结构）、`myproc`（获取当前进程结构体）。目标项目无文件描述符表抽象（或已内联），无 `myproc` 进程上下文依赖；参考项目强依赖进程结构体 `proc` 中的 `ofile` 数组。文件系统底层实现完全不同：ext4 extent 树 vs xv6 inode 直接/间接块。

* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_close`
  - 参考项目: `sysfile.c#sys_close`

### 【设备驱动/UART 初始化路径】

* **底层技术析理**: 目标项目 `uart_init` 调用 `uart_write`，说明其初始化过程包含直接寄存器写入操作（可能为 MMIO 方式配置 UART 控制寄存器）。参考项目 `uart_init` 调用 `initlock`（初始化自旋锁）、`uart_read_reg`、`uart_write_reg`（通过封装函数访问 UART 寄存器），说明其初始化包含锁机制初始化和间接寄存器访问。目标项目无锁初始化步骤（裸机/RTOS 环境可能使用关中断同步），参考项目强依赖自旋锁保护 UART 访问。两者在同步策略和硬件抽象层次上不同。

* **代码溯源链接**:

  - 参考项目: `uart.c#uart_init`

### 降权项说明

以下项目不作为强同源证据，已降权处理：

1. **C 标准库同名函数** (`memcmp`、`memcpy`、`strncmp`、`strchr`、`memmove`): 均为 C 标准库泛型函数，任何 C 项目均可独立实现，调用图谱均为“内部闭环逻辑”，不构成同源证据。

2. **POSIX 标准系统调用同名** (`sys_close`、`sys_exit`、`sys_read`、`sys_write`): 这些名称由 POSIX 规范定义，是操作系统领域的标准接口命名，同名是规范遵循的结果而非抄袭。双方实现调用图谱完全不同。

3. **UART 驱动同名函数** (`uart_init`、`uart_putc`、`uart_getc`): UART 是通用外设，这些函数名是行业惯例命名，双方调用图谱和同步策略不同。

4. **“内部闭环逻辑”标记**: 此标记仅表示函数无外部调用，是叶子函数或内联操作的正常现象，不能作为 AST 结构相同的证据。控方多次以“内部闭环”推断复用，属于以缺失证据作为证据的逻辑谬误。

5. **跨板块硬凑项**: 控方将 `RCC_WaitForHSEStartUp`（硬件时钟启动）与 `sys_clock_gettime`（系统调用获取时间）、`FLASH_ProgramWord`（Flash 写入）与 `_vsnprintf`（格式化字符串）、`CH340_Comm_Init`（USB-串口初始化）与 `trapframedump`（异常帧打印）等跨领域函数强行配对，已全部降权，不纳入主证据链。
