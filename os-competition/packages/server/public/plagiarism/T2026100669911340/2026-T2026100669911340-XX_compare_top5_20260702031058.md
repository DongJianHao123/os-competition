### 4. 核心代码调用流对比剖析
以下选取双方均存在的核心系统调用路径，论证其在架构设计上的根本性异构。

* **【系统调用/文件访问路径】**
* **底层技术析理**: 双方在文件访问的系统调用处理上展现了完全不同的抽象层次和对象管理模型。被告项目采用**薄封装层**模式，`sys_openat` 函数直接调用 `ext4_lookup` 和 `ext4_open` 等底层文件系统具体实现，路径短且紧耦合。原告项目则构建了完整的 **VFS（虚拟文件系统）层**，`sys_openat` 调用 `argstr` 解析路径，再通过 `ext4_dir_open`、`ext4_elock` 等一系列复杂的 inode 锁和目录缓存机制，最终通过 `fileexists` 和 `get_absolute_path` 完成操作。两者在内存抽象（直接操作 ext4 结构 vs 多层 VFS 对象）、同步策略（无锁 vs `ext4_elock` 细粒度锁）上存在本质差异。
* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_openat`
  - 参考项目: `kernel/src/fs/sysfile.c#sys_openat`

* **【任务调度/异常分发路径】**
* **底层技术析理**: 这是操作系统的核心脉搏，双方的实现截然不同。被告的 `trap_handler` 是一个**C语言层面的系统调用分发器**，其调用图谱为 `printf, sys_brk, sys_close, sys_exit...`，直接根据系统调用号跳转到具体的 `sys_*` 函数，是一个扁平的路由表结构。原告的 `kerneltrap` 是一个**RISC-V 架构相关的底层陷阱入口**，其调用图谱为 `r_scause, r_sepc, r_sstatus, devintr...`，它需要读取硬件 CSR 寄存器来区分中断、异常和系统调用，然后再进行分发。一个运行在高层抽象，一个紧耦合于特定 CPU 架构，AST 结构不可能同源。
* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#trap_handler`
  - 参考项目: `kernel/src/sys/trap.c#kerneltrap`

### 5. 单方模块与降权项
以下模块或证据因仅单方存在或属于公共资源，不构成同源证据，予以降权或排除：

- **网络协议栈（仅目标项目）**: 被告项目包含完整的 `secret_net` 加密串口通信协议栈（`Crypto_AES_GCM_Decrypt`, `Proto_BuildHandshake` 等），而原告项目无此模块。这是被告独立开发的强有力证据。
- **STM32 标准外设库（仅目标项目）**: 被告项目中 `Libary/stm32f10x_*.c` 路径下的所有函数（如 `RCC_WaitForHSEStartUp`, `GPIO_Init`）均来自 STMicroelectronics 官方标准外设库，是特定硬件平台的公共依赖，与原告项目无关。
- **第三方库 `lwext4`**: 双方文件系统模块中出现的同名函数（`ext4_mount`, `ext4_open` 等）均源于此开源库，属于公共依赖，必须从原创性比较中剔除。
- **标准 C 库函数**: `memcpy`, `atoi`, `strncmp` 等函数实现方式高度收敛，其 AST 相似性是算法必然性，不构成抄袭证据。
- **跨板块同名函数**: `main` 函数在裸机固件、Windows 应用和用户态程序中普遍存在，其同名不具备任何特异性，已降级为弱信号。

### 6. 保守结论
综合全模块 AST 指纹比对，双方代码库的客观相似度为 0.96%，覆盖率不足 3%。14 个共同指纹经拆解，全部指向标准 C 库、第三方开源库（lwext4）或 POSIX 通用接口，无一属于原告项目独有的私有逻辑或非标准宏。在进程管理、任务调度、内存管理等操作系统核心领域，双方的调用图谱、抽象层次和同步模型存在根本性架构差异。特别是被告独有的网络协议栈模块和基于 STM32 标准库的驱动层，进一步印证了其独立开发的特性。因此，现有技术证据无法支持“抄袭”或“系统性复用”的指控，应认定为两个独立开发的、面向不同硬件平台和设计目标的操作系统项目。
