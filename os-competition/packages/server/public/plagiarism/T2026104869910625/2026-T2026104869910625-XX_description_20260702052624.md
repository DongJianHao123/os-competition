## 第四部分：分模块架构说明

### MM（内存管理）
**证据较强**。核心实现分布在：
- `kernel/pmem.c`：buddy 页分配器（`buddy_safe_check`、`buddy_free_tracked`）
- `kernel/vma.c`：虚拟内存区域管理（`mmap`、`find_mmap_vma`）
- `kernel/exec.c`：ELF 加载与地址空间构建（`exec`、`alloc_vma_stack`、`loadaux`）
- `hsai/hsai_trap.c`：缺页处理（`pagefault_handler` 调用 `pmem_alloc_pages`、`mappages`）
- `kernel/syscall.c`：`sys_brk` → `growproc`

未体现：swap、页面置换、COW（写时复制）等高级策略的底层证据未明确。

### PROC（进程管理/任务调度）
**证据中等偏强**。传统 Unix-like PCB 与调度存在：
- `kernel/process.c#scheduler`：主调度循环，含上下文切换 `hsai_swtch`
- `kernel/process.c#exit` / `sched` / `yield` / `reparent`：进程生命周期
- `hsai/hsai_trap.c#forkret`：新进程首次返回，初始化 futex、文件系统挂载
- `kernel/futex.c#futex_wait`：futex 等待
- `kernel/signal.c` / `kernel/process.c#tgkill`：信号/线程组杀死
- `kernel/timer.c#refresh_process_timer`：进程计时
- `kernel/thread.c#alloc_thread`：内核线程/线程池分配

“异步运行时”底层证据未体现：无 task/channel/waker/ring_fifo/async-await 等轻量运行时符号，现有证据均为传统 scheduler + sleep + futex。

### FS（文件系统）
**证据强，但大量为第三方代码**：
- `kernel/fs/ext4*.c`（`ext4_balloc.c`、`ext4_ialloc.c`、`ext4_extent.c`、`ext4_dir_idx.c`、`ext4_journal.c`、`ext4_inode.c`、`ext4.c`、`ext4_fs.c`）构成完整 EXT4 实现，函数命名与 lwext4 等第三方库高度相似，应视为引入的第三方/公共依赖。
- `kernel/fs/vfs_ext4.c`、`kernel/fs/file.c`、`kernel/fs/fs.c`：VFS 胶水层与文件描述符管理。
- `kernel/syscall.c#sys_mount`、`sys_openat`、`sys_readv`：系统调用层对接。
- `kernel/fs/file.c` 与 `kernel/syscall.c` 中的 `busybox_virtual_*`：/proc 伪文件系统支持（oom_score_adj、stat、status 等）。

未体现：除 EXT4 与 /proc 伪文件外，其他文件系统（如 FAT、tmpfs、NFS）底层证据未体现。

### DEV（设备驱动）
**证据中等**：
- `hal/riscv/uart.c` / `hal/loongarch/uart.c`：串口驱动（`uart_init`、`uartputc`、`uartstart`）
- `kernel/console.c`：控制台字符设备（`chardev_init`、`consoleread`、`consoleintr`）
- `kernel/driver/loongarch/virtio_disk.c#la_virtio_disk_init`：VirtIO 块设备初始化
- `kernel/driver/riscv/virt.c#virtio_disk_intr`：VirtIO 磁盘中断
- `kernel/driver/loongarch/virtio_pci.c`：VirtIO PCI 配置访问

未体现：非 VirtIO 块设备、GPU、USB、网卡等驱动底层证据未体现。

### SYSCALL（系统调用）
**证据强**。`kernel/syscall.c` 为系统调用中枢：
- `syscall`：统一分发器，调用 `hsai_get_arg` 取参，分发至 110+ 个 `sys_*`
- 文件：`sys_openat`、`sys_read`、`sys_write`、`sys_pread`、`sys_readv`、`sys_lseek`、`sys_close`、`sys_dup3`、`sys_fchmod`、`sys_ftruncate`、`sys_fsync`、`sys_mount`
- 进程：`sys_kill`、`sys_brk`、`sys_getpid`、`sys_getppid`、`sys_setpgid`、`sys_getpgid`、`sys_getsid`、`sys_tkill`、`sys_epoll_pwait`、`sys_epoll_ctl`、`sys_listen`、`sys_ioctl`

### TRAP/IRQ（中断异常）
**证据强**：
- `hsai/hsai_trap.c#hsai_trap_init`：初始化 LoongArch/RISC-V 异常入口 CSR（`w_csr_eentry`、`w_stvec` 等）与定时器
- `hsai/hsai_trap.c#usertrap` / `kerneltrap`：用户态/内核态陷入处理
- `hsai/hsai_trap.c#pagefault_handler`：缺页异常
- `hsai/hsai_trap.c#hsai_usertrapret`：返回用户态
- `kernel/timer.c#timer_init` / `timer_tick`：时钟中断与调度计时
- `kernel/spinlock.c#push_off` / `pop_off`：中断屏蔽临界区

### NET（网络协议栈）
**底层证据未体现**。仅发现：
- `kernel/socket.c#sock_bind`（1 个函数，调用链仅 `memmove`）

无 socket 创建、bind/listen/accept 完整路径，无 TCP/UDP/IP/Ethernet 实现，无网卡驱动证据。

## 第六部分：边界与未体现内容

1. **网络协议栈**：仅发现 `kernel/socket.c#sock_bind`，无 socket 全生命周期、无 TCP/UDP/IP/Ethernet/网卡驱动证据，应判定为“底层证据未体现”。
2. **异步运行时**：模块画像含“任务调度/异步运行时”，但代码证据仅为传统 `scheduler` + `sleep_on_chan` + `futex` + `timer`，无 task/channel/waker/ring_fifo/async-await 等运行时结构，不能判定为实现了轻量异步运行时。
3. **第三方/公共依赖**：`kernel/fs/ext4*.c` 函数数量占绝对优势（534），命名与接口风格接近 lwext4 等第三方 EXT4 库，应单独说明；原创内核主体应重点考察 `vfs_ext4.c`、`syscall.c`、`file.c`、`process.c`、`hsai_trap.c` 等胶水层。
4. **未体现模块/特性**：
   - 高级内存管理：swap、页面置换、COW、NUMA 等无证据
   - 安全机制：LSM、SELinux、Capabilities 等无证据
   - 网络驱动与协议栈无证据
   - 非 EXT4 文件系统无证据
   - 多核 SMP 调度细节（如 IPI、核间负载均衡）证据不足
5. **用户目录性质**：`user/riscv/user.c`、`user/loongarch/user.c` 为测试/初始化用户程序，不属于内核主体。
6. **方向性说明**：本文档中“可能的系统方向”“接近 lwext4”等仅为基于代码路径的推断，不是最终身份、原创性或合规性结论；不做抄袭/违规判断。
