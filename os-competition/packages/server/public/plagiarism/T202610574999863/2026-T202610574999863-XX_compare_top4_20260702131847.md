#### 4. 核心代码调用流对比剖析

* **【进程管理/调度器核心路径】**
* **底层技术析理**: 目标项目`schedule`函数调用流为`proc_current -> proc_set_current -> proc_table_ptr -> switch_to`，直接操作全局进程表进行轮转。参考项目`__schedule`调用流为`get_current_task -> simple_sched_class.dequeue_task -> simple_sched_class.pick_next_task -> switch_to`，采用调度类抽象和运行队列模型。两者在同步策略上差异显著：参考项目包含`irq_disable`/`schedule_debug`/`schedule_tail`等完整关中断与调试尾操作，目标项目完全缺失这些路径，表明调度架构独立设计。
* **代码溯源链接**:
  - 目标项目: `kernel/src/sched.c#schedule`
  - 参考项目: `kernel/process/schedule.c#__schedule`

* **【任务调度/异步运行时/陷阱分发路径】**
* **底层技术析理**: 目标项目`trap_handler`调用流包含`csr_read`（直接CSR读取）、`loongarch_exc_name`/`riscv_exc_name`（双架构异常名解析）、`sched_tick`（时钟滴答）、`uart_put_hex`/`uart_puts`（原始UART输出）。参考项目`do_exception`调用流包含`is_interrupt_fault`（中断/异常判断）、`handle_plic_irq`（PLIC中断处理）、`handle_timer`（时钟处理）、`printk`（格式化输出）。目标项目无PLIC中断控制器交互，输出使用底层UART而非`printk`子系统，且支持双架构，硬件抽象层与日志系统完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/trap.c#trap_handler`
  - 参考项目: `kernel/trap/trap.c#do_exception`

* **【系统调用/分发与执行路径】**
* **底层技术析理**: 目标项目`syscall_handler`直接调用`sys_brk`、`sys_execve`、`sys_clone`等30余个内核态系统调用实现。参考项目`do_exception`虽调用同名`syscall_handler`，但其用户态系统调用通过`usr/user_syscall.c`中的`SYSCALL_1`、`SYSCALL_3`宏封装，如`sleep_proc`调用`SYSCALL_1`。两者系统调用ABI、封装层次和实现位置均不同，目标项目无参考项目的用户态syscall包装层。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall.c#syscall_handler`
  - 参考项目: `kernel/trap/trap.c#do_exception`

* **【内存管理/页表与分配路径】**
* **底层技术析理**: 目标项目`vmm_arch_enable`调用`csr_read`/`csr_write`/`tlb_flush_all`直接操作MMU和TLB，输出使用`uart_put_hex`/`uart_puts`。参考项目`FAT32_lookup`调用`alloc_pgtable`/`page_free_addr`/`read_FAT_Entry`，依赖FAT32文件系统元数据操作。两者内存管理对象模型完全不同：目标项目为直接页表硬件操作，参考项目为文件系统驱动的页面分配。

- **通用命名与公共接口**：`syscall_handler`、`switch_to`、`schedule`、`trap_init`等均为操作系统领域通用命名，不能作为同源证据。`内部闭环逻辑`是AST分析对无外部调用函数的通用标注，不代表代码相似。

#### 6. 保守结论

本次审计的两个操作系统内核项目，AST相似度仅0.96%，共同指纹仅3个，覆盖率均低于3%。所谓“复用铁证”实为通用控制流（PID查找闭环）、硬件原语巧合（`csr_write` vs `plic_switch_irq`）和标准命名（`syscall_handler`）的误读。双方在调度模型（直接进程表轮转 vs 调度类抽象）、内存管理（直接页表操作 vs FAT32驱动分配）、文件系统（VFS抽象 vs FAT32实现）、硬件抽象（双架构 vs 单架构+PLIC）和同步策略（无关中断锁 vs 完整锁子系统）等核心维度上存在根本性架构差异。目标项目独有的双架构支持、VFS层和缺失的设备驱动/锁机制，进一步支持独立开发结论。综合判定：无代码复用证据，两个项目为独立开发。
