#### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建与执行路径】**
* **底层技术析理**: 目标项目 `sys_execve` 通过 `elf_load` -> `proc_create_user` -> `vfs_find_inode` 的调用流，展现了以 VFS 和 ELF 加载器为核心的用户进程创建模型。参考项目 `init_pcb_exec` 则是一个内部闭环的 PCB 初始化函数，其进程创建逻辑分散在 `copy_parent_all_and_set_sp` 等函数中，基于内存拷贝和栈设置。两者在进程创建的抽象层次、文件系统交互方式和内存对象管理模型上完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall.c#sys_execve`
  - 参考项目: `kernel/sched/init.c#init_pcb_exec`

* **【任务调度/异步运行时/核心调度器路径】**
* **底层技术析理**: 目标项目 `schedule` 函数基于数组型进程表 `proc_table_ptr` 进行遍历，通过 `proc_set_current` 切换当前进程指针。参考项目 `do_scheduler` 基于双向链表 `list_head`，通过 `list_add_tail`/`list_del` 维护就绪队列，并使用 `list_entry` 宏获取 PCB。两者在就绪队列的数据结构（数组 vs 链表）、进程选择算法和状态管理模型上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/sched.c#schedule`
  - 参考项目: `kernel/sched/sched.c#do_scheduler`

* **【内存管理/物理内存初始化路径】**
* **底层技术析理**: 目标项目 `pmm_num_free` 是一个查询空闲物理页数量的内部闭环函数，体现了以物理页管理为核心的内存模型。参考项目 `directmap` 是建立直接内存映射的初始化函数，同样为内部闭环逻辑。两者虽同属内存管理模块，但一个服务于运行时查询，一个服务于启动时映射建立，在内存抽象和操作阶段上截然不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/pmm.c#pmm_num_free`
  - 参考项目: `arch/riscv/kernel/boot.c#directmap`

* **【系统调用/分发入口路径】**
* **底层技术析理**: 目标项目 `ksyscall6` 作为内核态系统调用分发器，通过 `syscall_handler6` 解析调用号并跳转。参考项目 `sys_clone` 是用户态库函数，通过 `invoke_syscall` 发起系统调用。两者分别位于系统调用的两端（内核分发 vs 用户封装），其 AST 结构相似性源于系统调用 ABI 的通用模式，而非内部实现逻辑的复用。
* **代码溯源链接**:

  - 参考项目: `tiny_libc/syscall.c#sys_clone`
