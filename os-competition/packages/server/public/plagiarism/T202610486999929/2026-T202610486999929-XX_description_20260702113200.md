## 第四部分：分模块架构说明

### MM（内存管理）

底层证据主要体现为内核堆分配与连续物理页帧管理：

- `crates/scarlet/src/mm/heap.rs#dealloc`：通过 `HEAP.0.lock()` 访问全局堆，使用 `buddy.free` 释放，并调用 `free_contiguous_frames`。
- `crates/scarlet/src/mm/heap.rs#mixed_alloc_dealloc`：测试/使用 `GLOBAL_ALLOC.alloc/dealloc`，说明存在全局分配器。
- `crates/scarlet/src/mm/memory_set.rs`：函数密集（81），是地址空间/内存集管理的主要文件。

未体现：完整页表遍历、TLB 刷新、按需分页（demand paging）、swap、COW 完整实现（仅在 `write_user_bytes_cow` 等名称中出现 COW 线索）。

### PROC（进程管理/任务调度）

有较多底层证据：

- `crates/scarlet/src/process/mod.rs#thread_new_all_types`：创建 `Process::new_user`、`Thread::new_idle/kernel/user`，说明存在进程/线程控制块。
- `crates/scarlet/src/sched/mod.rs#spawn_thread`：将线程加入运行队列，支持 `enqueue_on_hart` / `enqueue_rt_on_hart`，并设置 home hart。
- `crates/scarlet/src/sched/mod.rs#schedule`：包含 `__switch`、zombie 清理队列 `ZOMBIE_CLEANUP_QUEUES`、当前线程缓存切换，说明实现了上下文切换与调度循环。
- `crates/scarlet/src/syscall/sched.rs#sys_clone_fork` 与 `sys_execve`：实现了 fork/clone 与 execve 系统调用。

未体现：完整信号投递、进程组/会话、ptrace、fork 后写时复制细节、完整僵尸进程回收机制（仅有清理队列线索）。

### FS（文件系统）

证据最强：

- `crates/scarlet/ext4_rs_fixed/src/ext4_impls/balloc.rs#balloc_alloc_block_batch`：EXT4 块组、块位图、校验和、批量分配。
- `crates/scarlet/ext4_rs_fixed/src/ext4_impls/extents.rs#ext_grow_indepth` / `insert_new_extent`：extent 树深度增长与插入。
- `crates/scarlet/ext4_rs_fixed/src/ext4_impls/ialloc.rs#ialloc_alloc_inode`：inode 分配与位图操作。
- `crates/scarlet/ext4_rs_fixed/src/ext4_impls/file.rs#read_at` / `write_at`：基于 extent/block 的文件读写。
- `crates/scarlet/src/fs/ext4.rs#lookup` / `mkdir` / `create_file` / `rename` / `unlink` / `readdir` / `truncate`：VFS 层对 EXT4 的包装。
- `crates/scarlet/src/fs/ramfs.rs`、`crates/scarlet/src/fs/devfs.rs`、`crates/scarlet/src/fs/procfs/task_dir.rs`：ramfs、devfs、procfs 实现。
- `crates/scarlet/src/syscall/mount.rs#sys_mount`：挂载系统调用入口。

未体现：日志（journaling）/ext4 journal 恢复、块设备缓存策略、 quotas、ACL。

### DEV（设备驱动）

证据较弱，主要体现为抽象层：

- `crates/scarlet/src/fs/devfs.rs`：设备文件系统入口。
- `crates/scarlet/src/fs/ext4_adaptor.rs#write_offset`：块设备读写适配器，调用 `read_block` / `write_block`。
- `crates/scarlet/src/net/ethernet::send`：网络发送抽象。

未体现：具体磁盘控制器驱动（NVMe/SATA/SD）、网卡 MAC 驱动、UART/PLIC/CLINT 等中断控制器细节、DMA 管理。

### SYSCALL（系统调用）

证据强，覆盖 POSIX 风格系统调用：

- `crates/scarlet/src/syscall/fd.rs`：`sys_openat`、`sys_newfstatat`、`sys_readlinkat`、`sys_chdir`、`sys_mkdirat` 等。
- `crates/scarlet/src/syscall/io.rs`：`sys_read`、`sys_write`、`sys_pread64`、`sys_pwrite64`。
- `crates/scarlet/src/syscall/sched.rs`：`sys_clone_fork`、`sys_execve`、`sys_sched_getaffinity`、`sys_sched_setaffinity`。
- `crates/scarlet/src/syscall/socket.rs`：`sys_socketpair`、`sys_sendmsg`、`sys_recvmsg`、`sys_recvfrom`。
- `crates/scarlet/src/syscall/mount.rs#sys_mount`。

未体现：完整信号系统调用（sigaction/sigreturn 等）、mmap/munmap/mprotect 系列（`memory_set.rs` 存在但调用链未充分展开）、时间/时钟系统调用。

### TRAP/IRQ（中断异常）

证据集中在初始化与基本保护：

- `crates/scarlet/src/arch/riscv64/trap/init.rs#trap_init`：分配 trap page、写入异常入口、初始化 emergency stack。
- `crates/scarlet/src/arch/loongarch64/trap/init.rs#trap_init`：同功能，LoongArch64 版本。
- `crates/scarlet/src/mm/heap.rs#dealloc` 使用 `IrqGuard::new`，说明存在中断屏蔽原语。

未体现：完整异常分发处理函数、中断向量表、PLIC/CLINT/AIA 等中断控制器驱动、定时器中断处理。

### NET（网络协议栈）

证据中等：

- `crates/scarlet/src/net/tcp.rs#connect` / `receive`：维护 `TCP_CONNECTIONS` 全局表，处理 seq、payload。
- `crates/scarlet/src/net/arp.rs#send_request` / `send_arp_reply`：构造 ARP 请求/应答包。
- `crates/scarlet/src/syscall/socket.rs`：socketpair、sendmsg、recvmsg、recvfrom 系统调用。
- `crates/scarlet/src/net/ethernet::send`：以太网帧发送。

未体现：完整 IP 层（仅能从 TCP/ARP 推断其下存在 IP 支撑）、ICMP、UDP、路由表、TCP 状态机完整实现、校验和卸载、网卡接收路径。

## 第六部分：边界与未体现内容

### 没有底层证据支撑的模块/能力

- **完整中断/异常处理**：仅有 `trap_init` 初始化与 `IrqGuard` 屏蔽原语，具体异常分发、中断控制器（PLIC/CLINT/AIA）、定时器中断处理未体现。
- **原始硬件设备驱动**：无具体磁盘控制器、网卡 MAC、UART、时钟等驱动代码，仅有 `devfs`、`ext4_adaptor`、以太网发送抽象。
- **完整虚拟内存子系统**：`memory_set.rs` 存在但调用链未充分展开；`mmap/munmap/mprotect`、页错误处理、页表遍历、TLB 管理未直接体现。
- **信号机制**：仅在 `sys_read` 调用链中出现 `current_has_unblocked_signal`，完整信号投递、sigaction/sigreturn 未体现。
- **IPC（除 socketpair 外）**：管道（pipe）、消息队列、共享内存、信号量等未体现。
- **安全/权限框架**：仅见 inode mode/perm 位操作，无 capabilities、ACL、LSM 等证据。
- **网络协议栈深层模块**：IP 层、ICMP、UDP、路由表、TCP 完整状态机、校验和计算细节未直接体现。
- **日志与恢复**：EXT4 journal、fsck、崩溃恢复未体现。
- **构建/测试工具**：`crates/sakuya` 是 QEMU 测试与构建脚本工具，不属于内核运行时主体。
