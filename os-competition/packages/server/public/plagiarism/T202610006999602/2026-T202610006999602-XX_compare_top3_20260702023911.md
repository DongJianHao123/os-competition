### 4. 核心代码调用流对比剖析
* **【进程管理 / sys_clone 系统调用】**
* **底层技术析理**: 两者均实现了 `clone` 系统调用，但底层对象模型和内存管理策略截然不同。目标项目以 `TaskControlBlock` 为一级实体，在克隆时通过 `address_space.exclusive_access().memory_set.translate` 主动检查用户栈映射的有效性，体现了更严谨的安全设计。参考项目则以 `ProcessControlBlock` 为一级实体，直接操作其内部的 `tasks[0]` 数组，模型更简单。目标项目使用 `VirtAddr::from(stack - 1).floor()` 进行地址对齐，参考项目则无此操作。两者在核心流程上无共同调用链。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_clone`
  - 参考项目: `os/src/syscall/process.rs#sys_clone`

* **【内存管理 / fork 核心流程】**
* **底层技术析理**: 双方 `fork` 函数都调用了 `MemorySet::from_existed_user`，这源于同源的 rCore 教学内存管理组件。然而，目标项目的 `fork` 函数（位于 `task.rs`）调用链包含 60+ 个外部函数，引入了 `AddressSpaceState::new`、`FpuContext::new` 等复杂状态管理；参考项目的 `fork`（位于 `process.rs`）调用链约 30 个，包含 `RecycleAllocator::new` 等不同特性。目标项目以任务为中心管理资源，参考项目以进程为中心，这种架构差异导致其内存克隆的上下文准备和资源分配流程完全不同。
* **代码溯源链接**:
  - 目标项目: `os/src/task/task.rs#fork`
  - 参考项目: `os/src/task/process.rs#fork`

* **【文件系统 / easy-fs 底层布局操作】**
* **底层技术析理**: `easy-fs/src/layout.rs` 中的 `increase_size`、`write_at` 等函数在双方项目中 AST 结构几乎完全一致。这是因为 `easy-fs` 本身是一个独立的、公开的教学用 Rust 文件系统库，双方都将其作为外部组件引入。这种相似性属于对公共第三方库的直接使用，而非一方对另一方的代码复用。在更上层的文件系统适配层，目标项目实现了复杂的 `ext4` 适配，参考项目则使用 `fatfs`，证明双方在独立构建自己的文件系统生态。
* **代码溯源链接**:
  - 目标项目: `easy-fs/src/layout.rs#increase_size`
  - 参考项目: `easy-fs/src/layout.rs#increase_size`

### 5. 单方模块与降权项
*   **网络协议栈（仅目标项目）**: 目标项目包含 `os/src/net/` 目录，实现了完整的 TCP/UDP socket 管理、物理层数据包收发等功能，这是参考项目完全不具备的独立功能模块，是独立开发的强有力证据。
*   **第三方库（仅参考项目）**: 参考项目包含 `dependencies/k210-soc` 等大量平台特定第三方库，目标项目无此模块。这源于双方目标硬件平台（K210 vs LoongArch/RISC-V）的根本不同，不构成同源证据。
*   **跨板块同名函数（降权项）**: 如 `getpid`、`thread_count`、`add_initproc` 等函数，虽名称相同，但分属不同模块（`task.rs` vs `process.rs`），其内部实现因对象模型（`thread_group` vs `tasks` 数组）不同而存在本质差异。根据审计指令，此类跨板块同名只能作为弱信号，不构成强同源证据。
*   **公共接口与泛型函数名**: `sys_clone`、`sys_exit`、`fork`、`wait` 等均为 POSIX 标准或操作系统领域的通用概念，其命名不具备独创性。`Some`、`unwrap`、`clone`、`drop` 等为 Rust 标准库的惯用法，任何 Rust 项目都会高频使用，不能作为抄袭证据。

### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度仅为 2.33%，处于极低水平。双方在进程管理、内存管理、文件系统等核心模块虽有少量共同指纹，但几乎全部可归因于对 rCore 公共教学范式、POSIX 标准语义、Rust 语言惯用法以及 `easy-fs` 等第三方公共组件的共同遵循。在更深层的架构设计上，目标项目展现出以 `TaskControlBlock` 为核心、支持线程组、futex、ext4 及完整网络协议栈的独立演进路径，与参考项目以 `ProcessControlBlock` 为核心的简单模型存在系统性差异。因此，现有证据无法支持“抄袭”或“换皮复用”的指控，应认定为基于公共领域知识的独立开发项目。
