### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建】**
* **底层技术析理**: 双方在 `sys_clone3` 系统调用的实现上展现出根本性的架构差异。目标项目采用了更现代的 `pidfd` 机制，通过 `PF_TABLE` 全局表管理进程文件描述符，并处理 `Sig` 信号。参考项目则采用传统的 `sys_clone` 转发方式，直接操作子进程的 trap context (`child_trap_cx`) 来设置栈指针和线程指针。这体现了双方在进程创建抽象层次和资源管理模型上的不同设计哲学。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_clone3`
  - 参考项目: `os/src/syscall/process.rs#sys_clone3`

* **【任务调度/异步运行时/任务派发】**
* **底层技术析理**: 双方执行器的 `fetch` 方法虽然都从队列中取出任务，但调度策略完全不同。目标项目的 `fetch` 逻辑是简单的优先级优先：先尝试 `fetch_prio`，失败后再从普通队列 `pop_front`。参考项目的 `fetch` 则实现了更复杂的多级回退策略：优先从 `prior` 队列取，失败后从 `normal` 队列取，再失败后尝试 `fetch_idle`。这种调度算法的差异直接反映在 AST 调用链的复杂度上，是架构异构的明证。
* **代码溯源链接**:
  - 目标项目: `lib/executor/src/lib.rs#fetch`
  - 参考项目: `os/src/task/executor.rs#fetch`

* **【内存管理/用户态内存访问】**
* **底层技术析理**: 在 `sys_prlimit64` 系统调用中，双方访问用户空间内存的策略截然不同。目标项目使用了类型安全的 `UserReadPtr<RLimit>` 和 `UserWritePtr<RLimit>` 抽象，将权限检查和内存访问封装在安全接口内。参考项目则直接使用 `core::ptr::write` 等裸指针操作，依赖开发者手动保证安全性。这反映了双方在内存安全模型和抽象层次上的根本性差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_prlimit64`
  - 参考项目: `os/src/syscall/process.rs#sys_prlimit64`

### 5. 单方模块与降权项
* **单方特有模块/函数**: 目标项目中的 `lib/osfs/src/proc/*/inode.rs` 下的多个 `get_meta` 函数，以及参考项目中的 `os/src/fs/procfs/*/dir.rs` 下的多个 `metadata` 函数，均为各自 procfs 文件系统的内部实现，对方项目中不存在对应物。这些“内部闭环逻辑”无法形成跨项目证据。
* **公共依赖与泛型函数**: 双方共同使用的 `async_task::spawn`、`ext4_frename` 等属于对第三方 crate 或 C 库的依赖。`core::hint::spin_loop`、`VecDeque::push_back` 等属于 Rust 语言标准库的公共接口。这些项在 AST 层面的相似是**库 API 复用**，而非项目间代码复用，必须降权处理。
* **跨板块弱信号**: 任何跨模块的同名函数（如文件系统模块的 `ioctl` 与设备驱动模块的 `ioctl`）均不构成同板块主证据，仅能说明命名习惯相似，已在分析中排除。

### 6. 保守结论
本次审计基于 AST 调用图谱的客观比对，双方项目整体相似度仅 3.18%，处于独立开发的典型噪声区间。控方提出的“复用铁证”经技术剖析，均属于 Linux ABI 强制命名、Rust 语言标准库惯用法、同型号硬件驱动编程规范或第三方库 API 约束下的必然同构，而非原创逻辑的抄袭。双方在进程创建模型（pidfd vs trap context）、内存访问策略（类型安全指针 vs 裸指针）、调度器架构（双模块 vs 单模块多级队列）及网络协议栈实现（零共同指纹）等核心子系统上展现出根本性的设计差异。综上，现有证据无法支持“同源性”或“代码复用”的指控，目标项目应被认定为独立开发作品。
