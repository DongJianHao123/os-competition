### 4. 核心代码调用流对比剖析
* **【进程管理/execve 系统调用】**
* **底层技术析理**: 两者均实现了 `execve`，但文件加载与解析路径存在根本性架构差异。目标项目采用动态派发的 `<dyn File>::open` 结合 `Path::resolve_symlink_through` 进行路径解析，体现了 VFS 抽象层的多态设计。参考项目则硬编码了 `busybox` 和 `sh` 的路径字符串，并使用具体的 `FileReader` 类型。这表明两者在文件系统集成和可执行文件定位上采用了完全不同的设计策略。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_execve`
  - 参考项目: `os/src/syscall/process.rs#sys_execve`

* **【内存管理/fork 系统调用】**
* **底层技术析理**: 在 `fork` 的核心逻辑中，双方对用户空间内存的访问模型截然不同。目标项目使用泛型封装 `UserReadPtr<T>` / `UserWritePtr<T>` 来安全访问用户指针，这是一种带类型约束的抽象。参考项目则使用 `UserPtrRaw` 结合 `.ensure_read(vm)` / `.ensure_write(vm)` 的卫语句模式，这是一种更底层、与地址空间强绑定的模型。这两种内存安全抽象的设计哲学和实现路径是异构的。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/taskf.rs#fork`
  - 参考项目: `os/src/task/task.rs#fork`

* **【任务调度/执行器队列操作】**
* **底层技术析理**: 目标项目 `lib/executor/src/lib.rs` 实现了双队列调度，区分 `tasks` (普通) 和 `pritasks` (优先)，通过 `push_prio` 和 `fetch_prio` 独立管理。参考项目 `os/src/executor/mod.rs` 则使用单一 `queue`，通过 `push_preempt` (前插) 和 `pop_back` (后取) 在同一队列上模拟优先级。这是两种互斥的调度队列架构设计。
* **代码溯源链接**:
  - 目标项目: `lib/executor/src/lib.rs#push_prio`
  - 参考项目: `os/src/executor/mod.rs#push_preempt`

### 5. 单方模块与降权项
* **单方模块**: 无。双方覆盖的顶层模块类别基本一致，这是由操作系统内核的通用功能需求决定的。
* **降权项说明**:
  * **POSIX 系统调用同名函数**: `sys_getpid`, `sys_clone`, `sys_exit` 等函数名由 ABI 强制规定，同名是合规要求，不构成抄袭证据。
  * **Rust 惯用法 (Idiom)**: `iter().map().reduce()` 聚合、`VecDeque::push_back()`、`spin_loop` 自旋等是 Rust 标准库的通用编程模式，不能作为代码同源的指纹。
  * **硬件强制序列**: 设备驱动中 MMC 卡初始化、UART 寄存器配置等是硬件手册规定的标准操作序列，任何驱动实现都必然相同。
  * **第三方库调用**: 对 `ext4`、`smoltcp` 等第三方库的 API 调用相似，是库使用的正常现象。
  * **跨板块证据**: 控方将 `lib/mutex` (同步原语)、`lib/osfs` (文件系统) 等模块的证据强行归入“进程管理”板块，本庭不予采信，已按实际模块归属进行裁决。

### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度仅为 5.29%，远低于认定同源性所需的阈值。控方提出的“大规模复用”指控，其证据基础多为 POSIX 标准接口、Rust 语言惯用法、硬件强制序列及第三方库调用，这些均不能作为独立内核实现的原创性判别依据。相反，双方在核心系统调用（如 `execve`, `fork`）的实现、内存管理抽象（`UserReadPtr` vs `UserPtrRaw`）、任务调度架构（双队列 vs 单队列）等关键路径上，展现出清晰且根本性的设计差异。因此，本庭裁定，无证据表明目标项目对参考项目存在系统性抄袭或代码复用，两者为独立开发的内核实现。设备驱动模块的高相似度应关注其第三方代码来源的合规性，而非双方之间的抄袭关系。
