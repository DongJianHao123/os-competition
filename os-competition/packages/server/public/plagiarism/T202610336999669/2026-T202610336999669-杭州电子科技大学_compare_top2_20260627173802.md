## 4. 核心代码调用流对比剖析

### 4.1 【进程管理/系统调用路径】`sys_kill`

- **底层技术析理**：
  目标项目与参考项目在信号发送的系统调用实现上存在根本性架构差异。参考项目采用简单的 PID 定向信号模型：通过 `pid2process` 查找单个进程，直接操作 `process.inner_exclusive_access` 修改信号状态，使用 `SignalFlags::from_bits` 解析信号类型。目标项目则实现了更复杂的信号子系统：使用独立的 `StellarSig::from_raw` 信号抽象，支持按进程组查找（`get_tasks_by_pgid`），通过 `send_signal` 接口统一派发，并涉及 `current.inner.exclusive_session` 会话管理。两者在信号类型系统、进程查找策略、派发机制上均不同，调用图谱无重合。

- **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_kill`
  - 参考项目: `os/src/syscall/process.rs#sys_kill`

### 4.2 【进程管理/调度器路径】`schedule` / `take_current_task`

- **底层技术析理**：
  这是双方调度器架构差异最显著的证据。参考项目采用全局单例 `PROCESSOR` 模型，通过 `PROCESSOR.exclusive_access(file!(), line!())` 获取全局锁后操作 `take_current` 或 `current`，这是单处理器或全局锁模型的典型实现。目标项目采用 SMP 每 CPU 变量模型，通过 `cpu::current_cpu()` 获取当前 CPU 本地数据结构，然后访问 `cpu.current.take` 或 `cpu.current.as_ref`，无需全局锁。前者是集中式调度，后者是分布式每 CPU 调度，核心数据结构和同步策略完全不同。若目标项目是“换皮复用”，不可能将全局单例重构为 SMP 每 CPU 抽象而不引入大量新的 AST 结构。

- **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#take_current_task`
  - 参考项目: `os/src/task/processor.rs#take_current_task`

### 4.3 【内存管理/ELF加载路径】`MemorySet::from_elf`

- **底层技术析理**：
  双方在 ELF 加载逻辑上存在结构重合（59 个共同指纹中的核心贡献者），但映射策略存在差异。目标项目使用 `load_base` 偏移量计算加载地址（`(ph.virtual_addr() + ph.mem_size()) as usize + load_base`），参考项目直接使用虚拟地址（`(ph.virtual_addr() + ph.mem_size()) as usize`）。目标项目调用 `Self::new_bare` 创建空内存集，参考项目调用 `Self::new_process`。这些差异反映了不同的地址空间管理策略。部分相似性源于 ELF 格式规范（Program Header 遍历、`MapArea::new` 等）和 rCore 教学框架的共同影响。

- **代码溯源链接**:
  - 目标项目: `os/src/mm/memory_set.rs#from_elf`
  - 参考项目: `os/src/mm/memory_set.rs#from_elf`

### 4.4 【中断/异常处理/陷阱入口】`trap_handler`

- **底层技术析理**：
  双方陷阱处理入口存在结构重合，但功能扩展方向不同。目标项目的 `trap_handler` 包含网络中断轮询（`crate::net::poll_interfaces`）、TLB shootdown 处理（`crate::cpu::handle_tlb_shootdown`）、多核调度标记（`crate::cpu::set_need_resched`）等 SMP 和网络相关逻辑。参考项目的 `trap_handler` 包含信号检查（`check_signals_of_current`）、定时器检查（`check_timer`）、信号添加（`current_add_signal`）等单核信号处理逻辑。两者在陷阱分发的基础结构上相似（由 RISC-V 特权架构决定），但上层处理逻辑因功能需求不同而分叉。
