## 4. 核心代码调用流对比剖析

### 【进程管理/`sys_clone` 系统调用路径】

**底层技术析理**：
双方均实现了 Linux ABI 规定的 `sys_clone` 系统调用，但底层进程派生机制存在实质差异：

- **标志位解析**：目标项目使用 `CloneFlags::from_bits_truncate`（截断未知标志位），参考项目使用 `CloneFlags::from_bits(flags).unwrap`（遇未知标志位 panic）。错误处理策略不同。
- **任务注册**：目标项目调用 `add_task_local` + `register_task`（两步注册），参考项目仅调用 `add_task`（一步注册）。调度器接口不同。
- **克隆实现**：目标项目调用 `current.kernel_clone`，参考项目调用 `current_task.fork`。内核克隆入口的抽象层次不同。
- **额外检查**：参考项目使用 `clone_flags.contains` 进行标志位条件判断，目标项目无此步骤。

**代码溯源链接**：
- 目标项目：`os/src/syscall/process.rs#sys_clone`
- 参考项目：`kernel/src/syscall/process.rs#sys_clone`

### 【内存管理/`exit_current_and_run_next` 进程退出路径】

**底层技术析理**：
双方均实现了进程退出时的子进程回收和 INITPROC 收养机制，但同步原语和回收逻辑不同：

- **锁机制**：目标项目使用 `inner.exclusive_access`（互斥访问），参考项目使用 `acquire_inner_lock`（获取内部锁）。同步原语 API 不同。
- **子进程回收**：目标项目使用 `children_to_reparent.is_empty` 判断并批量转移子进程，参考项目逐个 `child.clone` 并 push 到 INITPROC 的子进程列表。回收策略不同。
- **INITPROC 访问**：目标项目通过 `INITPROC.gettid` 获取 TID，参考项目通过 `INITPROC.acquire_inner_lock` 获取锁。访问模式不同。

### 【中断/异常处理/`set_user_trap_entry` 陷阱入口设置】

**底层技术析理**：
双方均实现了用户态陷阱入口的设置，直接写入 RISC-V CSR 寄存器 `stvec`。这是 RISC-V 特权架构的**强制性硬件接口**——`stvec` 寄存器必须在陷阱处理初始化时被写入用户态陷阱处理函数的地址。任何 RISC-V 操作系统都必须执行此操作，不存在替代实现方式。目标项目额外设置了 `sscratch` 寄存器（`set_kernel_trap_entry`），参考项目未设置。

### 【文件系统/`make_pipe` 管道创建】

**底层技术析理**：
双方均实现了匿名管道的创建，返回读写端文件描述符。核心结构均为 `PipeRingBuffer` + 等待队列，但同步原语不同：

## 6. 保守结论

本次审计基于客观 AST 指纹数据（相似度 5.04%，共同指纹 85，覆盖率 8.0%/11.99%）和双方模块画像进行综合研判。目标项目与参考项目在进程管理、内存管理、文件系统等核心板块存在少量共同 AST 指纹，但均集中于 POSIX/Linux ABI 强制命名的系统调用包装层、RISC-V 架构强制性硬件接口（如 `stvec`、`sie` 寄存器操作）以及 rCore 架构的通用基础设施（如 `translated_refmut`、`UserBuffer`）。这些重合属于**接口语义强制收敛**和**公共架构模式**，不构成代码抄袭的实质性证据。

目标项目在锁机制（`exclusive_access`/`SpinNoIrqLock` vs `acquire_inner_lock`/`Mutex`）、时间表示（纳秒 vs 微秒）、进程派生（`kernel_clone` vs `fork`）、调度器接口（`sched_add_task` vs `add_task`）等底层实现细节上与参考项目存在**系统性差异**。更重要的是，目标项目独立实现了完整的**网络协议栈**（155 函数）和 **ext4 文件系统**（85 函数），这些模块在参考项目中完全缺失，是目标项目独立开发能力的有力证明。

综合判定：**双方为基于 rCore 公共架构的独立实现，不构成代码抄袭或系统性复用。**
