## 4. 核心代码调用流对比剖析

### 4.1 【进程管理/调度器架构】

**底层技术析理**:  
目标项目采用 **per-CPU 调度器架构**，通过 `cpu::current_cpu` 获取当前 CPU 的局部数据结构，调度入口 `run_tasks` 包含 `crate::net::poll_interfaces`（网络轮询）、`crate::timer::check_timer`（定时器检查）等扩展逻辑，任务切换使用 `__switch_load_only` 和 `__switch_with_hook` 支持 hook 机制。参考项目采用 **全局处理器数组架构**，通过 `PROCESSORS[hartid()].lock` 获取当前处理器互斥锁，调度入口仅包含 `fetch_task` 和 `__switch`，无网络轮询或 hook 机制。两者在调度器的抽象层次（per-CPU vs 全局锁数组）、扩展能力（hook 机制 vs 无 hook）、中断集成（网络轮询 vs 纯任务切换）上存在根本性架构差异。
