#### 4. 核心代码调用流对比剖析

* **【进程管理 / `sys_execve` 系统调用】**
* **底层技术析理**: 两者在程序加载的核心路径上展现出截然不同的设计哲学。目标项目采用**同步、基于VFS上下文**的加载模型，直接操作内核态VFS节点(`ctx.lookup` -> `inode.read_at`)读取ELF文件，随后跳转至测试框架(`runner::enter_user_elf`)。参考项目则采用**用户空间指针封装**模型，通过`UserReadPtr`安全地读取用户态内存中的路径和参数，体现了更严谨的用户/内核隔离设计。两者在内存抽象、对象管理模型上无任何重合。

* **底层技术析理**: 这是系统调用分发的核心枢纽，两者的实现路径完全异构。目标项目的`syscall`函数是一个**测试框架感知的状态机**，充斥着对`PARENT_TF`、`RUNNER_JB`等全局状态的检查和恢复，其目标是支持测试用例的连续执行与故障回跳。参考项目的`syscall`函数是一个**标准的trap上下文操作器**，通过`self.cx`（Context）对象规范地读取系统调用号、参数，并在调用返回后设置用户态寄存器（`set_user_a0`）和推进PC。两者在同步策略和核心职责上完全不同。
