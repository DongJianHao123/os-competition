## 4. 核心代码调用流对比剖析

### 【进程管理 / sys_wait4 系统调用路径】

**底层技术析理**：

目标项目 `sys_wait4` 采用**全局状态锁 + 调度器委托**模型：
- 通过 `WAIT_STATUS_PTR.lock` 获取全局等待状态指针
- 调用 `scheduler::try_run_child` 尝试运行子进程
- 调用 `scheduler::wait_for_child` 阻塞等待子进程退出
- 通过 `user_write_u32` 向用户态写入退出码

参考项目 `sys_wait4` 采用**进程向量直接操作**模型：
- 通过 `cprocess_vec[0].borrow` 直接借用进程控制块
- 通过 `cprocess.exit_code.unwrap` 获取退出码
- 通过 `inner.process.clone` 操作进程内部状态
- 大量使用 `debug!` 宏进行调试日志

**差异本质**：目标项目将等待逻辑下沉至调度器模块，配合其 LoongArch 基准测试 runner 的父进程跳转机制（`switch_to_parent` 中的 `core::arch::asm!` 与 `arch_shutdown`）；参考项目为通用教学内核的进程向量模型。两者在进程状态管理、同步原语、用户态回写机制上完全不同。

**代码溯源链接**:
- 目标项目: `kernel/src/syscall/mod.rs#sys_wait4`
- 参考项目: `kernel/src/sys_call/task/fork.rs#sys_wait4`

### 【进程管理 / sys_fork 系统调用路径】

**底层技术析理**：

目标项目 `sys_fork` 采用**调度器中心架构**：
- 通过 `scheduler::current_process` 获取当前进程
- 通过 `scheduler::fork_process` 完成进程创建、内存映射、上下文初始化
- 内存管理封装在调度器内，配合 `Sv39PageTable`、`BumpFrameAllocator`

参考项目 `sys_fork` 采用**进程-任务弱引用模型**：

- 通过 `Rc::downgrade` 建立弱引用
- 通过 `child_process.pmm.add_mapping_by_set` 直接操作物理页管理器
- 通过 `add_task_to_scheduler` 将任务加入调度队列

**差异本质**：目标项目将 fork 流程封装为调度器的原子操作，使用 `Arc` + `SpinMutex` 所有权模型；参考项目暴露了 `Rc` + `RefCell` 的内存映射细节。两者在抽象层级、所有权模型、内存管理子系统上完全不同。
