#### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建】**
* **底层技术析理**: 目标项目的 `sys_fork` 直接委托给 `scheduler::fork_process`，进程管理深度集成在调度器中。参考项目的 `sys_fork` 则通过 `current_task.fork` 创建一个新的 `TaskControlBlock`，然后调用 `spawn_user_task` 将其提交给异步执行器。前者是同步的函数调用链，后者是异步任务生成，两者在对象管理模型和并发模型上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/mod.rs#sys_fork`
  - 参考项目: `os/src/syscall/process.rs#sys_fork`

* **【任务调度/异步运行时/文件系统调用】**
* **底层技术析理**: 以 `sys_unlinkat` 为例，目标项目使用 `read_user_str` 从用户态拷贝路径后，直接调用 `ctx.unlink` 操作内存文件系统。参考项目则使用 `UserPtrRaw::new` 创建用户态指针，再通过 `abs_path_to_name` 和 `at_helper` 进行路径解析和权限检查，最后操作 VFS 层。目标项目通过 `ctx` 上下文对象封装操作，而参考项目通过独立的函数和 `UserPtrRaw` 安全抽象处理用户态内存，两者的安全模型和抽象层次完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/mod.rs#sys_unlinkat`

* **底层技术析理**: 目标项目的 `alloc` 函数实现了一个简单的 bump allocator，通过 `heap.next.saturating_sub` 和 `align_up_to` 管理堆指针。参考项目的 `alloc` 函数则基于 slab 分配器，通过 `SlabBlock` 和 `FrameAllocator` 进行多层内存管理。一个是线性增长的单向分配器，一个是面向对象的 slab 分配器，内存管理策略截然不同。
