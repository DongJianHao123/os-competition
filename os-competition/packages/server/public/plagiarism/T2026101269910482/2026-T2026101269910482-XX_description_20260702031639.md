## 第四部分：分模块架构说明

### MM（内存管理）

底层证据较强。核心机制包括：

- **页表**：`Sv39PageTable`（`kernel/src/memory/pagetable.rs#walk_create`），支持 Sv39 三级页表遍历与创建
- **物理页分配**：`BumpFrameAllocator`、`alloc_page`、`alloc_contiguous_pages`
- **堆分配**：`kernel/src/memory/heap.rs#alloc` 实现简单 bump/对齐分配
- **用户态映射**：`enter_user_elf` / `enter_user_elf_direct` / `run_disk_elf` 将 ELF segment 按页映射到用户地址空间
- **动态内存接口**：`sys_mmap`、`sys_brk`、`handle_user_page_fault` 均调用 `BumpFrameAllocator`、`Sv39PageTable`、`alloc_page` 完成按需映射
- **架构相关映射**：`kernel/src/arch/loongarch64.rs#map_page`、`test_la_paging`、`setup_kernel_page_table`

### PROC（进程管理/任务调度）

存在传统进程模型证据，但“异步运行时”证据不足：

- **PCB 与调度**：`kernel/src/task/scheduler.rs` 维护 `PROCESSES`、`CURRENT`、`PARENT_TF`、`PENDING_CHILDREN`、`WAIT_STATUS_PTR`
- **进程创建**：`fork_process` 创建新进程并复制用户页表（`copy_loongarch_user_pages`）；`sys_clone` / `sys_fork` 调用 `scheduler::fork_process`
- **进程退出与等待**：`exit_process` 设置 `Zombie`；`sys_wait4` → `wait_for_child` / `try_run_child`
- **执行新程序**：`sys_execve` → `ctx.lookup` 读取 inode → `enter_user_elf`
- **说明**：`runner.rs` 中的 `run_group`、`collect_cases` 等是测试用例编排逻辑，不是内核级异步运行时；未观察到 `waker`、`future`、`task queue` 等异步运行时原语

### FS（文件系统）

证据较强，分为三层：

- **VFS 抽象**：`kernel/src/fs/vfs.rs` 定义 `Inode`、`FileHandle`、`FileType`、`SeekFrom` 等
- **内存文件系统**：`kernel/src/fs/ramfs.rs#mkdir/rename/add/lookup` 提供 ramfs
- **系统调用上下文**：`kernel/src/syscall/context.rs` 负责路径解析（`normalize_path`、`parent_dir_and_leaf`、`lookup`）、文件安装（`install_file`、`install_file_at_root`、`install_minimal_procfs`）
- **ext4 只读解析**：`kernel/src/fs/ext4.rs#open/read_inode/resolve/read_dir/read_file/extent_lookup` 实现 ext4 元数据与数据读取，底层通过 `BlockDevice::read_sector` 访问 VirtIO 块设备
- **注意**：`ext4.rs` 被标记为“文件系统/第三方库”，其原创性边界需单独审查

### DEV（设备驱动）

VirtIO 块设备驱动证据较完整：

- **入口**：`kernel/src/driver/virtio_mmio.rs#init` / `init_pci` / `init_at` / `init_pci_legacy_io`
- **两种传输路径**：MMIO（`read_sector_mmio_once`）与 PCI legacy I/O（`read_sector_pci_legacy_io_once`）
- **DMA 描述符队列**：`VirtIoDma` / `VirtIoPciDma`，使用 `core::ptr::read_volatile`、`dma_fence` 进行内存屏障
- **块读取**：`read_sector` 统一调用 MMIO 或 PCI 路径

### SYSCALL（系统调用）

系统调用层是项目最厚的接口之一：

- **分发入口**：`kernel/src/syscall/mod.rs#syscall` 规模 6022，调用 64 个目标
- **文件类**：`sys_read/write/openat/close/lseek/pread64/pwrite64/readv/writev`
- **目录类**：`sys_getdents64/statx/fstat/fstatat/chdir/getcwd/renameat/unlinkat/mkdirat`
- **进程类**：`sys_clone/fork/execve/wait4`
- **内存类**：`sys_mmap/brk`
- **其他**：`sys_nanosleep/uname`
- **用户地址访问**：`read_user_str`、`read_user_str_array`、`user_write_bytes`、`user_write_u32`

### TRAP/IRQ（中断异常）

证据较弱，仅观察到异常/系统调用入口：

- `kernel/src/trap/mod.rs#handle_trap` → `handle_loongarch_trap`
- `kernel/src/trap/mod.rs#recover_loongarch_user_exception`
- `kernel/src/arch/riscv64.rs#init_trap`
- 功能：分发 syscall、处理用户页故障、runner 故障恢复
- **未体现**：中断控制器（PLIC/IOAPIC/LS7A）初始化、外部中断处理、定时器中断、 preemptive 调度

### NET（网络协议栈）

底层证据未体现。唯一相关函数为：

- `kernel/src/runner.rs#print_compat_netperf`：仅输出 netperf 兼容性报告字符串

不存在 socket、TCP/IP、NIC 驱动、协议栈分层等证据。

## 第六部分：边界与未体现内容

### 没有底层证据支撑的模块/特性

- **完整中断/IRQ 子系统**：仅见异常入口与系统调用分发，未见 PLIC/LS7A/IOAPIC、外部设备中断、定时器中断、 preemptive 调度
- **网络协议栈**：仅有一个 netperf 输出函数，无 socket、TCP/IP、NIC 驱动
- **异步运行时**：统计标签含“任务调度/异步运行时”，但代码中未见 `Future`、`Waker`、`async/await` 运行时原语，实际为测试运行器 + 传统进程调度
- **高级内存特性**：无 swap/换出、copy-on-write、共享内存、mprotect、madvise 等证据
- **进程间通信**：无 pipe、shm、signal、semaphore、message queue 等证据
- **安全与权限**：无用户/组权限检查、capabilities、SELinux 等证据
- **多核/SMP**：无 CPU 启动、核间中断、自旋锁以外的 SMP 证据
- **设备树/ACPI/固件抽象**：无相关解析层证据

### 需要单独说明的代码边界

- `kernel/src/fs/ext4.rs` 被模块画像标记为“文件系统/第三方库”，其 ext4 解析代码可能包含硬件协议约束或第三方实现，不应直接等同于原创内核主体
- `kernel/src/driver/virtio_mmio.rs` 属于 VirtIO 协议强约束代码，主要体现对标准队列/PCI 配置空间的操作
- `runner.rs` 大量函数用于竞赛测试用例（busybox、libcbench、lua、netperf 等）的兼容性与结果输出，是测试框架而非内核核心服务

### 判断性质说明

- “多架构支持（LoongArch64 / RISC-V64）”仅为基于 `arch/loongarch64.rs`、`arch/riscv64.rs`、`Sv39`、`LA_*` 环境变量的可能推断，不是最终架构范围结论
- “传统进程管理存在”基于 `fork_process`、`wait_for_child`、`Zombie` 等证据；“异步运行时存在”证据不足
- 本文档不做抄袭、违规或原创性最终判定，仅陈述可观察的技术事实
