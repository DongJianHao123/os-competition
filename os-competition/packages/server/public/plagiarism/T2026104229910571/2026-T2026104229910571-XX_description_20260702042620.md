## 第四部分：分模块架构说明

### MM（内存管理）

有中等强度证据，主要体现在用户态地址空间接口与写时复制机制：

- `kernel/kernel/src/syscall/mm.rs#sys_mmap`：实现 `mmap` 系统调用，处理 `MapFlags`、`MmapProt`、用户地址映射。
- `kernel/kernel/src/tasks/stack.rs#init_task_stack`：初始化任务用户栈，构造 `TrapFrame`、参数映射。
- `kernel/kernel/src/tasks/task.rs#cow_fork`：基于 `map_area.mtrackers` 实现 COW fork。
- `kernel/kernel/src/user/mod.rs#user_cow_int`：用户态访问触发的 COW 中断处理。
- `kernel/kernel/src/main.rs#dealloc`：页表帧回收，使用 `PAGE_TABLE_FRAMES` 与 `frame_unalloc`。

**未体现**：完整的物理页分配器（伙伴系统/SLAB）、内核堆分配器、页错误内核侧完整处理链、TLB/Cache 一致性管理的底层证据未体现。

### PROC（进程管理/任务调度）

证据较强，具备传统进程管理雏形：

- `kernel/kernel/src/tasks/initproc.rs#initproc` / `command_in_dir`：init 进程逻辑，调用 `add_user_task_with_cwd` 创建用户任务。
- `kernel/kernel/src/tasks/exec.rs#exec_with_process` / `resolve_interpreter` / `resolve_program`：ELF/解释器解析与程序加载。
- `kernel/kernel/src/tasks/task.rs#thread_exit` / `finish_process_exit` / `cleanup_resources_after_exit`：线程/进程退出、资源清理、向父进程发信号。
- `kernel/kernel/src/syscall/task.rs#sys_clone` / `sys_fork` / `sys_wait4` / `sys_exit_group`：Linux 风格进程控制。
- `kernel/crates/executor/src/executor.rs#run_ready_task`：异步运行时，维护 `TASK_QUEUE` 与 `TASK_MAP`。
- `kernel/kernel/src/watchdog.rs`：子任务预算/看门狗截止时间。

### FS（文件系统）

证据最强，呈现“VFS + 多后端”结构：

- `kernel/filesystem/fs/src/file.rs#open` / `async_write` / `getdents`：VFS 层统一文件操作，支持阻塞/非阻塞写。
- `kernel/filesystem/ramfs/src/lib.rs#mkdir` / `symlink`：内存文件系统后端。
- `kernel/filesystem/ext4fs/src/lib.rs#create` / `readat` / `seek` / `symlink`：ext4 后端，调用 `ext4_mknod`、`ext4_read` 等 C 接口。
- `kernel/filesystem/fs/src/fatfs_shim.rs#open`：FAT 文件系统适配层。
- `kernel/filesystem/procfs/src/lib.rs`：procfs，提供 `/proc` 静态文件、pid 目录、`meminfo`、`mounts`、`interrupts`。
- `kernel/kernel/src/syscall/fd.rs#sys_mount` / `sys_umount2`：挂载/卸载接口。

### DEV（设备驱动）

中等证据，以 VirtIO 与基础平台设备为主：

- `kernel/driver/kvirtio/src/lib.rs#enumerate_pci` / `init_mmio` / `virtio_device_probe`：VirtIO PCI/MMIO 枚举与设备探测。
- `kernel/driver/kvirtio/src/virtio_net.rs#recv`：VirtIO-Net 接收路径。
- `kernel/driver/ns16550a/src/lib.rs#init_driver`：NS16550A UART 初始化。
- `kernel/driver/general-plic/src/lib.rs#register_irq` / `try_handle_interrupt`：PLIC 中断认领/完成。
- `kernel/driver/kgoldfish-rtc/src/lib.rs#init_rtc`：Goldfish RTC。

### SYSCALL（系统调用）

证据较强，接口覆盖较广：

- `kernel/kernel/src/syscall/mod.rs#syscall`：中央系统调用分发器，按 `Sysno` 路由。
- `kernel/kernel/src/syscall/fd.rs`：文件描述符相关调用（open/read/write/mount/epoll/ioctl 等）。
- `kernel/kernel/src/syscall/task.rs`：进程/线程/futex 相关。
- `kernel/kernel/src/syscall/mm.rs`：mmap。
- `kernel/kernel/src/syscall/socket.rs`：socket 相关。
- `kernel/kernel/src/syscall/signal.rs` / `time.rs` / `shm.rs` / `sys.rs`：信号、时间、共享内存、系统信息。

### TRAP/IRQ（中断异常）

证据较弱，底层入口多来自第三方库：

- `kernel/crates/polyhal-trap/src/trap/loongarch64.rs#loongarch64_trap_handler`
- `kernel/crates/polyhal-trap/src/trap/aarch64.rs#handle_exception`
- `kernel/crates/polyhal-trap/src/trap/riscv64.rs#kernel_callback`
- `kernel/crates/polyhal-trap/src/trap/x86_64.rs#kernel_callback`
- `kernel/driver/general-plic/src/lib.rs`：PLIC 中断控制。
- `kernel/kernel/src/syscall/types/signal.rs#store_ctx` / `restore_ctx`：信号上下文保存/恢复。

**说明**：架构相关 trap 入口、页错误/非法指令分发逻辑主要位于 `polyhal-trap` 这一第三方/公共依赖库中，内核侧直接可见的异常处理代码有限。

### NET（网络协议栈）

**底层证据未体现**独立网络协议栈。现有证据仅为 Socket 系统调用封装：

- `kernel/kernel/src/socket.rs#new` / `readat` / `reuse` / `drop`：通过 `NET_SERVER` 获取/释放 TCP/UDP 连接。
- `kernel/kernel/src/syscall/socket.rs#sys_sendto` / `sys_recvfrom` / `sys_connect` / `sys_accept4`：Socket 系统调用入口。

`NET_SERVER` 的具体实现未在提供的证据中出现，因此不能判定为实现了自研 TCP/IP 协议栈。

## 第六部分：边界与未体现内容

1. **网络协议栈**：仅有 Socket 系统调用与 `NET_SERVER` 封装，独立的 TCP/IP 协议栈、ARP/IP/路由/拥塞控制等底层证据未体现。
2. **中断/异常底层实现**：架构相关的 trap 入口、异常分类、页错误初步分发主要位于 `kernel/crates/polyhal-trap` 第三方库；内核侧除 PLIC 和信号上下文保存外，具体异常处理链底层证据未充分体现。
3. **物理内存管理**：仅见 `frame_unalloc`、`PAGE_TABLE_FRAMES`、mmap、COW 等高层逻辑，完整的伙伴系统、SLAB/对象分配器、内核堆管理底层证据未体现。
4. **安全与权限**：`sys_capget` 等仅为系统调用封装，完整的 capability、ACL、SELinux 等安全机制底层证据未体现。
5. **多核/SMP**：`executor.rs#init` 提到 `cores`，但核间中断（IPI）、 per-CPU 调度、锁粒度等底层证据未体现。
6. **第三方/公共依赖**：`polyhal-trap`、`executor`、`ext4` C 绑定、`fatfs` 等应视为外部依赖或硬件协议强约束代码，不应夸大其为原创内核主体。
7. **模块加载/热插拔/电源管理/图形/音频**：底层证据未体现。
8. 本说明中的“可能的系统方向”“可能方向”仅基于 AST Call Map、函数名与目录结构的技术推断，不是对作品最终身份、原创性或参赛资格的结论。
