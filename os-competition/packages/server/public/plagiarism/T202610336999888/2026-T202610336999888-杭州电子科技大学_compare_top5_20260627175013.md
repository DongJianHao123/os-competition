### 4. 核心代码调用流对比剖析
* **【进程管理 / fork 实现】**
* **底层技术析理**: 两项目的 `fork` 实现展示了完全不同的进程创建模型。目标项目采用原地初始化大量原子变量（`AtomicBool::new`, `AtomicU32::new` 等）的方式来构建新进程状态，这是一种扁平化的、基于原子操作的模型。参考项目则采用工厂模式，调用 `create_process` 函数并传入 `generate_next_pid` 等参数，是一种结构化的、通过构造器创建对象的模型。两者在内存抽象和对象管理上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `examples/shell/src/uspace/process_lifecycle.rs#fork`
  - 参考项目: `process/src/process.rs#fork`

* **【进程管理 / 任务退出与电源管理】**
* **底层技术析理**: 在核心的 `exit_current` 调用流中，虽然任务队列清理逻辑（`EXITED_TASKS`, `WAIT_FOR_EXIT`）因共享 ArceOS 框架而一致，但在最终的硬件抽象层（HAL）调用上出现了明确分歧。目标项目调用 `axhal::power::system_off`，表明其设计目标是直接关闭系统电源。参考项目调用 `axhal::misc::terminate`，表明其设计是终止当前执行流并可能返回监控程序。这反映了双方在系统电源策略和运行环境上的不同设计考量。
* **代码溯源链接**:
  - 目标项目: `kernel/task/axtask/src/run_queue.rs#exit_current`
  - 参考项目: `arceos/modules/axtask/src/run_queue.rs#exit_current`

### 5. 单方模块与降权项
- **单方模块**: 无。所有宏观模块在双方项目中均存在。
- **降权项说明**:
  - **第三方库/公共依赖**: `ulib/axlibc` 下的 C 库函数（`printf`, `pow`, `string` 等）、`strftime` 等 Rust 库函数，以及 `smoltcp_impl` 下的网络协议栈封装，均属于对开源社区代码的直接复用或标准封装。其 AST 同构是上游代码一致性的体现，**不能作为项目间抄袭的证据**。
  - **公共接口与宏**: `syscall_body!` 宏、`sys_getpid` 等 POSIX 接口的简单封装，其相似性由 API 规范和框架宏决定，属于**领域命定效应**。
  - **桩函数**: `unimplemented!()` 桩函数是未完成功能的占位符，其 AST 结构在所有 Rust 项目中完全一致，**不构成任何实质性证据**。
  - **跨板块同名函数**: 控方已降级的 `fork`, `sys_execve` 等函数，虽同名但实现路径差异巨大，本庭确认其**不构成同源性证据**。

### 6. 保守结论
综合全模块 AST 指纹分析，目标项目与参考项目的整体相似度处于低位（18.82%）。在争议焦点的进程管理板块，双方共同指纹稀少且覆盖率低，核心相似性均可被“共享 ArceOS 上游基线”、“复用第三方开源库”及“操作系统领域惯例”合理解释。关键系统调用（如 `fork`, `execve`）的底层架构和调用流存在显著差异，而 `exit_current` 中 HAL 接口的不同更直接证明了双方进行了独立的平台适配。因此，本庭裁定，现有证据无法支持“抄袭”或“系统性代码复用”的指控，目标项目在进程管理板块的实现具备独立开发的原创性。整体项目风险极低。
