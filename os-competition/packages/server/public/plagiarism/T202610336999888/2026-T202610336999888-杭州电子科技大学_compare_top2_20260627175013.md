## 4. 核心代码调用流对比剖析

### 【进程管理 / 信号处理路径】
- **底层技术析理**: 目标项目 `sys_rt_sigprocmask` 采用 `all_application_signal_mask` + `clear_user_bytes` + `current_task_ext` + `ext.process.pending_exit_group` 的信号掩码管理模型，信号状态挂载于进程扩展结构；参考项目采用 `check_sigset_size` + `current().task_ext().thread_data().signal.with_blocked_mut` 的线程数据模型，信号掩码挂载于线程数据结构。两者在**信号子系统的抽象层次与数据归属**上存在根本性架构差异。
- **代码溯源链接**:
  - 目标项目: `examples/shell/src/uspace/signal_abi.rs#sys_rt_sigprocmask`
  - 参考项目: `api/src/imp/signal.rs#sys_rt_sigprocmask`

### 【进程管理 / 程序执行路径】
- **底层技术析理**: 目标项目 `sys_execve` 调用流为 `axtask::current().kernel_stack_top` → `context.enter_uspace` → `exec_program` → `make_uspace_context`，侧重**用户态上下文切换与内核栈顶操作**；参考项目调用流为 `arg.get_as_str` → `argv.get_as_null_terminated` → `args.to_vec`，侧重**参数解析与字符串处理**。两者在 execve 的实现策略上分属不同抽象层。
- **代码溯源链接**:
  - 目标项目: `examples/shell/src/uspace/process_lifecycle.rs#sys_execve`
  - 参考项目: `api/src/imp/task/execve.rs#sys_execve`

### 【进程管理 / 进程退出路径】
- **底层技术析理**: 目标项目 `exit_current`（run_queue）调用 `axhal::power::system_off` 执行系统关机，参考项目调用 `axhal::misc::terminate` 执行通用终止。两者在**HAL 层电源管理/终止策略**上选择了不同的抽象接口，反映独立的设计决策。
- **代码溯源链接**:
  - 目标项目: `kernel/task/axtask/src/run_queue.rs#exit_current`
  - 参考项目: `arceos/modules/axtask/src/run_queue.rs#exit_current`

### 【内存管理 / 进程克隆路径】
- **底层技术析理**: 目标项目 `sys_clone` 调用 `TaskInner::new` + `UserTaskExt::new` + `UspaceContext::from` + `all_application_signal_mask`，采用用户态上下文对象模型；参考项目调用 `Arc::default` + `CURRENT_DIR.deref_from(&process_data.ns).init_new` + `CURRENT_DIR.copy_inner`，采用命名空间目录复制模型。两者在**进程克隆的内存抽象与资源继承策略**上存在显著差异。
- **代码溯源链接**:
  - 目标项目: `examples/shell/src/uspace/process_lifecycle.rs#sys_clone`
  - 参考项目: `api/src/imp/task/clone.rs#sys_clone`

### 【文件系统 / procfs 挂载路径】
- **底层技术析理**: 目标项目 `procfs` 操作 `file_cpuinfo.write_at`、`file_meminfo.write_at`、`file_mounts.write_at` 等具体文件节点，采用**直接文件写入模型**；参考项目操作 `ProcInterruptsFile::new`、`ReadOnlyRamFileSystem::new`、`core_dir.create_readonly_file`、`proc_root.add_custom_node`，采用**只读文件系统构建器模型**。两者在 proc 文件系统的实现架构上完全不同。
- **代码溯源链接**:
  - 目标项目: `kernel/fs/axfs/src/mounts.rs#procfs`
  - 参考项目: `arceos/modules/axfs/src/mounts.rs#procfs`

## 5. 单方模块与降权项

### 仅参考项目存在的模块
- **第三方库（209 函数）**：包含 `axdriver_net/dwmac`（DWMAC 以太网驱动）、`visionfive2-sd`（昉·惊鸿 SD 卡驱动）等硬件特定驱动。目标项目未包含这些第三方库，说明双方在硬件支持范围与第三方依赖选型上存在显著差异，**不构成同源证据**。

### 降权项说明
以下证据类型因缺乏独创表达或属于公共基础设施，**降权为弱信号**，不作为同源性判定的核心依据：

1. **未实现桩函数**：`pthread_cond_wait`、`execve`、`wait3`、`waitpid` 等仅调用 `unimplemented` 的函数，属于标准脚手架代码，任何基于 ArceOS/axlibc 的项目均呈现相同形态。
2. **宏生成代码**：`sys_getpid`、`sys_epoll_wait`、`sys_socket` 等仅包含 `syscall_body!` 或 `debug!` + `syscall_body!` 的函数，其 AST 指纹由宏展开决定，非手写逻辑。
3. **trivial 字段访问封装**：`set_in_wait_queue`（`self.in_wait_queue.store`）、`in_wait_queue`（`self.in_wait_queue.load`）、`notify_exit`（`self.exit_code.store` + `self.wait_for_exit.notify_all`）等单行/双行函数，属于对原子变量的标准操作封装。
4. **上游官方工具链**：`raspi4/chainloader` 的 `print_jtag_info_and_wait_forever`、`panic_prevent_reenter` 以及 `bwbench_client` 的 `transmit_benchmark`、`receive_benchmark`，均为 ArceOS 官方仓库的标准工具，非参考项目原创。
5. **跨板块同名函数**：`kernel/fs/axfs/src/mounts.rs#procfs` 属于文件系统模块，`tools/raspi4/chainloader/src/main.rs#print_jtag_info_and_wait_forever` 属于设备引导工具，`ulib/axlibc/c/pow.c#exp_inline` 属于数学库，均被控方强行归入“进程管理”板块，**不构成同板块主证据**。

## 6. 保守结论

双方项目共同 AST 指纹（904 个，相似度 24.12%）主要源于共享 ArceOS 开源内核框架的上游代码库，包括 `axtask` 调度核心、`axlibc` C 标准库、`arceos_posix_api` 系统调用层、`smoltcp_impl` 网络协议栈封装以及 `raspi4/chainloader` 官方工具链。目标项目在进程管理模块中 79.85% 的函数具有唯一 AST 指纹，总体覆盖率不足 50%，独立开发内容占主导。双方在信号处理架构（进程扩展 vs 线程数据）、execve 实现策略（用户态上下文切换 vs 参数解析）、HAL 终止路径（`system_off` vs `terminate`）、procfs 构建模型（直接写入 vs 只读构建器）等核心子系统上呈现实质性架构差异。控方将 trivial 封装、未实现桩函数、宏生成代码及跨板块同名函数作为“系统性抄袭”证据，在软件相似性比对方法论上存在根本性缺陷。综合判定：**目标项目不构成对参考项目的恶意代码复用或抄袭，双方为基于同一开源框架的独立开发作品**。
