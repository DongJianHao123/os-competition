### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 两者均遵循“获取锁 -> 清理资源 -> 唤醒父进程 -> 调度”的通用模式，但资源清理的**对象管理模型**和**同步策略**存在显著差异。目标项目的`kexit`通过`begin_op`和`end_op`包裹文件系统操作，表明其将进程退出纳入全局日志事务管理，以保证文件系统一致性。参考项目的`exit`则直接进行`fileclose`和`iput`，无事务概念。此外，参考项目在`exit`内部直接调用`reparent`和`sched`，而目标项目将这些职责分离到其他函数中，体现了不同的内核模块化分解策略。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#kexit`
  - 参考项目: `proc.c#exit`

* **【文件系统/inode释放路径】**
* **底层技术析理**: 这是双方架构差异最悬殊的核心路径。参考项目的`iput`仅实现简单的引用计数递减和睡眠锁释放，是一个轻量级的内存操作。目标项目的`iput`则深入磁盘I/O层，涉及`bread`（读块）、`bwrite`（写块）、`ext4_itrunc`（ext4文件截断）和`iupdate`（更新inode元数据）。这证明目标项目实现了一个**有日志支持的、完整的ext4文件系统**，其`iput`语义和复杂度远超参考项目，绝非“换皮”或“伪装”所能实现。
* **代码溯源链接**:
  - 目标项目: `kernel/fs.c#iput`
  - 参考项目: `fs.c#iput`

### 5. 单方模块与降权项
- **单方模块（目标项目独有）**：`文件系统/第三方库`模块下的`ext4_write.c`、`ext4_bmap`等16个函数是目标项目独立开发的**铁证**。这些函数实现了复杂的ext4磁盘布局和写操作，参考项目完全不具备此能力，直接否定了“系统性抄袭”的指控。
- **跨板块硬凑（降权处理）**：控方将`fs.c#iput`、`trap.c#kerneltrap`、`console.c#consoleread`等函数归入“任务调度/异步运行时”板块，属于严重的**跨板块硬凑**。这些函数实际分属文件系统、中断处理和设备驱动，其与参考项目的相似性不能作为“任务调度”板块的同源证据，必须降权为弱信号。
- **公共接口/泛型函数名（降权处理）**：`printf`、`memset`、`memmove`、`bread`、`brelse`等函数是C标准库或操作系统内核中最基础、最通用的接口。它们的调用图谱高度确定，在任何独立实现中都会出现，不能作为抄袭证据。
- **内部闭环逻辑（降权处理）**：目标项目的`sys_mprotect`、`sys_munmap`与参考项目的`sys_sched_getscheduler`等函数虽同为“内部闭环逻辑”，但分属内存管理和调度两个完全不同的功能域。将它们并列比较是**无关联的硬凑**，不构成任何同源证据。

### 6. 保守结论
本次审计的宏观AST相似度仅为10.27%，远未达到可判定“系统性复用”的阈值。双方共有的AST指纹高度集中于操作系统内核教学的标准原语和硬件架构强制的操作序列，这可用“共同公开架构基线”（如MIT xv6）和RISC-V平台特性合理解释。目标项目在文件系统（特别是ext4实现）、进程退出模型、中断处理路径等核心子系统上展现出与参考项目截然不同的架构选型、实现复杂度和模块化分解策略。特别是目标项目独有的ext4文件系统模块，是其独立开发能力的有力证明。综上，现有证据链支持双方为独立开发的结论，整体代码同源性风险低。
