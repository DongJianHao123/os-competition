### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**

* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#kexit`
  - 参考项目: `kernel/proc.c#exit`

* **【任务调度/自旋锁原语】**
* **底层技术析理**: 双方均实现了 `acquire` 自旋锁函数，但底层原子操作指令集选择不同。目标项目使用 C11/GNU 现代原子内建函数 `__atomic_exchange_n`，而参考项目使用旧版 `__sync_lock_test_and_set` 和 `__sync_synchronize` 组合。这不仅是语法糖替换，更反映了编译器目标标准（C11 vs. 旧版 GNU）和内存序模型的独立选择，是底层实现路径分化的明确证据。
* **代码溯源链接**:
  - 目标项目: `kernel/spinlock.c#acquire`
  - 参考项目: `kernel/spinlock.c#acquire`

* **【文件系统/第三方库】**
* **底层技术析理**: 这是双方架构差异最大的模块。目标项目在 `kernel/ext4_write.c` 中实现 ext4 文件系统，调用流为 `bread -> brelse -> bwrite` 的传统 Unix 缓冲区缓存模型。参考项目则依赖 Rust 第三方库 `third/ext4_rs`，调用流为 `Block::load -> block.sync_blk_to_disk` 的面向对象模型。两者在语言生态、内存管理、抽象层次上完全不同，AST 指纹为 0，是独立开发的最有力证据。
