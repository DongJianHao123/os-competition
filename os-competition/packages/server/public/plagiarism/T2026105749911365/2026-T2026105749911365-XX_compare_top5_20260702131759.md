### 4. 核心代码调用流对比剖析
* **【进程管理/调度原语】**
* **底层技术析理**: 双方 `scheduler` 函数的调用流均为 `acquire -> intr_on -> mycpu -> release -> swtch`。此调用链并非某一项目的独创表达，而是实现一个基于自旋锁保护、关中断上下文切换的 Round-Robin 调度器的**唯一正确路径**。`acquire`/`release` 保护进程表，`intr_on` 恢复中断，`swtch` 执行上下文切换，任何独立开发者遵循 xv6 教科书模型都会写出逻辑相同的代码。真正的差异性应体现在调度算法、优先级、负载均衡等扩展上，而非此骨架流程。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#scheduler`
  - 参考项目: `kernel/proc.c#scheduler`

* **【文件系统/路径解析】**
* **底层技术析理**: 双方均有 `dirlookup` 函数，但底层文件系统架构截然不同。目标项目调用 `ext4_namecmp`、`iget` 等 ext4 特定接口，操作的是 inode 和 ext4 目录项；参考项目调用 `edup`、`eget`、`reloc_clus` 等 fat32 特定接口，操作的是 FAT 表项和簇。两者的**对象模型（inode vs. FAT entry）、内存管理（缓冲区缓存 vs. 簇管理）和同步策略（ext4日志 vs. fat32链锁）**完全不同，这是证明双方独立开发的最有力证据。
* **代码溯源链接**:
  - 目标项目: `kernel/fs.c#dirlookup`
  - 参考项目: `kernel/fat32.c#dirlookup`

* **【任务调度/同步原语】**
* **底层技术析理**: 双方 `acquire` 自旋锁函数的核心逻辑均包含 `holding` 检查、`push_off` 关中断和 `panic`。然而，目标项目使用 C11 标准的 `__atomic_exchange_n` 实现原子交换，而参考项目使用 GCC 传统的 `__sync_lock_test_and_set` 并显式调用 `__sync_synchronize` 内存屏障。**原子操作和内存顺序模型的选择差异**，是底层实现独立性的明确标志，驳斥了“逐字复制”的指控。
* **代码溯源链接**:
  - 目标项目: `kernel/spinlock.c#acquire`
  - 参考项目: `kernel/spinlock.c#acquire`

- **降权项**: 所有标记为“同板块同名”的微小函数（如 `sys_getpid`、`cpuid`、`kkill`），其调用图谱由功能唯一确定，属于**功能必然相似**，不构成独创性表达重合。`syscall` 函数中的 `syscalls[num]` 分发模式是所有系统调用表的通用实现范式。

### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度极低（9.46%），远未达到可判定为“抄袭”或“大规模复用”的阈值。控方所指控的“铁证”均属于基于 xv6 教学内核与 RISC-V 硬件规范的**公共架构收敛**，其调用图谱的相似性由功能正确性约束和硬件接口规范决定，而非对特定原创代码的复制。辩方成功论证了双方在核心创造性子系统（尤其是文件系统 ext4 vs fat32）上的**根本性架构差异**，以及在底层并发原语（原子操作）上的独立实现选择。因此，现有证据不足以支持控方指控，目标项目应被认定为独立开发，整体同源风险为**极低**。
