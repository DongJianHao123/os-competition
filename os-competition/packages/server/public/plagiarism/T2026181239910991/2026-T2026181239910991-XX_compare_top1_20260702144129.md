## 4. 核心代码调用流对比剖析

### **【进程管理 / Fork 语义实现路径】**

- **底层技术析理**: 
  目标项目的 Fork 相关逻辑位于用户态 Perfetto 追踪工具 `extract_process_fork` / `emit_fork_flow`，其调用流为：从 JSON 负载中提取 `p["child_pid_low"].as_u64` 和 `p["parent_pid_low"].as_u64` → 通过 `compute_flow_id` 生成追踪流标识 → 调用 `self.packets.push` 序列化数据包。这是一个**追踪数据反序列化-序列化**的管道。

  参考项目的 Fork 实现位于内核态 `sys_fork`，其调用流为：`CloneFlags::from_bits(flags as u32).unwrap` 解析克隆标志 → `current_process.fork` 触发内核进程控制块复制 → `new_process.getpid` 获取新进程标识。这是一个**内核进程创建**的原语。

  两者在抽象层次（用户态追踪 vs 内核态进程创建）、操作数据对象（JSON 字段 vs 内核 PCB）、外部依赖（`serde_json` vs `CloneFlags`）上完全不同。唯一的概念交集“父子 PID”是操作系统领域的通用概念，不构成结构同源。
