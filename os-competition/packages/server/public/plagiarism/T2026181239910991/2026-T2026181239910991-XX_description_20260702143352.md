## 第四部分：分模块架构说明

### MM（内存管理）
**有证据的内容**：板级页表初始化路径清晰，RISC-V 侧 `boards/tx-hal-riscv64-qemu-virt/src/pmap/mod.rs#map_kernel_high_alias` 构造 leaf/branch PTE；LoongArch 侧 `boards/tx-hal-loongarch64-qemu-virt/src/la64_pmap.rs#ensure_la64_kernel_pgdh_bootstrap_mapped` 设置内核 PGD 映射；`crates/tx-subsystems/src/vm/structure/recipe_tree.rs` 为高密度 VM 结构文件。

**未体现的内容**：用户态缺页异常处理、物理页分配器/SLAB/伙伴系统、`mmap/munmap/brk` 等系统调用、swap/页回收机制等底层证据未体现。

### PROC（进程管理/任务调度）
**传统进程管理**：底层证据未体现。AST Call Map 中的“进程管理”函数主要集中在 `xtask` 静态检查脚本、`tools/tx-trace-daemon` 的 Perfetto fork 流记录、以及 `tools/netfast` 的 `execvp/fork/waitpid` 探测程序，这些均非内核运行时实现。内核侧虽有 `crates/tx-subsystems/src/process/structure.rs` 与 `init_process_subsystem` 调用，但无 fork/exec/wait/zombie/PCB/父子进程关系等细节证据。

**任务调度/异步运行时**：有证据。`crates/tx-kernel/src/init/reactor_submit.rs#set_thread_reactor_affinity` 操作全局 `BOOT_REACTOR`，调用 `set_task_affinity` 与 `online_cpus`，表明存在基于 reactor 的轻量任务调度/异步运行时，而非传统进程调度器。

### FS（文件系统）
**有证据的内容**：VFS 抽象与挂载子系统较完整。`crates/tx-subsystems/src/mount/mod.rs#bootstrap_mount` 与 `#bootstrap_mount_registers_with_parent_mount_payload_key` 构造 `MountIdentity`、`DEntry`、`InodeMeta`、`FsObjectId`；`crates/tx-fs/src/procfs/mod.rs` 实现 procfs 的 `readdir`/`load_inode_meta`；`crates/tx-fat/src/namespace.rs` 实现 FAT namespace 的 `create_inode`；`crates/tx-fs/src/tx_ext4_bridge.rs` 提供 ext4 块读写桥接；`crates/tx-shims/src/linux_syscall/fs_*.rs` 封装了 `openat`、`name_to_handle_at`、`mount`、`syncfs`、`getdents64`、`fallocate`、`fsync` 等 Linux 兼容调用。

**未体现的内容**：日志型文件系统（如 ext4 journal）、块缓存/IO 调度、文件锁/fcntl 完整语义等未在证据中展开。

### DEV（设备驱动）
**有证据的内容**：`crates/tx-drivers/src/virtio/blk.rs#read_blocks_bootstrap` / `#write_blocks_bootstrap` 提供 virtio-blk 块读写；板级 UART（`uart_put_byte`、`uart_try_get_byte`）与 IRQ 分发表（`dispatch_irq`、`la64_pch_pic_write_u32`）有实现。

**未体现的内容**：PCI 枚举、DMA 子系统、定时器/时钟、RTC、GPU、音频、网络设备驱动（除 virtio 与 net 配置外）等底层证据未体现。设备驱动模块整体证据较窄。

### SYSCALL（系统调用）
**有证据的内容**：`crates/tx-shims/src/linux_syscall/` 下存在大量 Linux ABI 兼容封装，如 `fs_basic.rs#sys_openat`、`fs_handle.rs#sys_name_to_handle_at`、`fs_mut.rs#sys_mount`、`socket.rs#sys_getsockopt` / `#sys_socket_ioctl`；`crates/tx-kernel/src/trap.rs#try_direct_trap_syscall` 提供直接陷入式系统调用入口。

**未体现的内容**：进程控制类（`fork`、`execve`、`waitpid`、`exit`）、时间类、信号发送/处理等系统调用底层证据未体现。

### TRAP/IRQ（中断异常）
**有证据的内容**：RISC-V 与 LoongArch 板级均有 trap frame 抽象（`view_mut`）、异常栈转储（`console_write_stack_dump`）、非对齐访问模拟（`emulate_user_unaligned`）、IRQ 分发表（`dispatch_irq`）及 PCH PIC 掩码操作（`la64_pch_pic_write_u32`）。

**未体现的内容**：通用中断控制器抽象层、中断线程化/底半部机制、性能计数器异常等未在证据中体现。

### NET（网络协议栈）
**有证据的内容**：项目自身实现了 netlink 配置层（`rtnetlink.rs#handle_newlink/newaddr/newneigh`、`nfnetlink.rs#handle_newchain/newrule`）、网络 namespace（`namespace.rs`）、socket 系统调用封装（`sys_getsockopt`、`sys_socket_ioctl`）。

**需单独说明**：网络协议栈主体为第三方 `external/smoltcp-asterinas`（4,136 个函数），项目自身网络核心代码仅 63 个函数，应区分原创实现与外部依赖。

### 没有底层证据支撑的模块/能力
1. **传统进程管理**：`fork`/`execve`/`waitpid`/`exit`、进程控制块（PCB）、父子进程关系、进程组/会话、僵尸进程处理等底层证据未体现。
2. **完整设备驱动框架**：除 `virtio-blk` 与板级 UART/IRQ 外，PCI 枚举、DMA、定时器/时钟、RTC、GPU、音频、其他存储控制器等未体现。
3. **内存管理高级机制**：物理页分配器、SLAB/伙伴系统、用户态缺页处理、`mmap/munmap/brk`、swap/页回收等未体现。
4. **安全/权限/LSM**：除 `fs_handle` 中少量 capability 检查外，完整的安全模块、LSM、审计等未体现。
5. **电源管理、虚拟化/hypervisor、SMP 进程调度、文件系统日志与块缓存调度**等未体现。
