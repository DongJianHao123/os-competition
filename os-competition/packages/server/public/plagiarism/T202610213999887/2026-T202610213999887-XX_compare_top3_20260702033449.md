#### 5. 单方模块与降权项
- **单方模块**: `文件系统/第三方库` 模块仅存在于目标项目（200 个函数），包含 `ext4` 文件系统的完整实现（如 `mknod`、`read`、`insert_extent`、`write_direct` 等），参考项目无此模块。这属于目标项目的独立扩展，不构成同源证据。
- **跨板块降权项**: 控方指控中大量使用 `os/src/fs/proc/` 路径下的 `getattr`、`seek` 等函数作为“进程管理”板块证据。这些函数属于文件系统/VFS 板块，且参考项目未提供同板块对应实现。根据审计指令，跨板块同名或同 hash 只能作为弱信号，不构成同板块主证据。
- **公共接口降权项**: `sys_execve`、`sys_getpid`、`sys_getppid`、`sys_exit` 等系统调用函数名由 POSIX/Linux ABI 强制规定，任何兼容内核都必须使用这些名称。`Kstat::new`、`checked_add_signed`、`Arc::upgrade` 等属于 Rust 标准库或内核编程的通用惯用法，不能作为抄袭指纹。

基于 5.64% 的客观 AST 相似度及双方极低的模块覆盖率，目标项目与参考项目在底层架构上存在显著差异。共同指纹高度集中于 LA64 硬件规范强制接口（如 SBI 控制台、ESTAT 寄存器解析）和 POSIX 系统调用命名，属于领域共性和平台约束。在进程管理、内存管理、文件系统等核心子系统上，双方在同步原语、内存模型、调度策略和实现层次上展现了独立的架构设计。目标项目独有的 Ext4 文件系统实现进一步佐证了其独立开发能力。整体评估，现有证据不支持代码复用指控，应判定为独立开发。
