### 4. 核心代码调用流对比剖析

* **【进程管理/调度器核心】**
* **底层技术析理**: 双方均实现了 `run_tasks` 调度循环，但抽象层次与同步策略截然不同。目标项目将 `Processor` 作为结构体，通过 `self.current.clone()` 获取任务，使用 `PROCESSOR.write`（写锁）保护状态，并通过内联汇编 `asm!` 进行上下文切换。参考项目则将 `PROCESSOR` 作为全局静态变量，通过 `PROCESSOR.lock`（互斥锁）保护，并调用外部汇编函数 `__switch`。目标项目显式操作 `idle_task`，而参考项目通过 `do_wake_expired` 处理。两者在对象模型、锁原语和底层切换机制上存在根本性差异，是同一功能目标的独立架构实现。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

* **【内存管理/缺页异常处理】**
* **底层技术析理**: 双方在 `trap_handler` 中处理缺页异常，但内存模型和错误处理路径不同。目标项目（RISC-V）通过 `stval` 寄存器获取错误地址，并使用 `PageFaultCause::from` 解析异常原因，随后调用 `PageTable::from_token` 进行页表修复。参考项目（LoongArch）则通过 `BadV` 寄存器获取地址，并直接操作 `ERA` 寄存器准备返回。目标项目在异常处理中集成了 `clean_dentry_cache`，显示了其 VFS 与内存管理的紧耦合设计，而参考项目则无此调用。两者在架构相关寄存器操作、错误解析流程和子系统交互上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `os/src/arch/riscv64/trap/mod.rs#trap_handler`
  - 参考项目: `os/src/hal/arch/loongarch64/trap/mod.rs#trap_handler`

* **【文件系统/Ext4 实现】**
* **底层技术析理**: 双方均实现了 Ext4 的 `create` 操作，但对象管理和并发控制模型不同。目标项目通过 `dentry.flags.write().update_type_from_negative` 直接操作目录项的负状态标志，并调用 `Ext4Inode::new` 创建 inode。参考项目则通过 `Arc::new(Mutex::new(...))` 包装新创建的 inode，并显式设置 `InodeFileType` 和 `InodePerm`。目标项目的逻辑更侧重于通过 dentry 状态转换来管理创建流程，而参考项目则侧重于 inode 对象的权限与类型初始化。两者在 VFS 与具体文件系统的交互细节上体现了不同的设计权衡。
* **代码溯源链接**:
  - 目标项目: `os/src/ext4/mod.rs#create`
  - 参考项目: `os/src/fs/ext4/layout.rs#create`

### 5. 单方模块与降权项
*   **单方模块**:
    *   **网络协议栈/第三方库** 与 **第三方库**: 仅存在于参考项目中，包含 `virtio-drivers`、`dep_iso`、`dep_pci` 等多个外部依赖。目标项目未包含这些模块，这是证明双方代码基础完全不同的强有力证据，直接否定了“整体项目复用”的可能性。
*   **降权项说明**:
    *   **公共接口与泛型函数名**: 控辩双方提及的大量 `Ok`、`Err`、`Some`、`drop`、`clone`、`unwrap`、`asm!`、`copy_from_user`、`copy_to_user` 等，均为 Rust 标准库或操作系统内核领域的通用原语。它们在不同项目中产生相同的 AST 指纹是必然事件，不能作为抄袭证据。
    *   **跨板块证据**: 控方将 `sys_rt_sigtimedwait`（信号）与 `sys_clock_nanosleep`（定时器）等跨板块调用链进行比对，并将其降级为弱信号。本庭采纳此降级处理，并重申跨板块证据不构成同板块主证据。
    *   **Proc 文件系统**: 目标项目实现了完整的 procfs，而参考项目未体现。控方将 procfs 中 `getattr`、`seek` 等标准 VFS 接口的实现作为证据，但这些是实现任何 procfs 的必经之路，属于 Linux VFS 规范决定的公共模式，不能证明抄袭。

### 6. 保守结论
综合宏观覆盖率（4.5%）、全模块矩阵分析及核心调用流对比，本庭裁定目标项目与参考项目在代码实现层面构成**独立开发**。双方在进程管理、内存管理、文件系统等核心板块的底层架构、同步策略、对象模型上存在实质性差异。控方提出的“铁证”几乎全部源于对 POSIX 标准接口、硬件规范强制逻辑及 Rust 语言通用原语的误判。虽然双方在内存管理和中断处理等硬件强相关模块存在一定相似度，但这属于遵循相同技术规范（如 LA64 架构、RISC-V SBI、Ext4 磁盘布局）的必然结果，而非代码抄袭。整体来看，该项目不存在违规复用参考项目代码的实质性风险。
