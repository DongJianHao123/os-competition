* **底层技术析理**: 双方均实现了 POSIX `execve` 的系统调用，核心流程均为“解析路径 -> 加载可执行文件 -> 替换地址空间”。然而，在应用加载策略上存在架构差异：参考项目硬编码了针对 LoongArch 和 RISC-V 的 musl 库路径（`“/loongarch/musl/busybox”`），表明其与特定工具链深度绑定；而目标项目则采用统一的 `FileReader::new(app.clone())` 抽象，将路径解析与加载逻辑解耦，展现出更通用的设计。
