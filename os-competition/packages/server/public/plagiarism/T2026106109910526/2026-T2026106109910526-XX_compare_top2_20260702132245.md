### 4. 核心代码调用流对比剖析
*   **【进程管理/进程创建】**
    *   **底层技术析理**: `sys_clone` 系统调用。目标项目使用 `VirtAddr::from` 和 `current_task_arc` 等内部抽象来处理用户态指针和任务控制块，并通过 `CloneFlags::LEGACY_FLAGS.bits` 处理标志位。参考项目则直接使用 `UserWritePtr::from(child_tid.bits()).write` 进行用户态内存写入，并采用不同的标志位处理逻辑 `CloneFlags::from_bits(flags as u64 & !0xff)`。两者在内存访问模型和 API 封装上存在根本性差异。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/process.rs#sys_clone`
        - 参考项目: `kernel/src/syscall/process.rs#sys_clone`

*   **【进程管理/进程退出】**
    *   **底层技术析理**: `do_exit` 函数。目标项目的退出流程涉及 `os_is_shutting_down` 和 `os_send_shutdown` 等全局状态检查与关机逻辑，并调用 `TASK_MANAGER.remove_task`。参考项目则侧重于共享内存的清理（`SHARED_MEMORY_MANAGER.detach`）和通过 `UserWritePtr` 向用户态写入退出状态。两者的资源清理范围和同步策略完全不同。
    *   **代码溯源链接**:
        - 目标项目: `os/src/task/task.rs#do_exit`
        - 参考项目: `kernel/src/task/task.rs#do_exit`

*   **【中断/异常处理/架构抽象】**
    *   **底层技术析理**: `auto_sum` 函数。目标项目通过架构无关的 `Instruction::clear_sum` / `Instruction::set_sum` 指令抽象来操作处理器状态，这属于其 HAL 层设计。参考项目则直接调用 RISC-V 架构特定的 `riscv::register::sstatus::clear_sum` / `set_sum` 寄存器。这是双方在架构抽象层次上最根本的差异，证明了独立设计。
    *   **代码溯源链接**:

### 5. 单方模块与降权项
*   **单方模块**: 参考项目中的 `第三方库` 模块（包含 `crates/ring-buffer`, `crates/range-map`）在目标项目中无对应实现。这是双方在软件架构和组件选型上独立决策的直接证据，表明目标项目并非对参考项目的全盘复制。
*   **降权项**: 所有同名的 POSIX 系统调用（如 `sys_getpid`, `sys_execve`）、标准 C 库用户态封装（如 `execve`）、以及输出 Linux 标准格式的 `/proc` 文件接口（如 `meminfo.rs::serialize`），因其命名和功能由外部规范强制定义，均被降权为弱信号或不作为证据。对第三方库 `smoltcp` 的标准 API 调用同样被降权。

### 6. 保守结论
综合全模块证据，目标项目与参考项目的客观 AST 相似度仅为 7.18%，且重合点高度集中在硬件规范、POSIX 标准、Rust 语言惯用法和第三方库 API 调用上。控方提出的“复用铁证”均为极简的 getter/setter 或委托模式，其 AST 指纹过短，无法承载独创性表达。更为关键的是，双方在架构抽象（HAL vs 裸寄存器）、内存访问模型（`VirtAddr` vs `UserWritePtr`）、驱动框架和核心子系统（如调度器、VFS 对象模型）上存在根本性、系统性的差异。这些差异远大于表面上的函数同名，有力地支持了独立开发的结论。因此，本案整体抄袭风险极低，应认定为两个遵循相同领域规范但独立演进的操作系统内核项目。
