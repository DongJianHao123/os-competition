### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出】**
* **底层技术析理**: 目标项目采用封装式设计，`sys_exit` 直接调用 `task.do_exit`，将退出逻辑内聚在任务控制块方法中。参考项目则采用内联式设计，在系统调用处理函数中直接操作 `task.set_exit_code`、`task.set_state` 等。这反映了不同的职责划分策略：目标倾向于面向对象的消息传递，参考倾向于过程式的状态操作。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_exit`
  - 参考项目: `kernel/src/syscall/process.rs#sys_exit`

* **【文件系统/虚拟文件系统挂载】**
* **底层技术析理**: 目标项目使用 `CNXFS::create_sys_dir` 和 `DCACHE` 构建 procfs，其 VFS 层基于自定义的 `CNXFS` 和全局 `DCACHE`。参考项目则使用 `ExeDentry::new`、`InterruptsInode::new` 等，其 VFS 层基于 `osfs` 库的 `Dentry`/`Inode` 抽象。两者在文件系统对象模型上完全不同，表明 VFS 是独立设计的。
* **代码溯源链接**:
  - 目标项目: `os/src/fs/procfs/mod.rs#init_procfs`
  - 参考项目: `lib/osfs/src/proc/mod.rs#init_procfs`

* **【设备驱动/VirtIO 网络接收】**
* **底层技术析理**: 双方都遵循 VirtIO 规范进行 RX 队列操作，调用 `rx_buf.as_mut_slice` 或 `rx_buf.raw_buf_mut` 获取缓冲区。但目标项目使用 `log::trace!` 进行细粒度跟踪，参考项目则使用 `rx_buf.set_header_len` 等更具体的 VirtIO 头部操作。相似性源于 VirtIO 标准协议，但缓冲区管理和日志策略存在差异。

### 5. 单方模块与降权项
*   **单方模块**: 目标项目中的 `os/src/task/task.rs#exit_robust_list`（robust futex 处理）和 `os/src/syscall/io.rs#sys_epoll_pwait2`（`epoll_pwait2` 系统调用）在参考项目中未找到对应证据，表明目标项目有独立的特性开发。
*   **降权项**: 所有跨板块同名函数（如目标在 `syscall/net.rs` 的 `get` 与参考在 `task/sig_members.rs` 的 `get`）、通用宏（`println!`, `format!`）、标准库类型（`Ok`, `Err`, `Some`, `Arc`, `VecDeque`）以及 Linux ABI 强制命名（`sys_execve`, `sys_clone` 等）均被降权为弱信号或不作为证据。它们无法证明逻辑同源，只能说明双方遵循相同的语言规范和接口标准。

### 6. 保守结论
本次审计的核心结论是**独立开发**。4.47% 的 AST 相似度与不足 9% 的双向覆盖率，在操作系统内核这类高度依赖标准接口与硬件规范的领域中，属于典型的“领域同构噪声”。控方提出的“复用指控”所依赖的证据，经剖析后均指向 Linux ABI 强制格式、Rust 语言惯用法或硬件标准操作序列，而非对参考项目独创性代码逻辑的抄袭。双方在 VFS 架构、用户指针抽象、任务调度模型等核心子系统上展现出根本性的设计差异，这进一步排除了基于同一代码库进行“洗稿”的可能性。综合判定，目标项目不存在违规复用行为，其原创性应予认可。
