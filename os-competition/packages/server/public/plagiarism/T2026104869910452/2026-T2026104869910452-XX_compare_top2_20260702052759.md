- **底层技术析理**: 目标项目 `sys_execve` 实现了完整的路径解析与符号链接跟随流程，外部调用图谱包含 `<dyn File>::open`、`Path::new`、`Path::resolve_symlink_through`、`String::from_utf8` 等 67 个调用，体现了通用的 ELF 加载器设计。参考项目 `sys_execve` 仅硬编码了 `"/busybox"`、`"busybox"`、`"sh"` 等 13 个调用，为针对特定测试用例的简化实现。两者在文件打开策略（动态路径解析 vs 硬编码字符串）、参数处理（`Arc::clone` 传递 vs `Vec::new` + `argv.insert`）上存在根本性架构差异，AST 结构已显著分叉。

- **底层技术析理**: 目标项目采用显式互斥锁模式：通过 `self.parent_mut().lock()` 获取父进程引用，再通过 `parent.sig_handlers_mut().lock()` 获取信号处理器表，最后调用 `handlers.get(Sig::SIGCHLD).flags.contains` 检查标志位。参考项目采用闭包封装模式：通过 `parent.with_sig_handlers(|handlers| handlers.get(Sig::SIGCHLD))` 将信号处理器访问封装在闭包内。这是两种截然不同的 Rust 并发 API 设计风格——前者将锁管理暴露给调用方，后者将锁管理内聚到方法中。两者在错误处理上也不同：目标使用 `and_then`/`unwrap_or` 链式处理，参考使用 `expect`/`unwrap` 直接断言。
