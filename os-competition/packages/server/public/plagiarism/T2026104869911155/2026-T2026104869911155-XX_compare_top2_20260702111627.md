#### 4. 核心代码调用流对比剖析
* **【进程管理/信号等待路径】**
* **底层技术析理**: 目标项目 `sys_rt_sigtimedwait` 采用“系统调用入口 + 调度器状态机”两层架构。入口函数计算超时截止时间后，将任务置为 `TaskState::SignalWaiting` 并移交调度器；调度器 `handle_signal_timedwait` 通过 `monotonic_time_us` 判定超时，最终由 `complete_signal_timedwait` 通过 `write_user_bytes` 写回结果。参考项目则在单一 syscall 函数内通过 `pending.dequeue_expect` 直接从信号挂起队列取信号，使用 `UserWritePtr` 写回。两者在**同步原语**（显式状态机 vs 异步队列）、**用户态内存访问**（`aspace` 原始读写 vs `UserPtr` 封装）和**超时管理**（调度器层 vs 系统调用层）三个维度上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `src/task/mod.rs#sys_rt_sigtimedwait` / `src/task/riscv64.rs#handle_signal_timedwait`
  - 参考项目: `kernel/src/syscall/signal.rs#sys_rt_sigtimedwait`

* **【任务调度/文件控制路径】**
* **底层技术析理**: 目标项目 `sys_fcntl` 围绕 `FileLockOwner` 枚举（`OpenFileDescription` / `Process`）构建文件锁逻辑，锁等待返回 `SyscallResult::RecordLockWait` 并进入调度器状态机。参考项目 `sys_fcntl` 使用 `FcntlOp::from_repr` 解析操作码，通过 `FdFlags` 和 `OpenFlags` 位操作管理文件描述符属性。两者在**对象管理模型**（文件锁所有者 vs 描述符标志位）和**同步策略**（调度器阻塞 vs 位标志状态）上完全不同。
* **代码溯源链接**:
  - 目标项目: `src/task/mod.rs#sys_fcntl`
  - 参考项目: `kernel/src/syscall/fs.rs#sys_fcntl`

* **【内存管理/内存映射路径】**
* **底层技术析理**: 目标项目 `sys_mmap` 直接操作 `crate::mm::align_up`/`align_down` 进行页对齐计算，通过 `chunk.len` 管理映射长度。参考项目 `sys_mmap` 使用 `MapPerm::from`、`MmapProt::from_bits_truncate`、`MmapFlags::from_bits_truncate` 等类型安全抽象，并通过 `(addr + length).round_up` 进行地址对齐。两者在**抽象层次**（原始算术 vs 类型安全枚举）和**内存模型**（chunk 管理 vs 权限/标志位组合）上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `src/task/mod.rs#sys_mmap`
  - 参考项目: `kernel/src/syscall/mm.rs#sys_mmap`

* **【文件系统/读写路径】**
* **底层技术析理**: 目标项目 `write_at` 操作文件块缓存，通过 `FILE_CHUNK_SIZE` 分块、`chunk.copy_from_slice` 写入数据。参考项目 `write_at` 操作网络 socket，通过 `self.sk.sendto` 发送数据。两者虽同名，但分属**文件系统**与**网络协议栈**两个不同子系统，底层操作对象（块缓存 vs socket 缓冲区）和 I/O 路径（本地存储 vs 网络传输）完全不同。
