#### 4. 核心代码调用流对比剖析
* **【进程管理/信号等待机制】**
* **底层技术析理**: 两者均实现了 `sys_rt_sigtimedwait`，但采用了完全不同的同步/异步模型。目标项目在 `sys_rt_sigtimedwait` 中直接调用 `crate::arch::monotonic_time_us` 获取当前时间并计算绝对超时点，随后通过 `first_signal_in_sigset` 同步检查挂起信号集，若未命中则返回 `SyscallResult::SignalTimedWait` 让出 CPU。这是一个典型的同步内核状态机模型。参考项目则通过 `current_task().unwrap().clone()` 获取任务控制块，结合 `SumGuard` 和 `SigInfo` 指针写入，其底层依赖于异步运行时调度。两者在时间源获取、信号集查询方式、以及等待状态的表达上无共同外部调用对象。
* **代码溯源链接**:
  - 目标项目: `src/task/mod.rs#sys_rt_sigtimedwait`
  - 参考项目: `os/src/syscall/signal.rs#sys_rt_sigtimedwait`

* **【进程管理/程序执行】**
* **底层技术析理**: 在 `sys_execve` 的实现上，目标项目展示了更复杂的程序加载路径。它调用 `self.resolve_user_path` 解析用户态路径，随后在 `load_exec_image` 中通过 `ElfImage::parse` 解析 ELF 文件，并依据解析结果分别调用 `UserAddressSpace::load_dynamic_elf_with_args` 或 `load_static_elf_with_args`。参考项目的实现则极为简化，其调用图谱中出现了 `“/riscv/musl/busybox”.to_string` 等硬编码路径，直接使用 `FileReader::new` 加载。目标项目支持动态/静态 ELF 的区分加载及脚本解析（`exec_script_plan`），而参考项目则针对特定测试用例硬编码，两者在加载器抽象层次和功能完备性上差异巨大。

* **底层技术析理**: `sys_mmap` 的实现揭示了两者在内存管理后端上的根本差异。目标项目的调用链围绕 `crate::mm::align_up`、`chunk.len` 和 `error.errno` 展开，表明其使用自有的 `mm` 模块进行对齐和错误处理。参考项目则调用了 `MmapFlags::from_bits_truncate`、`MmapProt::from_bits_truncate` 和 `SHM_MANAGER.alloc`，显示其依赖一个独立的共享内存管理器 (`SHM_MANAGER`) 进行物理页分配。一个通过 chunk 管理，一个通过 SHM_MANAGER 分配，两者在物理内存分配策略和对象模型上无任何 AST 重合。
