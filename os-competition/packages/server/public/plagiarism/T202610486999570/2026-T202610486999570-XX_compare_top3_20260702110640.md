## 4. 核心代码调用流对比剖析

### 【进程管理 / `sys_fork` 系统调用路径】

**底层技术析理**: 
目标项目 `sys_fork` 调用链为 `current_process.fork` → `current_trap_cx` → `new_task.with_trap_cx_mut`，通过 `with_trap_cx_mut` 闭包修改 trap 上下文，体现了**封装式上下文操作**的设计。参考项目调用链为 `current_process.fork` → `new_process.inner_exclusive_access` → `tasks[0].as_ref().unwrap()` → `task.inner_exclusive_access().get_trap_cx`，直接通过 `inner_exclusive_access` 锁访问内部数据结构并硬编码索引 `tasks[0]`，体现了**直接内部访问**的设计。两者在**锁抽象层次**（封装闭包 vs 显式锁获取）和**数据结构访问路径**（`with_trap_cx_mut` vs `tasks[0]` 硬索引）上存在根本性架构差异。

**代码溯源链接**:
- 目标项目: `os/src/syscall/process.rs#sys_fork`
- 参考项目: `os/src/syscall/process.rs#sys_fork`

### 【进程管理 / `sys_getppid` 系统调用路径】

**底层技术析理**:
目标项目 `sys_getppid` 调用链为 `current_process` → `process.parent`，直接访问进程结构体的 `parent` 字段，数据结构**扁平化**。参考项目调用链为 `current_process().inner_exclusive_access().parent.as_ref().unwrap().upgrade().unwrap().getpid`，通过 `inner_exclusive_access` 锁 → `Weak` 指针升级 → `getpid` 的**深层嵌套访问**。两者在**数据封装层次**（直接字段访问 vs 多层锁+弱引用升级）上截然不同。

**代码溯源链接**:

### 【进程管理 / `current_trap_cx` 上下文获取路径】

**底层技术析理**:
目标项目 `current_trap_cx` 调用链为 `current_task().unwrap().trap_cx`，将 `trap_cx` 作为 Task 的直接字段访问，**数据结构内聚**。参考项目调用链为 `current_task().unwrap().inner_exclusive_access().get_trap_cx`，需要通过内部锁 `inner_exclusive_access` 间接获取，**访问控制分层**。两者在**对象模型**（直接字段 vs 内部锁保护）上设计哲学不同。

### 【进程管理 / `get_user_token` 页表令牌获取路径】

**底层技术析理**:
目标项目 `get_user_token` 调用链为 `self.memory.read().memory_set.token`，对 `memory` 字段使用 `RwLock` 的 `.read()` 获取读锁后访问，**读写锁同步**。参考项目调用链为 `self.memory_set.token`，直接访问 `memory_set` 字段，**无锁同步**。两者在**同步策略**（`RwLock` 读写锁 vs 无锁直接访问）上存在显著差异。
