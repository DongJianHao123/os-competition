### 4. 核心代码调用流对比剖析
* **【进程管理/系统调用路径】**
* **底层技术析理**: 目标项目采用**线程级调度模型**，参考项目采用**进程级调度模型**。以 `sys_sleep` 为例，目标项目调用流为 `mythread -> proc_killed -> thread_killed -> thread_sleep`，显式处理线程的睡眠与唤醒；参考项目调用流为 `sleep`，直接使用进程级睡眠。这体现了在调度粒度、状态管理和同步原语上的根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `kernel/sysproc.c#sys_sleep`
  - 参考项目: `src/kernel/sysproc.c#sys_sleep`

* **【内存管理/程序加载路径】**
* **底层技术析理**: 两者 `execve` 函数虽同名，但调用流差异显著。目标项目调用流包含 `ADD_AUXV, copyout, freeall` 等，侧重于辅助向量和内存拷贝；参考项目调用流包含 `EXT4close, EXT4read, MAKE_SATP` 等，直接耦合了 ext4 文件系统读取和页表构建。这表明两者在文件系统抽象层和内存管理流程上存在独立设计。
* **代码溯源链接**:

本次审计的目标项目与参考项目整体 AST 相似度为 31.36%，该数值主要由双方共用的第三方开源文件系统库（如 lwext4）贡献。在操作系统核心板块（进程管理、内存管理、任务调度等），双方的共同指纹覆盖率极低（均低于 15%），且重合点多为硬件接口、POSIX 语义或通用同步原语等“必然表达”或“公共领域惯用法”。深层架构剖析显示，目标项目采用线程级调度模型，并独立实现了 procfs、信号子系统等复杂特性，与参考项目的进程级模型存在根本性差异。综上，现有证据无法支持“系统性抄袭”的指控，项目整体风险低，应认定为独立开发。
