#### 4. 核心代码调用流对比剖析
* **【内存管理/exec 系统调用】**
* **底层技术析理**: 两项目在加载可执行文件时采用了完全不同的内存与文件对象模型。目标项目基于传统 Unix inode 与事务日志（`begin_op`/`end_op`），通过 `ilock`/`iunlockput` 管理文件元数据。参考项目则基于 FAT32 文件系统，使用 `elock`/`eunlock` 管理 FAT 目录项，并引入 `ADD_AUXV` 宏处理 ELF 辅助向量。调用流的差异反映了底层文件系统架构的根本性分歧。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/exec.c#exec`
  - 参考项目: `kernel/exec.c#exec`

* **【文件系统/目录查找】**
* **底层技术析理**: 目标项目使用 `iget` 获取 inode，并通过 `readi` 读取目录内容，是典型的 inode 遍历模型。参考项目使用 `eget` 获取 FAT32 目录项，并通过 `enext` 遍历簇链，是典型的 FAT 遍历模型。两者在对象命名、分配和访问接口上完全不同，证明文件系统模块是独立设计的。
* **代码溯源链接**:
