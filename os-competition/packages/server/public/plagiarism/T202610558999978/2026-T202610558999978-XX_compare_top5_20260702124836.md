#### 4. 核心代码调用流对比剖析
* **【进程管理/调度入口】**
* **底层技术析理**: 双方 `sched` 函数的调用图谱均为 `holding, intr_get, mycpu, myproc, panic, swtch`。这是 RISC-V 多核操作系统上下文切换的标准流程：检查锁持有状态、关闭中断、获取当前 CPU 和进程、验证一致性、执行汇编上下文切换。该调用序列由 xv6 的调度器设计文档和 RISC-V 特权架构规范共同决定，任何基于 xv6 的独立实现都会呈现相同的依赖集合。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#sched`
  - 参考项目: `kernel/proc.c#sched`

* **【内存管理/程序加载】**
* **底层技术析理**: 双方 `exec` 函数均负责加载可执行文件，但底层存储模型截然不同。目标项目调用 `begin_op, end_op, ilock, iunlockput`，表明其使用日志文件系统进行事务性磁盘操作。参考项目调用 `fat32_init, ename, eput`，表明其直接操作 FAT32 文件系统的目录项和簇链。这种文件系统后端的根本分歧，证明两个 `exec` 实现是独立开发的。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/exec.c#exec`
  - 参考项目: `kernel/exec.c#exec`

* **【文件系统/目录查找】**
* **底层技术析理**: 双方 `dirlookup` 函数同名，但实现路径完全不同。目标项目使用 `iget` 获取 inode、`readi` 读取 inode 数据块，是典型的 inode 式文件系统。参考项目使用 `eget` 获取目录项、`enext` 遍历目录、`reloc_clus` 处理 FAT 簇链，是典型的 FAT 文件系统。两者在磁盘数据结构、缓存机制、路径遍历算法上均存在本质差异。
* **代码溯源链接**:
  - 目标项目: `kernel/filesystem/fs.c#dirlookup`
  - 参考项目: `kernel/fat32.c#dirlookup`
