#### 4. 核心代码调用流对比剖析
* **【进程管理/上下文切换】**
* **底层技术析理**: 双方均遵循 xv6 的上下文切换模型。`sched` 函数作为切换核心，其调用流 `holding -> intr_get -> mycpu -> myproc -> panic -> swtch` 是 xv6 教材中规定的标准步骤：检查锁、关中断、获取当前 CPU 和进程、最后调用汇编 `swtch`。此调用流由 RISC-V 特权架构和 xv6 的调度器设计共同决定，是教学内核的通用范式，而非个人独创。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#sched`
  - 参考项目: `kernel/proc.c#sched`

* **【文件系统/目录查找】**
* **底层技术析理**: 双方均有 `dirlookup` 函数，但底层文件系统模型截然不同。目标项目基于 inode，调用 `iget, namecmp, readi` 等传统 UNIX 风格的函数。参考项目基于 FAT32，调用 `edup, eget, enext, eput, reloc_clus` 等簇链操作函数。这种在核心数据结构和访问方法上的根本性差异，是双方独立开发的决定性证据。
* **代码溯源链接**:
  - 目标项目: `kernel/filesystem/fs.c#dirlookup`
  - 参考项目: `kernel/fat32.c#dirlookup`

* **【任务调度/异步运行时/自旋锁】**
* **底层技术析理**: `acquire` 函数的调用流 `__sync_lock_test_and_set -> __sync_synchronize -> holding -> mycpu -> panic -> push_off` 是使用 GCC 内建原子操作实现自旋锁的标准模式。原子交换、内存屏障、递归持锁检查、关中断这一系列步骤是保证多核环境下自旋锁正确性的必要条件，任何基于 RISC-V 和 GCC 的内核实现都会呈现高度相似的调用序列。
* **代码溯源链接**:
  - 目标项目: `kernel/lock/spinlock.c#acquire`
  - 参考项目: `kernel/spinlock.c#acquire`

综合评估，目标项目与参考项目的客观 AST 相似度（18.52%）和覆盖率（约30%）均处于低位，不支持“大面积代码抄袭”的指控。双方在进程调度、系统调用等基础模块的相似性，完全可以用“均基于 xv6-RISC-V 教学内核进行开发”来解释，这些是操作系统课程设计的标准实践。决定性证据在于，双方在文件系统（inode vs FAT32）、网络协议栈（有 vs 无）、第三方库支持（无 vs ext4）等核心扩展领域做出了截然不同的架构选择与实现。这些重大差异无法通过简单的复制与修改达成，有力地证明了两个项目是独立规划与演进的。因此，整体抄袭风险极低，应认定为独立开发。
