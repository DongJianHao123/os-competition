### 4. 核心代码调用流对比剖析
* **【进程管理/退出流程】**
* **底层技术析理**: 目标项目将进程退出逻辑封装在 `Task` 对象的方法中（`task.do_exit` / `task.do_group_exit`），体现了面向对象的职责分离。参考项目则在系统调用处理函数中直接操作线程组锁和线程状态（`thread_group.lock`， `thread.set_state`），逻辑更过程化。两者的抽象层次和对象管理模型存在根本差异。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_exit`
  - 参考项目: `kernel/src/syscall/process.rs#sys_exit`

* **【内存管理/进程 fork】**
* **底层技术析理**: 目标项目的 `fork` 实现通过 `Arc::new` 和 `AtomicUsize::new` 等原子操作创建新的地址空间，并调用 `PROCESS_GROUP_MANAGER.add_task_to_group` 进行组管理。参考项目则通过 `(*self.children_mut()).clone()` 等模式手动克隆父进程的各种内存映射和权限结构。两者在内存共享和进程组管理上采用了完全不同的同步策略和数据结构。
* **代码溯源链接**:
  - 目标项目: `os/src/task/task.rs#fork`
  - 参考项目: `kernel/src/task/taskf.rs#fork`

* **【系统调用/用户指针抽象】**
* **底层技术析理**: 这是双方架构差异最显著的领域。目标项目使用 `UserPtrRaw` / `UserSliceRaw` 体系，通过 `ensure_read` / `ensure_write` 方法进行权限校验。参考项目则使用泛型化的 `UserReadPtr<T>` / `UserWritePtr<T>` 体系，在类型层面区分读写。这两种安全抽象在 Rust 生态中代表了两种截然不同的设计哲学，不可能通过简单“换皮”得来。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/mm.rs#sys_process_vm_readv`
  - 参考项目: `kernel/src/syscall/signal.rs#sys_rt_sigtimedwait`

### 5. 单方模块与降权项
*   **单方模块**: 目标项目的 `os/src/timer/recoder.rs` 和参考项目的 `kernel/src/task/time_stat.rs` 虽功能相似，但模块组织方式不同，属于单方实现。控方指控的 `update_child_time` 函数因无外部调用图谱（内部闭环逻辑），无法进行有效的 AST 比对，其相似性仅为功能语义上的收敛，不能作为强同源证据。
*   **降权项**: 所有跨板块同名函数（如目标项目在 `syscall/net.rs` 的 `get` 与参考项目在 `task/sig_members.rs` 的 `get`）均已被降权。它们属于不同子系统，函数名仅为通用动词，其内部调用图谱完全不同，不构成同板块主证据。所有 POSIX 系统调用同名函数（如 `sys_execve`， `sys_epoll_pwait`）也已被降权，因为它们是由接口标准强制规定的。

### 6. 保守结论
综合评估，目标项目与参考项目的客观 AST 相似度仅为 5.04%，处于极低水平。控方提出的“复用指控”所依赖的证据，经深度析理，均属于硬件规范、POSIX/Linux ABI 标准、Rust 语言惯用法或第三方库（如 `smoltcp`， `ext4`）使用范式的必然收敛。双方在核心架构设计上存在显著且根本的差异，尤其体现在**用户指针安全抽象、进程退出模型和内存管理策略**上，这些差异绝非简单的“换皮”或“删日志”可以解释。因此，本次审计判定两个项目为**独立开发**，不存在足以构成抄袭的代码同源性。建议驳回所有复用指控。
