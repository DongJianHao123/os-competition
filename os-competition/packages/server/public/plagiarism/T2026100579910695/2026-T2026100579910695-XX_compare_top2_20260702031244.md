#### 4. 核心代码调用流对比剖析
* **【进程管理/系统调用路径】**
* **底层技术析理**: 以 `sys_execve` 为例，目标项目构建了多架构兼容的程序加载路径，通过 `AtFlags` 控制路径解析行为，并使用 `FileReader` 抽象进行文件读取。参考项目则采用更简单的路径拼接和 `file.read_all` 一次性读取。两者在内存管理上，目标项目使用 `UserPtrRaw` 系列进行严格的用户态指针校验，而参考项目使用 `UserWritePtr`。同步策略上，目标项目在 `sys_clone` 中显式调用 `task.get_vm_space().lock()` 管理地址空间锁，参考项目则将此逻辑内聚在 `UserWritePtr` 的实现中。对象管理模型存在根本差异。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_execve`
  - 参考项目: `kernel/src/syscall/process.rs#sys_execve`

* **【任务调度/异步运行时】**
* **底层技术析理**: 在任务派生 (`spawn`) 上，目标项目展现了 SMP 架构下的多核调度设计。它通过 `select_run_queue_index` 选择运行队列，并将任务推入特定 CPU 的 `PROCESSORS` 队列中，包含抢占逻辑 (`push_preempt`)。参考项目则采用全局任务队列 `TASK_QUEUE`，仅通过 `push_normal` 和 `push_prior` 区分优先级。这反映了两种截然不同的调度架构：分布式 per-CPU 队列 vs 全局统一队列。
* **代码溯源链接**:
  - 目标项目: `os/src/executor/mod.rs#spawn`
  - 参考项目: `modules/executor/src/lib.rs#spawn`

* **【内存管理/硬件驱动】**
* **底层技术析理**: 在 MMC 卡初始化 (`card_init`) 中，双方都遵循了相同的硬件协议，发送 `CMD0`、`CMD8`、`ACMD41` 等命令序列。AST 指纹重合是因为对硬件寄存器 (`CMD`, `CMDARG`, `RESP`) 的操作序列是固定的，这是硬件手册的强制性规范，而非软件架构的复用。任何针对该硬件的驱动都必然产生此调用流。

  - 参考项目: `driver/src/blk/dw_mshc/mmc.rs#card_init`

* **底层技术析理**: 在 `init_devfs` 中，双方都创建了 `null`, `zero`, `tty` 等标准设备节点。然而，目标项目使用 `CNXFS::create_sys_dir` 和 `DCACHE` 进行集中式节点管理，而参考项目直接实例化 `NullDentry::new` 等对象并插入 VFS。文件系统的挂载、inode/dentry 管理模型存在本质区别。

#### 6. 保守结论
综合评估，目标项目与参考项目的整体 AST 相似度仅为 8.18%，双向覆盖率均低于 20%，远未达到“实质性相似”的阈值。观察到的大部分 AST 指纹重合可归因于三大非抄袭因素：1) 遵循 Linux/POSIX 系统调用标准与 procfs 等公共 ABI 规范；2) 使用相同的第三方库（如 `smoltcp`、`ext4`）和 Rust 语言惯用法；3) 硬件规范对设备驱动（UART、MMC）初始化序列的强制性约束。在核心架构层面，双方在进程调度（per-CPU vs 全局队列）、用户态内存安全抽象（`UserPtrRaw` vs `UserWritePtr`）、多架构支持（LoongArch/RISC-V vs 单 RISC-V）等方面存在根本性差异。因此，现有证据强力支持“独立开发”的结论，整体同源性风险极低。
