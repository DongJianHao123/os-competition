#### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建系统调用】**
* **底层技术析理**: 双方均实现了Linux `clone` 系统调用，但底层对象管理和内存模型截然不同。目标项目采用全局单例 `TASK_MANAGER` 管理所有任务，并通过自研的 `UserPtrRaw` 对象进行用户态指针的安全读写，其调用流为 `CloneFlags::from_bits -> UserPtrRaw::new(...).ensure_write`。参考项目则通过 `self.lproc` 成员方法访问当前进程，并使用 `UserReadPtr/UserWritePtr` 范型进行指针操作，调用流为 `CloneFlags::from_bits_truncate -> VirtAddr::from -> checked_write_u32`。两者在进程管理器、用户态内存抽象和具体操作函数上完全不同，不存在逻辑搬运。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_clone`
  - 参考项目: `src/syscall/process.rs#sys_clone`

* **【内存管理/堆内存调整】**
* **底层技术析理**: 双方均实现了`brk`系统调用以调整进程堆空间，但底层内存映射模型完全不同。目标项目使用 `UserVmArea` 和 `page_table` 进行显式的物理页帧映射（`map_zero_frames`），并操作 `range_vpn` 进行虚拟地址区间管理。参考项目则基于 `RangeMap` 数据结构进行区间管理，通过 `extend_back` 和 `reduce_back` 方法调整堆边界，不涉及直接的页表操作。这体现了“显式页表管理”与“区间映射抽象”两种根本不同的内存管理哲学。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/vm/uvm.rs#reset_heap_break`
  - 参考项目: `src/process/user_space/user_area.rs#reset_heap_break`

* **【设备驱动/UART串口驱动】**
* **底层技术析理**: 双方均实现了UART8250兼容串口的初始化与发送逻辑，调用流高度相似，例如`send_u8`均为“等待发送器空 -> 写入THR寄存器”。此相似性源于UART8250芯片的公开数据手册，其寄存器定义和操作流程是标准化的，无独创空间。尽管如此，目标项目在`init`函数中区分了`init_u32`和`init_u8`，体现了对不同硬件配置的适配，与参考项目存在细节差异。
* **代码溯源链接**:
  - 目标项目: `os/src/drivers/serial/uart.rs#send_u8`
  - 参考项目: `src/drivers/serial/uart8250.rs#send_u8`
