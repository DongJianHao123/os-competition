#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出与回收】**
* **底层技术析理**: 两者均遵循 Linux 的僵死进程回收模型。当子进程退出时，父进程通过 `sys_waitpid` 进行回收。目标与参考项目在 `handle_zombie` 函数中展现了相同的处理时序：首先通过 `assert!(child.is_zombie)` 确认状态，然后获取父进程锁 `child.parent.lock`，清理子进程列表 `children.clear`，最后从全局任务管理器移除 `TASK_MANAGER.remove_task`。这一精确的锁时序和状态清理步骤，特别是 `SigSet::all` 的使用，是特定于 Linux 信号处理与任务管理模型的实现细节，具有高度同源性。
* **代码溯源**:
  - 目标项目: `os/src/task/task.rs#handle_zombie`
  - 参考项目: `os/src/task/task.rs#handle_zombie`

* **【进程管理/用户态内存读写】**
* **底层技术析理**: `process_vm_readv` 和 `process_vm_writev` 系统调用允许一个进程直接读写另一个进程的地址空间。两者的实现均采用了相同的安全封装模式：通过 `TASK_MANAGER.get_task(pid)` 获取目标进程控制块，然后使用 `UserSliceRaw::new(iov.base as *const u8, iov.len)` 构造用户态切片，并链式调用 `.ensure_read(&mut target_task.get_vm_space().lock())` 来在目标进程的地址空间锁保护下校验内存可读性。这种将指针转换、长度封装、锁获取和权限校验串联成一个调用链的编码风格，是 Rust 内核中处理跨进程内存访问的一种高度特异性的设计模式。
* **代码溯源**:
  - 目标项目: `os/src/syscall/mm.rs#sys_process_vm_readv`
  - 参考项目: `os/src/syscall/mm.rs#sys_process_vm_readv`

* **【任务调度/异步运行时/上下文切换】**
* **底层技术析理**: 在任务上下文切换 `switch_out_current_task` 中，两者遵循了相同的硬件操作时序：首先通过 `Instruction::disable_interrupt` 关闭中断以保证切换的原子性，然后操作内核虚拟内存空间 `KVMSPACE.lock().enable`，最后通过 `current.get_trap_cx().fx_yield_task` 保存当前任务的浮点/向量寄存器上下文。这个 `关中断 -> 切换地址空间 -> 保存硬件上下文` 的精确时序，是内核中非常核心且危险的“临界区”代码，其实现逻辑的高度一致是强同源信号。
