### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出与调度】**
* **底层技术析理**: 两者在进程组退出的核心路径上展现了截然不同的对象模型和同步策略。目标项目以 `Process` 为中心，信号表 (`sig_table`) 内嵌于进程结构，通过 `inner_exclusive_access` 进行独占访问。参考项目则以 `Task` 为中心，通过 `get_locked_sigtable()` 显式获取锁保护的信号表，并在退出时调用 `send_signal_to_thread_group` 进行显式的线程组信号广播。这反映了“进程即容器”与“任务即主体”两种不同的内核设计哲学。
* **代码溯源链接**:
  - 目标项目: `os/src/task/mod.rs#exit_current_group_and_run_next`
  - 参考项目: `os/src/task/mod.rs#exit_current_group_and_run_next`

* **【进程管理/调度器核心】**
* **底层技术析理**: 调度器的核心抽象存在根本性差异。目标项目采用全局单处理器模型，通过 `PROCESSOR.exclusive_access()` 获取唯一的处理器实例。参考项目则实现了多核（Hart）感知的调度，通过 `get_proc_by_hartid(hart_id())` 按核心ID获取对应的处理器结构。这直接导致其上下文切换、任务选择等后续调用流完全不同，是架构级异构的铁证。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#schedule`
  - 参考项目: `os/src/task/processor.rs#schedule`

* **【内存管理/陷阱处理】**
* **底层技术析理**: 陷阱处理是内核与硬件交互的最底层，两者实现路径完全不同。目标项目在 `trap_handler` 中直接调用 `check_signals_of_current` 和 `check_timer`，逻辑内聚。参考项目则通过 `Syscall::from` 和 `Trap::Exception` 进行复杂的分发，并调用 `check_futex_timer` 等特有逻辑。这证明双方在异常向量表、上下文保存与恢复等最核心的底层机制上是独立实现的。
* **代码溯源链接**:
  - 目标项目: `os/src/hal/trap/mod.rs#trap_handler`
  - 参考项目: `os/src/trap/mod.rs#trap_handler`

### 5. 单方模块与降权项
- **单方模块**:
  - **网络协议栈**: 目标项目拥有完整的网络协议栈模块（`os/src/net/`）并集成了 `smoltcp` 第三方库，函数量达 2512 个。参考项目完全缺失此模块。这是目标项目独立扩展和差异化设计的显著标志。
- **降权项**:
  - **第三方库**: `lwext4` 和 `smoltcp` 等第三方库的代码重合不计入原创性比对。
  - **POSIX 系统调用存根**: `sys_geteuid`、`sys_getegid` 等函数在参考项目中仅为 `Ok` 返回值，属于未实现状态，不能作为比对基准。
  - **标准 Rust 惯用法**: `Option::take()`、`Option::as_ref().map()`、`unwrap()` 等模式是 Rust 语言的标准实践，不具备代码同源的证明力。
  - **跨板块同名**: 任何跨模块的同名函数（如文件系统与进程管理中的同名函数）均不能作为强同源证据，仅能作为弱信号。

### 6. 保守结论
综合客观 AST 指纹数据与深度架构剖析，认定双方项目为独立开发。尽管存在 20.31% 的 AST 相似度，但这主要由共同引用的第三方 C 库（`lwext4`）和 Rust 语言的标准惯用法贡献。在操作系统内核的核心竞争力领域，如进程调度（全局单处理器 vs. 多核 Hart 抽象）、内存管理（内嵌信号表 vs. 加锁信号表）、陷阱处理及网络协议栈（有 vs. 无）等方面，双方展现了根本性的架构差异和独立的设计选择。核心模块低于 10% 的覆盖率无法支撑“抄袭”或“同源”的指控。因此，整体同源性风险极低，应认定为独立开发作品。
