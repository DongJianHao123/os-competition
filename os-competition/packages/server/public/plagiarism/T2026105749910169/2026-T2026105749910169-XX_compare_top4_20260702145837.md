### 4. 核心代码调用流对比剖析

* **【进程管理/进程退出】**
* **底层技术析理**: 目标项目`exit_current`的调用流为`ProcessAction::ExitToKernel` → `ProcessSlot::empty` → `RuntimeProcessState::Zombie` → `release_all_pipe_fds`，其核心是操作基于数组槽位的`ProcessRuntime`状态机，资源回收聚焦于管道fd。参考项目`sys_exit`的调用流为`inner.process.borrow_mut().exit` → `self.release` → `heap.mem_set.release` / `pmm.release`，其核心是操作`Rc<RefCell<Process>>`共享所有权模型，资源回收聚焦于物理内存页。两者在**内存模型**（数组槽位 vs Rc<RefCell>）、**同步策略**（无锁状态机 vs 运行时借用检查）、**资源域**（管道fd vs 物理页）上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `src/process.rs#exit_current`
  - 参考项目: `kernel/src/sys_call/task/exit.rs#sys_exit`

* **【进程管理/进程克隆】**
* **底层技术析理**: 目标项目`clone_current_with_budget`的调用流为`next_pid.saturating_add` → `free_process_slot` → `child.frame.set_user_stack`，其核心是操作`TestFrame`测试帧，克隆的是测试上下文。参考项目`sys_fork`的调用流为`Process::new` → `child_process.pmm.add_mapping_by_set` → `add_task_to_scheduler`，其核心是复制内核`Process`结构体和物理内存映射。两者在**对象管理模型**（TestFrame vs Process+pmm）上完全不同，目标项目不存在对参考项目物理页表复制逻辑的任何复用。
* **代码溯源链接**:
  - 目标项目: `src/process.rs#clone_current_with_budget`
  - 参考项目: `kernel/src/sys_call/task/fork.rs#sys_fork`

* **【内存管理/mmap系统调用】**
* **底层技术析理**: 目标项目`sys_mmap`的调用流为`align_up` → `ctx.read_file_data(node, offset, count, ...)` → `memory.write_bytes(map_addr, data)`，其核心是通过`SyscallContext`读取运行时文件节点数据并写入模拟内存。参考项目`sys_mmap`的调用流为`MapFlags::from_bits_truncate` → `MemMap::new` → `VirtAddr::from(start).into()`，其核心是构建内核态内存映射描述符。两者在**抽象层次**（用户态模拟器 vs 内核态映射器）上存在本质区别。
* **代码溯源链接**:
  - 目标项目: `src/syscall/memory.rs#sys_mmap`
  - 参考项目: `kernel/src/sys_call/mm.rs#sys_mmap`

### 5. 单方模块与降权项

*   **单方模块**:
    *   **网络协议栈** (`src/netperf.rs`): 仅目标项目存在，参考项目无此模块。属于目标项目的增量开发。
    *   **文件系统/第三方库** (`src/fs/ext4.rs`): 仅目标项目存在自实现的Ext4解析器，参考项目使用`fatfs`库。技术选型和实现完全不同。
*   **降权项（弱信号）**:
    *   **跨板块同名函数 `write_at`**: 目标项目`src/process.rs#write_at`（进程内存写入）、`src/syscall/mod.rs#write_at`（syscall测试辅助）与参考项目`kernel/src/fs/specials/proc_mounts.rs#write_at`（procfs写入，实现为`todo!`）、`kernel/src/sys_call/net.rs#write_at`（网络队列写入）同名。它们分属进程管理、系统调用、文件系统、设备驱动四个不同板块，操作的数据结构完全不同，不构成同板块主证据。
    *   **标准库/语言级泛型调用**: 双方证据中大量出现的`assert!`、`assert_eq!`、`Ok`、`Err`、`Some`、`to_le_bytes`、`copy_from_slice`、`str.split`等，是Rust语言的通用基础设施，任何实现相同POSIX语义的项目都会使用，不能作为同源证据。
    *   **Linux标准路径/语义**: `/proc/meminfo`、`/proc/mounts`、`/proc/self`等是Linux标准procfs路径，`wait4`、`fork`、`kill`等是POSIX标准系统调用语义。双方实现相同标准功能是领域共性，不构成代码复用。

### 6. 保守结论

本次审计的核心结论是**独立开发**。客观AST相似度仅为0.59%，在双方共存的8个核心模块中，7个模块的共同指纹为0，这从根本上排除了系统性代码复用的可能。控方指控主要集中在进程管理板块，但该板块的共同指纹为0，其证据实质上是将“实现相同POSIX语义”这一领域共性，与“复用相同AST结构”混为一谈。目标项目是一个基于数组槽位、`TestFrame`和`MockMemory`的**确定性测试运行时**，而参考项目是一个基于`Rc<RefCell<Task>>`和物理页管理的**传统宏内核**，两者在内存模型、同步策略、对象管理等底层架构上存在显著差异。尽管存在极少量噪声级重合，但远未达到认定抄袭的阈值，整体风险极低。
