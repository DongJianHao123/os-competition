### 4. 核心代码调用流对比剖析
*   **【进程管理/处理器调度核心】**
    *   **底层技术析理**: 这是控方指控的核心区域。目标项目采用**多核 SMP (Symmetric Multi-Processing) 架构**，通过 `get_proc_by_hartid(hart_id())` 根据当前硬件线程 ID 获取对应的处理器控制块，再从中取出当前任务。这是一种 per-CPU 的数据结构设计。参考项目采用**单核或全局锁架构**，通过 `PROCESSOR.exclusive_access()` 获取全局唯一的处理器实例的互斥锁，再访问当前任务。两者在**同步策略**（per-hart 无锁/低锁竞争 vs 全局互斥锁）和**内存模型**（多实例 vs 单实例）上存在根本性差异。`current` 和 `take_current` 函数内部的 `Option` 操作是 Rust 语言的标准模式，不能掩盖其上层架构的巨大鸿沟。
    *   **代码溯源链接**:
        - 目标项目: `os/src/task/processor.rs#current_task`
        - 参考项目: `os/src/task/processor.rs#current_task`

*   **【内存管理/程序加载】**
    *   **底层技术析理**: 在 `exec` 系统调用的核心流程中，双方都遵循“解析 ELF -> 创建地址空间 -> 加载段 -> 设置栈”的通用模式。然而，目标项目引入了 `MemorySetInner` 作为内部抽象层，将 ELF 解析逻辑封装其中，并通过 `MemorySet::new` 进行构建，显示出更复杂的分层设计。参考项目则直接在 `exec` 函数中调用 `MemorySet::new` 和 `MemorySetInner::from_elf`，其对象模型相对扁平。这种内部抽象的差异是独立设计的体现。
