## 第四部分：分模块架构说明

### MM（内存管理）

- **证据锚点**：
  - `os/src/mm/address.rs`
  - `os/src/mm/memory_set/mmap_ops.rs#munmap`
  - `os/src/mm/frame_alloc/page_cache.rs#dealloc`
  - `os/src/arch/loongarch64/qemu/page_table.rs#handle_mmap_write_page_fault`
  - `os/src/syscall/mm/mmap.rs#sys_shmget`

- **说明**：
  - 实现了虚拟地址/物理地址抽象、页表管理、mmap/munmap 区域操作。
  - 存在页缓存（page_cache）与 CMA（连续内存分配）相关释放路径。
  - 包含 LoongArch64 的缺页处理，例如 mmap 写缺页时对页表项标志位的修改。
  - 共享内存系统调用 `sys_shmget` 表明支持 System V 风格的共享内存。

### PROC（进程管理/任务调度）

- **证据锚点**：
  - `os/src/task/process/process.rs#new / remove_from_global_map / exit_and_reparent`
  - `os/src/task/task/task.rs#clone_process / exec`
  - `os/src/task/processor.rs#run_tasks / schedule / current_task`
  - `os/src/syscall/task/wait.rs#sys_waitpid / sys_waitid`
  - `os/src/syscall/task/execve.rs#sys_execve`
  - `os/src/task/futex.rs#futex_wait_bitset / sys_futex`

- **说明**：
  - 存在传统 PCB（`Process`）与任务控制块（`Task`），使用 `meta_lock`、`inner_lock`、`memory_set` 等锁分层。
  - 支持 `PID_2_PROCESS_ARC` 全局进程表、父子进程关系、进程组（pgid）、会话等概念。
  - 实现了 `fork/clone`、`execve`、`waitpid/waitid`、`exit/exit_group`、zombie 清理。
  - 调度器基于就绪队列（`READY_QUEUE`）、多 hart `Processor`、上下文切换 `switch`。
  - 通过 futex、timer、signal 等机制实现阻塞与唤醒，具备轻量级异步/事件驱动能力。

### FS（文件系统）

- **证据锚点**：
  - `os/src/fs/kernel_fs_ops/open.rs#open`
  - `os/src/fs/files/devfs.rs`
  - `os/src/fs/ext4_lw/inode.rs#read_all`
  - `os/src/syscall/fs/mount.rs#sys_mount / sys_umount2 / sys_fsmount`
  - `crates/lwext4_rust/c/lwext4/src/ext4.c` 等

- **说明**：
  - 实现了 VFS 层：`FileClass`、`OSFile`、`FdTable`、`FsIndex`、mount 表（`MNT_TABLE`）。
  - 支持 `devfs`、`procfs`（`/proc/*/stat`、`/proc/*/status`、`/proc/*/maps`）。
  - EXT4 文件系统通过 `lwext4_rust` 封装，底层为 C 语言 lwext4 库（含 extent、journal、dir_idx、balloc 等）。
  - 支持 `openat2`、`read/write/pread/pwrite`、`copy_file_range`、`sendfile`、`fcntl`、`epoll`、`poll/select` 等。

### DEV（设备驱动）

- **证据锚点**：
  - `os/src/drivers/virtio/loongarch/pci.rs#create_net_transport`
  - `os/src/drivers/virtio/net.rs#receive / recycle_rx_buffer`
  - `os/src/net/device/ethernet.rs#send / process_arp`
  - `os/src/fs/files/loopdev.rs#ioctl`

- **说明**：
  - 主要证据集中在 **VirtIO 网络设备**驱动（PCI 枚举、接收/发送缓冲区回收）。
  - 存在 loopback 块/文件设备（`loopdev`）的 ioctl 接口。
  - 块设备驱动（如 NVMe、SATA、SD/eMMC）、字符设备（如串口、键盘、鼠标）的底层证据未充分体现。

### SYSCALL（系统调用）

- **证据锚点**：
  - `os/src/syscall/mod.rs#syscall`
  - `os/src/syscall/fs/io.rs#sys_read / sys_write / sys_pread64 / sys_pwrite64`
  - `os/src/syscall/fs/mount.rs#sys_mount / sys_umount2`
  - `os/src/syscall/net/io.rs#recv_impl / send_impl`
  - `os/src/syscall/task/wait.rs#sys_waitpid`
  - `os/src/syscall/signal.rs#sys_rt_sigprocmask`

- **说明**：
  - 系统调用分发器规模较大（229 条调用链），覆盖文件、网络、进程、内存、信号、多路 IO。
  - 兼容 Linux 风格调用号与语义（`openat2`、`mount`、`futex`、`epoll`、`ppoll`、`pselect6`、`recvmsg/sendmsg` 等）。
  - 用户态/内核态数据拷贝通过 `copy_from_user` / `copy_to_user` 完成。

### TRAP/IRQ（中断异常）

- **证据锚点**：
  - `os/src/trap/mod.rs#trap_handler`

  - `os/src/arch/irq/handler_table.rs#handle / unregister_handler`
  - `os/src/arch/loongarch64/qemu/trap_interface.rs#get_trap_cause`

- **说明**：
  - 具备用户态/内核态异常入口、`estat` 读取、`Trap::Interrupt/Exception` 分类。
  - IRQ 通过 `IRQ_HANDLERS` 与 handler table 分发，支持注册/注销。
  - 定时器、futex 超时在 `trap_handler` 与 `run_tasks` 中被统一检查。
  - 中断控制器、时钟源、PLIC/LS7A 等底层硬件抽象证据较少。

### NET（网络协议栈）

## 第六部分：边界与未体现内容

### 1. 第三方/外部代码需单独说明
- `crates/lwext4_rust/c/lwext4/` 是 **lwext4** 第三方 C 库，707 个函数主要集中于此，不应视为该作品原创内核主体。
- `user/` 目录为用户态测试程序（`initproc`、`ltp`、`forktree`、`matrix` 等），不属于内核代码。

### 2. 没有底层证据支撑的模块
- **块设备驱动**：除 `loopdev` 外，未见 NVMe、SATA/AHCI、SD/eMMC、virtio-blk 等块设备驱动证据。
- **USB / PCIe 通用框架**：仅见 VirtIO net 的 PCI 枚举，未见通用 USB/PCIe 子系统。
- **图形/显示/音频**：无 DRM/GPU、声卡驱动证据。
- **安全/隔离机制**：无 LSM、seccomp、capabilities、cgroup 完整内核实现证据（`test_cgroup_fj` 仅为测试名）。
- **虚拟化/容器**：无 hypervisor、KVM、namespace 完整实现证据。
- **电源管理/ACPI/UEFI**：无相关代码证据。
- **启动加载器/Bootloader**：无 OpenSBI/UEFI/GRUB 等启动代码证据。
- **实时调度/多核同步原语**：虽有 `hart_id`、`Processor` 多核调度入口，但自旋锁、RCU、NUMA、SMP 详细实现证据不足。

### 3. “可能方向”而非结论
- 本作品“可能的系统方向”仅基于目录结构、函数名、调用链推断，**不能作为最终作品身份或原创性结论**。
- “轻量任务调度或异步运行时”仅表示存在 `Poll`、`Waker`、`futex`、`block_on` 等机制，不等同于完整 async/await 运行时。
- “传统进程管理”有 `fork/exec/wait/zombie/PCB` 等证据支撑，但具体调度算法、调度类（CFS/RT）实现细节未在证据中体现。

### 4. 不做抄袭/违规判断
- 本文档仅描述技术实现与证据分布，不对代码原创性、合规性、抄袭与否作出任何判断。
