## 第四部分：分模块架构说明

### MM（内存管理）

**证据较强**。内核实现了基于伙伴算法的物理页分配器，并通过 `MyPagingHandler` 适配 `page_table` 风格的 Sv39 页表。

- 物理页分配：`kernel/src/paging.rs#alloc` / `free` 维护 `free_area[order]` 数组；`alloc_frames` / `dealloc_frames` 提供按页分配接口。
- 页表建立：`kernel/src/main.rs#new_pt` 使用 `pt.cursor().map_region` 映射 MMIO（`0x10000000` 起 `0xa000`）与 DRAM 区域（`DRAM_START..DRAM_END`）。
- 用户态映射：`kernel/src/main.rs#map_pages` / `unmap_pages` 在加载 ELF/栈/数据时进行虚实映射。
- DMA 与设备：`kernel/src/paging.rs#dma_alloc` / `dma_dealloc` 为 VirtIO-Block 提供一致性 DMA 内存；`mmio_phys_to_virt` 完成 MMIO 物理地址到内核虚拟地址的映射。
- 页表复制：`kernel/src/paging.rs#deep_clone` / `copy_frame` 提供页表级复制能力，可能用于 `fork`/`clone`。

### PROC（进程管理/任务调度）

**证据有限，存在轻量级进程抽象**。

- 进程表示：`kernel/src/main.rs#current_process` 通过 `kernel_core().processes.get_mut(...)` 访问当前进程；进程结构包含 `pt`、`files`、`pipes`、`mmaps`、`programs`、`state` 等字段。
- 创建与加载：`kernel/src/main.rs#spawn_process_with_elf` 解析 ELF、建立用户栈、构造 `TrapFrame`；`spawn_process` 处理参数并调用文件系统。
- 退出：`kernel/src/main.rs#exit_program` 将状态置为 `ProcessState::Zombie`，并清理 pipe 的 readers/writers。
- 调度：`kernel/src/main.rs#schedule` 调用 `self.sched_in` 并包含 `unimplemented!` 分支，说明调度器可能尚未完整；`sched_in` 通过 `context_switch` / `context_reset` 切换页表与上下文。

**未体现**：完整的 `fork`/`wait`/`waitpid`、父子进程树、进程组、信号等 POSIX 进程管理语义。

### FS（文件系统）

**证据较强**。文件系统基于第三方 `rsext4` 实现 ext4，块设备层基于 VirtIO-Block。

- 挂载：`kernel/src/fs.rs#init_virtio_blk` 初始化 `MmioTransport`、`VirtIOBlk`，并调用 `rsext4::mount` 与 `Jbd2Dev::initial_jbd2dev`。
- 块读写：`kernel/src/fs.rs#read_sector` / `write_sector` 调用 `BLK_DEVICE.read_blocks` / `write_blocks`。
- 文件读写：`kernel/src/fs.rs#read` / `write` 以扇区为单位缓冲。
- 目录遍历：`kernel/src/main.rs#list_dir` 使用 `DirEntryIterator` 迭代 ext4 目录项。
- 元数据：`kernel/src/syscalls.rs#set_from` 读取 inode 的 `uid/gid/size`。

### DEV（设备驱动）

**证据较弱，仅覆盖控制台与块设备 MMIO**。

- 控制台：`kernel/src/main.rs#sbi_putchar` / `sbi_getchar` 通过 SBI 调用或 UART MMIO 实现字符输入输出。
- MMIO 映射：`kernel/src/paging.rs#mmio_phys_to_virt` 将物理地址映射到内核虚拟地址，供 VirtIO 使用。
- 块设备：`kernel/src/fs.rs#init_virtio_blk` 中使用了 `virtio-drivers` 风格的 `VirtIOBlk`，但驱动逻辑主要依赖第三方库。

**未体现**：中断驱动的设备模型、DMA 描述符环的原创实现、定时器、PLIC/CLINT 中断控制器等。

### SYSCALL（系统调用）

**证据中等，分发逻辑集中**。

- 入口：`kernel/src/main.rs#temporary_trap_handler` 识别 `ecall` 后调用 `SyscallOp::from_sys` 与 `dispatch_syscall`。
- 分发：`kernel/src/syscalls.rs#dispatch_syscall` 规模很大，覆盖 `read/write/close/open/clone/fcntl` 等。
- 读写：`kernel/src/syscalls.rs#syscall_read` / `syscall_write` 通过 `current_process().fd` 定位资源，并在必要时设置进程状态。
- 描述符管理：`kernel/src/syscalls.rs#alloc_fd` / `close` 管理文件描述符表与 pipe 引用。

### TRAP/IRQ（中断异常）

**证据弱，仅覆盖系统调用陷入**。

- 统一处理：`kernel/src/main.rs#temporary_trap_handler` 处理 trap，对 syscall 路径调用分发函数，对异常调用 `unrecoverable`。
- 返回：`kernel/src/main.rs#skip_ecall` 用于调整 `sepc` 以跳过 `ecall` 指令。

**未体现**：外部硬件中断（timer、PLIC、UART 中断）、异常恢复、嵌套中断、中断向量表等。

### NET（网络协议栈）

**底层证据未体现**。AST Call Map 与文件分布中均未出现 socket、TCP/IP、网卡驱动、协议缓冲区等相关函数或模块。

## 第六部分：边界与未体现内容

### 没有底层证据支撑的模块

1. **网络协议栈（NET）**：未出现 socket、TCP/IP、网卡驱动、协议缓冲区等任何证据。
2. **完整中断/异常子系统**：仅有 `temporary_trap_handler` 处理 `ecall`，无 timer 中断、PLIC、外部中断、异常恢复、中断向量表。
3. **完整进程管理语义**：无 `fork`/`wait`/`waitpid`、进程组、会话、信号、完整父子进程树。
4. **抢占式/多核调度**：`schedule` 含 `unimplemented!` 分支，调度器疑似占位；无 SMP、自旋锁、CPU 亲和性。
5. **原创设备驱动**：VirtIO-Block 与 ext4 主要依赖第三方库；无原创 DMA 环、中断驱动模型、块缓存层。
6. **同步原语**：无 mutex、semaphore、condition variable、futex 等证据。
7. **虚拟文件系统（VFS）抽象**：文件系统直接对接 `rsext4`，未体现独立 VFS 层。

### 需要谨慎的判断

- “实现了 RISC-V 操作系统内核”是**可能方向**，基于 SBI、Sv39、`ecall`、VirtIO 等线索推断，但不是对最终完成度的结论。
- “支持 ext4 文件系统”主要依赖 `rsext4` 第三方 crate，应区分库代码与原创内核主体。
- “支持进程/调度”仅基于轻量级抽象与上下文切换证据，不能等同于完整 POSIX 进程模型或成熟调度器。
- 本文档仅基于 AST Call Map、函数名、路径锚点与调用链描述技术事实，不做抄袭、违规或原创性判定。
