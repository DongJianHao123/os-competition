* **【内存管理/物理页分配】**
* **底层技术析理**: 目标项目 `alloc` 函数通过 `self.free_area[found].pop()` 直接从数组取出页帧，并调用 `buf.fill` 清零，体现了基于数组栈的分配策略。参考项目 `alloc_power_of_two` 通过 `self.free_list[class].iter().next()` 和 `remove` 进行分配，并包含伙伴分裂的 `insert` 操作，体现了基于链表和显式分裂的分配策略。两者在数据结构操作和算法流程上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/paging.rs#alloc`
  - 参考项目: `third/buddy_system_allocator/src/frame.rs#alloc_power_of_two`

* **【进程管理/获取当前进程】**
* **底层技术析理**: 目标项目 `current_process` 通过 `kernel_core().processes.get_mut` 获取进程可变引用，体现了 Rust 的全局单例模式和所有权检查。参考项目 `exit` 通过 `myproc()` 宏获取当前进程指针，并配合 `acquire/release` 进行自旋锁保护，体现了 C 语言的手动同步模型。两者在内存安全抽象和同步机制上截然不同。
* **代码溯源链接**:
  - 目标项目: `kernel/src/main.rs#current_process`
  - 参考项目: `kernel/proc.c#exit`

* **【系统调用/分发与参数处理】**
* **底层技术析理**: 目标项目 `dispatch_syscall` 使用 Rust 的枚举和 `match` 进行分发，参数通过 `TrapFrame` 寄存器直接转换。参考项目 `syscall` 通过函数指针表 `syscalls[num]` 进行分发，参数通过 `argraw`、`argint` 等函数从 trapframe 中逐次提取。两者在分发机制和参数获取的抽象层次上完全不同。
