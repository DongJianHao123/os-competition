### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 双方均实现了进程退出逻辑，但资源回收模型和同步策略存在显著差异。目标项目`proc_exit`的调用流为`myproc -> panic -> proc_reparent -> proc_sched -> spinlock_*`，将子进程托付（reparent）和调度（sched）逻辑内聚，并使用显式的`spinlock`原语。参考项目`exit`的调用流为`acquire -> fileclose -> iput -> myproc -> panic -> release -> reparent -> sched`，在退出过程中直接执行文件关闭（`fileclose`）和inode释放（`iput`），锁原语为通用的`acquire/release`。两者在**资源清理的时机和封装层次**上完全不同，体现了不同的内核设计哲学。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#proc_exit`
  - 参考项目: `proc.c#exit`

* **【任务调度/异步运行时/系统调用分发】**
* **底层技术析理**: 这是控方指控的核心，但双方实现结构截然相反。目标项目`syscall`函数采用**直接展开调用**模式，内部包含`sys_brk, sys_chdir, sys_clone...`等30余个具体函数的显式调用，并穿插`printf`调试逻辑。参考项目`syscall`采用**表驱动间接调用**模式，通过`syscalls[num]`数组查找函数指针进行分发。前者是静态分派，后者是动态分派，AST结构无任何同源性。控方“将间接寻址改写为直接调用”的指控不成立，因为这两种结构在编译器层面生成的抽象语法树和控制流图完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/syscall.c#syscall`
  - 参考项目: `syscall.c#syscall`

* **【内存管理/块设备驱动】**
* **底层技术析理**: 双方在`virtio_disk_rw`函数上表现出相似的调用图谱，均包含`alloc3_desc`, `free_chain`, `__sync_synchronize`等。这并非代码复用，而是因为双方都必须遵循**Virtio块设备标准协议**。对描述符链的分配、填充、提交和释放操作，是由硬件接口规范定义的，任何正确实现Virtio驱动的代码都必然包含这些步骤和内存屏障同步原语。这是**协议规定动作**，而非思想表达。
* **代码溯源链接**:
  - 目标项目: `kernel/dev/virtio.c#virtio_disk_rw`
  - 参考项目: `virtio_disk.c#virtio_disk_rw`
