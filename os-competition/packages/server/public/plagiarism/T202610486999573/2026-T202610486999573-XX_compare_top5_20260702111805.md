#### 4. 核心代码调用流对比剖析
* **【进程管理/进程等待】**
* **底层技术析理**: 目标项目的 `proc_wait4` 与参考项目的 `wait` 虽然功能相同，但底层同步策略存在根本差异。目标项目采用主动让出 CPU 的协作式模型，调用 `proc_yield` 进入调度器；而参考项目采用阻塞式模型，调用 `sleep` 放弃 CPU。此外，目标项目将状态拷贝逻辑封装为 `proc_wait_copy_status` 并调用 `uvm_copyout`，而参考项目直接调用 `copyout2`，反映了内存管理子系统的不同抽象。锁原语也完全不同（`spinlock_acquire/release` vs `acquire/release`）。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#proc_wait4`
  - 参考项目: `kernel/proc.c#wait`

* **【任务调度/异步运行时/系统调用分发】**
* **底层技术析理**: 目标项目的 `syscall` 函数采用展开式分发，显式枚举了 30 余个系统调用处理函数（如 `sys_brk`, `sys_clone`），这是一种扁平化的控制流结构。参考项目的 `syscall` 则采用表驱动模型，通过 `syscalls[num]` 数组索引进行分发，这是一种间接调用的控制流结构。两者在 AST 层面是完全不同的设计模式，不存在结构复用。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/syscall.c#syscall`
  - 参考项目: `kernel/syscall.c#syscall`

* **【文件系统/路径切换】**
* **底层技术析理**: 目标项目的 `sys_chdir` 基于内存文件系统（ramfs），通过 `uvm_copyin_str` 从用户空间拷贝路径字符串，然后直接操作进程控制块中的当前目录指针。参考项目的 `sys_chdir` 基于 FAT32 磁盘文件系统，通过 `argstr` 获取路径后，需要调用 `elock/ename/eput/eunlock` 等一系列复杂的文件系统锁和目录项操作函数。两者的对象管理模型和底层文件系统抽象完全不同。
* **代码溯源链接**:
  - 目标项目: `kernel/syscall/sysfunc.c#sys_chdir`
  - 参考项目: `kernel/sysfile.c#sys_chdir`

#### 5. 单方模块与降权项
- **单方模块**: 目标项目独有的 `kernel/fs/ext4.c` 等文件系统/第三方库模块，以及参考项目独有的 `kernel/fat32.c`、`kernel/spi.c`、`kernel/gpiohs.c` 等模块，直接证明双方面向不同的硬件平台（virtio vs K210）并采用了截然不同的软件架构，是独立开发的有力证据。
- **降权项**: 所有 POSIX 系统调用入口函数（如 `sys_exit`, `sys_wait4`, `sys_unlinkat`）的同名现象，因其由接口规范强制规定，不能作为代码复用的证据。`printf`, `memcmp`, `va_arg` 等 C 标准库函数或宏，属于公共知识领域，其调用图谱的相似性不构成侵权。控方将文件系统、中断处理函数归入“任务调度/异步运行时”板块进行对比，属于跨板块硬凑证据，应予以降权或排除。

#### 6. 保守结论
综合客观 AST 证据，双方项目总体相似度极低（3.31%），共同指纹仅 20 个，且大部分可归因于 POSIX 规范、C 标准库或 xv6 教学范式的公共影响。在进程管理、内存管理等核心子系统上，双方在同步策略、内存模型、对象管理上存在实质性架构差异。特别是双方拥有大量对方不存在的独立模块，反映了完全不同的硬件平台和软件设计目标。因此，现有证据无法支持“抄袭”或“系统性洗稿”的指控，目标项目更符合在公开设计范式下独立开发的特征。
