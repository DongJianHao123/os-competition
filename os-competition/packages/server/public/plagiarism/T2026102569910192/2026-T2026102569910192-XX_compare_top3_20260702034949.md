## 4. 核心代码调用流对比剖析

### 【进程管理 / 进程退出路径 `do_exit`】

**底层技术析理**：双方均实现了 Linux `do_exit` 语义，但资源清理策略与对象模型存在根本性差异。目标项目采用全局资源表集中清理模式（`FD_TABLE.clear`、`IPC_MANAGER.clear`），并通过 `FutexKey::new` 处理 futex 唤醒；参考项目采用逐文件关闭模式（`close_all_filelike`），并通过 `clear_child_tid.get_as`/`query_paddr` 处理 `set_tid_address` 语义。目标项目使用 `UserPtr::<Pid>` 泛型参数化，参考项目使用 `UserPtr::<Tid>`。两者仅共享 `SignalInfo::new`、`axtask::exit`、`axtask::yield_now` 等退出必经步骤，这些步骤由 Linux ABI 规范和 ArceOS 框架共同决定。

- 目标项目: `SWTC-la/xapi/src/task/exit.rs#do_exit`
- 参考项目: `api/src/imp/task.rs#do_exit`

### 【进程管理 / 信号屏蔽字设置 `sys_rt_sigprocmask`】

### 【进程管理 / 进程退出 `exit`（process 层）】

**底层技术析理**：双方均实现了进程退出逻辑，但复杂度与语义存在数量级差异。目标项目实现了完整的 init/reaper 子进程继承模型：通过 `Arc::downgrade` 获取父进程弱引用，通过 `INIT_PROC.get().unwrap()` 获取 init 进程作为孤儿进程的收养者，通过 `core::mem::take` 原子地取出子进程列表，通过 `reaper.children.lock` 将孤儿进程转移给 reaper。参考项目仅通过 `self.is_zombie.store(true)` 设置僵尸标志位，无子进程继承逻辑。目标项目的实现复杂度远超参考项目，不可能由参考项目简单改写而来。

### 【系统调用 / 系统调用分发 `handle_syscall`】

**底层技术析理**：双方均实现了系统调用分发入口，但错误处理与日志策略不同。目标项目包含完整的错误处理链：通过 `sysno.ok_or` 进行系统调用号校验，通过 `err.code` 提取错误码，通过 `result.unwrap_or_else` 进行结果解包，通过 `debug!` 进行调试日志输出。参考项目仅通过 `Sysno::new` 构造系统调用号，通过 `handle_syscall_imp` 直接分发，无错误处理与日志逻辑。目标项目的分发层更加健壮，包含参考项目不具备的防御性编程措施。

### 【文件系统 / 文件重命名 `sys_renameat2`】

**底层技术析理**：双方均实现了 Linux `renameat2` 系统调用，但路径解析与缓存架构完全不同。目标项目采用基于 inode 缓存的解析模式：通过 `INODE_CACHE.0.lock` 获取全局 inode 缓存锁，通过 `HashKey::new` 构造缓存键，通过 `INODE_CACHE.remove` 进行缓存失效，通过 `Arc::downgrade` 管理 inode 引用。参考项目采用基于路径的解析模式：通过 `Path::new` 构造路径对象，通过 `fs.resolve_parent`/`fs.resolve_nonexistent` 进行父目录与目标路径解析，通过 `new_path.get_as_str` 获取字符串表示。两者在 VFS 架构上存在根本性差异：目标项目以 inode 缓存为中心，参考项目以路径字符串解析为中心。

- 目标项目: `SWTC/kernel/src/syscall/fs.rs#sys_renameat2`

## 6. 保守结论

本次审计基于客观 AST 指纹比对，双方整体相似度仅 17.87%，目标项目 77.69% 的 AST 指纹为独有。控方指控的“大规模系统性复用”与覆盖率数据严重不符：进程管理板块目标覆盖率仅 11.3%，且所谓“11 处核心铁证”中 9 处位于双方共有的 ArceOS 上游框架目录，属于共同依赖而非抄袭。双方在进程管理、内存管理、系统调用、文件系统、网络协议栈等核心自研模块中存在系统性架构差异——目标项目采用 `XProcess`/`UserCheck`/`INODE_CACHE`/`IPC_MANAGER` 等独立抽象层，参考项目采用 `task_ext`/`nullable`/`Path`/`SHM_MANAGER` 等不同设计范式。第三方库（lwext4、smoltcp、page_table_multiarch）的完全一致属于正常公共依赖。综合判定：双方为基于共同上游框架（ArceOS）的独立开发项目，不构成代码抄袭。
