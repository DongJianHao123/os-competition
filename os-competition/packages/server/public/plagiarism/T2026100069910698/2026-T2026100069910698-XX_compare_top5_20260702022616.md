* **底层技术析理**: 参考项目的`exit`是一个重量级函数，直接处理文件关闭（`fileclose`）、父进程重新认领（`reparent`）、调度切换（`sched`）等全部逻辑，体现了xv6-like教学系统的单体式设计。目标项目将退出逻辑拆分为`task_set_exit`（设置状态）和`task_finish`（执行清理），职责边界清晰，体现了现代微内核/宏内核混合的模块化设计思想。两者在调用对象数量、功能拆分粒度上存在本质差异。
* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_exit` -> `src/kernel/task.c#task_set_exit`
  - 参考项目: `src/sysproc.c#sys_exit` -> `src/proc.c#exit`

* **【任务调度/异步运行时/文件描述符关闭路径】**
* **底层技术析理**: 参考项目使用统一的`fileclose`函数关闭所有类型的文件，体现了简单的文件描述符模型。目标项目采用多态fd模型，根据fd类型分别调用`event_fd_close`、`fs_close`、`open_state_close`、`pipe_fd_close`、`socket_fd_close`，最后通过`task_group_clear_fd`清理进程组fd表。这证明目标项目支持eventfd、pipe、socket等多种fd类型，其VFS抽象层远比参考项目复杂，是独立架构设计的体现。
* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_close`
  - 参考项目: `src/sysfile.c#sys_close`

* **【文件系统/路径解析路径】**
* **底层技术析理**: 参考项目的`sys_faccessat`调用`ename`和`argstr`进行FAT32文件系统的路径查找，失败时使用`printf`调试。目标项目调用`resolve_path_arg`和`fs_stat_path`，通过VFS接口进行路径解析和权限检查，支持多种文件系统后端（ext4、memfs等）。两者依赖的文件系统模型（FAT32 vs VFS）和错误处理机制（`printf` vs 静默返回）完全不同。
* **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_faccessat`
  - 参考项目: `src/sysfile.c#sys_faccessat`

#### 5. 单方模块与降权项
- **目标项目独有模块**:
  - **文件系统/第三方库**: 目标项目包含完整的ext4文件系统实现（`src/kernel/ext4.c`），以及memfs内存文件系统（`src/kernel/fs.c#memfs_compact_pool`）。参考项目使用FAT32，无任何ext4或memfs相关代码。这些模块的函数数量（35个）和复杂度表明目标项目具有独立开发的完整子系统。
  - **设备驱动**: 目标项目包含virtio-net（`src/drivers/virtio_net.c`）和loongarch64 virtio_pci_blk（`src/arch/loongarch64/virtio_pci_blk.c`）等现代驱动，参考项目仅有SD卡驱动。硬件抽象层完全不同。
- **降权项说明**:
  - **标准C库函数**: `memcpy`、`memmove`、`memset`、`strlen`等函数属于标准C库或教科书级实现，任何独立开发的内核都会包含。它们不能作为同源性证据。
  - **POSIX/Linux标准系统调用名**: `sys_getpid`、`sys_close`、`sys_dup3`等函数名由POSIX或Linux ABI强制规定，同名是遵循标准的必然结果，不代表代码复用。
  - **跨模块比对**: 控方将目标`trap_enable_timer`与参考`trapframedump`、目标`net_init`与参考`sd_read_blocks`等跨模块函数强行对齐，违反了“同板块比对”原则，已降级为弱信号，不构成主证据。

#### 6. 保守结论
本次审计基于双向AST调用图谱覆盖率（目标1.56%，参考2.17%）及7个共同指纹的客观证据。极低的覆盖率与稀疏的共同指纹，无法支撑“系统性平移抄袭”的指控。所有共同指纹均可由POSIX/Linux标准ABI强制命名、标准C库公共依赖或领域通用编程模式合理解释。双方在核心子系统上存在根本性架构差异：目标项目采用VFS+多态fd+ext4/memfs的现代设计，参考项目为xv6-like的FAT32+统一file的单体架构。目标项目还包含virtio-net、loongarch64支持、robust futex等大量独立特性。因此，综合判定两个项目为独立开发，不存在应受处罚的同源性复用行为。
