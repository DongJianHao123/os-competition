## 4. 核心代码调用流对比剖析

### 【进程管理 / wait4 系统调用路径】

- **底层技术析理**: 
  - **目标项目**: `sys_wait4` → `task_wait` → `task_wait_status_value` + `task_write_user_int` + `task_request_resched`。采用**任务对象模型**，通过 `current_task` 获取当前任务控制块，在任务内部维护子进程状态链表，使用 `task_wait_status_value` 遍历并提取退出状态，通过 `task_write_user_int` 直接写入用户态地址（经 `user_range_ok` 校验），最后调用 `task_request_resched` 触发调度。
  - **参考项目**: `sys_wait4` → `wait` → `LIST_FOREACH` + `proc_lock` + `mtx_lock`/`mtx_unlock`。采用**进程链表模型**，通过 `cpu_this` 获取当前 CPU 上下文，使用 BSD 风格的 `LIST_FOREACH` 宏遍历子进程链表，在 `proc_lock` 和 `mtx_lock` 的双重锁保护下操作进程状态，通过 `copy_out` 回写用户态。
  - **异构性判定**: 双方在**数据结构抽象**（任务对象 vs 进程链表）、**同步原语**（无显式锁 vs 自旋锁+互斥锁）、**用户态写入机制**（`task_write_user_int` vs `copy_out`）三个维度上存在根本性差异。AST 控制流图无重合节点。

### 【任务调度/异步运行时 / Futex 系统调用路径】

- **底层技术析理**:
  - **目标项目**: `sys_futex` → `task_wait_futex_until` + `task_wake_futex`。采用**内联任务调度模式**，在系统调用入口处直接完成用户态地址校验（`user_range_ok`）、参数读取（`read_user_int`）、时间转换（`timespec_to_ticks`），随后调用任务调度原语 `task_wait_futex_until`（挂起当前任务直到 Futex 值变化）和 `task_wake_futex`（唤醒等待队列中的任务）。Futex 等待逻辑与任务调度器深度耦合。
  - **参考项目**: `sys_futex` → `futex_wait` + `futex_wake` + `futex_requeue`。采用**辅助函数封装模式**，将 Futex 核心逻辑封装在独立的 `futex_wait`、`futex_wake`、`futex_requeue` 函数中，系统调用入口仅负责参数分发和错误处理（`warn`）。Futex 子系统与调度器通过 `tsleep`/`wakeup` 接口解耦。
  - **异构性判定**: 双方在**模块耦合策略**（紧耦合 vs 分层解耦）、**时间管理**（`timespec_to_ticks` vs 直接使用 `TS_USEC`）、**错误处理**（`printk` 内联日志 vs `warn` 统一错误报告）上存在显著设计差异。这是两种对立的实现哲学，而非“内联展开”的复用变体。

- **代码溯源链接**:
  - 目标项目: `src/kernel/syscall.c#sys_futex`
  - 参考项目: `kern/syscall/sys_futex.c#sys_futex`

### 【内存管理 / mmap 系统调用路径】

- **底层技术析理**:
  - **目标项目**: `sys_mmap` → `align_down` + `align_up` + `fs_read_file` + `mmap_perm_from_prot` + `sfence_vma`。采用**文件映射优先**策略，通过 `fs_read_file` 将文件内容预读入内存后再建立映射；使用 RISC-V 特有的 `sfence_vma` 指令刷新 TLB；权限转换通过独立的 `mmap_perm_from_prot` 函数完成。
  - **参考项目**: `sys_mmap` → `PGROUNDUP` + `file_map` + `getDirentByFd` + `cur_proc_fs_struct`。采用**惰性文件映射**策略，通过 `file_map` 建立文件与虚拟地址的映射关系而不立即读取内容；使用通用的 `PGROUNDUP` 宏进行页对齐；通过 `cur_proc_fs_struct` 获取当前进程的文件系统上下文。
  - **异构性判定**: 双方在**文件映射策略**（预读 vs 惰性映射）、**TLB 管理**（显式 `sfence_vma` vs 隐式依赖页表机制）、**权限模型**（独立转换函数 vs 内联宏）上存在根本性差异。

  - 参考项目: `kern/syscall/sys_mmap.c#sys_mmap`

### 【文件系统 / chdir 系统调用路径】

- **底层技术析理**:
  - **目标项目**: `sys_chdir` → `copy_user_cstr` + `fs_resolve_task_path` + `fs_is_dir_path` + `path_eq`。采用**路径字符串 VFS 模型**，通过 `fs_resolve_task_path` 将路径字符串解析为 VFS 节点，使用 `fs_is_dir_path` 判定是否为目录，通过 `path_eq` 进行路径比较。文件系统操作基于字符串路径而非文件描述符。
  - **参考项目**: `sys_chdir` → `copyInStr` + `cur_proc_fs_struct` + `getFile` + `IS_DIRECTORY` + `file_close`。采用**文件描述符表模型**，通过 `cur_proc_fs_struct` 获取进程的文件系统上下文（包含当前目录 inode），使用 `getFile` 通过路径查找文件描述符，判定后通过 `file_close` 释放引用。
