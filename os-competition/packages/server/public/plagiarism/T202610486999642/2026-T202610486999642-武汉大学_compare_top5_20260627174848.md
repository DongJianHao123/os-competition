### 4. 核心代码调用流对比剖析
* **【进程管理 / PID 查询路径】**
* **底层技术析理**: 双方均通过全局 `PID2PCB` 容器进行 PID 到进程控制块的映射。然而，目标项目使用 `exclusive_access()` 暗示其基于单核中断禁用的同步模型，而参考项目使用 `lock()` 表明其采用多核互斥锁。这是两种截然不同的并发控制架构。此外，目标项目在 `remove_from_pid2process` 中额外调用了 `unregister_task_linux_tid`，体现了其对 Linux 线程 ID 管理的独立实现。
* **代码溯源链接**:
  - 目标项目: `os/src/task/manager.rs#pid2process`
  - 参考项目: `kernel/src/task/manager.rs#pid2process`

* **【进程管理 / 获取当前进程】**
* **底层技术析理**: 参考项目通过 `current_task().unwrap().process.upgrade().unwrap()` 的弱引用升级路径获取当前进程，存在失败可能。目标项目则在 `PROCESSOR` 中直接维护 `current_process` 字段，通过 `PROCESSOR.exclusive_access().current_process()` 直接获取，路径更短且无失败分支。这体现了不同的对象生命周期管理策略。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#current_process`
  - 参考项目: `kernel/src/task/processor.rs#current_process`

* **【内存管理 / 进程 fork】**
* **底层技术析理**: 双方 `fork` 函数均遵循创建新进程、克隆地址空间、复制资源的通用流程。但目标项目的调用图谱中出现了 `ProcessCpuTimes::with_inherited_self_maxrss` 等资源统计初始化，而参考项目则包含 `RecycleAllocator::new` 等特定的内存分配器初始化。这表明双方在资源管理和内存分配策略上存在独立的、差异化的设计。
* **代码溯源链接**:
  - 目标项目: `os/src/task/process_lifecycle.rs#fork`
  - 参考项目: `kernel/src/task/process.rs#fork`

* **【任务调度 / 主调度循环】**
* **底层技术析理**: 双方 `run_tasks` 函数均包含“取任务 -> 切换上下文 -> idle”的调度骨架。但目标项目的调用流中注入了 `crate::fs::console_tty_drain_uart`（UART 控制台刷新）、`hart::enable_interrupt_and_wait`（RISC-V/LoongArch 核级中断等待）和 `perf` 性能监控调用。这些外部依赖证明目标项目是一个集成了设备驱动、多架构支持和性能分析子系统的复杂内核，而非对简单调度器的“换皮”。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#run_tasks`
  - 参考项目: `kernel/src/task/processor.rs#run_tasks`
