## 第四部分：分模块架构说明

### MM（内存管理）
底层证据体现较强：
- 物理分配：`kernel/thread.c#alloc_thread` 调用 `kalloc`；`kernel/pmem.c#buddy_safe_check` 暗示 buddy 物理页管理
- 虚拟内存：`kernel/vma.c#mmap` 实现 `mmap`，涉及 `alloc_mmap_vma`、文件映射、权限检查
- 加载与映射：`kernel/exec.c#exec` 调用 `alloc_vma_stack`、`alloc_aux`、`mappages`
- 页错误与 COW：`hsai/hsai_trap.c#pagefault_handler` 调用 `handle_cow_write`、`mappages`、`get_file_ops()->read`
- 共享内存：`user/riscv/user.c#test_shm` 使用 `sys_mmap`、`sys_munmap`、`sys_shmat`、`sys_shmctl`

**未体现**：swap、slab allocator、NUMA、大页、内存压缩、页表遍历/TLB  shootdown 的完整细节。

### PROC（进程管理/任务调度）
底层证据体现为传统 Unix 风格进程管理：
- PCB/调度：`kernel/process.c#scheduler`、`kernel/process.c#sched` 使用 `list_*` 遍历就绪队列，调用 `hsai_swtch` 上下文切换
- 生命周期：`kernel/process.c#exit`、`kernel/process.c#wait`/`waitpid`、`hsai/hsai_trap.c#forkret`
- 线程组/信号：`kernel/process.c#tgkill`、`kernel/signal.c#set_sigaction`、`kernel/signal.c#sigprocmask`
- 同步：`kernel/futex.c#futex_wait`、`futex_wait_bitset`
- procfs：`kernel/procfs.c` 提供 `/proc/pid/stat|status|task|cpuinfo|interrupts` 等

**注意**：虽然模块画像中标记为“任务调度/异步运行时”，但底层证据主要是传统 `sleep/wakeup` + 上下文切换，**独立 async runtime / waker / task channel 底层证据未体现**。

### FS（文件系统）
证据非常强，但需区分原创内核代码与第三方库：
- **内核 VFS/适配层**：`kernel/fs/vfs_ext4.c`、`kernel/fs/file.c`、`kernel/fs/inode.c`、`kernel/fs/fs.c`
- **ext4 实现/第三方库**：`kernel/fs/ext4*.c` 共 523 个函数，包括 `ext4_generic_open2`、`ext4_dir_rm`、`ext4_extent_remove_space`、`ext4_balloc_alloc_block`、`ext4_journal.c#jbd_journal_prepare` 等，疑似公共 ext4 库
- **procfs**：`kernel/procfs.c` 生成 `/proc` 下内容
- **管道/FIFO**：`kernel/fs/fifo.c#fifowrite`/`fiforead`、`kernel/fs/file.c#fileclose`
- **系统调用路径**：`sys_openat`、`sys_unlinkat`、`sys_readlinkat`、`sys_lseek`、`sys_fchdir`、`sys_fchmod`、`sys_fchown`

### DEV（设备驱动）
- **UART**：`hal/riscv/uart.c`、`hal/loongarch/uart.c`（`uart_init`、`uartputc`、`uartstart`）
- **Console**：`kernel/console.c#chardev_init`、`consoleintr`、`consolewrite`
- **磁盘**：RISC-V VirtIO（`kernel/driver/riscv/virt.c#virtio_disk_intr`）、LoongArch VirtIO PCI（`kernel/driver/loongarch/virtio_pci.c`）
- **定时器**：`hsai/timer.c#timer_init`、`kernel/timer.c#timer_init`、`timer_tick`

### SYSCALL（系统调用）
- 核心文件 `kernel/syscall.c`（145 个函数）为系统调用分发与实现主体
- 已体现：`openat`、`read`/`write`、`preadv`/`pwritev`、`readlinkat`、`lseek`、`dup3`、`close`、`exit`、`getpid`、`getppid`、`kill`、`tkill`、`sigreturn`、`ioctl`、`fchdir`、`fchown`、`fchmod`、`setpgid`、`unshare`、`mmap`、`munmap`、`shmat`、`select`/`pselect6`、`accept`、`bind` 等
- 分发入口：`kernel/syscall.c#syscall` -> `hsai_get_arg` -> 具体 `sys_*`

### TRAP/IRQ（中断异常）
- 初始化：`hsai/hsai_trap.c#hsai_trap_init` 设置异常入口、TLB 重填入口、`stvec`、定时器
- 内核态陷入：`hsai/hsai_trap.c#kerneltrap` 处理设备中断、异常、kill/exit/panic
- 用户态陷入：`hsai/hsai_trap.c#usertrap` -> `devintr` / `handle_exception` / `check_and_handle_signals` -> `hsai_usertrapret`
- 返回用户态：`hsai/hsai_trap.c#hsai_usertrapret` 设置 `sepc`、`satp`、用户态 trapframe
- 双架构 CSR 操作：LoongArch `csr_*` 与 RISC-V `sstatus`/`stvec` 等

### NET（网络协议栈）
**底层证据未体现完整网络协议栈**。仅发现：
- `kernel/socket.c#sock_bind`（1 个函数，调用 `memmove`）
- `kernel/syscall.c#syscall` 分发入口涉及 `sys_accept`、`sys_bind`，但无对应 TCP/IP、UDP、网卡驱动、协议层实现证据。

## 第六部分：边界与未体现内容

4. **安全与权限机制**：仅有 `check_file_access` / `check_root_access` 等简单检查，无 MAC、SELinux、capability、seccomp

7. **块设备/文件系统缓存策略**：ext4 journal、bcache 有代码，但原创性边界需与第三方库区分

### 需要单独说明的原创性边界
- **`kernel/fs/ext4*.c` 系列（523 个函数）** 属于文件系统/第三方库模块，疑似公共 ext4 库代码。内核主体对其的原创工作应主要体现在 `kernel/fs/vfs_ext4.c`、`kernel/fs/file.c`、`kernel/syscall.c` 等包装层与系统调用适配层。
- **双架构 HAL**：`hal/riscv/`、`hal/loongarch/`、`kernel/driver/riscv/`、`kernel/driver/loongarch/` 存在，但 HAL 层与 `kernel/` 核心之间的原创性边界需进一步区分。
- 本文档仅基于 AST Call Map、函数名、路径锚点与调用链描述技术事实，**不做抄袭/违规判断**，也不对最终身份或原创性作结论。
