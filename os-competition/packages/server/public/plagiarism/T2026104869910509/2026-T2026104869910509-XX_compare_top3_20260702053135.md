### 4. 核心代码调用流对比剖析
* **【进程管理/时间统计聚合】**
* **底层技术析理**: 控方认为此处的 `iter-map-reduce` 链是“最铁的证据”，但这恰恰是 Rust 语言特性的体现。目标项目将时间聚合逻辑拆分为 `get_process_ustime`, `get_process_utime`, `get_process_cputime` 三个独立函数，调用 `timer_mut()` 获取时间记录器；参考项目则使用 `process_time_pair`, `process_user_time`, `process_cpu_time`，调用 `time_recorder()`。两者在对象命名、方法选择上完全不同，唯一的共同点是使用了 Rust 标准库的 `Iterator::reduce` 进行求和，这是任何 Rust 开发者都会采用的最简洁、最地道的写法，属于语言惯用法而非逻辑复用。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/time.rs#get_process_ustime`
  - 参考项目: `os/src/task/task.rs#process_time_pair`

* **【任务调度/执行器设计】**
* **底层技术析理**: 双方调度器架构存在显著差异。目标项目采用**双队列设计**，将任务分为普通任务 (`tasks`) 和优先级任务 (`pritasks`)，`fetch` 操作会优先从 `pritasks` 中获取。参考项目采用**单队列抢占设计**，只有一个 `queue`，通过 `push_preempt` 将高优先级任务插入队首来实现抢占。这种调度策略上的根本不同，直接反驳了“同构复用”的指控。`push_back` 和 `pop_front` 等操作是 `VecDeque` 作为双端队列的标准 API，不具备独创性。
* **代码溯源链接**:
  - 目标项目: `lib/executor/src/lib.rs#fetch`
  - 参考项目: `os/src/executor/mod.rs#fetch`

* **【系统调用/信号处理】**
* **底层技术析理**: 在 `sys_rt_sigtimedwait` 的实现中，双方展示了完全不同的用户态内存安全抽象。目标项目使用了一套类型化的指针包装器 `UserReadPtr::<SigSet>::new` 和 `UserWritePtr::<SigInfo>::new`，在类型层面区分读写。参考项目则使用更底层的 `UserPtrRaw::new` 配合 `ensure_read`/`ensure_write` 方法，并手动进行原始指针转换 `(info_ptr as *mut SigInfo).write`。这证明双方在解决用户态内存访问这一核心安全问题上，进行了完全独立的抽象层设计。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/signal.rs#sys_rt_sigtimedwait`
  - 参考项目: `os/src/syscall/signal.rs#sys_rt_sigtimedwait`
