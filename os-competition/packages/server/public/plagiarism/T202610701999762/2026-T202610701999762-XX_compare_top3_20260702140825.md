### 4. 核心代码调用流对比剖析
*   **【进程管理 / 信号处理】**
    *   **底层技术析理**: 目标项目 `rt_sigprocmask` 通过 `need::syscall_args::read_value` 解析参数，直接操作 `info.signal_blocked` 对象进行掩码管理，并调用 `mask_without_unblockable` 处理不可阻塞信号。参考项目 `sigprocmask` 则通过 `SigMaskHow::from_bits` 解析参数，必须显式调用 `task.acquire_inner_lock` 获取锁后才能操作 `inner.sigmask`。两者在**参数解析机制、数据对象封装、锁同步策略**上完全不同，体现了异构的任务状态管理模型。
    *   **代码溯源链接**:
        - 目标项目: `kernel/src/l2/30_syscall_handlers/sys_signal/api.rs#rt_sigprocmask`
        - 参考项目: `os/src/task/signal.rs#sigprocmask`

*   **【任务调度 / Futex 机制】**
    *   **底层技术析理**: 目标项目 `wait_until` 使用 `WAITERS.with` 操作线程局部存储的等待队列，通过 `clock::now_ns` 和 `deadline_ns.is_some_and` 实现超时判断。参考项目 `do_futex_wait` 则使用 `Arc::downgrade` + `WaitQueue::new` 构建动态等待队列，通过 `TimeSpec::now` 计算时间差，并直接调用 `block_current_and_run_next` 触发调度。两者在**等待队列数据结构、时间源、调度触发方式**上存在根本性差异。
    *   **代码溯源链接**:
        - 目标项目: `kernel/src/l1/50_bridge/futex/api.rs#wait_until`
        - 参考项目: `os/src/task/threads.rs#do_futex_wait`

*   **【内存管理 / 内存映射】**
    *   **底层技术析理**: 目标项目 `mmap` 使用 `NEXT_MMAP.fetch_add` 原子操作分配虚拟地址，并通过 `data.resize` 预分配物理页。参考项目 `mmap` 则使用 `USR_MMAP_BASE` 静态基址，通过 `MapArea::new` 或 `MapArea::new_shared` 创建映射区域对象。两者在**虚拟地址空间管理策略（原子递增 vs 静态基址）、物理内存分配时机、映射对象抽象模型**上截然不同。
    *   **代码溯源链接**:
        - 目标项目: `kernel/src/l2/30_syscall_handlers/syscall_handlers/api.rs#mmap`
        - 参考项目: `os/src/mm/memory_set.rs#mmap`

### 5. 单方模块与降权项
*   **单方模块**: **网络协议栈**、**网络协议栈/第三方库**、**第三方库** 仅存在于参考项目中。这是两个项目在功能范围和架构设计上的重大差异，是独立开发的有力佐证。
*   **降权项说明**:
    *   **同名函数 `getpid`**: 目标项目 [[kernel/src/l2/30_syscall_handlers/syscall_handlers/api.rs#getpid]] 与参考项目 [[os/src/task/task.rs#getpid]] 虽同名，但实现逻辑极简，是 POSIX 标准接口的最小实现，不构成强同源证据。
    *   **跨板块配对**: 控方证据底座中，将目标项目的 `init_user_stack` 与参考项目的 `sys_shmctl` 配对，将 `rt_sigprocmask` 与 `sys_prlimit` 配对，属于跨功能、跨板块的硬凑，本庭不予采信。
    *   **第三方库重合**: 双方在 `ext4_rs` 等第三方库上存在大量共同指纹，这是双方相似度的主要来源。引用相同的开源库是正常开发行为，不能作为项目间代码抄袭的证据。

### 6. 保守结论
综合全模块 AST 指纹比对，目标项目与参考项目的客观相似度仅为 3.81%，远低于同源性阈值。控方对【进程管理】板块的指控，因该板块共同 AST 指纹为 0 而缺乏客观证据支撑，其论点建立在语义联想和跨板块硬凑之上。双方在任务调度、内存管理、文件系统等核心子系统上展现出截然不同的抽象层次、同步策略和对象管理模型。观察到的少量相似度主要集中于第三方公共库（如 ext4_rs）和硬件架构规定的标准操作（如 CSR、SBI），属于正常且不可避免的公共知识复用。因此，本庭裁定，无证据表明目标项目对参考项目存在系统性代码抄袭或逻辑复用，应认定为独立开发。
