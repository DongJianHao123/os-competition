### 4. 核心代码调用流对比剖析
* **【进程管理 / 信号掩码管理】**
* **底层技术析理**: 控方指控 `rt_sigprocmask` 与 `sigprocmask` 逻辑一致。然而，目标项目通过 `info.signal_blocked` 对象进行 `add/remove` 操作，并使用 `mask_without_unblockable` 过滤不可屏蔽信号；参考项目则通过 `inner.sigmask` 对象进行 `insert/remove` 操作，并使用 `SigMaskHow::from_bits` 解析操作类型。两者在数据结构、API 调用和错误处理（`log::warn!` vs `trace!`）上均存在显著差异。这种相似性源于 Linux `rt_sigprocmask` 系统调用的标准行为规范，而非代码复用。
* **代码溯源链接**:
  - 目标项目: `kernel/src/l2/30_syscall_handlers/sys_signal/api.rs#rt_sigprocmask`
  - 参考项目: `os/src/task/signal.rs#sigprocmask`

* **【进程管理 / Futex 等待机制】**
* **底层技术析理**: 目标项目的 `wait_until` 使用 `WAITERS.with` 闭包式线程局部存储管理等待队列，并通过 `clock::now_ns` 获取裸纳秒时间；参考项目的 `do_futex_wait` 则使用 `WaitQueue::new` 显式创建队列，并通过 `Arc::downgrade` 管理引用计数，时间获取使用 `TimeSpec::now` 结构体。两者在同步原语、内存管理模型和时间抽象上完全不同，体现了截然不同的内核设计哲学。
* **代码溯源链接**:
  - 目标项目: `kernel/src/l1/50_bridge/futex/api.rs#wait_until`
  - 参考项目: `os/src/task/threads.rs#do_futex_wait`

* **【进程管理 / Execve 栈构建】**
* **底层技术析理**: 目标项目将栈构建拆分为 `init_user_stack`、`push_bytes`、`push_usize` 等函数，使用 `user_memory::write(space, *sp, data)` 直接写入用户地址空间；参考项目则在庞大的 `sys_execve` 函数中通过 `copy_from_user` / `copy_to_user` 进行数据拷贝。目标项目使用 `sp.checked_sub` 进行栈指针边界检查，参考项目则无此显式操作。两者在内存访问抽象和错误处理模型上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/l1/50_bridge/exec/exec.rs#init_user_stack`
  - 参考项目: `os/src/syscall/process.rs#sys_execve`

* **【进程管理 / Procfs 文件系统】**
* **底层技术析理**: 目标项目采用扁平化设计，通过 `node_normalized` 函数将路径直接映射为 `Node::Dir` 或 `Node::File`，并使用 `file_content` 函数硬编码文件内容；参考项目则采用树形 `DirectoryTreeNode` 结构，通过 `ROOT.mkdir` 和 `FileSystem::new` 动态构建目录树。两者在 VFS 架构、路径解析和元数据管理上完全不同。
