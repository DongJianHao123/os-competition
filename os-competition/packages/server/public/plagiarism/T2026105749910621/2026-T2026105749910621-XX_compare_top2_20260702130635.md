### 4. 核心代码调用流对比剖析

* **【进程管理/调度器入口】**
  * **底层技术析理**: 双方均实现了 xv6 风格的调度器循环。然而，目标项目在 `scheduler()` 中显式调用了 `intr_off()`，而参考项目未调用。这并非简单的增删，而是反映了不同的中断管理策略：目标项目可能在调度入口采取了更保守的全关中断策略，而参考项目则依赖 `swtch` 内部的机制。这构成了同步策略上的架构差异。
  * **代码溯源链接**:
    - 目标项目: `kernel-riscv/kernel/proc.c#scheduler`
    - 参考项目: `kernel/proc.c#scheduler`

* **【文件系统/目录查找】**
  * **底层技术析理**: 双方均有 `dirlookup` 函数，但底层文件系统架构截然不同。目标项目基于 xv6 原始 inode 模型，调用 `iget`, `readi` 等函数操作 inode 缓存。参考项目则实现了 FAT32 文件系统，调用 `edup`, `eget`, `enext` 等函数遍历 FAT 簇链。这是内存模型和对象管理模型的根本性差异，排除了代码抄袭的可能。
  * **代码溯源链接**:
    - 目标项目: `kernel-riscv/kernel/fs.c#dirlookup`
    - 参考项目: `kernel/fat32.c#dirlookup`

* **【系统调用/进程退出】**
  * **底层技术析理**: 目标项目的 `sys_exit` 调用 `kexit`，而 `kexit` 的调用图谱包含了 `proc_close_linux_fds`, `contest_runner_finish` 等 Linux 兼容层和竞赛框架的清理逻辑。参考项目的 `sys_exit` 调用 `exit`，其调用图谱包含 `fat32_init` 等 FAT32 文件系统回调。两者在内核对象回收和子系统交互的抽象层次上完全不同。
  * **代码溯源链接**:
    - 目标项目: `kernel-riscv/kernel/sysproc.c#sys_exit` -> `kernel-riscv/kernel/proc.c#kexit`
    - 参考项目: `kernel/sysproc.c#sys_exit` -> `kernel/proc.c#exit`
