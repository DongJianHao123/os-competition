#### 4. 核心代码调用流对比剖析

* **【进程管理/调度原语】**
* **底层技术析理**: 双方 `sched` 函数的调用流均为 `holding→intr_get→mycpu→myproc→panic→swtch`。这是 RISC-V 多核调度器上下文切换的“标准答案”：必须检查锁持有、禁用中断、获取当前 CPU 和进程、通过 `swtch` 汇编切换上下文。任何基于 xv6-RISC-V 的正确实现都会收敛于此调用图，无法作为抄袭证据。目标项目独有的 `scheduler` 函数（调用 `intr_off/intr_on`）体现了其额外的 CPU 调度外层逻辑。
* **代码溯源链接**:
  - 目标项目: `kernel-riscv/kernel/proc.c#sched`
  - 参考项目: `kernel/proc.c#sched`

* **【任务调度/自旋锁】**
* **底层技术析理**: 双方 `acquire` 函数的核心调用流均为 `holding→mycpu→panic→push_off`，这是自旋锁协议的必然步骤。关键差异在于原子操作：目标项目使用 C11 标准的 `__atomic_exchange_n`，参考项目使用 GCC 旧版 `__sync_lock_test_and_set` 和 `__sync_synchronize`。这是编译器标准代际差异，非恶意伪装。`release` 函数同理（`__atomic_store_n` vs `__sync_lock_release`）。
* **代码溯源链接**:
  - 目标项目: `kernel-riscv/kernel/spinlock.c#acquire`
  - 参考项目: `kernel/spinlock.c#acquire`

* **【内存管理/文件系统后端】**
* **底层技术析理**: 双方 `sys_open` 均完成路径解析到文件描述符的映射，但底层对象模型截然不同。目标项目调用 `create→dirlookup→ialloc→ilock`，基于 xv6 原生 inode 和日志层（`begin_op/end_op`）；参考项目调用 `create→enameparent→ealloc→elock`，基于 FAT32 的 `ename`/`ealloc` 对象。这是两种互不兼容的文件系统架构，不可能通过简单重命名转换。
* **代码溯源链接**:
  - 目标项目: `kernel-riscv/kernel/sysfile.c#sys_open`
  - 参考项目: `kernel/sysfile.c#sys_open`

* **【中断/异常处理】**
* **底层技术析理**: 双方 `kerneltrap` 均处理 RISC-V 陷阱，调用 `devintr→intr_get→r_scause→r_sepc` 等寄存器读取函数。这是 RISC-V 特权架构手册规定的标准流程。目标项目调用图含 `+4` 项，参考项目含 `+5` 项，具体处理路径存在差异。该模块共同 AST 指纹为 0，表明中断处理逻辑独立实现。
* **代码溯源链接**:
  - 目标项目: `kernel-riscv/kernel/trap.c#kerneltrap`
  - 参考项目: `kernel/trap.c#kerneltrap`

#### 5. 单方模块与降权项

*   **目标项目独有模块**:
    *   **Linux 兼容层** (`kernel/linux_syscall.c`, `kernel/linux_exec.c`): 包含 `linux_proc_virtual_file`、`linux_sys_exit`、`collect_linux_exec_args` 等 37 个系统调用函数。参考项目无任何对应模块，是目标项目架构独立性的核心证据。
    *   **ext4_lite 文件系统** (`kernel/ext4_lite.c`): 提供 `ext4_mount`、`ext4_lite_read_file` 等轻量级 ext4 支持。参考项目使用 FAT32，双方文件系统后端完全不同。
    *   **竞赛运行器** (`contest_runner_finish`): 目标项目特有的竞赛管理逻辑，参考项目不存在。
*   **参考项目独有模块**:
    *   **FAT32 文件系统** (`kernel/fat32.c`): 参考项目使用 FAT32 作为主要文件系统，目标项目使用 xv6 inode/日志 + ext4_lite。
    *   **DMAC 驱动** (`kernel/dmac.c`): 参考项目包含 DMA 控制器驱动 (`dmac_wait_idle`, `dmac_wait_done`)，目标项目无此模块。
*   **跨板块归因降权项**:
    *   `user/sh.c` 中的 `parseexec`、`runcmd`、`gettoken`、`nulterminate`、`fork1` 属于**用户态 Shell**，被控方错误归入“进程管理”或“任务调度/异步运行时”。这些函数是 xv6 公开教学 Shell 的标准实现，任何基于 xv6 的项目都会包含，不构成跨项目复用证据。
    *   `kernel/console.c` 中的 `consoleintr`、`consoleread` 属于**字符设备驱动**，被控方错误归入“任务调度/异步运行时”。
    *   `kernel/trap.c` 中的 `kerneltrap` 属于**中断/异常处理**，被控方错误归入“任务调度/异步运行时”。
*   **公共架构必然收敛降权项**:
    *   `mycpu`、`cpuid`、`sys_getpid`、`sys_uptime` 等函数，其调用图由 RISC-V 硬件寄存器（`tp`）、CPU 数组索引、进程 PCB 指针等唯一确定，是任何正确实现的必然路径，不能作为抄袭证据。
    *   `zombie.c` 的 `main` 函数：目标使用 `pause`（Linux 信号语义），参考使用 `sleep`（xv6 原生延时），语义不同，非等价替换。

#### 6. 保守结论

本次审计的目标项目与参考项目之间，客观 AST 相似度仅为 12.27%，双方约 80% 的代码指纹是独立的。控方所指控的“大规模照搬”与“恶意混淆”，实质上是将基于同一公开教学内核（xv6-RISC-V）独立开发时不可避免的公共架构收敛，误读为抄袭证据。目标项目独有的 Linux 兼容层、ext4_lite 文件系统、竞赛运行器等模块，与参考项目的 FAT32、DMAC 等模块形成了根本性的架构差异，排除了通过简单重命名或 API 替换进行复用的可能。综合判定，两项目为基于公共开源架构的独立开发成果，不存在足以构成抄袭的同源性。
