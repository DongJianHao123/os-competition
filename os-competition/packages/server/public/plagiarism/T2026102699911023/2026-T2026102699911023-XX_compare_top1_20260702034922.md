#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 在 `do_exit` 这一核心函数中，双方均调用了 `FutexKey::new_current`, `SignalInfo::new_kernel`, `axtask::yield_now`, `clear_child_tid.vm_write` 等关键操作，证明两者都遵循了 Linux 进程退出的标准步骤。然而，关键的**同步与资源回收顺序**揭示了架构差异：目标项目在设置 `clear_child_tid` 之后才调用 `SHM_MANAGER.lock().clear_proc_shm` 清理共享内存，而参考项目则在函数体更早的位置执行此操作。这种核心流程中调用顺序的重新排列，是独立实现而非代码搬运的有力证明。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/ops.rs#do_exit`
  - 参考项目: `api/src/task.rs#do_exit`

* **【进程管理/PIDFd 构造路径】**
* **底层技术析理**: `sys_pidfd_open` 系统调用的实现展示了不同的对象构造哲学。目标项目通过 `PidFd::new_process` 和 `PidFd::new_thread` 两个独立路径，在构造阶段就区分了进程和线程文件描述符，并引入了 `PidFdFlags` 进行标志位解析。参考项目则使用统一的 `PidFd::new` 构造器，并依赖 `get_process_data` 获取数据。这种在对象创建模型上的根本性差异，是架构设计层面的异构。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/fs/pidfd.rs#sys_pidfd_open`
  - 参考项目: `api/src/syscall/fs/pidfd.rs#sys_pidfd_open`

* **【任务调度/异步运行时/Futex 等待路径】**
* **底层技术析理**: 在 `wait_if` 函数中，双方都遵循了 `Poll::Ready -> Some -> condition.take -> cx.waker().clone` 的异步等待标准模式。但目标项目调用 `block_on` 进行阻塞，而参考项目调用 `block_on_interruptible`。这并非简单的重命名，`block_on_interruptible` 意味着参考项目的运行时支持信号中断的阻塞等待，而目标项目的 `block_on` 则不支持，这指向了底层异步运行时在**可中断性**这一核心特性上的不同实现。
* **代码溯源链接**:
  - 目标项目: `kernel/src/task/futex.rs#wait_if`
  - 参考项目: `core/src/futex.rs#wait_if`

#### 5. 单方模块与降权项
- **单方模块**: **设备驱动**（120个函数）与**网络协议栈**（138个函数）仅存在于参考项目中。这是双方架构差异的最显著标志，直接证明目标项目并非对参考项目的“整体搬运”，而是进行了大幅度的裁剪或采用了完全不同的技术栈。
- **降权项**:
  - **第三方库/公共依赖**: `axfs-ng` 库中的函数（如 `inode.rs` 中的 `create`, `metadata` 等）呈现“同板块同名同结构”，覆盖率高达 89.19%。此为上游库继承，必须从原创性争议中剥离。

#### 6. 保守结论
综合全案证据，目标项目与参考项目 25.25% 的 AST 相似度是双方基于同一开源框架（ArceOS）和同一领域规范（POSIX/Linux）进行独立开发的合理结果。控方所指控的“进程管理”板块系统性复用，实为将 POSIX 标准接口、ArceOS 框架 API 调用序列以及 VFS、IPC 等不同子系统函数强行归并后制造的假象。双方在核心架构上存在根本性差异：目标项目缺失网络与驱动模块，采用了不同的错误处理体系（`AxError` vs `LinuxError`）、不同的 PIDFd 对象模型，并在核心退出流程中展现出不同的同步顺序。因此，现有证据远不足以支持“重度抄袭/复用”的判定，应认定为独立开发作品。
