#### 4. 核心代码调用流对比剖析

- **【进程管理/`sys_waitpid` 调用路径】**
  - **底层技术析理**: 目标项目通过 `current_task()` 获取当前任务控制块，然后遍历子进程列表，对每个 `child` 调用 `try_inner_borrow()` 尝试获取内部锁，再通过 `.map(|inner| inner.is_zombie())` 检查僵尸状态。这体现了 Rust 的 `Option` 和 `Result` 错误处理范式，以及基于 `RefCell` 或 `Mutex` 的内部可变性同步策略。参考项目则直接调用 `argaddr`、`argint` 提取参数，然后传入 `waitpid` 函数，其内部依赖全局进程表和 `acquire/release` 自旋锁。两者在内存模型（所有权 vs 全局变量）和同步原语（内部可变性 vs 显式锁）上存在根本性架构差异。
  - **代码溯源链接**:
    - 目标项目: `os/src/syscall/process.rs#sys_waitpid`
    - 参考项目: `kernel/syscall/sysproc.c#sys_waitpid`

- **【内存管理/`sys_mmap` 调用路径】**
  - **底层技术析理**: 目标项目的 `sys_mmap` 调用 `current_task().unwrap()` 获取任务，进而访问其 `memory_set` 进行区域分配，并处理文件映射（如 `DevZero`）。这体现了内存管理被封装在任务对象内的面向对象/所有权模型。参考项目的 `sys_mmap` 则使用 `argfd`、`argint` 解析参数，调用 `get_fops` 获取文件操作表，并使用 `PGROUNDUP` 等宏进行页对齐操作，是典型的 C 语言面向过程的内核实现。两者的抽象层次（对象封装 vs 函数操作）和内存模型（基于能力的内存集 vs 全局页表操作）截然不同。
  - **代码溯源链接**:
    - 目标项目: `os/src/syscall/process.rs#sys_mmap`
    - 参考项目: `kernel/syscall/sysproc.c#sys_mmap`

- **【文件系统/`sys_openat` 调用路径】**
  - **底层技术析理**: 目标项目在 `sys_openat` 中构造 `OpenFlags` 对象，并根据文件类型动态分发，例如为 `/dev/zero` 创建 `DevZero::new()` 设备文件对象，体现了 Rust 的 trait 对象和多态特性。参考项目则通过 `argstr` 获取路径字符串，调用 `get_absolute_path` 进行路径解析，然后使用 `filealloc` 和 `fdalloc` 进行文件描述符分配，是典型的 C 语言文件系统实现。两者在对象管理模型（trait 对象动态分发 vs 函数指针表）上存在显著差异。
  - **代码溯源链接**:
    - 目标项目: `os/src/syscall/fs.rs#sys_openat`
    - 参考项目: `kernel/syscall/sysfile.c#sys_openat`

#### 5. 单方模块与降权项

- **单方模块**:
  - **网络协议栈**: 仅目标项目具备，包含 `os/src/net/socket.rs`、`os/src/net/mod.rs` 等，基于 `smoltcp` 库实现。参考项目无此模块，是目标项目独立开发的有力证据。
  - **中断/异常处理**: 仅参考项目作为独立模块体现，包含 `kernel/trap/la/iointc-2k1000.c` 等。目标项目的中断处理可能分散在其他模块中，未形成独立指纹。

- **降权项**:
  - **第三方库/公共依赖 (`lwext4`)**: 双方 `文件系统/第三方库` 模块中 382 个共同指纹均来自开源库 `lwext4`。这是公共代码，必须从原创性比对中剔除。
  - **公共测试套件 (`basic/user/src/oscomp/`)**: 用户态测试文件如 `waitpid.c`、`fork.c` 等，其路径、宏 (`TEST_START`/`TEST_END`)、函数名和调用链均由 OS 竞赛测试框架统一规定，非任何一方原创，不能作为抄袭证据。
  - **POSIX 同名系统调用**: `sys_fork`、`sys_exit`、`sys_getpid` 等函数名是 Linux/POSIX 标准接口名称，任何实现该 ABI 的项目都必须使用。同名不代表同源。
  - **跨语言弱信号**: 所有内核态系统调用（目标 Rust vs 参考 C）因语言范式导致的 AST 结构差异巨大，即使存在少量共同指纹，也仅能反映语义收敛，不能证明代码复用。

#### 6. 保守结论

综合评估，目标项目与参考项目的整体 AST 相似度仅为 21.82%，且绝大部分重合（382/441）源于双方共同引入的第三方开源库 `lwext4`。在剔除第三方库和公共测试套件后，两项目在原创内核代码层面的 AST 重合度极低（不足 3.3%）。内核核心模块如内存管理、任务调度、设备驱动等，因语言范式（Rust vs C）和架构设计（所有权模型 vs 全局变量、trait 对象 vs 函数指针）的差异，呈现出根本性的异构特征。用户态测试代码的重合源于公共竞赛框架，内核系统调用的同名源于 POSIX 标准。因此，现有证据无法支持目标项目对参考项目存在原创代码复用的指控，整体抄袭风险极低。
