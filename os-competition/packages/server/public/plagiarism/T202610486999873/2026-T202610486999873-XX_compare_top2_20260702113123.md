### 4. 核心代码调用流对比剖析
* **【进程管理 / 任务调度核心】**
* **底层技术析理**: 双方调度器架构存在根本性差异。目标项目采用**单核全局调度器**模型，通过`PROCESSOR.exclusive_access()`获取全局唯一的管理器实例，再从中获取当前任务。参考项目则采用**多核（Hart）感知的调度器**模型，通过`hart_id()`获取当前核心ID，再从对应核心的处理器实例中获取任务。这是两种完全不同的并发与CPU抽象模型，非简单“洗稿”所能实现。
* **代码溯源链接**:

  - 参考项目: `os/src/task/processor.rs#current_task`

* **【进程管理 / 获取父进程ID】**
* **底层技术析理**: 双方在进程关系管理上采用了不同的数据模型。目标项目使用`Weak`（弱引用）指针管理父进程关系，调用`parent_weak.upgrade().map(...)`来安全地访问可能已退出的父进程。参考项目则在任务控制块中直接存储`ppid`字段，通过`current_task().unwrap().ppid`直接访问。弱引用升级模式与直接字段访问模式在所有权和生命周期管理上存在本质区别。
* **代码溯源链接**:

* **底层技术析理**: 双方在用户态-内核态数据交换上采用了不同的内存访问抽象。目标项目通过`crate::mm::translated_refmut`和`inner.memory_set.valid_user_buf`等接口，基于页表翻译和内存集进行访问控制。参考项目则使用`put_data`、`get_data`、`if_bad_address`等辅助函数进行地址有效性检查和数据拷贝。这是两套独立设计的内存安全抽象层。

* **单方模块**:

### 6. 保守结论
本次审计的客观AST相似度为22.19%，处于低位。该相似度主要由双方共同引用的第三方C库`lwext4`贡献，剥离该公共依赖后，内核原创代码的相似度极低。控方提出的“铁证”均可被公共教学模板、Rust语言惯用法或POSIX接口规范合理解释，不构成抄袭。双方在进程管理、任务调度、内存管理等核心模块的架构设计、同步原语和内存模型上存在根本性差异，例如单核全局锁与多核Hart分发模型、弱引用与直接字段访问等，这些是独立设计的典型特征。综上，现有证据无法支持代码复用指控，目标项目应被认定为独立开发，整体同源性风险低。
