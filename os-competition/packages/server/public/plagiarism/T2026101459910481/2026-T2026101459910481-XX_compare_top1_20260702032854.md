### 4. 核心代码调用流对比剖析
* **【进程管理/调度原语】**
* **底层技术析理**: 双方均实现了 `sleep` 函数，但抽象层次与同步策略截然不同。目标项目基于 `WaitQueue` 等待队列对象，通过 `self.lock.lock` 获取队列锁，`self.tasks.add_task` 将任务加入队列，最后调用 `sleep_task` 挂起。这是一个面向对象的、显式的等待队列模型。参考项目则直接操作全局进程表，通过 `acquire` 获取进程锁，`myproc` 获取当前进程指针，修改状态后调用 `sched` 进入调度器。这是典型的 xv6 风格进程睡眠模型。两者在对象管理、锁粒度、挂起机制上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `os/src/kernel/scheduler/wait_queue.rs#sleep`
  - 参考项目: `kernel/proc.c#sleep`

* **【进程管理/子进程调度返回】**
* **底层技术析理**: 双方均实现了 `forkret` 函数，用于 fork 后子进程的首次调度返回。目标项目的调用流围绕 Rust 的所有权模型展开：通过 `cpu.current_task.as_ref().expect(...)` 显式处理 `Option`，克隆任务指针，然后调用架构相关的 `crate::arch::forkret_restore` 恢复陷阱帧。参考项目的调用流则是典型的 C 过程式调用：`myproc` 获取当前进程，`fsinit` 初始化文件系统，`release` 释放锁，最后 `usertrapret` 返回用户态。目标项目无文件系统初始化步骤，且通过 `PreemptGuard` 管理抢占，体现了不同的初始化顺序和并发控制策略。
* **代码溯源链接**:
  - 目标项目: `os/src/kernel/task/mod.rs#forkret`
  - 参考项目: `kernel/proc.c#forkret`

* **【内存管理/内存映射】**
* **底层技术析理**: 双方均实现了 `mmap` 系统调用。目标项目的调用流展示了 Rust 的安全抽象：使用 `MapFlags::from_bits_truncate` 和 `ProtFlags::from_bits_truncate` 处理标志位，通过 `VA::from_usize` 和 `Vpn::from_addr_ceil` 进行地址计算，体现了强类型和内存安全的设计。参考项目的调用流则使用 `PGROUNDUP` 宏进行地址对齐，`kalloc` 分配物理页，`idup`/`ilock` 操作 inode，是典型的 C 语言手动内存管理风格。两者在地址空间抽象和内存分配路径上完全不同。
* **代码溯源链接**:
  - 目标项目: `os/src/kernel/syscall/mm.rs#mmap`
  - 参考项目: `kernel/proc.c#mmap`
