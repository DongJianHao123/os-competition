## 第四部分：分模块架构说明

### MM（内存管理）
底层证据较强。包含连续物理页框分配器（`alloc_contig_frames`/`dealloc_contig_frames`）、用户地址空间 `MemorySpace`、ELF 加载器、系统调用 `mmap`/`brk`、页表映射区域拆分与权限修改等。
- 证据锚点：`os/src/mm/frame_allocator/allocator.rs#dealloc_contig_frames`、`os/src/mm/memory_space/space/elf_loader.rs#from_elf`、`os/src/kernel/syscall/mm.rs#mmap`、`os/src/mm/memory_space/mapping_area/split_ops.rs#partial_change_permission`

### PROC（进程管理/任务调度）
传统进程管理证据较充分：存在 PCB（`TaskStruct`）、任务管理器（`TASK_MANAGER`）、父子关系与进程组、`fork`/`clone`/`exec`/`wait4`/`exit_group`、zombie 回收、`/proc` 文件系统暴露进程信息。调度方面以 RR 调度器（`rr_scheduler.rs`）和等待队列（`wait_queue.rs`）为主，配合 futex、信号、定时器队列。**轻量级 async runtime / waker / ring_fifo 等异步运行时原语底层证据未体现**。
- 证据锚点：`os/src/kernel/syscall/task/wait_ops.rs#wait4`、`os/src/kernel/syscall/task/clone_ops.rs#clone`、`os/src/kernel/task/process.rs#exit_process_with_status`、`os/src/kernel/scheduler/rr_scheduler.rs#next_task`、`os/src/kernel/scheduler/wait_queue.rs#wake_up`、`os/src/fs/proc/inode.rs#readdir`

### FS（文件系统）
VFS 层证据完整：挂载表 `MOUNT_TABLE`、目录项缓存 `DENTRY_CACHE`、mount/umount。具体文件系统包括 tmpfs、vfat、ext4（内核侧包装 + 第三方 `ext4_rs`）、procfs、simple_fs。第三方 `ext4_rs` 实现了块分配、索引节点分配、extent 管理等，但属于 `third_party`，不应视为原创内核主体。
- 证据锚点：`os/src/vfs/mount.rs#mount`、`os/src/vfs/mount.rs#umount`、`os/src/fs/tmpfs/inode.rs#rename`、`os/src/fs/vfat/inode.rs#rename`、`os/src/fs/ext4/inode.rs#rename`、`os/src/fs/proc/inode.rs#readdir`、`third_party/ext4_rs/src/ext4_impls/balloc.rs#balloc_alloc_block_batch`

### DEV（设备驱动）
设备框架证据中等。支持设备树解析、UART16550 串口、Goldfish RTC、PCIe 总线扫描、VirtIO-Blk 块设备、GPT 分区表读取，以及网络设备注册。未看到 USB、GPU、ACPI、NVMe、AHCI 等驱动证据。
- 证据锚点：`os/src/device/device_tree.rs#init`、`os/src/device/serial/uart16550.rs#init`、`os/src/device/rtc/rtc_goldfish.rs#init_dt`、`os/src/device/bus/pcie.rs#from_fdt`、`os/src/device/block/virtio_blk.rs#init_pci`、`os/src/device/block/partition.rs#read_gpt_partitions`、`os/src/net/mod.rs#register_net_device`

### SYSCALL（系统调用）
存在统一的系统调用分发器 `dispatch_syscall`，覆盖 fs、mm、network、task、signal、io、sys 等类别。`fcntl`、`syslog`、`getrandom`、`brk`、`mmap`、`mount`、`getifaddrs` 等均有实现。
- 证据锚点：`os/src/kernel/syscall/dispatch.rs#dispatch_syscall`、`os/src/kernel/syscall/sys.rs#syslog`、`os/src/kernel/syscall/mm.rs#mmap`、`os/src/kernel/syscall/fs/mount_ops.rs#mount`

### TRAP/IRQ（中断异常）
支持 RISC-V、LoongArch、mock 三套架构的 trap 入口。包含用户态/内核态 trap 处理、系统调用分发、定时器/设备中断、IPI、trap frame 初始化、中断嵌套深度管理（`intr_guard`）。PLIC 中断控制器通过设备树初始化。
- 证据锚点：`os/src/arch/riscv/trap/trap_handler.rs#trap_handler`、`os/src/arch/riscv/trap/trap_handler.rs#user_trap`、`os/src/arch/loongarch/trap/trap_handler.rs#user_trap`、`os/src/arch/loongarch/trap/trap_handler.rs#install_trap_entry`、`os/src/arch/riscv/trap/trap_frame.rs#set_exec_trap_frame`、`os/src/sync/intr_guard.rs#new`、`os/src/device/irq/plic.rs#init_dt`

### NET（网络协议栈）
网络栈以第三方 `smoltcp` 为核心，内核侧提供接口管理、socket 抽象、Unix Domain Socket、TCP/UDP socket 操作、`poll_smoltcp` 轮询、ifaddrs 与路由配置。原始网络驱动（如 e1000/virtio-net 具体实现）底层证据未直接体现，仅看到网络设备注册与接口封装。
- 证据锚点：`os/src/net/interface.rs#create_smoltcp_interface`、`os/src/net/stack/mod.rs#poll_smoltcp`、`os/src/net/stack/mod.rs#socket_read`、`os/src/net/unix_socket.rs#bind`、`os/src/net/socket.rs#udp_push`、`os/src/net/config.rs#set_interface_config`

## 第六部分：边界与未体现内容

### 1. 没有底层证据支撑的模块/特性
- **异步运行时原语**：现有证据为传统任务调度器、等待队列、futex，未体现 `waker`、`ring_fifo`、`async/await` 运行时等轻量异步机制
- **虚拟内存高级机制**：未体现页面置换（swap）、页缓存（page cache）、写时复制（COW）的显式证据
- **块设备/文件系统缓存**：未看到独立的块缓存或页缓存层证据
- **原始网络驱动**：仅看到网络设备注册与 `smoltcp` 封装，未看到 virtio-net/e1000/DMA 描述符等具体网卡驱动实现
- **安全/权限框架**：除基本 `FileMode`、`AtFlags`、`check_syslog_permission` 外，未看到 MAC、Capabilities、SELinux 等安全模块
- **高级总线与外设**：未体现 USB、GPU、NVMe、AHCI、SATA、ACPI 等驱动
- **SMP 多核细节**：`handle_ipi` 有调用，但多核调度、CPU 亲和性、跨核 TLB shootdown 等证据有限
- **EXT4 日志（JBD2）**：`ext4_rs` 第三方库内部是否支持日志底层证据未直接体现

### 2. 第三方与公共依赖边界
- `third_party/ext4_rs` 是独立的 EXT4 文件系统 Rust 实现，属于第三方库，不应夸大其为参赛作品原创内核主体
- 网络协议栈核心依赖 `smoltcp`，内核侧主要为接口封装与 socket 抽象
- `mock` 架构目录表明存在模拟/测试目标，不等同于真实硬件支持

### 3. 判断性质说明
- 本报告仅基于 AST Call Map、函数名、模块路径与调用链进行技术描述
- “RISC-V / LoongArch 支持”仅为根据 `os/src/arch/` 目录结构得出的可能方向，不代表最终硬件适配结论
- 未对代码原创性、抄袭或违规情况进行任何判断
