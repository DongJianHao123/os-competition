### 4. 核心代码调用流对比剖析
* **【进程管理/Procfs读取链路】**
* **底层技术析理**: 控方指控的`generate -> format! -> to_string() -> as_bytes() -> copy_from_slice`链路是Rust语言中进行字符串格式化和缓冲区操作的**标准范式**，不具备独创性。目标项目采用**“生成器模式”**，将`/proc`下各文件抽象为`Generator` trait，通过`ProcInodeContent::Dynamic`统一调度；参考项目则为每个文件实现独立的`read`方法。两者在架构抽象层次上完全不同。目标项目的`psmem.rs#generate`核心调用对象是`TASK_MANAGER`和`collect_user_vm_stats`，而参考项目的`meminfo.rs#read`核心调用是`MemInfo::generate`，两者获取数据的来源和对象模型截然不同。
* **代码溯源链接**:
  - 目标项目: `os/src/fs/proc/generators/psmem.rs#generate`
  - 参考项目: `os/src/fs/procfs/meminfo.rs#read`

* **【进程管理/Futex等待机制】**
* **底层技术析理**: Futex的“检查值-挂起等待-唤醒”流程由Linux/POSIX规范强制定义，属于**领域强制语义**。目标项目采用**全局Futex管理器（`FUTEX_MANAGER`）与定时器队列（`TIMER_QUEUE`）分离的两级锁设计**；参考项目则在每个任务结构体内部嵌入`futex`对象和`WaitQueue`。目标项目的`futex_wait_common`调用`fm.get_wait_queue(paddr).add_task`，参考项目的`do_futex_wait`调用`futex.inner.insert`，两者在同步原语粒度、对象生命周期管理上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `os/src/kernel/syscall/task/futex_ops.rs#futex_wait_common`
  - 参考项目: `os/src/task/threads.rs#do_futex_wait`

* **【进程管理/信号处理】**
* **底层技术析理**: `rt_sigprocmask`系统调用的`how, set, oldset`参数及null检查是Linux ABI的强制要求。目标项目使用`normalize_signal_mask`和`t.blocked.bits`操作信号掩码，并通过`read_from_user`/`write_to_user`进行用户态拷贝；参考项目使用`SigMaskHow::from_bits`和`inner.sigmask.insert/remove`，并通过`copy_from_user`/`copy_to_user`进行拷贝。两者在系统调用封装层、内部数据结构操作方式上完全不同。
* **代码溯源链接**:
  - 目标项目: `os/src/kernel/syscall/signal.rs#rt_sigprocmask`
  - 参考项目: `os/src/task/signal.rs#sigprocmask`

### 5. 单方模块与降权项
*   **第三方库/公共依赖**: 参考项目独有的`第三方库`模块（如`dependency/dep_pci`、`dependency/rustsbi`）及双方共有的`文件系统/第三方库`模块（如`ext4_rs`）中的高覆盖率，均源于对公共上游代码的依赖，**不构成项目间的抄袭证据**。
*   **通用接口与泛型函数名**: `init_procfs`、`unlink`、`add_task`、`pop_task`等函数名是操作系统领域的通用术语或POSIX标准接口名。`Arc::new`、`SpinLock::new`、`Vec::new`、`format!`等是Rust标准库的通用方法。这些元素出现在任何Rust语言编写的操作系统中都是正常现象，**不具备独创性指纹价值**。
*   **跨板块配对**: 控方将目标项目的`wait_for_signal`与参考项目的`sys_execve`进行配对，属于**跨功能、跨板块的强行比对**，此类证据无效。

### 6. 保守结论
综合全模块AST覆盖矩阵与核心调用流分析，目标项目与参考项目的整体相似度极低（3.77%）。控方提出的“进程管理”板块抄袭指控，因该板块共同AST指纹为零且覆盖率0.0%而缺乏客观证据支撑。双方在文件系统/第三方库模块的相似度源于对公共`ext4_rs`库的正常依赖。在其他核心模块中，观察到的少量相似性均源于POSIX/Linux规范、RISC-V/LoongArch硬件架构规范或Rust语言标准范式，属于领域强制语义或通用写法。双方在内核对象模型、同步机制、系统调用封装等关键架构决策上展现出根本性差异。因此，本庭裁定，**无证据表明目标项目对参考项目构成代码抄袭，认定其为独立开发。**
