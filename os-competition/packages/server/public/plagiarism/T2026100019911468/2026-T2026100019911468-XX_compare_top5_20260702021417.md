    *   **底层技术析理**: 双方 `exec()` 函数的实现存在**显著架构差异**。目标项目调用 `MAKE_SATP, PGROUNDUP, PX, elock, copyout`，而参考项目调用 `CopyString2Stack, CreateUserStack, __debug_info, copyout`。目标项目使用 `MAKE_SATP` 宏直接构造页表基址，而参考项目则通过 `CreateUserStack` 等辅助函数构建用户栈。两者在内存映射、栈构造和调试支持上采用了不同的抽象层次和实现策略，否定了“原样照搬”的指控。
    *   **代码溯源链接**:
        - 目标项目: `xv6-k210/kernel/exec.c#exec`
        - 参考项目: `kernel/exec.c#exec`

*   **【文件系统/初始化路径】**
    *   **底层技术析理**: 双方 `fat32_init()` 函数均调用 `bread, brelse, initlock...` 以初始化 FAT32 文件系统。然而，目标项目的调用流中**额外包含 `ext4_init, ext4_probe`**，这表明其文件系统架构是 FAT32 + ext4 的双模设计。参考项目则无此 ext4 调用。这一差异是目标项目独立集成第三方库、扩展系统功能的直接证据。

        - 参考项目: `kernel/fat32.c#fat32_init`

### 5. 单方模块与降权项
*   **单方独有模块（目标项目）**:
    *   **文件系统/第三方库 (`ext4.c` 等)**: 目标项目集成了完整的 ext4 文件系统支持，这是其相对于参考项目的重大独立扩展，是判定独立开发的关键证据。
    *   **进程管理扩展 (`run_next_init_test`, `try_exec_busybox_command` 等)**: 目标项目实现了独立的 init 测试框架与 Busybox 命令兼容层，参考项目无此实现。
*   **降权项说明**:
    *   **公共开源基线代码**: `xv6-user/sh.c`, `xv6-user/zombie.c`, `xv6-user/grind.c` 等文件属于 MIT xv6 官方用户态程序，不能作为指控项目间抄袭的证据。
    *   **公共硬件抽象层**: `dmac.c`, `spi.c`, `sdcard.c`, `uart.c`, `plic.c` 等文件中的函数高度相似，源于双方共同依赖的 Kendryte K210 官方 SDK，属于公共依赖，不构成指向参考项目的排他性同源证据。
    *   **跨板块弱信号**: 控方将 `dmac.c` 中的 DMA 等待函数归入【进程管理】板块进行指控，属于证据归类错误。DMA 驱动调用 `sleep` 是驱动使用内核 API 的正常行为，其相似性应归因于公共 SDK，已作降权处理。

### 6. 保守结论
综合全模块矩阵分析、关键证据指纹比对及核心调用流剖析，本庭认定，目标项目与参考项目 38.19% 的 AST 相似度主要源于双方对 MIT xv6-RISC-V 教学内核、RustSBI 引导程序及 Kendryte K210 硬件抽象层等**公共开源基线的共同继承**。控方提出的“系统性、大规模底层逻辑克隆”指控，因无法排除共同祖先、误将公共用户态代码与硬件 SDK 作为抄袭证据、且忽视了目标项目在 ext4 文件系统、MLFQ 调度、Busybox 兼容层等方面的大量独立架构设计与实现，故证据不足。目标项目整体呈现为在公共基线之上的独立演化与功能扩展，原创性风险较低。
