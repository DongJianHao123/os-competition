#### 4. 核心代码调用流对比剖析
* **【进程管理/调度原语】**
* **底层技术析理**: 双方均实现了 `sleep`、`sched`、`yield` 等 xv6 标准调度原语，其核心调用流骨架（`acquire -> [当前任务] -> release -> sched`）是 xv6 教学内核的经典范式。根本差异在于获取当前任务实体的抽象层：目标项目使用 `myproc`，表明其调度实体是进程；参考项目使用 `mythread`，表明其调度实体是线程。这导致两者在 `sys_exit_group`、`sys_clone` 等系统调用中衍生出完全不同的线程组管理逻辑。
* **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#sleep`
  - 参考项目: `kernel/proc.c#sleep`

* **【系统调用/参数解析】**

#### 6. 保守结论
本次审计的核心结论是独立开发。整体 38.35% 的 AST 相似度主要由第三方开源库 `lwext4` 贡献，剔除后核心内核模块的结构重合度极低。控方关于进程管理板块“系统性复用”的指控，混淆了公共基线（xv6）收敛、标准 OS 语义必然性以及架构设计差异。双方在核心架构上存在根本性分歧：目标项目采用进程中心模型与分层 VFS 设计，参考项目采用线程中心模型与扁平化文件系统。这些底层架构的显著差异，结合极低的模块内结构重合度，足以排除代码抄袭的嫌疑。建议将 `lwext4` 等第三方开源代码从抄袭评估中排除，并认定双方为独立开发。
