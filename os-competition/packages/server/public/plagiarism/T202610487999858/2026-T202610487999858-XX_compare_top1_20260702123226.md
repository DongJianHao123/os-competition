## 4. 核心代码调用流对比剖析

### 【进程管理 / Socket 接收路径】

- **底层技术析理**: 双方 `recvfrom` 函数的调用流完全一致：通过 `TAILQ_FIRST`/`TAILQ_FOREACH` 遍历 socket 消息队列，使用 `MIN` 宏计算可拷贝字节数，调用 `memmove` 将数据从内核缓冲区拷贝到用户空间，最后通过 `message_free` 将消息节点回收到空闲链表。该调用流体现了相同的对象管理模型（TAILQ 队列 + 空闲链表回收）和同步策略（`acquire`/`release` 包裹整个操作）。`message_free` 中“释放内存却向队列插入尾部”的缓冲池回收模式是双方共有的非平凡设计选择。

- **代码溯源链接**:
  - 目标项目: `kernel/proc/socket.c#recvfrom`
  - 参考项目: `kernel/proc/socket.c#recvfrom`

---

### 【进程管理 / 自旋锁原语】

- **底层技术析理**: 双方 `acquire` 和 `release` 的实现调用流完全一致。`acquire` 使用 `__sync_lock_test_and_set`（或等价内联汇编 `amoswap`）进行原子测试置位，`__sync_synchronize` 作为内存屏障，`push_off` 关闭中断防止死锁，`holding` 进行防御性断言。`release` 则反向执行 `__sync_lock_release`→`__sync_synchronize`→`pop_off`。这是 RISC-V 架构下自旋锁的标准实现模式，由硬件内存模型和 xv6 设计规范共同决定。

- **代码溯源链接**:
  - 目标项目: `kernel/proc/spinlock.c#acquire`
  - 参考项目: `kernel/proc/spinlock.c#acquire`

---

### 【进程管理 / 调度器核心】

- **底层技术析理**: 双方 `scheduler` 函数的调用流一致：`acquire` 获取进程锁→`intr_on` 开中断→`mycpu` 获取当前 CPU 结构→遍历进程表选择可运行进程→`release` 释放锁→`swtch` 执行上下文切换。`sched` 函数中 `holding`→`intr_get`→`mycpu`→`myproc`→`panic`→`swtch` 的调用序列体现了相同的上下文切换不变式检查（必须持有锁且关中断）。这是 xv6 调度器框架的标准实现。

- **代码溯源链接**:
  - 目标项目: `kernel/proc/proc.c#scheduler`
  - 参考项目: `kernel/proc/proc.c#scheduler`

---

### 【进程管理 / 管道同步】

- **底层技术析理**: 双方 `piperead_kernel` 和 `pipewrite_kernel` 的调用流一致：`acquire` 加锁→`killed` 检查进程是否被终止→`myproc` 获取当前进程→`sleep`/`wakeup` 进行睡眠/唤醒同步→`release` 解锁。该调用流体现了相同的管道同步策略：在持有锁的情况下检查进程状态、阻塞等待条件满足、被唤醒后重新获取锁。`killed` 检查的位置和方式完全一致。

- **代码溯源链接**:
  - 目标项目: `kernel/proc/pipe.c#piperead_kernel`
  - 参考项目: `kernel/proc/pipe.c#piperead_kernel`

### 【内存管理 / 伙伴系统】

- **底层技术析理**: 双方 `mark_parent` 函数均呈现“内部闭环逻辑”，即函数内部不调用任何外部函数，仅通过位运算和数组索引操作完成伙伴系统父节点的标记。这是伙伴系统算法的经典实现模式：通过块索引计算伙伴索引，向上递归标记父节点为已分裂状态。双方在该函数上的一致性源于算法本身的确定性，但函数被归入进程管理板块属于归类错误，实际属于内存管理。

## 6. 保守结论

本庭对目标项目 `[2026-T202610487999858-XX]` 与参考项目 `[2025-啊对的对的嗷不对不对-华中科技大学]` 进行了全面的 AST 级同源性审计。双方客观 AST 相似度为 80.43%，但其中约 55.8% 的共同指纹来自第三方库/公共依赖模块（lwext4、printf、qsort 等），剔除后核心原创模块的实际相似度显著降低。

在进程管理板块，socket 子系统、自旋锁原语、调度器核心、管道同步等底层逻辑呈现系统性同构，调用图谱高度一致，存在明显的同源痕迹。但目标项目在多个维度展现了独立开发特征：进程管理函数数多于参考项目（94 vs 87）、内存管理函数数显著多于参考项目（189 vs 164）、`forkret` 中引入 `filesystem2_init` 扩展、`wakeup2` 和 `sys_clone` 中增加 `myproc` 调用等。这些差异具有明确的功能语义，不是随机扰动。

综合判断，目标项目并非“镜像级”复制，而是在参考项目基础上进行了有意义的架构改动与功能扩展。操作系统内核受 xv6 教学骨架、RISC-V/LoongArch 硬件架构、POSIX 接口规范的三重约束，底层原语的调用图谱收敛是独立开发下的正常现象。本庭认定目标项目存在**部分逻辑借鉴**，但整体上构成了**存在架构改动的衍生作品**，建议根据赛事规则中关于“合理借鉴与独立扩展”的界定标准进行最终裁决。
