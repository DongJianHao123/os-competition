## 4. 核心代码调用流对比剖析

### 【进程管理 / sys_exit 退出系统调用】

- **底层技术析理**: 
  - **目标项目**: 采用批处理模型。`sys_exit` 调用 `batch::run_next_app` 直接切换到下一个应用，无进程控制块（PCB）、无任务状态管理、无资源回收逻辑。调度是静态的应用数组轮转。
  - **参考项目**: 采用多道程序模型。`sys_exit` 调用 `exit_current_and_run_next`，该函数涉及当前任务 PCB 的状态更新、僵尸进程处理、父进程唤醒等完整进程生命周期管理。
  - **异构性**: 目标项目无 `current_task` 抽象、无 `TaskControlBlock` 结构、无 `exit_code` 传递机制。两者在进程抽象层次上存在根本差异。

- **代码溯源链接**:
  - 目标项目: `src/syscall/process.rs#sys_exit`
  - 参考项目: `os/src/syscall/process.rs#sys_exit`

---

### 【中断/异常处理 / trap_handler 陷阱处理】

- **底层技术析理**:
  - **目标项目**: `trap_handler` 仅处理简单的系统调用分发（`syscall::syscall`）和缺省 panic，异常时直接调用 `batch::run_next_app` 切换应用，无信号机制、无页错误处理、无 COW 支持。
  - **参考项目**: `trap_handler` 包含完整的信号检查（`check_signals_error_of_current`）、信号注入（`current_add_signal`）、信号处理（`handle_signals`）、`scause.cause` 分支处理等。
  - **异构性**: 目标项目无 `SignalFlags`、无 `current_trap_cx` 上下文保存/恢复、无用户态信号递送机制。同步策略上，目标是简单的同步 panic 后切换，参考是异步信号驱动的复杂状态机。

- **代码溯源链接**:
  - 目标项目: `src/trap/handler.rs#trap_handler`
  - 参考项目: `os/src/trap/mod.rs#trap_handler`

### 【系统调用 / 用户态 syscall 底层封装】

- **底层技术析理**:
  - **目标项目**: 每个 syscall 函数（`sys_write`、`sys_exit` 等）直接内联 `asm!` 宏，寄存器分配和 ecall 指令在函数体内展开。这是 rCore-Tutorial 第二章的标准写法。
  - **参考项目**: 所有 syscall 函数统一调用 `syscall` 辅助函数，由该辅助函数集中执行 `asm!`。这是更高层次的抽象封装。
  - **异构性**: 两种封装策略反映不同的设计选择——目标项目选择零开销内联，参考项目选择代码复用和集中管理。AST 结构因此完全不同。

## 6. 保守结论

目标项目 `[2026-T202610574999903-XX]` 与参考项目 `[2022-S0S-哈尔滨工业大学]` 的客观 AST 相似度仅为 0.62%，共同指纹仅 3 个，且全部集中在 Rust 语言 trait 强制实现（`write_str`）和系统调用单行转发函数（`exit`/`getpid`/`yield_`）上——这些是实现 RISC-V OS 基础 I/O 和 syscall 接口的**标准惯用写法**，其实现空间被语言规范、硬件 SBI 接口和 syscall ABI 压缩至唯一路径，不具备独创性表达上的可复用性。
