* **【进程管理/退出系统调用】**

  - 目标项目: `src/syscall/process.rs#sys_exit`
  - 参考项目: `kernel/src/syscall/process.rs#sys_exit`

* **【内存管理/陷阱处理】**
* **底层技术析理**: 目标项目 `trap_handler` 直接调用 `batch::run_next_app` 进行任务切换，并调用 `syscall::syscall` 进行系统调用分发，其内存管理仅涉及简单的 Trap 上下文保存与恢复。参考项目 `user_trap_handler` 则涉及 `check_signals_error_of_current`、`handle_signals` 等信号处理机制，以及 `current_trap_cx` 等更复杂的上下文管理。两者在异常处理流程的深度和广度上存在根本差异。
* **代码溯源链接**:
  - 目标项目: `src/trap/handler.rs#trap_handler`
  - 参考项目: `kernel/src/trap/trap.rs#user_trap_handler`

* **【通用模块/其他/BSS段初始化】**
* **底层技术析理**: 目标项目 `clear_bss` 使用 `(a as *mut u64).write_volatile` 配合 `step_by` 进行循环清零，这是一种直接操作指针的底层实现。参考项目 `clear_bss` 使用 `core::slice::from_raw_parts_mut(...).fill(0)`，这是一种更安全、更符合 Rust 风格的切片填充方法。两者在内存操作的安全抽象层次和具体算法上不同。

#### 5. 单方模块与降权项
- **单方模块**: 参考项目独有的【设备驱动】模块（如 `sdcard.rs`）以及【文件系统】中的 `RunFS` 实现，目标项目均无对应代码。这体现了参考项目在硬件交互与持久化存储上的巨大复杂度优势，是两项目架构差异的核心体现。
- **降权项**: 双方存在大量同名函数（如 `sys_exit`, `sys_getpid`, `sys_yield`, `shutdown`, `_start`, `panic` 等），但这些均为操作系统/POSIX 标准接口或 Rust 语言约定名称。其底层 AST 调用图谱均存在显著差异，属于“同名异构”，依法降权为弱信号，不构成同源主证据。
- **第三方库/公共依赖**: 双方均使用了 Rust 核心库（`core`）和标准宏（`println!`, `asm!`），以及 RISC-V SBI 规范接口。这些是开发 RISC-V Rust OS 的公共基础，不能作为抄袭证据。

#### 6. 保守结论
