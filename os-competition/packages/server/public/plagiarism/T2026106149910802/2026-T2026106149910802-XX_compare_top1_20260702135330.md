### 4. 核心代码调用流对比剖析
本庭选取了双方均存在且控方提出指控的核心系统调用路径进行深度对比。

* **【进程管理 / sys_clone 进程创建】**
* **底层技术析理**: 这是揭示双方架构差异的关键证据。目标项目的调用流为 `current_task.fork`，表明其采用“任务(Task)”作为核心调度实体，进程是任务的容器。而参考项目的调用流为 `current_process.fork`，表明其以“进程(Process)”为核心调度实体。这直接导致了两者在资源复制、PID分配、信号处理等后续流程上的根本性分歧。目标项目使用 `add_task` 显式将新任务加入调度器，而参考项目则在 `fork` 内部完成。这是两种完全不同的内核对象管理模型，绝非简单的“变量名替换”。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_clone`
  - 参考项目: `os/src/syscall/process.rs#sys_clone`

* **【进程管理 / sys_sigprocmask 信号掩码处理】**
* **底层技术析理**: 此路径暴露了双方在内存访问和安全模型上的差异。目标项目使用 `safe_translated_ref` 和 `inner.memory_set.clone`，暗示其可能通过页表翻译或显式的内存集克隆来安全访问用户空间数据。参考项目则使用 `inner.memory_set.write_to_user_ptr` 和 `token.into`，表明其依赖于一个“Token”机制来授权内核访问用户内存。这两种内存访问策略在安全隔离和实现复杂度上截然不同。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_sigprocmask`
  - 参考项目: `os/src/syscall/signal.rs#sys_sigprocmask`

* **【任务调度 / run_tasks 调度主循环】**
* **底层技术析理**: 虽然调度循环的骨架（取任务、切换、空闲）是抢占式内核的通用设计，但具体实现细节体现了独立开发的特征。目标项目使用 `context_switch_pt` 进行上下文切换，并集成了 `crate::timer::check_futex_timer` 来支持futex定时器，这是参考项目所没有的特性。参考项目则使用 `__switch` 并直接访问 `task.inner.exclusive_session`。这些差异表明目标项目在调度器层面进行了功能扩展和独立的底层实现。
* **代码溯源链接**:
  - 目标项目: `os/src/task/processor.rs#run_tasks`
  - 参考项目: `os/src/task/processor.rs#run_tasks`

### 6. 保守结论
综上所述，16.74%的客观AST相似度在操作系统内核这一高度规范化的领域中，属于低水平。该相似度主要由共享的第三方开源库（如`virtio-drivers`、`ext4_rs`）贡献，而非核心内核逻辑。控方提出的“铁证”，经技术分析，均为Rust语言标准惯用法、硬件编程规范或POSIX系统调用语义下的必然实现，缺乏独创性。双方在进程/任务模型、内存访问安全策略、系统调用分发等核心架构层面展现出根本性差异。因此，本庭认为，现有证据远未达到认定抄袭或结构性复用的标准，整体风险极低，支持独立开发的结论。
