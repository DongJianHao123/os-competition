### 4. 核心代码调用流对比剖析
* **【进程管理/时间统计】**
* **底层技术析理**: 两者都实现了对线程组 CPU 时间的聚合，但采用了完全不同的时间抽象层。目标项目使用 `TimeRecorder` 对象，通过 `time_pair()` 方法直接返回 `(user, kernel)` 元组；参考项目使用 `TimerMut` 对象，通过 `user_and_system_time()` 方法返回。这证明双方的时间子系统是独立设计的，仅在最外层的聚合逻辑上因 Rust 的 `map-reduce` 惯用法而表现出相似性。
* **代码溯源链接**:
  - 目标项目: `os/src/task/task.rs#process_time_pair`
  - 参考项目: `kernel/src/task/time.rs#get_process_ustime`

* **【进程管理/进程模型】**
* **底层技术析理**: `pid()` 函数的实现是双方进程模型差异的缩影。目标项目区分了线程组长（leader）与普通线程，`pid()` 需判断 `is_leader()` 来决定返回自身 ID 还是组长的 ID。参考项目则采用更传统的 `process` 结构体，线程通过 `self.process().tid` 直接获取进程 ID。这体现了在进程管理这一核心子系统上的根本性架构异构。
* **代码溯源链接**:
  - 目标项目: `os/src/task/task.rs#pid`
  - 参考项目: `kernel/src/task/task.rs#pid`

* **【任务调度/异步运行时】**
* **底层技术析理**: 在任务派发函数 `spawn` 中，目标项目展现了复杂的负载均衡和抢占逻辑，包括计算 `cpu_mask_id`、选择运行队列 `select_run_queue_index()` 以及支持 `push_preempt`。参考项目的实现则相对简单，仅将任务推入一个可用的处理线。这种调度策略上的显著差异是独立架构设计的有力证明。
* **代码溯源链接**:
  - 目标项目: `os/src/executor/mod.rs#spawn`
  - 参考项目: `lib/executor/src/libt1.rs#spawn`

### 5. 单方模块与降权项
* **单方模块**: 无。双方覆盖的模块领域基本一致，这是由操作系统内核的通用功能需求决定的。
* **第三方库/公共依赖**: 网络协议栈（`smoltcp`）、ext4 文件系统（`ext4` 库）、部分同步原语（`SpinNoIrqLock`）等均依赖或参考了第三方 crate 或公共实现。这些库的 API 调用在 AST 指纹中表现为重合，但**不能作为双方代码同源的证据**，因为它们属于公共上游依赖。
* **降权项**: 所有跨板块同名函数（如不同模块下的 `new`、`get`、`validate`）、Rust 语言内置项（`Ok`、`Err`、`Some`、`asm!`）以及 POSIX 强制系统调用名（`sys_getpid`、`sys_execve`），均因不具备独创性或属于领域强制规范，被降权为弱信号或无效信号。
