### 4. 核心代码调用流对比剖析
以下剖析揭示了双方在核心子系统上不仅API调用相似，其内部的**数据流和控制流逻辑也完全一致**，这是“同源”判定的基石。

*   **【进程管理/进程创建】**
    *   **底层技术析理**: 在`sys_clone`的实现中，双方均遵循完全相同的私有处理流程：1. 使用 `CloneFlags::from_bits(flags & !0xff)` 屏蔽低8位信号，解析clone标志。2. 通过 `UserPtrRaw::new(child_tid.0 as *mut u32).ensure_write` 和 `UserPtrRaw::new(parent_tid.0 as *mut u32).ensure_write` 分别处理子线程和父线程的TID写入。这种对`flags`的特定屏蔽操作、对`child_tid`和`parent_tid`的统一封装写入模式，是高度特化的私有约定，绝非POSIX规范所能约束。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/process.rs#sys_clone`
        - 参考项目: `os/src/syscall/process.rs#sys_clone`

*   **【任务调度/异步运行时】**
    *   **底层技术析理**: 在`spawn`任务生成逻辑中，双方均采用相同的自定义SMP调度策略：1. 通过 `<Arc<TaskControlBlock> as Clone>::clone(&(&future.task.clone())).turn_cpu_mask_id` 获取任务指定的CPU亲和性掩码。2. 若未指定，则回退到 `crate::processor::schedule::select_run_queue_index()` 选择运行队列。3. 最终通过 `PROCESSORS[...].unwrap_with_mut_task_queue` 获取特定CPU的任务队列并`push`。这种将`turn_cpu_mask_id`与全局`PROCESSORS`数组相结合的调度模型，是项目特有的架构设计。
    *   **代码溯源链接**:
        - 目标项目: `os/src/executor/mod.rs#spawn`
        - 参考项目: `os/src/executor/mod.rs#spawn`

*   **【内存管理/用户态内存访问】**
    *   **底层技术析理**: 在`sys_process_vm_writev`跨进程内存写入中，双方的内存访问抽象层完全一致：1. 通过 `TASK_MANAGER.get_task(pid as usize).ok_or` 获取目标进程控制块。2. 使用 `UserSliceRaw::new(iov.base as *const u8, iov.len).ensure_read` 从调用者空间读取数据。3. 使用 `UserSliceRaw::new(iov.base as *mut u8, iov.len).ensure_write` 向目标进程空间写入数据。`UserSliceRaw` 及其 `ensure_read/ensure_write` 方法是项目自定义的用户态内存安全封装，其API设计和使用模式是强有力的同源指纹。
    *   **代码溯源链接**:
        - 目标项目: `os/src/syscall/mm.rs#sys_process_vm_writev`
        - 参考项目: `os/src/syscall/mm.rs#sys_process_vm_writev`

    *   **底层技术析理**: 在procfs的初始化中，双方构建目录树的逻辑完全一致：1. 使用 `CNXFS::create_sys_dir` 创建目录项。2. 使用 `CNXFS::create_sys_file` 创建文件项。3. 所有创建的dentry均通过 `DCACHE.lock().insert` 插入全局目录缓存。`CNXFS` 是一个高度非标准的、项目特有的文件系统创建辅助类型，它的存在和使用是双方共享同一代码基线的铁证。

### 6. 保守结论
本庭裁定，目标项目与参考项目在操作系统内核的宏观架构、核心数据结构和关键算法流程上存在**根本性的同源关系**。94.38%的AST相似度及超过96%的双向覆盖率，远超独立开发可能达到的界限。辩方提出的“领域共性”论点，无法解释如 `CNXFS`、`UserPtrRaw`、`SumGuard` 等大量非标准、项目特有的私有抽象在双方代码中呈现出镜像级的调用模式。因此，最合理的解释是双方项目共享了同一个核心代码基线。目标项目在该基线上进行了局部修改，如调整了`sys_execve`中的可执行文件路径、在`do_exit`中增加了`Instruction::shutdown`等，但这属于对共同基线的增量开发，而非颠覆性的架构创新。本次仲裁仅就代码同源性做出判断，不涉及对参赛规则或学术诚信的最终定性。
