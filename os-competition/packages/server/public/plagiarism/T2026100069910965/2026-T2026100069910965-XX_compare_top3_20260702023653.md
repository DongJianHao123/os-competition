### 4. 核心代码调用流对比剖析
* **【进程管理/上下文切换路径】**
* **底层技术析理**: 双方均实现了 `scheduler` -> `swtch` 的上下文切换流，这是操作系统调度器的数学最小骨架。然而，双方的**同步原语**存在根本性差异：目标项目采用基于 `arch_intr_off/on` 的中断控制策略，属于多架构抽象设计；参考项目采用基于 `acquire/release` 的自旋锁策略，属于 xv6 经典单核锁设计。目标项目在 `sleep` 中引入 `proc_exit_signal` 信号处理，表明其实现了更复杂的可中断睡眠语义，而参考项目无此路径。
* **代码溯源链接**:
  - 目标项目: `kernel/proc.c#scheduler`
  - 参考项目: `kernel/proc.c#scheduler`

* **【内存管理/内存映射路径】**
* **底层技术析理**: 双方均存在 `sys_mmap` 系统调用，但底层内存管理模型完全不同。目标项目调用 `kalloc_page`、`kfree_page` 等独立物理页帧分配器，并涉及 `clear_vma_range` 虚拟内存区域操作；参考项目则依赖 `argaddr`、`argint` 参数解析及 `__debug_error` 等调试宏，其内存分配与文件系统（fat32）紧耦合。目标项目展现了更现代、更独立的虚拟内存管理抽象。
* **代码溯源链接**:
  - 目标项目: `kernel/sysproc.c#sys_mmap`
  - 参考项目: `kernel/sysfile.c#sys_mmap`

### 5. 单方模块与降权项
- **单方独有模块**:
  - **网络协议栈**: 目标项目存在 `kernel/main.c#is_netperf_script`，参考项目无任何网络相关函数。此为独立功能扩展。
  - **文件系统/第三方库**: 目标项目包含完整的 ext4 实现（`fs/ext4/ext4.c`），参考项目使用 fat32。这是文件系统选型的根本性差异。
- **降权项说明**:
  - **标准 POSIX 接口**: `sys_getpid`、`sys_getppid` 等函数因接口规范强制同名且逻辑极简，其“内部闭环逻辑”或单节点调用不能作为同源证据。

  - **公共库函数**: `strncmp`、`memcmp`、`memcpy` 等标准 C 库函数，在任何 C 项目中实现都高度相似，属于公共领域知识。

### 6. 保守结论
综合客观 AST 覆盖率（1.0%）、双向指纹重合度（<2.1%）及核心模块调用流分析，目标项目与参考项目在进程管理、内存管理、文件系统等关键子系统上展现出**根本性的架构异构**。目标项目采用中断抽象、ext4 文件系统、virtio 驱动及独立网络栈，与参考项目的自旋锁、fat32 文件系统、SD 卡驱动形成清晰的技术分界线。双方仅存的少量同名函数均为 POSIX 标准接口或操作系统公共骨架，其内部实现路径与同步策略均不相同。因此，本次审计判定两项目为**独立开发**，不存在足以构成抄袭指控的同源性代码复用。
