#### 4. 核心代码调用流对比剖析
* **【进程管理/进程创建】**
* **底层技术析理**: 目标项目采用全局任务管理器（`MANAGER`）和中断屏蔽（`irq::disable/restore`）进行同步，通过 `TaskControlBlock::fork_from` 拷贝任务控制块。参考项目则基于 `current_task()` 获取当前任务指针，使用 `acquire_inner_lock` 任务内锁进行同步，并通过 `CloneFlags` 控制资源共享粒度。两者在并发模型、对象生命周期管理和资源隔离策略上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_fork` -> `kernel/src/sched/exec.rs#fork_current`
  - 参考项目: `os/src/syscall/process.rs#sys_clone`

* **【进程管理/信号处理】**
* **底层技术析理**: 目标项目的 `sys_kill` 通过遍历全局 `MANAGER.tasks` 数组并结合 `current_process_group` 查找目标进程，最终调用 `manager::send_signal`。参考项目则使用 `find_task_by_tgid` 进行精确查找，并直接操作任务内部结构 `inner.add_signal`。两者在进程查找算法和信号注入点上完全不同，体现了“全局数组遍历”与“哈希表查找”的设计差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/syscall/process.rs#sys_kill`
  - 参考项目: `os/src/syscall/process.rs#sys_kill`

* **【进程管理/定时器管理】**
* **底层技术析理**: 目标项目将定时器处理逻辑独立为 `sched` 模块下的 `process_posix_timers` 和 `process_itimer_real` 函数，直接读取硬件 ticks（`read_ticks`）并计算超时。参考项目则在陷阱处理路径 `update_process_times_leave_trap` 中，使用 `TimeVal::now` 抽象时间接口，并调用 `self.update_itimer_prof_if_exists` 等任务内方法。两者在定时器的检查时机（独立轮询 vs. 陷阱驱动）和时间源抽象上存在显著差异。
* **代码溯源链接**:
  - 目标项目: `kernel/src/sched/manager.rs#process_posix_timers`
  - 参考项目: `os/src/task/task.rs#update_process_times_leave_trap`

* **【内存管理/物理内存分配】**
* **底层技术析理**: 这是双方共同指纹最多的模块。两者均实现了基于位图的物理帧分配器（`alloc`/`dealloc`）。然而，目标项目的分配器用于 PID 管理（`kernel/src/sched/pid.rs`），而参考项目用于物理页帧管理（`os/src/mm/frame_allocator.rs`）。这种将同一数据结构应用于不同子系统（进程ID vs. 物理内存）的做法，恰恰说明其并非源自同一设计文档，而是各自独立地将经典位图算法应用于各自的内核需求。
* **代码溯源链接**:
  - 目标项目: `kernel/src/sched/pid.rs#alloc`
  - 参考项目: `os/src/mm/frame_allocator.rs#alloc`

#### 5. 单方模块与降权项
- **单方模块**: 参考项目的 `user/src/bin/initproc.rs` 属于用户态初始化程序，其环境变量字符串列表与内核进程管理无关，不能作为内核代码复用的证据。
- **第三方库/公共依赖**: 双方均使用了 Rust 标准库或核心库中的通用原语，如 `assert!`、`Some`、`drop`、`Vec`、`Box` 等。这些是 Rust 语言的共性，不构成项目间的特异性指纹。
- **公共接口**: `sys_getpid`、`sys_exit`、`sys_kill` 等 POSIX 标准系统调用，其函数签名和名称由标准规定，同名是兼容性要求，而非抄袭证据。其内部实现路径已在上文被证明是异构的。
- **跨板块弱信号**: 控方提及的 `procfs_read`（文件系统）与 `sys_uname`（系统调用）的相似性，属于跨板块的弱信号。两者分属不同的内核子系统，其“字符串拼接与 `copy_to_user`”的相似性是用户态数据交互的通用模式，不构成同源主证据。

#### 6. 保守结论
本次审计的目标项目与参考项目在全局 AST 相似度上仅为 1.08%，处于极低水平。尽管在内存管理等少数模块存在零星共同指纹，但这些指纹均指向操作系统领域的经典算法（如位图分配）或通用标准（如 ELF 加载、POSIX 接口），而非某一项目的独创性复杂逻辑。更为关键的是，双方在进程管理、任务调度等核心子系统上展现出根本性的架构分歧：目标项目采用全局任务数组与中断屏蔽的同步模型，而参考项目则基于任务内锁与线程局部指针。这种在并发控制、对象生命周期管理等核心设计决策上的显著差异，是独立开发的强有力证据。综合判断，目标项目不存在对参考项目的系统性代码复用，整体原创性风险极低。
