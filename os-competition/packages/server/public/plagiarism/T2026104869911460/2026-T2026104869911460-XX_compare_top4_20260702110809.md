#### 4. 核心代码调用流对比剖析
*   **【进程管理/系统调用】`sys_wait4` 实现**
    *   **底层技术析理**: 两者均实现了POSIX标准的`wait4`调用，在解析选项位掩码时使用了相似的`from_bits().ok_or()`模式。然而，其底层进程管理模型截然不同。目标项目使用`UserPtr<T>`进行用户态内存访问，并通过`PidSel`枚举选择进程；参考项目则使用`UserBufferWriter`，并通过`TASK_MANAGER.remove`直接操作任务管理器。这表明两者仅在POSIX接口定义层汇合，内部实现路径是独立的。
    *   **代码溯源链接**:
        - 目标项目: `NoAxiom/kernel/src/syscall/process.rs#sys_wait4`
        - 参考项目: `pantheon/kernel/src/syscall/impls/process.rs#sys_wait4`

*   **【内存管理/系统调用】`sys_openat` 实现**
    *   **底层技术析理**: 两者均实现了文件打开调用，但使用了完全不同的标志位和模式抽象。目标项目使用`FileFlags`和`InodeMode`，而参考项目使用`OpenFlags`和`CreateMode`。这表明双方从类型定义层面就走向了不同的设计，并非同一套代码的换皮。
    *   **代码溯源链接**:
        - 目标项目: `NoAxiom/kernel/src/syscall/fs.rs#sys_openat`
        - 参考项目: `pantheon/kernel/src/syscall/impls/fs.rs#sys_openat`

*   **【中断/异常处理】Trap入口设置**
    *   **底层技术析理**: 在设置内核/用户态Trap入口时，目标项目（LoongArch架构）调用`ecfg::set_vs`和`eentry::set_eentry`，而参考项目（RISC-V架构）调用`stvec::write`。这是由不同CPU指令集架构（ISA）决定的根本性差异，直接证明了二者不可能在核心Trap机制上同源。
    *   **代码溯源链接**:
        - 目标项目: `NoAxiom/lib/arch/src/la64/trap.rs#set_kernel_trap_entry`
        - 参考项目: `pantheon/kernel/src/trap/mod.rs#set_kernel_trap_entry`

#### 5. 单方模块与降权项
*   **单方模块/特性**:
    *   **目标项目独有**: 实现了完整的Futex同步机制（`sys_futex`， `sys_futex_waitv`）、进程命名空间（`task/namespace.rs`）、`prctl`系统调用等高级特性，这些在参考项目中未见对应实现。
    *   **参考项目独有**: 包含独立的第三方库模块（`virtio-drivers2`）、CPU亲和性设置（`sys_sched_getaffinity/setaffinity`）等，目标项目未实现。
    *   **结论**: 功能集的互补性是独立开发的强有力证明。

*   **降权项（不构成强同源证据）**:
    *   **POSIX标准接口同名**: `sys_getpid`， `sys_getuid`， `sys_exit`等函数名相同是遵循Linux ABI规范的必然结果，其内部实现（如目标项目`pid_in_namespace` vs 参考项目`tgid`）差异显著。
    *   **Trivial访问器**: `pcb`函数（`self.pcb.lock`）和`add_child`函数（`self.children.push`）是Rust语言中访问锁保护数据或向容器添加元素的极简惯用法，不具备可保护的独创性。
    *   **硬件驱动范式**: SD卡驱动中的`read_block`/`write_block`/`send_cmd`等函数的调用序列由硬件数据手册规定，不同项目的实现必然趋同。
    *   **跨板块错误归类**: 控方将用户态UI库函数（`libd`）归入“进程管理”板块，属于证据污染，已予纠正。

#### 6. 保守结论
本次审计基于4.86%的极低AST客观相似度，认定目标项目与参考项目在底层架构上存在显著差异，不构成代码复用或抄袭。控方提出的证据，一部分源于对POSIX标准接口和硬件驱动范式的误读，另一部分（如UI库）高度疑似源自公共框架，但均无法形成完整的同源性证据链。双方在进程模型、内存抽象、同步机制、系统调用实现及支持架构上的系统性差异，有力支持了独立开发的结论。综合判定，整体同源性风险极低。
