#### 4. 核心代码调用流对比剖析

* **【内存管理/sys_brk 系统调用】**
* **底层技术析理**: `sys_brk` 是用于调整进程堆空间的系统调用。目标项目与参考项目虽然都实现了该调用，但其底层内存模型和对象管理方式存在显著差异。目标项目采用 `VirtPageNum` 作为地址操作的核心抽象，通过 `ceil_to_page_size` 进行对齐，反映出其内存管理器以页为基本单位进行分配。参考项目则直接使用 `VirtAddr` 的 `floor()` 方法进行地址对齐，并通过 `inner.addrspace.alloc_heap_section` 在一个地址空间对象上直接操作。这表明目标项目拥有更独立的物理/虚拟页管理层次，而参考项目则将堆管理紧密耦合在地址空间结构中。两者的同步与容错策略也不同：目标项目在 `sys_brk` 中显式处理了 `exe.contains` 等与执行文件相关的逻辑，而参考项目则无此设计。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/mm.rs#sys_brk`
  - 参考项目: `kernel/src/syscall/process.rs#sys_brk`

* **【进程管理/进程退出逻辑】**
* **底层技术析理**: 在 `sys_exit` 的实现上，双方展现了完全不同的任务生命周期管理和调度策略。目标项目将退出逻辑清晰地拆分为 `kernel_exit`（执行退出清理）、`remove_task`（从任务集中移除）和 `switch_to_next_task`（触发调度）三个阶段，并在退出后通过 `core::hint::spin_loop` 进入自旋，体现了其调度器与任务管理器分离的模块化设计。参考项目则将清理与调度合并为一个原子操作 `exit_current_and_run_next`，并在其后调用 `panic!`，这是一种更简化且假设该路径不可达的设计。目标项目的调用流更长、职责更分离，而参考项目则更紧凑，两者在架构抽象层次上存在根本不同。
