#### 4. 核心代码调用流对比剖析
* **【进程管理/进程退出路径】**
* **底层技术析理**: 目标项目将进程退出逻辑拆分为`kernel_exit`（处理线程组退出）、`remove_task`（清理数据结构）和`switch_to_next_task`（显式调度）三个独立步骤，并在异常路径使用`core::hint::spin_loop`。参考项目则使用高度封装的`exit_current_and_run_next`函数，内部包含`panic!`。两者在调度原语粒度、错误处理策略（`log::debug!` vs `panic!`）和函数抽象层级上存在根本性差异，表明其调度与任务管理模型是独立设计的。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/task.rs#sys_exit`
  - 参考项目: `os/src/syscall/process.rs#sys_exit`

* **【内存管理/内核页表初始化】**
* **底层技术析理**: 两者均需将内核各段（`.text`， `.rodata`， `.data`， `.bss`）映射到虚拟地址空间。目标项目使用`KERNEL_BASE`作为基址，而参考项目使用`KERNEL_SPACE_OFFSET << PAGE_SIZE_BITS`。这不仅是常量命名差异，更反映了内核地址空间布局设计的根本不同。此外，目标项目映射了更多内存区域（如`MEMORY_END`），体现了更复杂的内存管理需求。
* **代码溯源链接**:
  - 目标项目: `os/src/mm/memory_set.rs#new_kernel`
  - 参考项目: `os/src/mm/memory_set.rs#new_kernel`

* **【中断/异常处理/Trap入口】**
* **底层技术析理**: 两者均在`trap_handler`中进行信号检查和定时器处理，但调用的底层对象完全不同。目标项目调用`check_all_real_timers`、`check_pending_signals`等函数，而参考项目调用`check_timer`、`current_add_signal`等函数，且参考项目直接操作`inner_exclusive_access`进行任务状态管理。这表明两者的信号分发机制和定时器管理模型是异构的。
