* **底层技术析理**: `sys_execve` 是双方共有的核心系统调用，但其内部调用流揭示了完全不同的内存与文件抽象模型。目标项目采用 Linux 风格的 VFS 层，通过 `copyinstr` 从用户态拷贝字符串，并调用 `create_elf_tables` 构建 ELF 辅助向量栈，文件操作通过 `dentry_put` 管理目录项引用计数。参考项目则采用 xv6 风格，直接使用 `fetchstr` 拷贝，并进入 `exec` 函数进行加载，内存分配直接调用 `kalloc`。两者在抽象层次、内存管理粒度上存在根本性差异。
* **代码溯源链接**:
  - 目标项目: `src/syscall/proc.c#sys_execve`
  - 参考项目: `kernel/src/proc/sysproc.c#sys_execve`

* **【进程管理/上下文获取】**
* **底层技术析理**: `sys_getpid` 的实现体现了双方在进程上下文管理上的核心架构差异。目标项目通过 `thiscpu` 宏/函数直接从 per-CPU 变量中获取当前进程指针，这是一种为多核和缓存友好性优化的现代设计（类似 Linux `current`）。参考项目则通过 `myproc()` 函数，通常涉及遍历进程表或访问全局变量，是 xv6 等教学系统的经典实现。这种底层机制的不同，决定了整个内核中获取当前进程的方式都截然不同，无法通过简单重命名实现。
* **代码溯源链接**:
  - 目标项目: `src/syscall/proc.c#sys_getpid`
  - 参考项目: `kernel/src/proc/sysproc.c#sys_getpid`

* **【中断/异常处理/PLIC 控制器】**
* **底层技术析理**: 在中断处理路径上，双方虽都有 `plic_claim` 和 `plic_complete` 函数，但其硬件交互模式完全不同。目标项目通过 `PLIC_CTX` 宏和 `assert` 进行内存映射寄存器操作，服务于 QEMU/Loongson 平台。参考项目则通过 `PLIC_MCLAIM`/`PLIC_SCLAIM` 宏区分机器态与监督态，并调用 `cpuid` 获取硬件线程 ID，服务于 K210 平台。这是两个完全不同的硬件抽象层实现。
* **代码溯源链接**:
  - 目标项目: `src/platform/qemu_virt/plic.c#plic_claim`
  - 参考项目: `kernel/src/sys/plic.c#plic_claim`

#### 5. 单方模块与降权项
- **单方模块/代码**: 目标项目独有的 `src/platform/loongson/` 目录（龙芯平台支持）、`src/hal/virtio/` 目录（VirtIO 驱动框架）；参考项目独有的 `kernel/src/driver/` 目录下 K210 相关驱动（`gpiohs`、`dmac`、`spi`、`sd_final`）、`kernel/src/fs/fat32.c`（FAT32 文件系统实现）。这些单方存在的、与特定硬件或文件系统绑定的代码，强有力地证明了两个项目是面向不同平台独立开发的。
- **降权项**: 所有 POSIX 标准系统调用同名（如 `sys_exit`, `sys_close`, `sys_execve`）、标准 C 库函数同名（如 `memcmp`, `strncpy`）、通用设计模式同名（如 `main`, `sched_yield`, `lookup_path`）以及所有跨模块的证据配对，均因语义强制、公共领域代码或逻辑不相关而降权为弱信号或不作为证据。

#### 6. 保守结论
本次审计的核心结论是目标项目为独立开发，不存在对参考项目的代码复用或抄袭。客观 AST 相似度（1.01%）极低，共同指纹稀少，且主要集中在可归于第三方公共库（如 ext4）或标准 C 库的部分。控方提出的“换皮抄袭”指控，在进程管理、任务调度等核心板块的共同 AST 指纹为零的事实面前无法成立。双方在硬件平台、进程模型、文件系统架构等底层设计上存在系统性差异，这些差异是贯穿整个内核的自洽选择，而非零散的伪装。虽然存在少量同名函数和相似调用链，但这完全在 POSIX 标准、公共算法和第三方库引用的合理解释范围内，不构成同源性风险。
