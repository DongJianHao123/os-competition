* **底层技术析理**: `usertrapret` 是 RISC-V 内核从 S 模式返回 U 模式的必经路径。两者均调用 `r_sstatus` 读取特权状态寄存器，这是 RISC-V ISA 规范强制步骤，非原创逻辑。目标项目使用 `ARCH_MMU_ROOT_MAKE` 构建页表基址并调用 `arch_user_return_prepare` 等可移植封装层，体现跨架构设计意图；参考项目直接使用 `MAKE_SATP` 宏和 `r_satp` 等寄存器操作，为 RISC-V 专用实现。目标项目增加了 `uart_out_flush` 刷新串口，参考项目增加了 `r_time` 读取时间戳，实现细节显著不同。两者在页表切换、中断状态恢复的抽象层次和辅助操作上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `src/trap/trap.c#usertrapret`
  - 参考项目: `src/kernel/trap.c#usertrapret`

* **【进程管理 / 进程标识获取】**
* **底层技术析理**: `sys_getpid` 和 `sys_getppid` 为 POSIX 标准系统调用，任何实现该接口的内核都必须获取当前执行上下文并返回进程 ID。目标项目通过 `thiscpu` 获取 per-CPU 结构指针，再间接索引当前进程；参考项目通过 `myproc` 直接获取当前进程结构体指针。两者在上下文获取的抽象模型上存在根本差异：per-CPU 模型 vs per-process 直接索引。调用图谱中仅单节点重合，且该节点语义为“获取当前上下文”，属于功能必然性，不构成代码复用证据。
* **代码溯源链接**:
  - 目标项目: `src/syscall/proc.c#sys_getpid`
  - 参考项目: `src/kernel/sysproc.c#sys_getpid`

* **【任务调度/异步运行时 / 文件关闭】**
* **底层技术析理**: `sys_close` 为 POSIX 标准系统调用。目标项目采用分层 VFS 架构，调用链为 `fdnum`（解析 fd 编号）→ `fd_get`（获取文件对象）→ `vfs_close`（VFS 层关闭）→ `fd_free`（释放 fd 槽位）；参考项目采用 xv6 风格单体封装，调用 `argfd` 一步完成解析与获取，再调用 `fileclose` 直接关闭。目标项目的四步分层调用链与参考项目的两步单体调用链在 AST 结构上完全不同，反映了 VFS 抽象层与单体文件表两种截然不同的文件系统架构设计。
* **代码溯源链接**:
  - 目标项目: `src/syscall/file.c#sys_close`
  - 参考项目: `src/kernel/sysfile.c#sys_close`

#### 5. 单方模块与降权项
- **单方模块**: 目标项目独有的 `src/hal/virtio/`、`src/platform/loongson/` 等驱动栈在参考项目中完全不存在；参考项目独有的 `src/driver/gpiohs.c`、`src/driver/fpioa.c`、`src/driver/spi.c`、`src/fs/fat32/` 等 K210 平台代码在目标项目中完全不存在。双方硬件平台、文件系统、驱动架构无交集。
- **第三方库/公共依赖**: lwext4 库函数（`ext4_dir_add_entry`、`ext4_mount`、`ext4_readlink`、`ext4_unlink`）及 C 标准字符串函数（`memcmp`、`strncpy`、`strncmp`、`strcat`）的共同指纹属于合法公共代码复用，已从原创性证据中排除。
- **跨板块弱信号**: 控方提出的 `sleep`（目标任务调度 vs 参考进程管理）、`pipe_read` vs `pipereadv`（跨板块、跨接口）等配对，因分属不同模块且函数签名/语义不同，不构成同板块主证据，已严格降权。
- **标准接口同名**: `sys_getpid`、`sys_exit`、`sys_close`、`usertrapret`、`sys_pipe2`、`sys_fstat`、`sys_unlinkat`、`sys_mkdirat` 等均为 POSIX/Linux 标准系统调用名或 RISC-V 内核社区通用命名，独立实现必然同名，不能作为抄袭证据。

#### 6. 保守结论
本案客观 AST 相似度仅 1.1%，共同指纹 13 个，目标覆盖率 2.95%，属于操作系统内核独立开发的正常背景噪声水平。13 个共同指纹中，多数可归因为 POSIX 标准系统调用同名、RISC-V 特权架构强制寄存器操作、lwext4 第三方开源库引用及 C 标准库公共实现。双方在硬件平台（QEMU virt/Loongson vs K210）、文件系统架构（VFS+ext4 vs FAT32）、内核抽象模型（per-CPU vs per-process、分层 VFS vs 单体文件表）、驱动栈（virtio-blk vs sdcard/spi）等核心维度上存在根本性差异。任务调度、中断处理、系统调用等关键板块共同 AST 指纹均为零。综合判定：无证据表明目标项目对参考项目存在实质性代码复用，原创性成立。
