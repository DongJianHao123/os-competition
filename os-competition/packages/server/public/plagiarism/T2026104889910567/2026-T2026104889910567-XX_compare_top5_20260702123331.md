## 4. 核心代码调用流对比剖析

### 4.1 【进程管理/进程调度路径】

- **底层技术析理**：
  目标项目的进程调度基于**显式运行队列 + 自旋锁**模型：`sched_yield` → `context_switch_yield` → `dequeue_proc`/`enqueue_proc` + `spinlock_acquire`/`spinlock_release` + `context_switch`。上下文获取通过 RISC-V 寄存器 `r_tp` 实现（`thiscpu` 宏）。
  参考项目的进程调度为**极简协作式**模型：`yield` → `my_proc` + `swtch`，无运行队列、无自旋锁、无显式出队入队操作。
  两者在**同步策略**（有锁 vs 无锁）、**调度数据结构**（运行队列 vs 直接切换）、**上下文获取机制**（`r_tp` vs `my_proc`）三个维度存在根本性架构差异，非“拆分/膨胀”所能解释。

- **代码溯源链接**：
  - 目标项目: `src/task/sched.c#sched_yield` → `src/task/sched.c#context_switch_yield` → `src/task/sched.c#dequeue_proc` / `src/task/sched.c#enqueue_proc`
  - 参考项目: `src/core/proc.c#yield` → `src/core/proc.c#swtch`

### 4.2 【内存管理/内存分配路径】

- **底层技术析理**：
  目标项目 `kmalloc` 采用 **buddy + slab 二级分配器**：`kmalloc` → `size2index` → `kmem_cache_alloc`（小内存）/ `buddy_alloc`（大内存），使用 `ALIGN_UP` 对齐，通过 `r_tp` 获取上下文。
  参考项目 `kmalloc` 采用 **buddy + slab 二级分配器但实现不同**：`kmalloc` → `kalloc_bytes_slab` / `kalloc_pages_buddy`，使用 `PFN_TO_PADDR` 转换。
  两者虽同为二级分配器架构（这是 OS 领域的通用设计模式），但内部 slab/buddy 接口命名与调用链不同（`kmem_cache_alloc` vs `kalloc_bytes_slab`，`buddy_alloc` vs `kalloc_pages_buddy`），属于**同领域独立实现**。

- **代码溯源链接**：
  - 目标项目: `src/mm/kalloc.c#kmalloc` → `src/mm/kalloc.c#kmem_cache_alloc` / `src/mm/kalloc.c#buddy_alloc`
  - 参考项目: `src/mm/mm.c#kmalloc` → `src/mm/mm.c#kalloc_bytes_slab` / `src/mm/mm.c#kalloc_pages_buddy`

### 4.3 【中断/异常处理/Trap 初始化路径】

- **底层技术析理**：
  目标项目 `trap_init` 通过 `arch_irq_mask_read/write` 等 HAL 层抽象操作中断控制器，通过 `set_kerneltrap` 设置内核 trap 入口。
  参考项目 `trap_init` 直接写 Loongson 架构 CSR 寄存器（`w_csr_ecfg`/`w_csr_eentry`/`w_csr_tlbrentry`），无 HAL 抽象层。
  两者在**抽象层次**（HAL 封装 vs 裸寄存器操作）和**目标架构**（RISC-V vs Loongson）上完全不同。

- **代码溯源链接**：
  - 目标项目: `src/trap/trap.c#trap_init` → `src/hal/timer.c#timer_init`
  - 参考项目: `src/core/trap.c#trap_init` → `src/core/trap.c#timer_init`

### 4.4 【文件系统/目录查找路径】

- **底层技术析理**：
  目标项目 `sys_chdir` 基于 **VFS + dentry** 抽象层：`vfs_path_lookup` → `dentry_get`/`dentry_put` → `S_ISDIR` 校验。

- **代码溯源链接**：
