## 第四部分：分模块架构说明

### MM（内存管理）

**已实现（有底层证据）**：

- 用户地址空间管理：`sys_mmap` 通过多种 `Backend`（`new_alloc`、`new_cow`、`new_file`、`new_linear`、`new_shared`）建立映射
- ELF 加载器：`kernel/src/mm/loader.rs#load_user_app` 调用 `ELF_LOADER.lock().load(...)` 并分配栈
- COW 后端：`kernel/src/mm/aspace/backend/cow.rs#alloc_new_at` 从文件读取并建立物理页映射
- 用户指针访问校验：`kernel/src/mm/access.rs` 提供 `vm_read` / `vm_write` / `vm_read_uninit` 等
- 页表抽象：`page_table_multiarch/src/bits64.rs`、`page_table_entry` 提供多级页表结构与位操作

**证据锚点**：

- `kernel/src/syscall/mm/mmap.rs#sys_mmap`
- `kernel/src/mm/loader.rs#load_user_app`
- `kernel/src/mm/aspace/backend/cow.rs#alloc_new_at`
- `kernel/src/mm/access.rs`
- `page_table_multiarch/src/bits64.rs`

### PROC（进程管理）

**已实现（有底层证据）**：

- 进程/线程控制块扩展：`kernel/src/task/mod.rs` 中的 `AxTaskExt` 包含 `proc_data`、`clear_child_tid`、`sigwait_set`、`vfork_state` 等
- 进程表与进程组表：`PROCESS_TABLE`、`PROCESS_GROUP_TABLE`
- 生命周期：`sys_clone` / `sys_fork` / `sys_execve` / `sys_exit` / `sys_exit_group` / `do_exit`
- 等待子进程：`sys_waitid`、信号化等待、`wait_siginfo`
- `pidfd`：`sys_pidfd_open`、`sys_pidfd_getfd`、`sys_pidfd_send_signal`
- 信号：`sys_rt_sigprocmask`、`send_signal_to_process`、`send_signal_to_process_group`
- `/proc` 伪文件系统：`kernel/src/pseudofs/proc.rs#lookup_child` 暴露 `stat`、`status`、`maps`、`mounts`、`cmdline`、`comm`、`exe`、`fd`、`task` 等

**证据锚点**：

- `kernel/src/task/ops.rs#do_exit`
- `kernel/src/syscall/task/clone.rs#do_clone`
- `kernel/src/syscall/task/execve.rs#sys_execve`
- `kernel/src/syscall/task/wait.rs#sys_waitid`
- `kernel/src/pseudofs/proc.rs#lookup_child`

### 任务调度/异步运行时

**已实现（有底层证据）**：

- 对 `axtask` 的包装与调用：`axtask::current`、`axtask::yield_now`、`axtask::spawn`、`axtask::future::sleep`、`axtask::set_current_affinity`
- 同步原语：`futex` 等待队列、`wait_if`、`block_on`
- IO 多路复用：`epoll` 注册 waker、`timerfd` 定时器、`pipe` 读写
- TTY 行规程：`ldisc.rs#read` 使用 `block_on`、`WaitPollable`

**证据锚点**：

### FS（文件系统）

## 第六部分：边界与未体现内容

### 明确没有底层证据支撑的模块

1. **中断/异常处理框架**：无 trap entry、IDT/向量表、异常栈帧保存与恢复、时钟中断处理等代码。
2. **底层设备驱动框架**：无 IRQ 注册、DMA、设备树解析、PCI/平台总线、块设备驱动等。

4. **SMP 启动与 CPU 管理**：`axhal::cpu_num` 仅被调用，无 CPU 启动、IPI、per-CPU 数据结构等证据。
5. **安全/LSM/_capabilities**：无相关证据。
6. **虚拟化/ hypervisor**：无相关证据。
7. **块存储与持久化文件系统**：`MemoryFs`、`tmpfs` 为内存文件系统，无 ext4/FAT/块缓存等持久化 FS 证据。

### 需要单独说明的第三方/公共依赖

- **`axtask`、`axhal`**：代码中大量以库函数形式调用，应视为底层调度与硬件抽象依赖，而非本项目原创内核主体。
- **`page_table_multiarch` / `page_table_entry`**：目录存在且函数密集，可能是项目自研或基于公共页表 crate 改造，需结合仓库历史判断原创性。
- **`tools/ltp-runner.c` 与 `scripts/ltp/`**：属于 LTP 兼容性测试运行器与辅助脚本，不是内核本体。

### 判断性质声明

- 本作品中“任务调度/异步运行时”当前证据主要表现为对 `axtask` 的包装与上层同步原语，**不能等同于从零实现的完整调度器**。
- “传统进程管理”有 `fork/exec/wait/zombie/PCB/父子进程/进程组` 等证据，但底层上下文切换仍依赖外部 crate。
- 所有关于“基于 ArceOS/axhal”的推断仅为**可能方向**，不是对项目身份或原创性的最终结论。
- 本文档不做抄袭、违规或排名判断。
