#### 4. 核心代码调用流对比剖析

* **【进程管理/信号处理】**
* **底层技术析理**: 目标项目采用“用户态内存拷贝”模型，通过 `process.read_user_usize` 和 `process.write_user_usize` 与用户空间交互，并使用 `LinuxError` 枚举返回错误码。参考项目则采用“内核任务扩展”模型，通过 `current().task_ext().thread_data().signal` 直接操作内核信号对象。两者在**抽象层次**（用户拷贝 vs. 内核对象）和**错误处理模型**（错误码 vs. Result）上存在根本性架构差异。
* **代码溯源链接**:
  - 目标项目: `pulse_syscalls/src/impls/misc.rs#sys_rt_sigprocmask`
  - 参考项目: `api/src/imp/signal.rs#sys_rt_sigprocmask`

* **【进程管理/进程退出】**
* **底层技术析理**: 目标项目实现了“线程退出”与“进程组退出”的分离，`sys_exit` 调用 `thread.exit_current`，而 `sys_exit_group` 额外调用 `thread.process().begin_group_exit`。参考项目则使用单一的 `do_exit` 函数处理所有退出场景。目标项目的**对象管理模型**更精细，将进程和线程的生命周期管理解耦。
* **代码溯源链接**:
  - 目标项目: `pulse_syscalls/src/impls/task/exit.rs#sys_exit_group`
  - 参考项目: `api/src/imp/task/exit.rs#sys_exit_group`

* **【内存管理/缺页中断处理】**
* **底层技术析理**: 目标项目在 `handle_page_fault` 中实现了独立的 **COW (Copy-On-Write)** 逻辑，通过 `core::ptr::copy_nonoverlapping` 进行物理页拷贝。参考项目则将 COW 处理委托给后端 `area.backend().handle_page_fault`。两者在**内存模型**和**职责划分**上不同，目标项目将策略内聚在 fault handler 中，而参考项目采用了更分层的设计。
* **代码溯源链接**:
  - 目标项目: `arceos/modules/axmm/src/backend/cow.rs#handle_page_fault`
  - 参考项目: `arceos/modules/axmm/src/aspace.rs#handle_page_fault`

#### 5. 单方模块与降权项
*   **单方模块**: 目标项目存在大量参考项目未体现的模块，如 `第三方库` (1354个函数)，包含 `crates/virtio-drivers`, `crates/page_table_multiarch` 等独立开发的组件。这是目标项目独立演进的强证据。
*   **降权项说明**:
    *   **跨板块同名函数**: 如目标项目 `pulse_core/src/task/thread.rs#exit_current` 与参考项目 `arceos/api/arceos_posix_api/src/imp/pthread/mod.rs#exit_current`，虽同名但分属 `pulse_core` 和 `arceos_posix_api` 不同抽象层，AST 结构完全不同，不构成同源证据。

#### 6. 保守结论
综合全模块证据，目标项目与参考项目的 AST 相似度（13.58%）处于低位，且共同指纹主要源于对 ArceOS 上游框架及 musl libc 等公共依赖库的复用。在操作系统核心竞争力的【进程管理】、【内存管理】等子系统上，双方在抽象层次、内存模型、同步策略和错误处理等关键架构设计上存在显著且根本的差异。目标项目包含大量独有模块和独立重构的核心逻辑。因此，现有证据无法支持“抄袭”或“大规模代码复用”的指控，应认定为基于同一开源框架的独立开发项目。
