#### 4. 核心代码调用流对比剖析
* **【进程管理/任务创建】**
* **底层技术析理**: 目标项目 `spawn` 函数展现了复杂的 SMP 负载均衡逻辑：通过 `Instruction::get_tp` 获取当前 CPU ID，计算 `cpu_mask_id` 以选择目标处理器，并操作 per-CPU 的 `PROCESSORS` 结构体和 `rt_hint` 进行实时调度提示，最终调用 `TASK_QUEUE.enqueue_round_robin`。参考项目 `spawn` 则依赖 `async_task` 第三方库进行异步任务生成，并通过全局 `TASK_QUEUE.push_normal` 或 `push_prior` 进行简单入队。两者在调度模型（多核主动均衡 vs 单核异步框架）、队列结构（per-CPU vs 全局）和优先级管理上完全不同。
* **代码溯源链接**:

  - 参考项目: `modules/executor/src/lib.rs#spawn`

* **底层技术析理**: 目标项目 `run_until_idle` 实现了复杂的多核空闲循环：包含 `IDLE_CHECK_COUNT` 计数器、遍历所有 `PROCESSORS` 进行任务队列长度查询、计算本地/实时/其他任务数量，并执行 `migrate_id` 任务迁移和 `crate::smp::tlb::quiescent_tick` TLB 静默操作。参考项目 `run_until_idle` 仅为 `TASK_QUEUE.fetch` 和 `task.run` 的简单轮询。目标项目在空闲态处理了 CPU 间负载均衡和 TLB 一致性，参考项目无此设计。
