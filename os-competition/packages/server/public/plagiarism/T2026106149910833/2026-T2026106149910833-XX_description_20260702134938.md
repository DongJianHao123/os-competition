## 第四部分：分模块架构说明

### MM（内存管理）
底层证据主要体现为用户态内存接口与可执行文件加载，而非完整的物理内存管理。
- 提供 `sys_mmap`（`source/core/syscall/memory.rs#sys_mmap`），支持用户映射标志解析与对齐。
- ELF 加载链：`prepare_interpreted_executable_image` → `UserAddressSpaceLoadPlan` → `ExecutableSegmentPlanSet`（`source/core/loader/elf.rs`、`source/core/loader/elf/stack.rs`）。
- 单任务内存运行时集中在 `source/core/task/single/memory_runtime.rs`。
- 页缓存相关操作在 VFS 后端有涉及（`active_file_page_cache_mut`）。
- **未体现**：物理页框分配器、页表遍历/TLB 刷新、swap、NUMA 等底层物理内存管理细节在提供的调用链中未直接出现。

### PROC（进程管理/任务调度）
传统进程管理证据充分。
- 进程模型采用全局 slot 数组：`CHILD_PID`、`CHILD_EXITED`、`CHILD_EXIT_CODE`、`CHILD_PARENT_PID`、`CHILD_THREAD_GROUP`、`CHILD_PROCESS_GROUP`、`CHILD_SESSION`、`CHILD_FILE_CREATION_MASK`、`CHILD_NICE` 等（`source/core/task/single/process_runtime.rs#single_begin_child`、`clear_child_slot`）。
- 支持 `clone3`/`fork`/`vfork`、`exec`（ELF 与 shebang）、`wait4`/`waitid`、pidfd、`prctl`、`kcmp`、`process_vm`、`capset` 等。
- 信号投递：`single_apply_signal_info_to_pid`、`signal_process_outcome_process` 等。
- 调度部分表现为**阻塞式等待通道**：`source/core/scheduler/blocking.rs` 维护 child_exit、eventfd、futex、flock、inotify、message_queue、pipe 等等待位图槽；阻塞 syscall 通过 `wait_task_on_channel_until` 与架构相关的 `block_switch` 完成上下文切换（`apply_wait_child_context_switch`、`enter_next_task_runtime`）。
- **注意**：虽然模块画像标签含“异步运行时”，但提供的 Call Map 更偏向阻塞调度与上下文切换，async/await 任务运行时证据不明显。

### FS（文件系统）
文件系统是代码量最大的模块，证据强。
- **VFS 层**：`source/core/fs/vfs.rs`、`source/core/fs/vfs/overlay.rs` 提供 `WritableOverlay`，支持 copy-up。
- **ext4**：`source/core/fs/ext4.rs` 实现 superblock、inode、extent、目录项解析等。
- **procfs**：`source/core/fs/vfs/proc.rs` 提供 `/proc/[pid]/...`、`/proc/mounts`、`/proc/cpuinfo` 等虚拟目录项。
- **fd 管理**：`source/core/fs/fd.rs`、`source/core/fs/fd/socket.rs` 管理 fd 表与打开文件表。
- **挂载与路径解析**：`source/core/syscall/fs_path/mount.rs#sys_mount` / `sys_umount2`、`source/kernel/syscall_runtime/path_resolution.rs#resolve_runtime_path_components_in_scope`。
- **特殊文件抽象**：eventfd、epoll、signalfd、timerfd、inotify、pipe、message queue 均通过 fd 表与 VFS 后端交互。

### DEV（设备驱动）
设备驱动证据非常有限。
- 仅发现 **Virtio-blk** 块设备驱动：
  - RISC-V64：`source/arch/riscv64/block.rs#read_sector_inner`、`ensure_ready_modern`
  - LoongArch64：`source/arch/loongarch64/block.rs#read_sector_inner`、`find_virtio_block_device`、`ensure_ready`、`reset_device`
- `source/core/block.rs#read_sector` 提供跨架构的统一块读取接口。
- **未体现**：显示、输入、网络、USB、NVMe、SATA 等其他设备驱动。

### SYSCALL（系统调用）
- 中央分发器：`source/core/syscall/dispatch.rs#dispatch_with_runtime` / `dispatch_single_with_runtime`，规模超过 2 万 token，调用数 250+。
- 按功能分散在 `source/core/syscall/` 下：`fs_path`、`fd_read`、`fd_write`、`process_lifecycle`、`process_control`、`memory`、`sched`、`signal`、`socket`、`sysv_msg`、`sysv_sem`、`keyring`、`poll`、`epoll`、`mqueue` 等。
- 错误映射：`source/kernel/syscall_runtime/path_resolution.rs#map_runtime_errno` 将内部错误映射为 Linux errno。

### TRAP/IRQ（中断异常）
- 架构相关 trap 帧：`RiscvUserTrapFrame`、`LoongArchTrapFrame`。
- fork 子进程首跳：`source/arch/riscv64/trap/fork.rs#start_child_after_fork`、`source/arch/loongarch64/trap/fork.rs#start_child_after_fork`。
- 信号栈：`source/arch/riscv64/trap/signal.rs#signal_stack_top`、`source/arch/loongarch64/trap/signal.rs#signal_stack_top`。
- 定时器中断：`source/arch/riscv64/timer.rs#poll_timer_interrupt_target`。
- 阻塞 syscall 唤醒与恢复：`source/arch/riscv64/trap/block_dispatch.rs#consume_blocked_syscall` 及其 `consume_*_wait_block` 系列。
- 证据集中在 syscall 入口、fork、信号、定时器与阻塞唤醒，完整 IRQ 控制器、中断向量表、DMA 等细节未充分展现。

### NET（网络协议栈）
**底层证据未体现**。
- 仅有 socket 相关系统调用表层：`source/core/syscall/socket.rs#sys_getsockopt`、`encode_socket_endpoint`、`try_recvmsg_at`。
- 这些函数主要处理地址编码、fd 可读可写状态与缓冲区拷贝，没有 TCP/IP 协议栈、路由、ARP、网卡驱动等证据。

## 第六部分：边界与未体现内容

- **完整网络协议栈底层证据未体现**：仅有 socket 系统调用表层与地址编码，没有 TCP/IP、UDP、路由、ARP、网卡驱动等。
- **设备驱动范围极窄**：除 Virtio-blk 外，显示、输入、网络、USB、NVMe、SATA、声卡等设备驱动均未体现。
- **物理内存管理细节未直接体现**：页框分配、页表操作、TLB 刷新、swap、NUMA 等在提供的调用链中未出现，证据集中在用户映射、ELF 加载与页缓存。
- **SMP/NUMA/多核调度证据不足**：虽有 per-core active runtime snapshot 与定时器同步，但不足以证明完整 SMP 调度或 CPU 迁移。
- **安全与高级文件系统特性未体现**：LSM、SELinux、AppArmor、磁盘配额、文件加密、快照等未体现。
- **电源管理、ACPI/DT 高级初始化、虚拟化支持未体现**。
- **文件系统/第三方库部分需单独审视**：`source/core/fs/ext4.rs` 等被归入“文件系统/第三方库”模块（55 函数），无法从调用链判断其原创性或依赖关系。
- **方向判断仅为技术推测**：本文档基于 AST Call Map、函数名、目录结构与调用链描述技术事实，对“实现了什么、证据来自哪里、哪些模块没有体现”做出判断，不做抄袭/违规/最终身份认定。
