#### 4. 核心代码调用流对比剖析
本庭选取控方重点指控的【进程管理】板块，对其核心系统调用路径进行深度析理，以论证其架构异构性。

*   **【进程管理 / wait 系统调用路径】**
*   **底层技术析理**: 这是控方“指控二”的核心。目标项目的 `sys_waitid` 函数在判断无子进程可回收时，返回一个 `SyscallOutcome::Blocked(SyscallBlock::waitid_child(...))` 状态。这是一个**同步的、基于状态机的阻塞模型**。该状态随后被架构相关的 trap handler（如 `consume_waitid_child_block`）处理，通过 `prepare_blocked_context_switch` 直接操作 CPU 上下文进行任务切换。相比之下，参考项目的 `syscall_wait4` 在需要阻塞时，会创建一个 `WaitFuture` 并调用 `axtask::block_task`。这是一个**异步的、基于 Future 的阻塞模型**。它将阻塞逻辑交给 `async-task` 运行时，通过 `spawn_kernel_task` 和 `Future::poll` 机制在调度器层面处理。两者从**抽象层次**（同步状态机 vs. 异步 Future）、**阻塞原语**（`SyscallBlock` vs. `WaitFuture`）到**调度入口**（架构相关 trap handler vs. 通用 `axtask` 运行时）均完全不同。控方所谓的“拓扑重合”是对“等待-唤醒”这一通用计算机科学概念的过度抽象，忽略了具体的实现机制。
*   **代码溯源链接**:
    - 目标项目: `source/core/syscall/process_lifecycle.rs#sys_waitid`
    - 参考项目: `api/linux_syscall_api/src/syscall_task/imp/task.rs#syscall_wait4`

*   **【进程管理 / procfs 文件系统】**
*   **底层技术析理**: 这是控方“指控四”的核心。目标项目通过 `VfsDirEntry::new` 静态构建目录项，每个条目都绑定了一个预定义的 `VFS_BUILTIN_PROC_*_INODE` 常量。这是一种**静态定义、基于常量的查找表**实现。参考项目则通过 `RamFileSystem::new` 动态创建一个内存文件系统实例，然后调用 `proc_root.create` 动态创建文件和目录。这是一种**动态创建、基于对象**的实现。两者的**对象管理模型**截然不同。至于 `b"cpuinfo"` 等文件名，它们是 Linux `/proc` 文件系统的**标准接口名称**，任何要实现兼容性的内核都必须使用，不构成私有埋点或抄袭痕迹。
*   **代码溯源链接**:
    - 目标项目: `source/core/fs/vfs/proc.rs#proc_root_dir_entry`
    - 参考项目: `modules/axfs/src/mounts.rs#procfs`
