## 第四部分：分模块架构说明

### MM（内存管理）

- **证据较强**。物理内存分配依赖离线引入的 `buddy_system_allocator`，其 `alloc`/`dealloc` 按 2 的幂对齐分配；帧分配器在 `frame.rs` 中维护 `free_list` 并支持分裂/合并。
- 内核提供 `sys_mmap`、`sys_msync`、`sys_mprotect` 等 POSIX 内存映射接口；`sys_mmap` 中可见 `VirtAddr`、`align_up`、`collect_shared_file_writes` 等处理。
- 页故障处理在 `polyhal-trap` 各架构实现中可见，例如龙芯 `loongarch64_trap_handler` 区分指令/加载/存储页故障，并调用 `emulate_load_store_insn` 处理非对齐访问。
- **未明确体现**：完整的虚拟内存页表映射代码、swap、COW（写时复制）细节在提供的证据中未直接出现。

### PROC（进程管理/任务调度）

- **证据强**。实现了传统 Unix 进程模型：进程控制块 `TaskControlBlock`、线程组 `ThreadGroup`、父子关系、PID/TGID。
- `sys_clone` 创建线程/进程；`sys_execve` 解析 ELF、选择解释器、构造用户栈；`sys_wait4_thread_group`/`sys_waitid` 实现 wait 语义并回收僵尸子进程。
- 退出路径 `exit_current_and_run_next` → `finish_task_exit` → `finish_process_exit` 处理 fd 刷新、futex 唤醒、robust list、信号通知、等待队列清理等。
- 调度以阻塞式等待队列为主（`wait_queue.rs`），配合 timer、futex、signal wait queue；**async runtime / futures executor 的底层证据未体现**，模块名中的“异步运行时”应理解为传统任务阻塞/唤醒机制。

### FS（文件系统）

- **证据最强**。`os/src/fs/` 下实现了 VFS 统一入口（`vfs.rs#open_path`、`metadata`、`rename_path`），并同时支持：
  - 内存文件系统 `MEM_FS`（`MEM_FS.lock` 频繁出现）；
  - ext4 真实块设备文件系统，通过 `os/src/fs/ext4_vol.rs` 封装，提供 `mkdir_ext4_with_mode`、`create_regular_ext4_with_mode`、`rename_ext4`、`unlink_non_dir`、`start_writeback_worker_explicit` 等。
- 文件描述符层 `os/src/fs/fd.rs` 实现 `read`/`write`、pipe、offset、文件锁等；`os/src/syscall/fs.rs` 提供 `sys_read`/`sys_write`/`sys_pread64`/`sys_pwrite64`/`sys_splice`/`sys_copy_file_range` 等。
- 支持 mount/umount、`procfs` 魔术链接（`/proc/self/exe`、`/proc/[pid]/fd/*`）、`epoll_ctl`、文件记录锁 `sys_fcntl_file_lock`。
- **未明确体现**：除 ext4 与 memfs 外的其他文件系统（如 FAT、NTFS、NFS），以及块设备缓存/JBD2 日志的完整自实现证据不足（ext4 依赖第三方 crate）。

### DEV（设备驱动）

- **证据弱**。仅 20 个函数，集中在：
  - VirtIO 块设备：`os/src/drivers/virtio_mmio_blk.rs#probe_first_virtio_disk_from_dt`、`os/src/drivers/virtio_pci_blk.rs`；
  - HAL 共享内存 DMA：`os/src/drivers/hal.rs#share`；
  - debug console：`patch/polyhal/src/debug_console/{riscv64,aarch64}.rs`、`os/src/main.rs#early_la_putchar`。
- **未体现**：网络设备驱动、GPU、USB、SATA/NVMe、字符设备框架、设备树/ACPI 之外的 BSP 驱动。

### SYSCALL（系统调用）

- **证据强**。`os/src/syscall/mod.rs#syscall` 是统一分发入口，调用数达 172，覆盖 fs/process/mm/signal/net 各大类。
- 文件类：`sys_read`、`sys_write`、`sys_mount`、`sys_umount2`、`sys_newfstatat`、`sys_pread64`、`sys_pwrite64`、`sys_fallocate`、`sys_splice`、`sys_copy_file_range`、`sys_epoll_ctl`、`sys_fcntl_file_lock` 等。
- 进程类：`sys_clone`、`sys_execve`、`sys_wait4`、`sys_waitid`、`sys_exit`/`sys_exit_group`、`sys_getpid`/`sys_getppid`、`sys_sched_yield`。
- 内存类：`sys_mmap`、`sys_msync`、`sys_mprotect`（证据中可见 `sys_mmap`、`sys_msync`）。
- 信号类：`sys_sigtimedwait`、`sys_sigprocmask`。
- 其他：`sys_nanosleep`、`sys_clock_nanosleep`、`sys_futex_stub`。

### TRAP/IRQ（中断异常）

- **证据强**。`polyhal-trap` 提供多架构 trap handler：
  - RISC-V：`init_trap_only` 设置 `stvec`，`handle_syscall` 进入系统调用分发；
  - LoongArch64：`loongarch64_trap_handler` 处理页故障、非对齐访问模拟；
  - x86_64：`idt.rs#new` 初始化 IDT，`apic.rs#init` 初始化 Local APIC 与 IO APIC。
- os 层：`os/src/trap/mod.rs#init` 调用 HAL 初始化；`handle_syscall` 为 syscall 入口；`handle_exit` 处理退出。
- **未明确体现**：完整中断向量表、PIC/IOAPIC 中断分发后半部（top/bottom half）、时钟中断与调度 tick 的完整链路。

### NET（网络协议栈）

- **底层证据未体现**。仅在 `os/src/syscall/net.rs` 中看到 socket 选项设置（`sys_getsockopt`、`sys_setsockopt`）、Unix domain socket 路径解析（`sockaddr_un_path`）、端口绑定记录（`register_bound_socket`）等系统调用表层封装。
- **未体现**：TCP/IP 协议栈、NIC 驱动、sk_buff、路由、socket 文件系统完整实现。

## 第六部分：边界与未体现内容

### 1. 没有底层证据支撑的模块/能力

- **完整网络协议栈**：仅有 socket 系统调用表层封装，无 TCP/IP、NIC 驱动、协议状态机。
- **GPU / 图形 / 声卡 / USB / SATA / NVMe 等驱动**：仅见 VirtIO 块设备与 debug console。
- **虚拟化 / Hypervisor**：无 VMX/SVM、VM exit、virtio-net backend 等证据。
- **安全模块 / LSM / 强制访问控制**：无 SELinux/LSM hook、capabilities 完整实现证据。
- **实时调度 / 多核 SMP 调度完整实现**：APIC 初始化存在，但 SMP 调度、核间中断、负载均衡证据不足。
- **电源管理 / ACPI 高级功能**：仅见 ACPI RSDP/MADT/MCFG 解析，未体现睡眠、热插拔等。
- **块设备缓存 / ext4 日志（JBD2）自实现**：ext4 操作大量依赖第三方 crate，内核层主要是封装。
- **异步运行时（async/await executor）**：模块名含“异步运行时”，但证据为传统阻塞式调度与等待队列。

### 2. 需要单独说明的第三方/公共依赖

- `patch/polyhal*` 系列：硬件抽象层与启动代码，属于 patch 后的公共依赖，不应视为本项目原创内核主体。
- `offline_crate_sources/buddy_system_allocator`：离线引入的伙伴分配器 crate。
- `offline_crate_sources` 中可能还包含 ext4 解析/操作 crate（证据集中在 `os/src/fs/ext4_vol.rs` 的封装调用）。
- 这些代码在评价原创性时应与 `os/src/` 内核主体区分。

### 3. 判断性质说明

- 本报告中“可能的系统方向”“保守系统方向提示”仅为基于函数分布与调用链的技术推测，**不是对作品最终身份、原创性或违规情况的结论**。
- 所有模块强弱判断均基于给定 AST Call Map 与函数统计；未在证据中出现的功能，均标记为“底层证据未体现”，不做夸大或编造。
