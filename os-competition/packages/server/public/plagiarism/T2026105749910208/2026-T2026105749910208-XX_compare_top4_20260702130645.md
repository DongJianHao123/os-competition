#### 4. 核心代码调用流对比剖析
* **【内存管理/伙伴分配器】**
* **底层技术析理**: 这是双方唯一的 AST 指纹重合点。两者均实现了伙伴系统算法，其核心操作如计算阶数（`size.trailing_zeros()`）、向上查找空闲块（`(class+1..i+1).rev`）、块的分割与合并等，调用图谱必然高度一致。然而，目标项目将此代码置于 `offline_crate_sources` 目录下，是典型的对开源 crate `buddy_system_allocator` 的 vendoring 使用方式；参考项目则将其直接内联进 `kernel/src/memory/allocator/`。这更像是双方共同依赖了同一公共算法或上游组件，而非一方抄袭另一方。在其他内存管理核心路径（如 `map_kernel`）上，目标项目使用 `PTE::new_page` / `TLB::flush_vaddr` 抽象，参考项目使用 `FrameArea::new_without_clear` / `pgtable.map`，架构完全不同。
* **代码溯源链接**:
  - 目标项目: `offline_crate_sources/buddy_system_allocator/src/lib.rs#alloc`
  - 参考项目: `kernel/src/memory/allocator/mod.rs#alloc`

* **【进程管理/系统调用】**
* **底层技术析理**: 目标项目的 `sys_waitid` 实现了 POSIX 标准的进程等待，涉及 `WaitTarget` 枚举、`thread_group` 遍历、`sleep_on_child_exit` 等复杂同步机制。参考项目的 `sys_clone` 则是基础的线程创建调用，通过 `my_thread!().fork` 实现。两者在进程模型上存在根本差异：目标项目采用线程组（thread group）模型，参考项目采用简单的线程池（THREAD_POOL）模型。AST 调用图谱无任何重合。
* **代码溯源链接**:
  - 目标项目: `os/src/syscall/process.rs#sys_waitid`
  - 参考项目: `kernel/src/syscall/proc.rs#sys_clone`

* **【文件系统/系统调用】**
* **底层技术析理**: 两者均有 `sys_openat` 函数，但实现路径完全不同。目标项目通过 `resolve_host_path` 和 `open_resolved_path` 进行路径解析，背后是复杂的 VFS 和 ext4 支持。参考项目则通过 `get_user_string` 获取路径字符串，然后操作 FAT32 的 `FileInner`。两者的文件系统抽象层次和底层数据结构无任何 AST 级别的重合。

  - 参考项目: `kernel/src/syscall/fs.rs#sys_openat`

#### 6. 保守结论
本案客观 AST 相似度极低（1.11%），项目整体应定性为**独立开发**。唯一的代码重合点——伙伴系统分配器——是操作系统内核中广泛使用的经典算法，且目标项目以 `offline_crate_sources` 形式透明引用了开源组件 `buddy_system_allocator`，属于正常的第三方代码使用或算法级收敛，而非对参考项目的抄袭。在进程管理、文件系统、设备驱动等核心子系统上，双方 AST 指纹覆盖率为零，架构设计与实现路径存在根本性、系统性的差异。控方提出的跨板块“同板块证据”因功能无关且无 AST 重合，不构成有效指控。综上，本次审计未发现足以判定代码抄袭的同源性证据。
