## 第四部分：分模块架构说明

### MM（内存管理）

- **有证据**：`platform/riscv64-virt/src/main.rs#alloc` / `dealloc` 与 `platform/loongarch64-virt/src/main.rs#alloc` / `dealloc` 实现了基于阶（order）的伙伴类物理分配器，涉及 `next_power_of_two`、`pop`/`push`、对齐与大小取最大等操作。
- **有证据**：`crates/mm/src/lib.rs`（97 函数）被 `kernel_core::bootstrap` 以 `MemoryManager::from_hal` 形式初始化。
- **未体现**：页表、虚拟内存映射（mmap）、缺页异常处理、swap、用户/内核地址空间切换等高级内存管理底层证据未体现。

### PROC（进程管理/任务调度）

- **有证据**：`crates/proc/src/lib.rs` 144 函数；`crates/kernel-core/src/lib_*.inc.rs#bootstrap` 中创建了 `ProcessTable::new` 与 `Scheduler::new`。
- **有证据**：`crates/syscall/src/lib.rs#sys_execve` 规模较大，说明存在程序加载/执行相关逻辑。
- **未体现**：`fork`/`wait`/`exit`/`zombie`、父子进程关系、PCB 字段细节、调度算法（如 CFS/优先级/时间片）等底层证据未体现。  
  另外，模块画像中“任务调度/异步运行时”的 87 个函数实际来自 `tools/xtask/src/main.rs`，是**构建与测试脚本**，不能作为内核调度器证据。

### FS（文件系统）

- **有证据**：`crates/vfs/src/lib.rs` 提供 VFS 层，包括 `mount_ext4`、`chdir`、`bind_socket`、`connect_socket`、`ext4_root_mount_supports_open_and_getdents`。
- **有证据**：`crates/fs-ext4/src/lib.rs` 实现 ext4 相关读写与测试镜像构造，如 `build_test_image`、`read`、`Ext4Mount::probe`、`VecBlockDevice::from_image`。
- **说明**：文件系统栈以 ext4 为主，块设备通过内存镜像或 virtio 块设备承载。

### DEV（设备驱动）

- **极弱证据**：模块画像中设备驱动仅 1 个函数，实际为平台 panic 等辅助代码。
- **依赖第三方**：块设备、网络设备主要依赖 `third_party/virtio-drivers`（`queue.rs`、`transport/mod.rs`、`device/socket/...`）。
- **未体现**：原创字符设备驱动、中断控制器驱动、时钟驱动、PCI/DT 总线扫描等底层证据未体现。

### SYSCALL（系统调用）

- **强证据**：`crates/syscall/src/lib.rs` 324 函数，覆盖 `sys_mount`、`sys_openat`、`sys_read`、`sys_write`、`sys_readv`、`sys_writev`、`sys_pread64`、`sys_pwritev`、`sys_readlinkat`、`sys_execve` 等。
- **有证据**：`crates/syscall/src/fs_domain.rs#dispatch` 对文件类系统调用进行统一分发。
- **说明**：系统调用实现较完整，是本项目原创代码最密集的区域之一。

### TRAP/IRQ（中断异常）

- **有证据**：`crates/kernel-core/src/lib_riscv.inc.rs#handle_trap` 与 `lib_loongarch.inc.rs#handle_trap` 是陷阱入口，处理 `SyscallArgs`、定时器编程（`hal().timer.program_oneshot`）等。
- **较弱**：平台目录 `platform/*/src/main.rs` 中中断/异常相关函数仅 2 个，且为入口转发。
- **未体现**：具体 IRQ 控制器（PLIC/LS7A）初始化、中断向量表、中断嵌套等底层证据未体现。

### NET（网络协议栈）

- **有证据**：`crates/vfs/src/lib.rs#bind_socket` / `connect_socket` 实现了内核态 socket 文件表示与连接通道。
- **有证据**：`crates/syscall/src/lib.rs#eventfd_epoll_and_socketpair_smoke`、`inet_loopback_socket_smoke` 等测试说明 socket API 存在。
- **依赖第三方**：`third_party/virtio-drivers/src/device/socket/...` 提供 virtio-vsock 设备驱动。
- **未体现**：原创 TCP/IP 协议栈、IP 路由、ARP、UDP/TCP 状态机等底层证据未体现。

## 第六部分：边界与未体现内容

- **任务调度/异步运行时证据边界**：模块画像中 87 个“任务调度/异步运行时”函数全部位于 `tools/xtask/src/main.rs`，属于构建、QEMU 启动与 OSCOMP 自测脚本，**不是内核调度器实现**。内核中仅能通过 `ProcessTable::new`、`Scheduler::new` 推断存在进程/调度抽象，但具体算法与数据结构未体现。

- **传统进程管理未体现**：没有 `fork`/`wait`/`exit`/`zombie`、父子进程关系、PCB 字段、进程组/会话等底层证据。`sys_execve` 存在但实现细节未展开。

- **设备驱动原创证据极弱**：除平台 panic 与 HAL 封装外，没有原创字符设备、块设备、中断控制器、时钟、总线扫描等驱动代码。块/网络设备主要依赖 `third_party/virtio-drivers`。

- **网络协议栈未体现**：没有原创 TCP/IP、UDP、ARP、IP 路由、拥塞控制等协议栈证据。网络功能表现为 socket API（`bind`/`connect`）与 virtio-vsock 设备封装。

- **虚拟内存高级特性未体现**：没有页表、页故障、mmap、用户/内核地址空间隔离、swap 等证据。内存管理主要表现为物理堆/页式分配器。

- **SMP、同步原语、安全模块未体现**：没有多核调度、自旋锁/信号量实现、权限检查、Capabilities 等证据。

- **第三方依赖占比高**：`third_party/` 与部分 `crates/kernel-core` 代码被归入“第三方库”统计，共 1785 个函数，评审时应将其与原创内核主体严格区分。

- **“可能的系统方向”仅为推断**：本作品更可能是一个面向 QEMU `riscv64-virt`/`loongarch64-virt` 的轻量级内核，支持 ext4、基础系统调用与 virtio 设备，但上述方向判断基于目录与调用链线索，**不是最终身份或原创性结论**。本文档不做抄袭/违规判断。
