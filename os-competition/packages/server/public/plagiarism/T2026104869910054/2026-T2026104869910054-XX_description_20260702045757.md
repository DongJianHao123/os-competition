## 第四部分：分模块架构说明

### MM（内存管理）

底层证据主要位于 `crates/mm/`。`crates/mm/src/aspace.rs#map_loongarch_kernel_direct` 使用 `PageTable::from_space`、`map_loongarch_megapage_range`、`map_mmio_range` 建立内核直接映射；`crates/mm/src/lib.rs` 函数密度高（78 函数），涉及 `kernel_token`、`phys_to_virt`、`virt_to_phys` 等地址转换与 TLB 刷新。用户地址空间创建可见于 `crates/kernel/src/init_proc.rs#spawn` 调用的 `mm::AddressSpace::new_user`。

**未体现**：页错误处理、swap、mmap 子系统、slab/伙伴分配器等细节在提供的 Call Map 中无直接证据。

### PROC（进程管理/任务调度）

进程生命周期证据分散在 `crates/task/` 与 `crates/kernel/`。`crates/task/src/fork.rs#sys_clone` 实现 `Process::new`、pid/tid 分配、子进程 trap frame 设置；`crates/task/src/spawn.rs#init_trap_context_in_place` 初始化用户/内核栈、TLS、trampoline；`crates/task/src/execve.rs#check_execve_redirect` 构建用户栈并处理 execve 重定向；`crates/kernel/src/init_proc.rs#spawn` 负责 ELF 解析与解释器加载。用户态侧有 `user/libc/src/lib.rs#fork`（调用 `clone`）与 `user/apps/user_init/src/main.rs#wait_exit_code` / `reap_adopted_zombies`（调用 `wait4`）。

调度证据可见 `crates/kernel/src/entry.rs#spawn_init_task` 中的 `task::sched_init`、`task::sched_add`、`task::sched_mark_idle`，以及 `crates/syscall/src/imp/task.rs#sys_sched_setaffinity`。

**未体现**：完整的 PCB 字段、父子进程关系维护、僵尸进程回收内核侧细节、SMP 调度算法等底层证据不足。

### FS（文件系统）

文件系统是该作品证据最强的模块之一。`crates/fs/src/impls/ext4/` 下实现了 EXT4 的 inode、superblock、adapter、file 等子模块：`crates/fs/src/impls/ext4/adapter.rs#lookup_at_with_final_follow` 与 `lookup_inner_with_final_follow` 完成带符号链接跟随的路径解析；`alloc_block`、`add_dir_entry`、`add_extent_to_inode` 等实现块分配与目录项管理；`load_page_cache_slot_for_write`、`alloc_page_cache_slot_for_write`、`flush_page_cache_slot` 构成页缓存写回路径。`crates/fs/src/impls/ramfs/super_block.rs#link_inode` 与 `dir_entry` 提供内存文件系统；`crates/fs/src/impls/procfs/procfs.rs` 提供 `/proc` 内容。`crates/fs/src/open.rs#sys_open_follow`、`sys_read`、`sys_getdents` 为高层入口。`crates/syscall/src/imp/fs.rs` 中的 `sys_openat`、`sys_read`、`sys_write`、`sys_mount`、`sys_mkdirat` 等将用户态请求路由到上述实现。

**注意**：EXT4 / RAMFS / PROCFS 相关函数在统计中被归入“文件系统/第三方库”，其原创范围需与第三方 crate 区分。

### DEV（设备驱动）

设备层证据较少，主要以抽象接口形式出现：`crates/syscall/src/imp/fs.rs#sys_read` / `sys_write` 中调用 `block_device_read` / `block_device_write` 与 `console_getchar` / `console_putchar`；`crates/syscall/src/imp/fs.rs#fs_stat_counts` 引用 `devfs.*` 统计块设备信息。未见到具体硬件控制器（磁盘、NIC、中断控制器）驱动实现。

**未体现**：底层设备驱动、DMA、中断分发、时钟硬件细节等。

### SYSCALL（系统调用）

系统调用层证据极强。`crates/syscall/src/dispatch.rs#handle_syscall` 是巨型分发器（168 条调用链），覆盖 fs/net/task/signal 等子系统。`crates/syscall/src/imp/fs.rs`（160 函数）实现 open/read/write/close、mount、mkdirat、chdir、stat、chmod/chown、epoll、eventfd 等；`crates/syscall/src/imp/net.rs`（123 函数）实现 socketpair、UDP 收发、pipe-backed socket；`crates/syscall/src/imp/task.rs`（102 函数）实现 clone、sched_setaffinity 等；`crates/syscall/src/imp/signal.rs`（79 函数）实现信号与时钟。

### TRAP/IRQ（中断异常）

证据位于 `crates/hal/src/la64/trap.rs#rust_trap_handler`，其解码 `TrapCause::UserEcall` 等异常原因并调用 `registers::clear_timer_interrupt`。`crates/kernel/src/trap_handler.rs#kernel_trap_handler` 进一步提取 sepc/sp、系统调用参数、设置返回值、推进 PC，并处理致命错误 `fatal_shutdown`。`crates/task/src/spawn.rs#init_trap_context_in_place` 初始化 trap 上下文。

**未体现**：外部设备中断处理、IRQ 路由、PLIC/HTVEC 等 LoongArch 中断控制器细节。

### NET（网络协议栈）

网络证据集中在 `crates/syscall/src/imp/net.rs`：`sys_socketpair` 创建 pipe-backed socket；`udp_send_iov_queued` / `udp_recv_iov_queued` / `udp_enqueue_iov` / `udp_dequeue_iov` 维护 `BOUND_SOCKETS` 表与 UDP 报文队列；`pipewrite_socket` / `piperead_socket` 处理 socket 的阻塞/唤醒。`user/apps/user_init/src/main.rs#run_netperf_cases` 仅为用户态测试编排。

**未体现**：IP 层、TCP 协议、ARP/ICMP、NIC 驱动、DMA 收发等底层网络协议栈证据未直接体现。

- **虚拟内存高级子系统**：页错误处理、swap、mmap、copy-on-write、共享内存等细节未体现。
- **SMP/多核调度**：仅见 `sys_sched_setaffinity`，缺少 CPU 初始化、核间中断、自旋锁/RCU 等证据。
- **安全与权限模型**：仅见 `chmod`/`chown`/`check_access_mode` 等零散调用，完整 capability、ACL、审计等未体现。
- **时钟与定时器**：仅通过 `check_itimers` 间接引用，硬件定时器驱动、高精度计时器子系统未直接体现。
- **图形/声音/其他外设**：无证据。

### 需要单独说明的边界

- **第三方库占比极高**：统计中“第三方库”达 1805 个函数，占绝对多数；“文件系统/第三方库”254 个函数、“网络协议栈/第三方库”137 个函数也包含大量实现。文档中将其列为证据，但不代表它们均为该作品的原创内核主体，其原创范围需进一步与依赖库区分。
- **“可能的系统方向”仅为推断**：基于文件路径（如 `la64`、`ext4`、`ramfs`、`procfs`）、函数名与调用链得出，不是最终作品身份或原创性结论。
- **用户态代码与内核代码需区分**：`user/apps/user_init/`、`user/libc/` 中的函数属于用户态测试框架与 libc 封装，不能直接等同于内核实现。
- 本文档不做抄袭、违规或原创性判定，仅依据提供的 AST Call Map 与路径锚点描述技术事实。
